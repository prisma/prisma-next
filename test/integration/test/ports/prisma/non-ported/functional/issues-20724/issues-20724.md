# Non-ported — issues-20724

Every test asserts the shape of Prisma's `PrismaClientKnownRequestError`: `.name === 'PrismaClientKnownRequestError'`, `.code` (`P2002`/`P2010`), and specifically whether `.meta.modelName` is present. prisma-next surfaces a `SqlQueryError` with `sqlState` (`'23505'`), `constraint` (`'user_email_key'`), and `table` (`'user'`) — but no `.code`, no `.meta`, and no `.modelName`. The suite's differential subject (modelName IS present on ORM-level errors vs NOT present on raw-query errors) has no prisma-next analogue: ORM and raw errors both surface the same structured `SqlQueryError` shape.

- `packages/client/tests/functional/issues/20724/tests.ts` › `modelName is returned on error.meta › should return modelName on error.meta when performing prisma.model.create` — prisma-next unique-violation error is `SqlQueryError` (sqlState 23505 / constraint / table), carries no `.code`/`.meta.modelName`
- `packages/client/tests/functional/issues/20724/tests.ts` › `modelName is returned on error.meta › should return modelName on error.meta when performing prisma$transaction with the client` — same; additionally array/batch `$transaction([...])` unsupported
- `packages/client/tests/functional/issues/20724/tests.ts` › `modelName is not returned on error.meta › should not return modelName when performing queryRaw` — asserts Prisma's P2010 raw-query error carries no modelName; prisma-next raw errors are `SqlQueryError` with no `.code`/`.meta`, so the P2010-vs-modelName distinction is inexpressible
- `packages/client/tests/functional/issues/20724/tests.ts` › `modelName is not returned on error.meta › should not return modelName when performing executeRaw` — same; P2010 error-shape distinction inexpressible
- `packages/client/tests/functional/issues/20724/tests.ts` › `modelName is not returned on error.meta › should not return modelName when performing transactions with raw queries` — same; additionally array/batch `$transaction([...])` unsupported
