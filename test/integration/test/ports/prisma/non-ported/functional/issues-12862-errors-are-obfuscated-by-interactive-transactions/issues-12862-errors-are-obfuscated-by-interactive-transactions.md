# Non-ported — issues-12862-errors-are-obfuscated-by-interactive-transactions

The subject is that a **CHECK-constraint** violation (`viewCount >= 0`) propagates the correct DB error message through create / batch-tx / interactive-tx. Upstream injects the CHECK via `alterStatementCallback` (raw `ALTER TABLE ... ADD CONSTRAINT ... CHECK (...)`). prisma-next's PSL/contract has no CHECK-constraint surface (confirmed: `packages/2-sql/9-family/src/core/migrations/contract-to-schema-ir.ts` notes numeric-enum CHECK constraints are "not yet supported"), and the harness pushes schema via plan→apply from the contract only — there is no way to inject an arbitrary raw ALTER. Without the CHECK constraint the `viewCount: -1` insert simply succeeds, so the check-constraint-violation subject cannot be reproduced.

- `packages/client/tests/functional/issues/12862-errors-are-obfuscated-by-interactive-transactions/tests.ts` › `should propagate the correct error when a method fails` — CHECK constraint (`viewCount >= 0`) not expressible in prisma-next PSL/contract; cannot inject raw ALTER via the plan→apply push
- `packages/client/tests/functional/issues/12862-errors-are-obfuscated-by-interactive-transactions/tests.ts` › `should propagate the correct error when a method fails inside an transaction` — same CHECK-constraint gap; additionally array/batch `$transaction([...])` unsupported
- `packages/client/tests/functional/issues/12862-errors-are-obfuscated-by-interactive-transactions/tests.ts` › `should propagate the correct error when a method fails inside an interactive transaction` — same CHECK-constraint gap (the interactive-tx facade exists, but there is no CHECK constraint to violate)
