New: Referential Actions

Act on your rows
as relations change

Define ON DELETE and ON UPDATE behavior directly in the contract. The database enforces it — no application code needed.

ON DELETE: SET NULL

Delete a user? Their posts survive — the FK column becomes NULL.

ON UPDATE: CASCADE

Update a user's PK? Every referencing post automatically follows.

All five PostgreSQL actions supported:

cascade setNull restrict setDefault noAction
contract.ts
// Before — no referential action config defineContract() .table('post', t => t .column('userId', { type: int4Column }) .foreignKey(['userId'], { table: 'user', columns: ['id'], }) )
contract.ts (with referential actions)
defineContract() .table('post', t => t .column('userId', { type: int4Column, nullable: true }) .foreignKey(['userId'], { table: 'user', columns: ['id'], }, { onDelete: 'setNull', onUpdate: 'cascade', }) )

Referential Actions

What happens
at query time

A user has many posts. The FK is configured with onDelete: 'setNull' and onUpdate: 'cascade'.

The database handles the referential integrity for you — no manual cleanup queries, no orphan-hunting.

Validation built in. Setting setNull on a non-nullable column? The contract rejects it at emit time, before it ever reaches the database.

Scenario A: Delete a user

onDelete: 'setNull'
-- Before user │ id=1 email='ada@prisma.io' post │ id=10 userId=1 title='Hello' post │ id=11 userId=1 title='World' DELETE FROM "user" WHERE id = 1; -- After user │ (deleted) post │ id=10 userId=NULL title='Hello' post │ id=11 userId=NULL title='World' Posts preserved. FK set to NULL.

Scenario B: Update a user's PK

onUpdate: 'cascade'
-- Before user │ id=1 email='ada@prisma.io' post │ id=10 userId=1 UPDATE "user" SET id = 42 WHERE id = 1; -- After user │ id=42 email='ada@prisma.io' post │ id=10 userId=42 <-- cascaded FK automatically updated.

How It Works

Contract-first reconciliation

One command diffs your contract against the live database, computes a plan, and applies it with full verification.

M
Read Marker
I
Introspect DB
D
Diff & Plan
A
Apply Ops
W

CONTRACT

Deterministic JSON with cryptographic hash. The single source of truth for what the schema should be.

PLAN

Immutable migration object. Each operation is classified as additive, widening, or destructive. Previewable before execution.

MARKER

Hash stored in the database. Anchors the plan origin so the runner verifies the DB is at the expected state.

Scenario: Drift Detection

Catch what
doesn't belong

Someone ran manual SQL against the database: added a legacy_code column on account and created a legacy_audit table.

The contract knows nothing about these. When you run db update, it detects the drift and classifies the cleanup as destructive.

Scenario: Irreconcilable Drift

When the plan
can't reconcile

A more extreme case: someone changed the primary key of the account table from id to email directly in the database.

The contract still expects id as the PK. This isn't just a column to drop — it's a structural conflict the planner cannot resolve automatically.
PLANNING_FAILED

No silent corruption. The command halts before touching anything and tells you exactly what conflicts.

prisma-next db update
✔ Connecting to database... (437ms) ✔ Reading contract marker (96ms) ✔ Introspecting database schema (799ms) ✘ Planning migration (failed) ✘ Migration planning failed (PN-CLI-4020) Why: Table "account" has primary key mismatch: expected columns [id], got [email] Fix: Use db schema-verify to inspect conflicts, or ensure the database is empty Conflicts (showing 1 of 1): - [indexIncompatible] Table "account" has primary key mismatch: expected columns [id], got [email]

Takeaways

Why this matters

01

Referential actions in the contract

ON DELETE and ON UPDATE behavior is declared alongside your schema. The database enforces it, validated at emit time.

02

Preview every migration before applying

Plan mode shows operations, classes, DDL, and destination hash. No changes touch the database until you say so.

03

Destructive changes are gated

Drift is detected and classified. Drops require --accept-data-loss. Irreconcilable conflicts halt with structured errors.

04

Machine-readable by default

JSON output with deterministic operation IDs, hash anchoring, and timings. Built for CI pipelines and AI agents.
