Production app flow

How a Stripe dispute becomes an approved evidence response.

This tour follows the complete app shape: Stripe posts a signed webhook, Prisma Compute durably ingests it, a worker collects evidence from provider APIs, finance reviews high-value disputes, then the workflow submits evidence and records what was approved.

Source, runtime, worker, approval, outbox, Studio

The schema makes Stripe ingestion explicit.

The workflow is selected by source and event type. The dedupe expression points at the Stripe dispute id, so retries for the same dispute collapse into one case. 

workflow DisputeEvidence {
  trigger stripeDisputeCreated {
    source = "stripe"
    event = "charge.dispute.created"
    dedupeBy = "event.data.object.id"
  }

  approval humanApproval {
    when = "state.amount > 50000"
    assignees = ["role:finance_ops"]
    onApprove = submitEvidence
  }
}
  • Production route Stripe posts to `/api/prisma-workflows/ingest/stripe/:accountId?`.
  • Dedupe policy `event.data.object.id` makes the dispute the unit of work, not the delivery attempt.
  • Threshold `50000` is cents, so disputes over $500 pause for review.

Generation turns the schema into a Compute app.

`workflow generate` emits a manifest, typed client helpers, Studio artifacts, and `compute.ts`. In production, `DATABASE_URL` selects `PostgresWorkflowStore`. 

const connectionString = process.env['DATABASE_URL'];
const store = connectionString
  ? new PostgresWorkflowStore({
      connectionString,
      schemaName: "_prisma_workflows",
    })
  : undefined;

export function createApp(options = {}) {
  return createWorkflowHttpApp({
    manifest,
    steps,
    connectors: { ...connectors, ...(options.connectors ?? {}) },
    ...(store ? { store } : {}),
  });
}
  • Generated imports Step modules are imported from `src/steps/*` and wired by step name and file path.
  • Connector override Deployment code can pass a real Stripe connector and secrets without changing the schema.
  • Runtime routes The app exposes ingest, worker, approval, replay, inspect, and Studio endpoints.

Webhook work ends after durable ingest.

The HTTP app preserves the raw request body for signature debugging, normalizes the event if a connector is present, stores the ingest event, creates a queued run, and returns. 

curl -X POST https://app.example.com/api/prisma-workflows/ingest/stripe/acct_live_123 \
  -H "stripe-signature: t=...,v1=..." \
  -H "content-type: application/json" \
  -d '{
    "type": "charge.dispute.created",
    "data": { "object": { "id": "du_001", "amount": 72500 } }
  }'

# Response for new work
{ "matchedWorkflows": ["DisputeEvidence"], "duplicate": false }
  • Signature first Failed connector verification returns `401` and writes no ingest event.
  • Atomic store The Postgres store inserts the ingest event, trigger match, run, and first timeline entry together.
  • Fast acknowledgement Provider calls happen later through `POST /api/prisma-workflows/run`.

The first step builds the case file.

The step reads `event.data.object`, extracts the dispute id, customer id, amount, currency, and reason, then calls the connected systems in parallel. 

const [hubspotHistory, shopifyOrders, zendeskTickets, stripeMetadata] =
  await Promise.all([
    providers.hubspot.customerHistory(customerId),
    providers.shopify.orderHistory(customerId),
    providers.zendesk.priorTickets(customerId),
    providers.stripe.paymentMetadata(disputeId),
  ]);

return {
  disputeId,
  customerId,
  customerEmail: stripeMetadata['receiptEmail'],
  amount,
  currency,
  reason,
  hubspotHistory,
  shopifyOrders,
  zendeskTickets,
  stripeMetadata,
};
  • Mocked example The repository uses mocked providers so tests run without external accounts.
  • Production replacement Real clients use account-scoped secrets, provider SDKs, rate limits, and typed responses.
  • State contract Every returned field is merged into workflow state and becomes visible in Studio snapshots.

Large disputes block until a human decides.

The draft response is created before the approval gate. If the amount is over $500, the run waits with an approval record assigned to finance operations.

{

  • Run status The workflow status changes to `waiting_for_approval`, not failed or completed.
  • Audit trail The approval decision, approver, reason, and state at the time of review are persisted.
  • Timeout path The schema can define timeout behavior for queues that need escalation.

External effects are marked and keyed.

The Stripe evidence submission and Slack summary are external side effects. Their idempotency expression uses `state.disputeId`, so retries do not submit a second evidence packet for the same dispute. 

step submitEvidence {
  run = "./src/steps/submit-evidence.ts"
  checkpoint = true
  sideEffects = "external"
  idempotency = "state.disputeId"
}

step postSummary {
  run = "./src/steps/post-summary.ts"
  sideEffects = "external"
  idempotency = "state.disputeId"
}
  • Stripe evidence The production step calls Stripe's dispute evidence API with the approved response.
  • Slack summary Finance and support can see the outcome without opening the workflow run.
  • Replay guard External side effects require explicit confirmation during replay.

Studio is the operations console.

Studio reads the same runtime data the worker writes: definitions, ingest events, runs, step attempts, state snapshots, approvals, outbox entries, and dead letters.

What changes between this example and production

The workflow definition stays the same. The production app swaps mocks for real connectors, deploys with Postgres, and configures operational controls.

Production checklist

These are the pieces a real app wires before accepting live Stripe events.

Before switching Stripe to live mode

Database

Provision Postgres, run workflow DDL, and set `DATABASE_URL` so Compute uses `PostgresWorkflowStore` with `_prisma_workflows`.

Stripe connector

Implement signature verification, event normalization, account scoping, and the dispute-level dedupe policy.

Secrets

Store Stripe, HubSpot, Shopify, Zendesk, and Slack credentials outside code and resolve them by tenant or account.

Worker cadence

Run `POST /api/prisma-workflows/run` from a worker loop or scheduler so webhook requests stay fast.

Approval queue

Connect `role:finance_ops` to the real user directory and expose approve or reject actions in the operator UI.

Observability

Watch dead letters, failed attempts, approval timeouts, duplicate events, and provider rate-limit responses in Studio.
