# Project Plan

## Summary

The project ships in eight milestones sequenced foundation â†’ consumers â†’ mechanism â†’ exemplars â†’ docs. The opening foundation milestone introduces the framework SPI interfaces (`ContractSerializer<TContract>`, `SchemaVerifier<TContract, TSchema>`), the framework `SchemaNodeBase` abstract class, the `Storage` interface with `namespaces` shape, and the per-SPI family abstract bases. The user-facing capabilities this project delivers are: multi-schema Postgres contracts with namespace declarations in PSL (top-level `namespace { â€¦ }` blocks; namespaces do not recursively nest) and the TS builder; cross-namespace FK references within a contract space via dot-qualified type references in `@relation` (the FL-02 fix that unblocks Supabase's `auth.users` story); connection-bound multi-tenancy via `__unspecified__` + `search_path`. ## Delivery sequencing â€” split into two PRs

The project ships across **two PRs** to keep each PR's review burden manageable and to isolate risk between the proven mechanism (M1-M4) and the heavier greenfield namespace work (M5a + M5b). The project itself stays intact at `projects/target-extensible-ir/` across both PRs; only the delivery is split.

- **PR1 â€” `target-extensible-ir` branch (current).** Ships M1 + M2 + M2.5 + M3 + M3.5 + M4 + M6. Decision recorded at the M5a-pause point: M5a's pre-flight reconnaissance found the namespace work was substantial enough that bundling it into one PR with M1-M4 would double the diff and put the proven mechanism at risk if M5a hit a deeper architectural issue. ## Milestones

The eight milestones below correspond to the eight PRs (M1, M2, M2.5, M3, M3.5, M4, M5a, M5b, M6). Each milestone is a single PR. **M3 is not splittable into Postgres-then-SQLite:** the two SQL targets share family-level abstract bases (`SqlTable`, `SqlColumn`, `SqlForeignKey`, â€¦), and lifting one to the class-hierarchy shape without the other would leave the family bases in a dual-shape state â€” incompatible with NFR1 ("no dual-shape transition window"). The Postgres + SQLite lifts therefore land in one PR; SQLite's share of the work is mechanical within that scope.

### M1 â€” Foundation (framework interfaces + family abstract bases)

**Goal:** declare the SPI interfaces and per-SPI family abstract bases. No targets touched yet; the framework + family layers compile but have no consumers. This is the largest single design surface and lands before any consumer commits to the shape.

**Tasks:**

- [ ] Declare framework types in `1-framework/`:

  - `abstract class SchemaNodeBase implements SchemaNode` â€” centralised `freeze()` helper for subclasses (mirrors today's frozen-AST pattern in `OpFactoryCall` and `MongoSchemaNode`). Subclasses call `this.freeze()` in their constructors.
  - `interface Namespace extends SchemaNode { id: string; â€¦ }`.
  - `abstract class NamespaceBase implements Namespace`.
  - `interface Storage { readonly namespaces: Record<string, Namespace> }` â€” the framework-level promise that every IR carries namespaces (enforces FR15 at the type level).

### M2.5 â€” Mirror SQL's adapter-SPI dispatch pattern in Mongo

**Goal:** restore cross-target consistency (Principle 2 from the architectural-principles ADR) before SQL exercises the same SPI surface in M3. Today, Mongo's familyâ†’target dep edge is inverted (`@prisma-next/family-mongo` depends on `@prisma-next/target-mongo`, `@prisma-next/adapter-mongo`, and `@prisma-next/driver-mongo`). The inversion forced M2 R1 to place Mongo's SPI bases at `mongo-contract` (foundation) as a workaround. M3 will land SQL's SPI bases at `family-sql` (family layer) â€” without M2.5, the project ships with a Mongo-at-foundation / SQL-at-family asymmetry that future readers trip over and that any future Mongo target (Atlas, multi-tenant) inherits.

The SQL pattern (verified on disk in pre-M2.5-R1 reconnaissance, recorded in `wip/unattended-decisions.md Â§ 11`): `family-sql` declares a `SqlControlAdapter` SPI interface; the per-target adapters (`PostgresControlAdapter`, etc.) implement the SPI with method bodies that execute queries directly via `driver.query(...)` â€” the adapter is the natural home because it has both the connection (`driver`) and the dialect-specific query vocabulary. `family-sql/control-instance.ts` dispatches via `getControlAdapter().readMarker(driver, APP_SPACE_ID)` rather than importing target/adapter symbols directly. The result: `family-sql` carries zero `target-* / adapter-* / driver-*` runtime deps. M2.5 mirrors this in Mongo.

**Original "relocation only" framing (corrected pre-R1).** M2.5 was originally specified as a flat file relocation of `mongo-target-descriptor.ts` from family to target. Pre-R1 reconnaissance found that the dep severance the original spec promised could not be reached by the file move alone â€” `family-mongo/control-instance.ts` carries seven additional runtime imports of target/adapter symbols (marker ops, schema verifier, operation formatter, target contract type, introspection helper) that the descriptor file relocation does not touch. The corrected spec rewrites M2.5 as the architecturally correct shape (SPI dispatch); the descriptor relocation falls out as one of nine orchestrated changes. See `wip/unattended-decisions.md Â§ 11` for the full reasoning trail and the user-validated call to land the corrected scope.

**Sequencing.** Runs AFTER the post-M2 R2 PR 494 rebase (so the multi-space runner code added by PR 494 to `mongo-target-descriptor.ts` is in our tree before we move the file) and BEFORE M3 (so SQL's SPI base placement is symmetric with Mongo's from the start, and so the lessons from this Mongo dispatch refactor inform M3's SPI surface where applicable).
