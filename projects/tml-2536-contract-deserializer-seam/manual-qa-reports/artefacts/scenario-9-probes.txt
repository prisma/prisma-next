PROBE 2 — delete storage.types entirely

mutation: del d['storage']['types']
exit: 0
envelope:
{
  "ok": true,
  "noOp": true,
  "from": "sha256:f7a8eb5...",
  "to":   "sha256:f7a8eb5...",
  "operations": [],
  ...
}

judgement: SILENT ACCEPTANCE. The planner returns "no changes" against a snapshot whose storage.types is missing entirely. Cause: the planner appears to use migration.json's `to`/`from` hashes for diff detection, not the actual deserialized end-contract.json contents. So a semantically-inconsistent snapshot (here: missing required polymorphic types) passes without surfacing the inconsistency. Not a TML-2536 regression — the original bug was about CRASHING on untagged entries, which is now fixed — but a diagnostic-surface gap.

SUMMARY

- The strict deserializer fires loudly on structurally-malformed shapes (Probes 1, 3, 6) with named entry / named value / named file path — the diagnostic is actionable.
- TWO classes silently pass: deleting `storage.types` entirely (Probe 2) and renaming a type (Probe 4). The planner uses migration.json's stored hashes for diff detection rather than re-comparing the deserialized snapshot's storage shape against the live contract's. This isn't a TML-2536 regression but is a diagnostic-surface gap worth tracking.
- File-permission failures on a contract read (Probe 5) surface as the generic `PN-CLI-4999 Unexpected error` envelope rather than a dedicated I/O-failure diagnostic. The EACCES message is in the body, but the envelope summary + fix string aren't informative.

No probe reproduced the original TML-2536 bug class (untagged storage.types crashing the planner).
