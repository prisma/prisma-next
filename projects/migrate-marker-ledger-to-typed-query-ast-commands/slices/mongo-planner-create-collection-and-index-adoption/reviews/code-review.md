## Findings

### Minor

**M1 — The lowering oracle is a subset matcher, not the "exact wire command" / "byte-parity" the spec and D1 dispatch claim.**
`packages/3-mongo-target/2-mongo-adapter/test/ddl-lowering-oracle.test.ts` uses `toMatchObject` throughout (e.g. lines 34–38, 123–128). `toMatchObject` asserts the listed keys are present but does **not** assert the absence of extra keys, so an arm that accidentally wrote an unintended option field would still pass. The "omits undefined options" cases compensate partially: createCollection checks `Object.keys(opts).toHaveLength(0)` (line 111), but the createIndex equivalent only spot-checks three of ~10 fields (lines 245–247). The spec (§ Cross-cutting, "the exact wire command produced per node") and the D1 dispatch ("byte-parity oracle", "assert the exact wire command object") overstate what the test pins.
*Fix:* either switch the option-object assertions to `toEqual` on `wire.options`, or add a "no extra keys" check (`expect(Object.keys(opts).sort()).toEqual([...])`) to one representative case per kind. Field-by-field parity against the retired executor is in fact complete — this is about the test's strength matching its stated claim, not a real parity gap.

**M2 — Redundant `unique` re-assignment in the createIndex factory (inherited, not new).**
`packages/3-mongo-target/1-mongo-target/src/core/migration-factories.ts:159-163` spreads `...options` and then re-sets `unique: options?.unique ?? undefined` to the same value it already spread. This is dead noise. It is **pre-existing on main** (the old factory had the identical line), so it is out of slice scope and does not block — but the slice rewrote these exact lines through the builder and could have dropped it. Flagging for a one-line cleanup.
*Fix:* delete the `unique: options?.unique ?? undefined,` line; `...options` already carries it.

**M3 — `voidToAsyncIterable` runs the DDL operation eagerly, breaking the laziness symmetry with the DML generator arms.**
`packages/3-mongo-target/3-mongo-driver/src/mongo-driver.ts:30-37,78-97`. The DML arms are `async *` generators: the DB call does not fire until the iterator is pumped. The DDL arms call `executeCreateCollectionCommand(...)` (etc.) **eagerly** to produce the `Promise<void>` passed to `voidToAsyncIterable`, so the operation starts the moment `execute()` is called, before any iteration. Two consequences: (a) if a caller ever calls `execute()` on a DDL command and discards the iterable without iterating, the operation still runs and its rejection becomes an unhandled promise rejection (no `.catch`); (b) it is an asymmetry a future reader will not expect from a function named like the lazy arms. In practice both call sites — the runner DDL loop (`mongo-runner.ts:154-159`) and the data-transform loop — always iterate, so there is no live bug today. Re-iteration is safe (`op` is captured once; `.then` on a settled promise does not re-run the operation), so there is no double-execution. Open question 2 (error fidelity) is correctly answered and covered by the negative tests.
*Fix (optional):* make the DDL arms `async function*` that `yield` nothing after `await this.executeX(cmd)`, matching the DML idiom and restoring laziness — then `voidToAsyncIterable` and the five `castAs` wrappers disappear. Acceptable to leave as-is given both call sites iterate; recommend the generator form for consistency.

### Nit

**N1 — D3 helper widened the framework miss-check from `=== undefined` to falsy.**
`packages/1-framework/1-core/framework-components/src/shared/resolve-codec.ts:14` checks `if (!descriptor)`. The framework `extractCodecLookup` call site previously checked `if (d === undefined)` (control-stack.ts); the runtime side already used `!descriptor`. The unification standardises on falsy. A descriptor that is falsy-but-defined (e.g. `null`) would now throw where the framework side previously did not — but descriptor indexes return `undefined` for misses and a descriptor object is always truthy, so this is behavior-preserving in practice. No action needed; noting the semantic widening for the record.

**N2 — Two `createCollection` behavioral tests soft-pass on MMS capability gaps.**
`packages/3-mongo-target/2-mongo-adapter/test/command-executor.test.ts:258-268` (timeseries) and `:275-285` (clustered) wrap `execDdl` in `try { … } catch { return; }`, so the test passes whether the op succeeds **or** throws. That hides a real lowering regression for those two option fields behind the MMS-support escape. The lowering oracle still pins both fields structurally (oracle lines 81-92, 68-79), so parity is not unguarded — but the behavioral assertion is effectively disabled on these runners. Consider asserting the error is specifically a capability error rather than swallowing all throws.

**N3 — Inherited `Buildable` / bare `as { build: unknown }` smell is pre-existing, not entered here.**
The review brief lists this as "entered with D1". It is not: `interface Buildable`, `isBuildable`, `resolveQuery`, and the bare `value as { build: unknown }` cast all exist on `origin/main` (`migration-factories.ts:32-46` on main) and `git log -S "interface Buildable" origin/main..HEAD` returns empty — this branch neither adds nor touches them. So there is nothing to judge in this slice; the smell is correctly out of scope. If it is to be addressed, it belongs in a separate cleanup (the cast could be replaced with a `blindCast`/type-predicate, but the whole `Buildable` indirection predates this work).

## Conformance checklist (all pass)

- **Five DDL kinds lower; runner path uniform.** `lowerDdlCommand` (mongo-adapter.ts:56-118) handles all five; `lower`'s `'collection' in plan` discriminator (mongo-adapter.ts:355-360) correctly routes DDL (`{command}`, no top-level `collection`) vs DML (`MongoQueryPlan` carries `collection`). Runner DDL loop (mongo-runner.ts:154-159) is identical in shape to the data-transform loop (330-335).
- **`MongoCommandExecutor` fully retired.** File deleted; `grep` for the class and `core/command-executor` import path returns empty; runner-deps `commandExecutor` slot gone; export switched to `MongoInspectionExecutor` from the new `inspection-executor.ts`. `MongoInspectionExecutor` is byte-identical to the old inspection half, including the NamespaceNotFound (code 26) → `[]` handling.
- **ops.json serialization untouched.** Serializer revival path is the only remaining `new CreateIndexCommand(`/`new CreateCollectionCommand(` (mongo-ops-serializer.ts:491,513) — explicitly exempt. `toOp()` stays sync; no lowerer threading.

Each field is covered by an oracle case and (for the major ones) a behavioral case. Explicit index `name` is always on the wire for the production path: the factory passes `name = defaultMongoIndexName(keys)` unconditionally (migration-factories.ts:139,162) and the arm writes it whenever defined (mongo-adapter.ts:84).
- **Construction via the builder.** `collection(name).createCollection(options)` / `.createIndex(keys, options)` are real terminals returning the canonical frozen nodes (collection.ts:175-184) — no parallel option shapes (reuse `CreateCollectionOptions`/`CreateIndexOptions`/`MongoIndexKey`), no second AST layer. - **`.and()` combinator** added beside `.not()` (filter-expressions.ts:20-22) — accepted fact.
- **D3** is its own commit (`cd4cda3ba`), single-sources the find→throw→materialize logic into `resolveCodecDescriptorOrThrow`, leaves the two descriptor indexes unmerged across the plane split (deliberate, accepted fact), and pins both sides with tests.
- **Repo conventions.** No new bare `as` in production source (the DDL arms use `castAs`/`blindCast` with reasons consistent with the existing switch defaults; the one bare `as` in the driver's exhaustive default at mongo-driver.ts:101 is pre-existing).
