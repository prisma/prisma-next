# Walk-schema audit

> **Stability note.** This audit is consumed by Phase 1 (walk-schema → class-flow IR) and Phase 4 (planner collapse). If either phase discovers the audit is wrong for a branch, the audit is updated before proceeding.

## Scope and conventions

The audit covers the two walk-schema pipelines reachable from `createPostgresMigrationPlanner().plan(...)`:

- `packages/3-targets/3-targets/postgres/src/core/migrations/planner.ts` — the private `buildX` methods on `PostgresMigrationPlannerImpl`. These are the additive walk-schema branches (create table, add column, add index/constraint, etc.) plus the two "lift external op" branches (`buildDatabaseDependencyOperations`, `buildStorageTypeOperations`).
- `packages/3-targets/3-targets/postgres/src/core/migrations/planner-reconciliation.ts` — `buildReconciliationCallFromIssue` and its helpers, the `SchemaIssue`-driven reconciliation branches (drop table/column/index, ALTER TYPE, etc.).

`planner-recipes.ts` exports only two helpers — `buildAddColumnOperationIdentity` and `buildAddNotNullColumnWithTemporaryDefaultOperation`. They are pure composition helpers used from `planner.ts:buildAddColumnItem`; they do not gate their own branch and are not rows below.

`planner-strategies.ts` is out of scope by the spec (already issue-based). Its strategies are referenced in the "Covered by existing strategy?" column to show where Phase 4 would dedupe walk-schema logic against the issue pipeline. The strategies in play are:
- `notNullBackfillStrategy` — `nullability_mismatch` where the contract wants `NOT NULL`.
- `typeChangeStrategy` — `type_mismatch`.
- `nullableTighteningStrategy` — `nullability_mismatch` where the contract wants nullable.
- `enumChangeStrategy` — `enum_values_changed`.

`SchemaIssue` kinds referenced below are the union defined at [`packages/1-framework/1-core/framework-components/src/control-result-types.ts`](../../../packages/1-framework/1-core/framework-components/src/control-result-types.ts) (`BaseSchemaIssue.kind` plus `EnumValuesChangedIssue`). The shipped Phase 1 class-flow IR lives at [`packages/3-targets/3-targets/postgres/src/core/migrations/op-factory-call.ts`](../../../packages/3-targets/3-targets/postgres/src/core/migrations/op-factory-call.ts).

Each walk-schema branch that produces at least one `PostgresOpFactoryCall` (or a `SqlPlannerConflict`) is one row. The "Corresponding `OpFactoryCall` class(es)" column reflects what the shipped Phase 1 implementation already constructs — the audit is retrospective against the shipped code, not prescriptive for an unshipped design.

## `NONE` summary — branches whose trigger is not surfaced by `verifySqlSchema`

| Branch | Why it's `NONE` | Proposed minimal extension |
|---|---|---|
| `buildFkBackingIndexOperations` (`planner.ts`) | `verifySqlSchema` emits index issues only for user-declared indexes; it has no concept of "FK backing index missing". | Add an `fk_backing_index_missing` variant of `BaseSchemaIssue` carrying `table`, `indexOrConstraint` (the default name), and the FK `columns`. Alternative: extend `index_mismatch` with an optional `fkBacking?: true` flag so the variant union stays flat. |
| `buildAddColumnItem` shared-temp-default / empty-table branches (`planner.ts`) | The branch choice depends on schema-shape joins (`SchemaTableLookup`, row-count safety), not on a distinct issue kind. | No extension needed; the retarget re-walks `options.schema` inside the `missing_column` handler. Only add an issue field if Phase 4 prefers to pre-compute the branch decision in the verifier. |
| `buildStorageTypeOperations` (when the codec hook needs to fire for a fresh type) | `type_missing` carries only `typeName`, not the codec-planned ops. | No extension needed if the handler is allowed to re-invoke `codecHooks.get(...).planTypeOperations(...)` keyed off `type_missing`. Add `plannedOps?: readonly SqlMigrationPlanOperation[]` on `type_missing` only if the retarget intentionally avoids re-dispatching through the codec at reconcile time. |
| `buildDatabaseDependencyOperations` (`planner.ts`) | `dependency_missing` carries only `dependencyId`, not the install ops that `PlannerDatabaseDependency.install` exposes. | No extension needed if the handler re-invokes `collectInitDependencies(options.frameworkComponents)` keyed off `dependency_missing`. Add `installOps?: readonly SqlMigrationPlanOperation[]` only if the retarget intentionally decouples the handler from framework-component collection. |

## Summary

- Total walk-schema branches catalogued: **20** (10 in `planner.ts`, 9 in `planner-reconciliation.ts`'s reconciliation switch + 1 in `convertIssueToConflict`).
- Absorbable-as-is (`yes`): **14**.
- Absorbable with an issue extension (`yes-with-issue-extension`): **6** — the four `NONE`-column branches above, plus the two flagged with silent-no-op / autoincrement caveats (`default_missing`, `default_mismatch`).
- Needs deeper investigation (`no-investigate`): **0**.

The shipped Phase 1 class-flow IR (`PostgresOpFactoryCall` hierarchy in [`op-factory-call.ts`](../../../packages/3-targets/3-targets/postgres/src/core/migrations/op-factory-call.ts)) already covers every concrete call class named in the tables above: `CreateTableCall`, `AddColumnCall`, `DropTableCall`, `DropColumnCall`, `AddPrimaryKeyCall`, `AddUniqueCall`, `AddForeignKeyCall`, `CreateIndexCall`, `DropIndexCall`, `DropConstraintCall`, `SetNotNullCall`, `DropNotNullCall`, `AlterColumnTypeCall`, `SetDefaultCall`, `DropDefaultCall`, `CreateEnumTypeCall`, `AddEnumValuesCall`, `DropEnumTypeCall`, `RenameTypeCall`, `CreateExtensionCall`, `CreateSchemaCall`, `RawSqlCall`, `DataTransformCall`. No row reads "(new call class)" — Phase 1's IR inventory is complete relative to walk-schema.

`RawSqlCall` is used as the single escape hatch for the two `buildAddColumnItem` non-default branches and for codec-hook-planned / dependency-install ops that do not match the structured call shapes. Phase 4's absorption work is therefore not gated on new call classes; it is gated on the issue-extension decisions flagged in the `NONE` summary above.
