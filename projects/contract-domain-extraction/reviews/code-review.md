## Summary

This branch extracts shared domain-level `.d.ts` generation utilities into the framework emitter, implements a Mongo target-family emitter hook that consumes them, and builds a Mongo demo app whose contract artifacts are produced by the emitter pipeline. 76 new tests (34 shared utilities + 38 Mongo emitter + 4 full-stack integration), all passing. SQL emitter regression verified (116 tests pass, unchanged). The emitter → runtime loop is closed: the demo's `contract.json` and `contract.d.ts` are generated by `emit()` + `mongoTargetFamilyHook`. Two infrastructure fixes (canonicalization and contract schema) were needed to support the pipeline.

All actionable review findings from the prior round have been addressed in `d520bafa3`. Findings from jkomyno's PR review have been triaged and integrated below.

## What Looks Solid

- **Clean separation of shared vs. family-specific code.** The 8 extracted functions are pure, well-scoped, and compose naturally. The Mongo hook consumes them cleanly while adding only Mongo-specific storage/validation logic.
- **Thorough test coverage.** 34 tests for shared utilities covering edge cases (special characters, empty inputs, nested objects). 38 tests for the Mongo hook across validation, generation, and end-to-end scenarios.
- **ADR conformance is explicit.** Tests assert the *absence* of `strategy` on relations (ADR 177). The generated type structure matches ADR 172's domain-storage separation.
- **No SQL regression.** All 116 SQL emitter tests pass unchanged. The shared extraction is purely additive.
- **Good fixture design.** The blog contract IR fixture covers the key Mongo patterns: root models, embedded/owned models, `storage.relations`, domain-level relations with join details.
- **Emitter pipeline end-to-end.** The demo's `contract.json` and `contract.d.ts` are generated by `emit()` with `mongoTargetFamilyHook`, then consumed by the full Mongo runtime stack.
- **Supporting infrastructure fixes are well-targeted.** The canonicalization and contract schema changes are minimal and scoped to the exact issues that blocked the pipeline.
- **Hook type safety.** `mongoTargetFamilyHook` now uses `as const satisfies TargetFamilyHook`, catching SPI drift at compile time.
- **Shared test utility.** `createMongoIR` extracted to `test/fixtures/create-mongo-ir.ts`, eliminating duplication across 3 test files.
- **Consistent `readonly` in generated types.** Model parts (`fields`, `relations`, `storage`, `owner`, etc.) now use `readonly` consistently throughout the generated `.d.ts`.
- **`OperationTypes` and `executionHash` included.** Generated output now includes `OperationTypes` export, `MongoTypeMaps<CodecTypes, OperationTypes>`, and optional `executionHash` on `ContractBase` — aligned with the SQL hook's output.
- **Validation completeness.** `storage.relations` keys are now verified against model-level relations, preventing orphaned storage relation entries.
- **Codec injection guard.** The generation script asserts the string replacement succeeded, preventing silent failures if the emitter output format changes.

## Blocking Issues

_None remaining._

### ~~F11 — No Mongo control plane for codec type contribution~~ (Resolved)

**Location:** [examples/mongo-demo/scripts/generate-contract.ts](../../../examples/mongo-demo/scripts/generate-contract.ts)

**Issue:** The generation script patched `CodecTypes` in the emitted `.d.ts` via string replacement because there was no Mongo descriptor/control-plane machinery to contribute `TypesImportSpec` entries for codec types.

**Resolution:** Added `@prisma-next/mongo-core/codec-types` subpath export with the `CodecTypes` type. The demo now passes `codecTypeImports` to `emit()`, eliminating the string replacement hack. Full descriptor-based control plane extraction deferred to M5/M6.

**Source:** jkomyno P2 (codec type injection hack), promoted from our F11.

## Non-Blocking Concerns

### F05 — Type casts on `ContractIR` generic fields

**Location:** [packages/2-mongo-family/3-tooling/emitter/src/index.ts](../../../packages/2-mongo-family/3-tooling/emitter/src/index.ts) — lines 149–150, 231

**Issue:** `ContractIR`'s generic type parameters require casts like `ir.storage as unknown as MongoStorageIR`. The `as unknown as` bypasses the type system completely. Comments explaining the necessity have been added (lines 149, 230), which satisfies the repo rule.

**Status:** Acceptable — the cast is documented and `validateStructure` performs runtime checks first. A proper fix requires narrowing `ContractIR` generics (M5 IR alignment).

### F09 — `_id: 'u1' as never` cast in test seeding

**Location:** [examples/mongo-demo/test/blog.test.ts](../../../examples/mongo-demo/test/blog.test.ts) — lines 50–53

**Issue:** Tests seed documents with string `_id` values using `as never`. A comment has been added explaining the cast is needed because the MongoDB driver expects `ObjectId` for `_id`.

**Status:** Acceptable — comment explains the intent.

### F12 — `_generated.regenerate` message references wrong command

**Location:** [examples/mongo-demo/src/contract.json](../../../examples/mongo-demo/src/contract.json) — `_generated` section

**Issue:** The `_generated` metadata says `"regenerate": "To regenerate, run: prisma-next contract emit"`, but the actual command is `pnpm emit`. This is baked into the `emit()` function, not the demo's fault.

**Status:** Informational — no action needed on this branch.

### F13 — Triplicate `ModelStorageField` in SQL JSON schema

**Location:** `packages/2-sql/2-authoring/contract-ts/schemas/data-contract-sql-v1.json`

**Issue:** The `ModelStorageField` key previously appeared three times in the same JSON object. Duplicate keys in JSON are a data integrity issue (last key wins). Pre-existing, not introduced by M3.

**Source:** jkomyno P1.

**Status:** Resolved — the file now has a single `ModelStorageField` entry (fixed during rebase conflict resolution).

### F14 — Misleading error in `resolveIncludeRelation`

**Location:** [packages/3-extensions/sql-orm-client/src/collection-contract.ts](../../../packages/3-extensions/sql-orm-client/src/collection-contract.ts) — `resolveIncludeRelation`

**Issue:** Error message is misleading when a relation can't be resolved. Introduced by M2 (now merged to main).

**Source:** jkomyno P2.

**Status:** Resolved — error messages now distinguish "relation not found" from "incomplete join metadata."

### F15 — `resolveModelTableName` throws instead of falling back

**Location:** [packages/3-extensions/sql-orm-client/src/collection-contract.ts](../../../packages/3-extensions/sql-orm-client/src/collection-contract.ts) — `resolveModelTableName`

**Issue:** Function now throws when `storage.table` is missing instead of falling back to a convention. Intentional change from M2, but the error path may be too aggressive.

**Source:** jkomyno P2.

**Status:** Resolved — throw behavior kept (correct for domain-storage design); error messages improved to distinguish "model not found" from "missing storage.table."

### F16 — Repeated `Record<string, unknown>.pipe(iterate-and-validate)` in `contract-schema.ts`

**Location:** [packages/2-mongo-family/1-core/src/contract-schema.ts](../../../packages/2-mongo-family/1-core/src/contract-schema.ts) — 5 occurrences

**Issue:** The loop-parse-collect pattern repeats 5 times for `fields`, `relations`, `variants`, `collections`, and `models`. Arktype natively supports `type({ '[string]': SubSchema })` for record validation, which would replace each occurrence with a one-liner.

**Source:** jkomyno P3.

**Status:** Resolved — replaced with Arktype native `{ '[string]': SubSchema }`.

### F17 — No canonicalization tests for Mongo collections

**Location:** [packages/1-framework/1-core/migration/control-plane/src/emission/canonicalization.ts](../../../packages/1-framework/1-core/migration/control-plane/src/emission/canonicalization.ts)

**Issue:** The `isRequiredCollections`/`isCollectionEntry` preservation rules added for Mongo have no dedicated tests. A regression would silently change Mongo contract hashes.

**Source:** jkomyno P3.

**Status:** Resolved — canonicalization tests added for `storage.collections` preservation.
