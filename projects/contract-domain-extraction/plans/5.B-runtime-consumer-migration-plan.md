# M5 Phase B — Runtime consumer migration

## Goal

Migrate the read side of the contract pipeline (validation → runtime) to the unified `Contract<TStorage, TModels>` type. Wire SQL validation through the framework's `validateContract` with a SQL-specific `StorageValidator`, redefine `SqlContract` as `Contract<SqlStorage, ...>`, update all runtime consumers to accept `Contract` instead of `ContractBase`, and remove `ContractBase`.

Phase A (write side) is complete: the emitter and authoring surfaces produce `Contract` directly, and `ContractIR` is eliminated. `SqlSemanticContractDefinition` remains as an authoring-level intermediate but now lowers to `Contract` instead of `ContractIR`. The contract JSON format is unchanged — it's in the ADR 172 shape. This phase updates the TypeScript types on the read side of the serialization boundary.

**Spec:** [projects/contract-domain-extraction/spec.md](../spec.md)
**ADR:** [ADR 182 — Unified contract representation](../../../docs/architecture%20docs/adrs/ADR%20182%20-%20Unified%20contract%20representation.md)
**Linear:** [TML-2191](https://linear.app/prisma-company/issue/TML-2191)

## Current state

### Two `validateContract` implementations

The framework and SQL families each have their own `validateContract`:

| | Framework (`@prisma-next/contract/validate-contract`) | SQL (`@prisma-next/sql-contract/validate`) |
|-|------------------------------------------------------|-------------------------------------------|
| **Signature** | `validateContract<T extends Contract>(value, storageValidator): T & ValidateContractResult` | `validateContract<T extends SqlContract<SqlStorage>>(value): T` |
| **Structural** | Arktype `ContractSchema` on stripped JSON | `normalizeContract()` + `validateSqlContract()` (arktype) |
| **Domain** | `validateContractDomain()` | Same `validateContractDomain()`, but result (warnings) discarded |
| **Storage** | Delegates to `storageValidator(contract)` | Inlined: `validateContractLogic()`, `validateModelStorageReferences()`, `validateStorageSemantics()` |
| **Post-processing** | None | `constructContract()` (strips `_generated`), `decodeContractDefaults()` (bigint literals) |
| **Used by** | Tests only | All SQL runtime consumers (via `@prisma-next/sql-contract/validate`) |

### Two contract types

| Type | Definition | Role |
|------|-----------|------|
| `ContractBase` | Framework `types.ts` — carries `schemaVersion`, `sources`, top-level `storageHash`, optional `execution`/`profileHash`, `roots`, `models` | Legacy JSON-aligned base type |
| `Contract<TStorage, TModels>` | Framework `contract-types.ts` — carries `storage: TStorage`, `models: TModels`, `roots`, `capabilities`, `execution?`, `profileHash` (required) | ADR 182 unified in-memory type |
| `SqlContract<S, TModels, ...>` | `Omit<ContractBase, 'models'> & { storage: S; models: TModels; execution?: ExecutionSection }` | SQL contract type (extends `ContractBase`) |

Key structural differences between `ContractBase` and `Contract`:

| Field | `ContractBase` | `Contract` |
|-------|---------------|-----------|
| `storageHash` | Top-level `contract.storageHash` | `contract.storage.storageHash` (on `StorageBase`) |
| `schemaVersion` | Present | Absent (persistence detail) |
| `sources` | Present | Absent (persistence detail) |
| `profileHash` | Optional | Required |
| `models` | `Record<string, DomainModel>` | `TModels extends Record<string, ContractModel>` |

## Design decisions

### 1. Redefine `SqlContract` as a `Contract` alias

`SqlContract` becomes a type alias for `Contract<SqlStorage, TModels>`, preserving its generic parameters:

```typescript
export type SqlContract<
  S extends SqlStorage,
  TModels extends Record<string, ContractModel<SqlModelStorage>>,
> = Contract<S, TModels>;
```

`SqlContract` has no default type parameters. Without literal types from `contract.d.ts`, a defaulted `SqlContract` (e.g. `SqlContract<SqlStorage, Record<string, ContractModel<SqlModelStorage>>>`) provides no useful model-specific type information. Forcing callers to provide type arguments makes intent clear: use `SqlContract<MyStorage, MyModels>` when you have concrete types from `contract.d.ts`, use `Contract<SqlStorage>` for generic bounds.

This is a breaking change at the type level — `SqlContract` no longer carries `schemaVersion`, `sources`, or a top-level `storageHash`. Consumer code that reads these fields must be updated.

The existing generic parameters (`TStorageHash`, `TExecutionHash`, `TProfileHash`) are dropped. These hash parameters were used for type-level hash branding but are no longer needed — `StorageBase<THash>` carries `storageHash`, and the unified `Contract` type doesn't parameterize execution/profile hashes at the top level.

### 2. SQL `validateContract` delegates to framework `validateContract`

The SQL `validateContract` is refactored to call the framework's `validateContract` with a SQL-specific `StorageValidator`:

```typescript
export function validateContract<TContract extends Contract<SqlStorage>>(
  value: unknown,
): TContract {
  const contract = frameworkValidateContract<TContract>(value, validateSqlStorage);
  return decodeContractDefaults(contract);  // SQL-specific post-processing (bigint literals)
}
```

The design becomes:

1. **Framework validation** (three-pass): structural (arktype — strips persistence fields including `_generated`), domain (`validateContractDomain`), storage (`validateSqlStorage`).
2. **Post-processing** (SQL-owned): `decodeContractDefaults()` (bigint literal decoding).

The `validateSqlStorage` function encapsulates the existing SQL validation logic:

```typescript
function validateSqlStorage(contract: Contract): void {
  validateContractLogic(contract as SqlContract);          // PK/FK/index column references
  validateModelStorageReferences(contract as SqlContract); // model.storage → storage.tables
  const errors = validateStorageSemantics((contract as SqlContract).storage);
  if (errors.length > 0) {
    throw new Error(`Contract semantic validation failed: ${errors.join('; ')}`);
  }
}
```

### 3. Remove `normalizeContract` — no legacy format support

After Phase B, all contract JSON artifacts are in the ADR 172 shape (produced by the updated emitter from Phase A). There is no need to support legacy JSON formats. `normalizeContract` is removed entirely.

The responsibilities it currently handles are distributed:

- **Persistence field stripping** (`schemaVersion`, `sources`, `_generated`): moved to the framework `validateContract`'s `stripPersistenceFields` (which already strips `schemaVersion` and `sources`; `_generated` is added).
- **Column/FK/nullable defaults**: the emitter produces well-formed JSON with all defaults explicit. If any defaulting is still required, it belongs in the framework's arktype schema `.pipe()` transforms or the SQL `StorageValidator`, not in a separate normalizer.
- **`storageHash` resolution**: contracts always carry `storageHash` on the `storage` block (emitter stamps it there since Phase A). No derivation from top-level `storageHash` needed.

`constructContract` is also removed — its only job was stripping `_generated`, which the framework now handles.

### 4. `storageHash` migration: `contract.storageHash` → `contract.storage.storageHash`

`Contract` has `storageHash` on `contract.storage.storageHash` (via `StorageBase`), not at the top level. All ~10 runtime sites that read `contract.storageHash` must migrate.

The migration is mechanical:
- `contract.storageHash` → `contract.storage.storageHash`
- Type constraints update from `{ storageHash: string }` to `{ storage: { storageHash: string } }`

The framework `RuntimeCore` (which accepts `TContract = unknown`) narrows the contract inline — its hash read at line ~173 changes from `contract.storageHash` to `contract.storage.storageHash` with an updated type narrowing.

### 5. `ContractBase` removal is the final step

`ContractBase` is removed only after:
1. `SqlContract` is redefined as a `Contract` alias (no longer `Omit<ContractBase, 'models'> & ...`)
2. All runtime consumers compile against `Contract` (no remaining `ContractBase` imports outside tests)
3. The `DomainRelationOn` import in `SqlContract`'s file is switched to `ContractRelationOn`

The remaining deprecated `Domain*` type aliases are removed at the same time — grep confirms they are consumed only in domain types re-exports, validate-domain, and test files that can be updated.

### 6. `ExecutionSection` vs `ContractExecutionSection`

`ContractBase` uses `ExecutionSection` (optional `executionHash`). `Contract` uses `ContractExecutionSection` (required `executionHash` when the section is present). SQL consumers that read `contract.execution?.executionHash` need no change — both types make it available, just with different optionality. The `SqlContract` alias carries `Contract`'s `ContractExecutionSection` automatically.

### 7. No changes to the contract JSON format

The emitted `contract.json` is unchanged. Phase B only updates TypeScript types on the read side. The framework `validateContract` strips persistence fields and validates the remaining structure against the `Contract` schema.

### 8. `PostgresContract` becomes a `Contract` narrowing

`PostgresContract` changes from `SqlContract<SqlStorage> & { target: 'postgres' }` to `Contract<SqlStorage> & { target: 'postgres' }` (equivalent after `SqlContract` is aliased). The adapter's `Adapter<AnyQueryAst, PostgresContract, ...>` signature is unchanged.

## Steps

### Step 1: Redefine `SqlContract` as a `Contract` alias

**File:** `packages/2-sql/1-core/contract/src/types.ts`

Replace the current `SqlContract` definition:

```typescript
// Before
export type SqlContract<
  S extends SqlStorage = SqlStorage,
  TModels extends Record<string, unknown> = Record<string, unknown>,
  TStorageHash extends StorageHashBase<string> = StorageHashBase<string>,
  TExecutionHash extends ExecutionHashBase<string> = ExecutionHashBase<string>,
  TProfileHash extends ProfileHashBase<string> = ProfileHashBase<string>,
> = Omit<ContractBase<TStorageHash, TExecutionHash, TProfileHash>, 'models'> & {
  readonly targetFamily: string;
  readonly storage: S;
  readonly models: TModels;
  readonly execution?: ExecutionSection;
};

// After
export type SqlContract<
  S extends SqlStorage,
  TModels extends Record<string, ContractModel<SqlModelStorage>>,
> = Contract<S, TModels>;
```

**Impact:** `SqlContract` loses `schemaVersion`, `sources`, top-level `storageHash`, and the `TStorageHash`/`TExecutionHash`/`TProfileHash` generic parameters. Consumer code that:
- Passes >2 generic arguments to `SqlContract<...>` must be updated
- Reads `contract.schemaVersion` or `contract.sources` must be updated (these are persistence-only fields)
- Reads `contract.storageHash` must switch to `contract.storage.storageHash`

**Tests:** Update type tests in `packages/2-sql/1-core/contract/test/` and `packages/1-framework/1-core/shared/contract/test/` to reflect the new `SqlContract` shape. Remove assertions about `SqlContract extends ContractBase` (it no longer does). Add assertions about `SqlContract extends Contract`.

### Step 3: Wire SQL `validateContract` through framework `validateContract`

**File:** `packages/2-sql/1-core/contract/src/validate.ts`

Refactor to compose with the framework validation:

1. **Extract `validateSqlStorage`** — a `StorageValidator` that encapsulates the existing SQL-specific validation logic (`validateContractLogic`, `validateModelStorageReferences`, `validateStorageSemantics`). These functions already exist; the new `validateSqlStorage` is a thin wrapper that calls them in sequence.

2. **Refactor `validateContract`** — call `frameworkValidateContract(value, validateSqlStorage)` instead of inlining structural/domain validation.

3. **Remove `normalizeContract`** — all legacy format handling is eliminated (see design decision 3). The framework `validateContract` handles persistence field stripping.

4. **Remove `constructContract`** — `_generated` stripping moves to the framework's `stripPersistenceFields`.

5. **Keep `decodeContractDefaults`** as SQL-specific post-processing (bigint literal decoding).

6. **Surface domain warnings** — the framework `validateContract` returns `ValidateContractResult` with `warnings`. The SQL `validateContract` should surface these. This is a new capability — currently domain warnings are silently discarded.

```typescript
import {
  validateContract as frameworkValidateContract,
  type StorageValidator,
} from '@prisma-next/contract/validate-contract';

const validateSqlStorage: StorageValidator = (contract) => {
  const sqlContract = contract as Contract<SqlStorage>;
  validateContractLogic(sqlContract);
  validateModelStorageReferences(sqlContract);
  const semanticErrors = validateStorageSemantics(sqlContract.storage);
  if (semanticErrors.length > 0) {
    throw new Error(`Contract semantic validation failed: ${semanticErrors.join('; ')}`);
  }
};

export function validateContract<TContract extends Contract<SqlStorage>>(
  value: unknown,
): TContract {
  const validated = frameworkValidateContract<TContract>(value, validateSqlStorage);
  return decodeContractDefaults(validated);
}
```

**Framework `stripPersistenceFields` update:** Add `_generated` to the fields stripped by the framework's `validateContract`. Currently it strips `schemaVersion` and `sources`; `_generated` is the same kind of emission-time metadata.

**Deletions:** `normalizeContract` and `constructContract` (and `construct.ts`) are removed entirely.

### Step 4: Update runtime consumer type bounds

Update all consumers from `TContract extends SqlContract<SqlStorage>` to `TContract extends Contract<SqlStorage>`. Since `SqlContract` is now an alias for `Contract<SqlStorage, ...>`, these are equivalent — but switching to `Contract` directly makes the intent clearer and removes the dependency on the SQL-specific alias where the framework type suffices.

**Framework-level consumers** (should use `Contract`, not `SqlContract`):
- `RuntimeCore<TContract>` in `runtime-core.ts` — already generic over `TContract = unknown`, just update the hash read
- `RuntimeFamilyAdapter<TContract>` in `runtime-spi.ts` — already generic

**SQL-level consumers** (can use either `SqlContract` or `Contract<SqlStorage>`):

- `createExecutionContext()` — same
- `SqlFamilyAdapter<TContract>` — same
- `sql()` / `Db<C>` — same

**Extensions** (depend on SQL):

Since `SqlContract<S, M>` = `Contract<S, M>` after Step 1, most consumer sites just need a recompile — they already satisfy `Contract`. The explicit type annotations are updated for clarity and to remove the unnecessary indirection.

**Generic bound migration:** `SqlContract` no longer has default type parameters. Sites that used `SqlContract<SqlStorage>` as a generic bound (e.g. `TContract extends SqlContract<SqlStorage>`) must switch to `Contract<SqlStorage>`. `SqlContract` is reserved for sites that have concrete type arguments from `contract.d.ts`.

## Sequencing

Steps can be partially parallelized, but the core dependency chain is:

```
Step 1 (redefine SqlContract)
  ↓
Step 2 (migrate storageHash reads) ←── can start in parallel with Step 3
  ↓
Step 3 (wire SQL validateContract through framework)
  ↓
Step 4 (update consumer type bounds) ←── mostly a recompile; can overlap with Step 3
  ↓
Step 5 (remove ContractBase)
  ↓
Step 6 (remove Domain* aliases)
  ↓
Step 7 (tests + verification)
```

In practice, Steps 1–4 are likely done together as a single coherent change since redefining `SqlContract` immediately cascades type errors to all consumers. Steps 5–6 are cleanup that can be a separate commit.

## Risks

1. **No default type parameters on `SqlContract`.** The old `SqlContract` defaulted `TModels` to `Record<string, unknown>`, allowing `SqlContract<SqlStorage>` without specifying models. The new alias has no defaults — callers must provide both type arguments. Sites that used `SqlContract<SqlStorage>` as a generic bound must switch to `Contract<SqlStorage>` (which has sensible defaults). This surfaces compilation errors at sites that relied on the defaults, but each error points to a site that should be using `Contract<SqlStorage>` for generic bounds or providing concrete types.

2. **Hash branding.** `ContractBase` parameterized hash types (`StorageHashBase<THash>`, etc.) via generic parameters. `Contract` does not expose these as top-level generics — `StorageBase<THash>` carries the hash branding within `TStorage`. This means `SqlStorage<THash>` naturally propagates the branded hash into `Contract`. Consumer code that explicitly references `TStorageHash` type parameters on `SqlContract` must be simplified.

3. **Framework `RuntimeCore` is untyped.** It uses `TContract = unknown` and narrows inline. The `storageHash` migration (Step 2) updates the narrowing cast. This is safe but relies on runtime correctness rather than type checking. Future work could add a minimal contract structural bound to `RuntimeCore`.
