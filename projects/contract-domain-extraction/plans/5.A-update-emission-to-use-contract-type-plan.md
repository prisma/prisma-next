# M5 Phase A — Update emission to use `Contract` type

## Goal

Migrate the write side of the contract pipeline (authoring → emitter) from `ContractIR` to the unified `Contract<TStorage, TModels>` type. Authoring surfaces produce `Contract` directly, the emitter accepts `Contract` instead of `ContractIR`, and hash computation moves from the emitter to the authoring/storage layer. `ContractIR` and `SqlSemanticContractDefinition` are eliminated.

The read side is untouched: `SqlContract` (which extends `ContractBase`) stays as-is, `validateContract()` continues returning `ContractBase`-derived types with top-level `storageHash`, and no runtime consumers are modified. That's Phase B.

**Spec:** [projects/contract-domain-extraction/spec.md](../spec.md)
**ADR:** [ADR 182 — Unified contract representation](../../../docs/architecture%20docs/adrs/ADR%20182%20-%20Unified%20contract%20representation.md)
**Linear:** [TML-2190](https://linear.app/prisma-company/issue/TML-2190)

## Design decisions

### 1. Phase A is write-side only

Phase A changes what goes *into* `emit()` and what authoring surfaces *produce*. It does not change what `validateContract()` *returns* or what runtime consumers *read*.


| Boundary                    | Before Phase A                          | After Phase A                           |
| --------------------------- | --------------------------------------- | --------------------------------------- |
| Authoring output            | `ContractIR`                            | `Contract<SqlStorage, SqlModelStorage>` |
| Emitter input               | `ContractIR`                            | `Contract`                              |
| Emitter JSON output         | Same                                    | Same (byte-identical)                   |
| `validateContract()` return | `SqlContract` (extends `ContractBase`)  | Unchanged                               |
| Runtime consumers           | Read `contract.storageHash` (top-level) | Unchanged                               |


`SqlContract` stays defined as `Omit<ContractBase, 'models'> & { ... }`. It gains `contract.storage.storageHash` as a side effect of `SqlStorage` extending `StorageBase`, but consumers don't need to change — both paths work. The `SqlContract` = `Contract<S,M>` redefinition and consumer migration happen in Phase B.

### 2. Hash computation moves to authoring

Currently the emitter computes all three hashes (`storageHash`, `executionHash`, `profileHash`) from canonical JSON. This contradicts ADR 182's stated principle ("family-owned hashing") and creates a design problem: `Contract` carries `storage.storageHash`, but authoring can't compute it without serializing first.

The cycle breaks trivially: hash the content *excluding* the hash field, then stamp it in.

```
1. Build storage block without hash:  { tables: { ... } }
2. Compute hash:                      hash(canonicalize({ target, targetFamily, storage }))
3. Create storage block with hash:    { storageHash: hash, tables: { ... } }
4. Create Contract with full storage
```

Common `canonicalize()` and `hash()` utilities make this available to any authoring surface. The canonicalization semantics:

- Sort keys lexicographically at every level
- Strip keys with `undefined` values (absent key = `undefined` key → same hash)
- Strip semantically-default values (e.g. `nullable: false`, `onDelete: 'noAction'`)
- JSON-serialize with `bigintJsonReplacer`

These are the same semantics the emitter uses today. Extracting them into shared utilities means authoring and emission produce identical hashes.

The emitter stops computing hashes — it reads them from the `Contract` it receives (`contract.storage.storageHash`, `contract.execution?.executionHash`, `contract.profileHash`).

### 3. `schemaVersion` is a persistence detail

`schemaVersion` is not on the `Contract` type. The emitter hardcodes `"schemaVersion": "1"` when writing JSON. The hashing utilities also hardcode it internally (it's a constant — we only support one schema version). If we ever bump it, we update the utilities.

### 4. `sources` is removed

`sources` is not on the `Contract` type. The emitter already excludes it from JSON output. Removing `ContractIR` removes `sources` from the type system. The framework `validateContract` already strips it during parsing.

### 5. Roots are authored, not derived

The emitter does not derive `roots`. Authoring surfaces (PSL interpreter, staged DSL, chain builder) produce `roots` as part of the contract. The Mongo PSL interpreter already does this (`roots[collectionName] = modelName`). The SQL authoring surfaces currently hardcode `roots: {}` — this phase populates them.

### 6. `SqlSemanticContractDefinition` — eliminate if straightforward

`SqlSemanticContractDefinition` is an authoring-level intermediate used by 4 files. The primary goal is `ContractIR` elimination. If `SqlSemanticContractDefinition` can be eliminated without significant complexity (the staged DSL and PSL interpreter produce `Contract` directly), do it. Otherwise, keep it as an internal intermediate and change `buildSqlContractFromSemanticDefinition` to return `Contract<SqlStorage, SqlModelStorage>` instead of `ContractIR`.

## Structural audit: what changes

### Emitter (`emit.ts`)


| Current                                                                         | After                                                                                                         |
| ------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| Input: `ContractIR`                                                             | Input: `Contract`                                                                                             |
| `validateCoreStructure(ir)` — runtime field checks                              | Removed (type system enforces structure)                                                                      |
| Constructs `canonicalContract` by picking fields from IR                        | Serializes `Contract` directly + stamps `schemaVersion: '1'`                                                  |
| `computeStorageHash(canonical)` / `computeExecutionHash` / `computeProfileHash` | Reads hashes from `contract.storage.storageHash`, `contract.execution?.executionHash`, `contract.profileHash` |
| `stripRelationStrategies(models)`                                               | Removed (`Contract` relations don't have `strategy`)                                                          |
| `CanonicalContractSchema` (arktype)                                             | Updated to match `Contract` shape (no `sources`, required `roots`)                                            |


### `TargetFamilyHook` SPI

All three methods change parameter type from `ContractIR` to `Contract`:

- `validateTypes(contract: Contract, ctx: ValidationContext): void`
- `validateStructure(contract: Contract): void`
- `generateContractTypes(contract: Contract, ...): string`

Both SQL and Mongo hooks update in lockstep.

### SQL authoring (`contract-ir-builder.ts`)

`buildContractIR(state)` → `buildContract(state): Contract<SqlStorage, SqlModelStorage>`

Key changes:

- Removes `schemaVersion`, `sources` from output
- Moves `storageHash` into `storage` block (computed via shared utility)
- Populates `roots` from models
- Returns typed `Contract<SqlStorage, SqlModelStorage>` instead of `ContractIR`
- Adds `meta: {}`, `profileHash` (computed via shared utility)

`buildSqlContractFromSemanticDefinition(def)` → returns `Contract<SqlStorage, SqlModelStorage>` instead of `ContractIR`. If `SqlSemanticContractDefinition` is being eliminated, this function is replaced by direct contract construction in the staged lowering and PSL interpreter.

### SQL PSL interpreter

`interpretPslDocumentToSqlContractIR` → `interpretPslDocumentToSqlContract`

Return type: `Result<Contract<SqlStorage, SqlModelStorage>, ...>` instead of `Result<ContractIR, ...>`.

Populates `roots` from models (matching Mongo PSL interpreter pattern).

### Mongo authoring

The Mongo PSL interpreter already produces a `ContractIR`-shaped object with `roots`. It changes to produce `Contract<MongoStorage, MongoModelStorage>` (which is `MongoContract` after the type alias).

## Steps

### Step 1: Common canonicalization and hashing utilities

Extract canonicalization and hashing from `packages/1-framework/1-core/control-plane/src/emission/` into a shared location accessible by authoring surfaces.

**New exports from `@prisma-next/contract`:**

- `canonicalize(value: unknown): string` — recursively sort keys, strip undefined and semantically-default values, JSON-serialize. Carries the existing `omitDefaults` and `sortObjectKeys` semantics.
- `computeStorageHash(args: { target: string; targetFamily: string; storage: Record<string, unknown> }): StorageHashBase<string>` — canonicalize the input (with hardcoded `schemaVersion: '1'`), SHA-256 hash.
- `computeExecutionHash(args: { target: string; targetFamily: string; execution: Record<string, unknown> }): ExecutionHashBase<string>`
- `computeProfileHash(args: { target: string; targetFamily: string; capabilities: Record<string, Record<string, boolean>> }): ProfileHashBase<string>`

The emitter's existing hashing module (`emission/hashing.ts`) delegates to these utilities (or is replaced by them). Existing emitter tests verify identical hash output.

**Location:** `packages/1-framework/0-foundation/contract/src/hashing.ts` (foundation layer — no framework-domain dependencies). The canonicalization logic moves alongside or stays in the existing module and is re-exported.

**Tests:** Port or extend existing canonicalization and hashing tests. Verify that calling the new utilities with the same input produces identical hashes to the current emitter pipeline.

### Step 2: `SqlStorage extends StorageBase`

Add `storageHash` to `SqlStorage` so it satisfies `StorageBase`:

```typescript
export type SqlStorage<THash extends string = string> = StorageBase<THash> & {
  readonly tables: Record<string, StorageTable>;
  readonly types?: Record<string, StorageTypeInstance>;
};
```

**Impact on `SqlContract`:** `SqlContract` already has `storage: S` where `S extends SqlStorage`. After this change, `contract.storage.storageHash` is available alongside the existing `contract.storageHash` (from `ContractBase`). No consumer changes needed.

**Test fixture updates:** Every inline `SqlStorage` construction (`{ tables: { ... } }`) needs `storageHash`. Provide a factory:

```typescript
function sqlStorage(tables: Record<string, StorageTable>, opts?: { target?: string; types?: ... }): SqlStorage {
  const storage = { tables, ...ifDefined('types', opts?.types) };
  return { ...storage, storageHash: computeStorageHash({ target: opts?.target ?? 'postgres', targetFamily: 'sql', storage }) };
}
```

`validateContract()` already populates the storage block from JSON — update it to include `storageHash` in the storage object it returns.

### Step 3: `MongoStorage extends StorageBase`, `MongoContract` = `Contract`

`**MongoStorage`:**

```typescript
export type MongoStorage<THash extends string = string> = StorageBase<THash> & {
  readonly collections: Record<string, MongoStorageCollection>;
};
```

`**MongoContract`:**

```typescript
export type MongoContract<
  S extends MongoStorage = MongoStorage,
  M extends Record<string, ContractModel<MongoModelStorage>> = Record<string, ContractModel<MongoModelStorage>>,
> = Contract<S, M>;
```

`**MongoModelDefinition`:**

```typescript
export type MongoModelDefinition = ContractModel<MongoModelStorage>;
```

Update Mongo test fixtures to include required `Contract` fields (`target`, `capabilities`, `extensionPacks`, `meta`, `storage.storageHash`). Provide a factory similar to SQL.

### Step 4: Contract construction factories

Replace `contractIR()`, `irHeader()`, `irMeta()` with `Contract`-based equivalents.

```typescript
function createContract<S extends StorageBase, M extends Record<string, ContractModel>>(opts: {
  target: string;
  targetFamily: string;
  roots: Record<string, string>;
  models: M;
  storage: S;
  capabilities?: Record<string, Record<string, boolean>>;
  extensionPacks?: Record<string, unknown>;
  execution?: ExecutionSection;
  profileHash?: ProfileHashBase<string>;
  meta?: Record<string, unknown>;
}): Contract<S, M>
```

This is the test-facing factory. It fills in defaults (`capabilities: {}`, `extensionPacks: {}`, `meta: {}`) and computes `profileHash` if not provided.

For SQL-specific test construction:

```typescript
function createSqlContract(opts: {
  target?: string;
  models: Record<string, ContractModel<SqlModelStorage>>;
  storage: Omit<SqlStorage, 'storageHash'>;
  // ... optional overrides
}): Contract<SqlStorage, typeof opts.models>
```

This computes `storageHash` and assembles the full contract. Tests that currently use `contractIR({ header: irHeader({...}), meta: irMeta(), ... })` switch to these factories.

### Step 5: Update emitter to accept `Contract`

`**emit()` signature:**

```typescript
export async function emit(
  contract: Contract,
  options: EmitOptions,
  targetFamily: TargetFamilyHook,
): Promise<EmitResult>
```

**Changes:**

- Remove `validateCoreStructure()` — the type system enforces structure
- Remove `stripRelationStrategies()` — `Contract` relations don't have `strategy`
- Read hashes from `contract.storage.storageHash`, `contract.execution?.executionHash`, `contract.profileHash` instead of computing them
- Construct canonical JSON by serializing `contract` fields + stamping `schemaVersion: '1'`
- Update `CanonicalContractSchema` (arktype): remove `sources`, make `roots` required
- Keep `_generated` metadata in JSON output
- Keep Prettier formatting of `.d.ts` output

`**TargetFamilyHook` SPI:** Change all three method signatures from `ir: ContractIR` to `contract: Contract`.

**SQL hook (`sqlTargetFamilyHook`):** Update `validateTypes`, `validateStructure`, and `generateContractTypes` to read from `Contract` instead of `ContractIR`. The data is structurally the same — this is a type annotation change with minor field access updates (e.g., no `ir.schemaVersion` reads).

**Mongo hook (`mongoTargetFamilyHook`):** Same treatment.

**Emitter tests:** Update all test files under `packages/1-framework/3-tooling/emitter/test/` and hook test files under `packages/2-sql/3-tooling/emitter/test/` and `packages/2-mongo-family/3-tooling/emitter/test/`. Replace `contractIR()`/`irHeader()`/`irMeta()` usage with contract factories from Step 4.

### Step 6: Update authoring to produce `Contract`

**SQL chain builder / `buildContractIR` → `buildContract`:**

Rename `buildContractIR` to `buildContract` in `contract-ir-builder.ts`. Change return type to `Contract<SqlStorage, SqlModelStorage>`. Changes:

- Remove `schemaVersion` and `sources` from output
- Compute `storageHash` via `computeStorageHash()` and include in storage block
- Compute `profileHash` via `computeProfileHash()`
- Derive `roots` from models: for each model with a `storage.table`, add a root entry (e.g., `User` → `users: 'User'`, using a pluralization utility matching the Mongo pattern)
- Compute `executionHash` via `computeExecutionHash()` when execution section is present
- Return typed `Contract<SqlStorage, SqlModelStorage>`

`**buildSqlContractFromSemanticDefinition`:** Change return type to `Contract<SqlStorage, SqlModelStorage>`. If `SqlSemanticContractDefinition` can be eliminated without significant complexity, inline the semantic-to-contract conversion into the staged lowering and PSL interpreter. Otherwise keep the semantic intermediate and only change the output type.

**SQL PSL interpreter:** Rename `interpretPslDocumentToSqlContractIR` → `interpretPslDocumentToSqlContract`. Return `Result<Contract<SqlStorage, SqlModelStorage>, ...>`. Populate `roots` from models.

**Mongo PSL interpreter:** Change return type from `ContractIR` to `Contract<MongoStorage, MongoModelStorage>`. The structure is already correct — this is a type change plus adding `storageHash` to the storage block and removing `schemaVersion`/`sources`.

**Staged DSL (`contract-builder.ts`):** `buildStagedContract()` returns `Contract<SqlStorage, SqlModelStorage>` instead of calling `buildContractIR`.

**Provider types:** Update `ContractConfig`, `ContractSourceTypes`, and any `ContractIR` references in config/provider types to use `Contract`.

### Step 7: Remove `ContractIR` and cleanup

- Delete `ContractIR` interface and factory functions (`contractIR`, `irHeader`, `irMeta`) from `packages/1-framework/0-foundation/contract/src/ir.ts`
- Delete or empty `@prisma-next/contract/ir` export
- Update all ~83 files that reference `ContractIR` — by this point they should all be using `Contract` (from steps 4–6)
- Remove deprecated `Domain`* type aliases from `domain-types.ts` if no consumers remain (check with grep)
- Remove `sources` from any remaining types (`ContractBase` still has it but that's Phase B)
- Remove the old hashing module from the emitter if fully replaced by shared utilities

## ADR impact

**ADR 182:** The current implementation contradicts the "family-owned hashing" principle — the emitter computes hashes, not the family. This phase brings the implementation in line with the ADR. No ADR update needed.

Verify no other ADRs specify "the emitter computes hashes." If found, update them to reflect the new design: authoring/storage layer computes hashes using shared canonicalization utilities; the emitter reads pre-computed hashes from the contract.
