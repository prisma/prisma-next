# Task 5.8 — Complete control flow inversion for descriptor assembly

## Goal

Complete the inversion of descriptor assembly so that the framework drives assembly, not individual family packages. After this task:

- Descriptor assembly happens once in the framework layer (`createControlStack()`) and the assembled state flows to whoever needs it — emitter, runtime, family-specific operations.
- The framework calls `emit()` directly — family instances no longer own emission.
- Family packages at layer 9 provide descriptors, hooks, and family-specific operations — not framework-level orchestration.
- The operation registry is assembled at the framework level, since operations are a cross-family concept (both SQL and Mongo need them).

This task picks up where 3.9 and 3.10 left off. Task 3.9 extracts the assembly functions to the framework and builds the Mongo family package using the inverted pattern. Task 3.10 extracts framework component types into `@prisma-next/framework-components`, renames `assembleComponents()` to `createControlStack()`, and defines a parallel `createExecutionStack()`. This task migrates the SQL family to the same pattern and updates the framework orchestration code (CLI, ControlClient) to drive assembly and emission directly.

## Design

### Current flow (family drives assembly and emission)

```
User config
  → createControlStack({ family, target, adapter, driver, extensions })
    └─ extracts: codecTypeImports, operationTypeImports, extensionIds, ...
    └─ does NOT assemble operationRegistry (SQL-specific)
  → config.family.create(stack)
    └─ SqlFamilyDescriptor.create(stack)
       └─ createSqlFamilyInstance({ target, adapter, extensions })
          → assembleOperationRegistry(descriptors)       ← DUPLICATES extraction
          → extractCodecTypeImports(descriptors)          ← DUPLICATES extraction
          → extractOperationTypeImports(descriptors)      ← DUPLICATES extraction
          → ... all extraction from descriptors
          → returns instance with preassembled state

Later: familyInstance.emitContract({ contract })
         → emit(contract, preassembledEmitOptions, sqlTargetFamilyHook)
```

### Target flow (framework drives assembly and emission)

```
User config
  → createControlStack({ family, target, adapter, driver, extensions })
    └─ extracts: codecTypeImports, operationTypeImports, extensionIds, ...
    └─ assembles: operationRegistry (from ComponentMetadata.operationSignatures)
  → config.family.create(stack)
    └─ SqlFamilyDescriptor.create(stack)
       └─ reads pre-assembled state from ControlStack
       └─ does SQL-specific assembly only (typeMetadataRegistry, mutation defaults)

Later: framework calls emit(contract, stack, stack.family.hook) directly
         ← no familyInstance.emitContract()
```

### Key design decisions

#### 1. Operation registry moves to the framework

`OperationSignature` and `OperationRegistry` are already framework types (`@prisma-next/operations`). The pattern of data-type-specific operations is cross-family — both SQL and Mongo will need operation registries. Only the operation specifics (e.g., SQL's `lowering` field on `SqlOperationSignature extends OperationSignature`) are family-specific.

- Add `operationSignatures?: () => ReadonlyArray<OperationSignature>` to `ComponentMetadata`.
- SQL descriptors already implement `operationSignatures()` returning `SqlOperationSignature[]`, which satisfies `ReadonlyArray<OperationSignature>` via covariance.
- `createControlStack()` assembles the registry from all descriptors.
- `ControlStack` gains `operationRegistry: OperationRegistry`.

The SQL-specific `SqlOperationRegistry` (typed to `SqlOperationSignature` with `lowering`) is still needed at runtime for query lowering. The SQL family instance can construct it separately for that purpose, or downcast from the framework registry.

#### 2. `emit()` accepts `ControlStack` instead of `EmitOptions`

`EmitOptions` is a manually-destructured subset of `ControlStack` — every field in `EmitOptions` (except the unused `outputDir`, which is always `''` and never read) already exists on `ControlStack`. Replace `EmitOptions` with `ControlStack` as the second parameter to `emit()`. This eliminates the manual destructuring at every call site.

Before:
```typescript
emit(contract, { outputDir: '', operationRegistry, codecTypeImports, ... }, hook)
```

After:
```typescript
emit(contract, stack, hook)
```

#### 3. `emitContract()` removed from `ControlFamilyInstance`

The framework calls `emit()` directly using `stack` and `stack.family.hook`. There's no reason to retain `emitContract()` on the family instance — it's just a passthrough that rebuilds what the stack already has.

#### 4. `frameworkComponents` derived from stack

`ControlClientImpl` currently builds `frameworkComponents` (a flat `TargetBoundComponentDescriptor[]` array) separately from the stack, from the same descriptors. This can be derived from the stack, eliminating the duplication in both `ControlClientImpl` and `executeContractEmit()`.

## Steps

### Step 1: Add `operationSignatures` to `ComponentMetadata`

**Files:**
- `packages/1-framework/1-core/shared/framework-components/src/framework-components.ts`
- `packages/1-framework/1-core/shared/operations/src/index.ts` (import `OperationSignature`)

Add to `ComponentMetadata`:
```typescript
readonly operationSignatures?: () => ReadonlyArray<OperationSignature>;
```

SQL descriptors (postgres target, postgres adapter, pgvector extension) already implement `operationSignatures()` via `SqlControlStaticContributions`. This interface declares `operationSignatures: () => ReadonlyArray<SqlOperationSignature>`. Since `SqlOperationSignature extends OperationSignature`, no changes needed to the SQL descriptors — they already satisfy the new `ComponentMetadata` field. Verify this by typechecking.

**Tests:** Type test asserting `ComponentMetadata` accepts an `operationSignatures` field.

### Step 2: Assemble `operationRegistry` in `createControlStack()`

In `packages/1-framework/1-core/control-plane/src/types.ts`:
- `ControlFamilyDescriptor.create()` receives the `ControlStack` (which contains both descriptors and derived state) instead of a raw `ControlPlaneStack`
- Or: the framework creates the family instance by passing the control stack to a factory
- Make the control stack a **required** parameter (currently optional from task 3.9 for backward compatibility with the SQL descriptor). Once the SQL family is migrated to receive the control stack (Step 4), both families require it and the parameter should no longer be optional.
- **Consolidate `ControlStack` and `ControlPlaneStack`** (review finding F12): currently `ControlStack` is the aggregated metadata (type imports, renderers, authoring contributions) while `ControlPlaneStack` is the bundle of descriptors (target, adapter, driver, extensions). These should be merged into a single `ControlStack` type carrying both the component descriptors and their derived aggregations, consistent with how `ExecutionStack` works.

Add `operationRegistry: OperationRegistry` to the `ControlStack` interface.

In `createControlStack()`, assemble the registry from all descriptors:
```typescript
const operationRegistry = createOperationRegistry();
for (const descriptor of allDescriptors) {
  if (descriptor.operationSignatures) {
    for (const signature of descriptor.operationSignatures()) {
      operationRegistry.register(signature);
    }
  }
}
```

This replaces the SQL-specific `assembleOperationRegistry()` for the emit path. The SQL package's `assembleOperationRegistry()` can be removed or retained for SQL-specific `SqlOperationRegistry` construction if needed at runtime.

**Tests:** Test that `createControlStack()` assembles an `operationRegistry` from descriptor operation signatures. Test duplicate detection (already covered by `OperationRegistry.register()`).

### Step 3: Change `emit()` to accept `ControlStack`

**Files:**
- `packages/1-framework/1-core/migration/control-plane/src/emission/emit.ts`
- `packages/1-framework/1-core/migration/control-plane/src/emission/types.ts`

Change the `emit()` signature:
```typescript

  contract: Contract,
  options: EmitOptions,
  targetFamily: TargetFamilyHook,
): Promise<EmitResult>

// After:
export async function emit(
  contract: Contract,
  stack: ControlStack,
  targetFamily: TargetFamilyHook,

```

The execution plane (`packages/1-framework/1-core/execution-plane/`) also assembles component descriptors at runtime. Task 3.10 defines `createExecutionStack()` in `@prisma-next/framework-components/execution`. If the SQL runtime needs additional derived state beyond what `createExecutionStack()` provides, extend it or add SQL-specific assembly on top (same pattern as the control stack).

Inside `emit()`, read fields from `stack` instead of destructuring `options`:
```typescript
const {
  operationRegistry,
  codecTypeImports,
  operationTypeImports,
  extensionIds,
  parameterizedRenderers,
  parameterizedTypeImports,
  queryOperationTypeImports,
} = stack;
```

Remove or deprecate `EmitOptions`. Remove the vestigial `outputDir` field.

**Tests:** Existing emitter tests updated to pass a `ControlStack` (or a test double satisfying the interface) instead of `EmitOptions`.

## Sequencing recommendation

Steps 1–2 (add `operationSignatures` to `ComponentMetadata`, assemble in `createControlStack()`) are self-contained and can land first.

Steps 3–5 (change `emit()` signature, framework-driven emission, remove `emitContract()`) form a coherent unit.

Step 6 (refactor SQL instance) can be done with or after steps 3–5.

Step 7 (update Mongo) is trivial after step 5.

Step 8 (move SQL family to layer 9) is a mechanical rename — best done as a separate commit after the behavioral changes land, to keep diffs reviewable.

Step 9 (derive `frameworkComponents` from stack) is optional cleanup that can be done with or after the other steps.

## Relationship to Milestone 6

M6 and 5.8 are conceptually orthogonal but both modify `emit()`:

- **5.8** changes **where assembly state comes from** — `emit()` receives `ControlStack` instead of `EmitOptions`.
- **M6** changes **how hooks contribute type generation** — `emit()` stops calling `targetFamily.generateContractTypes()` and uses shared utilities + hook-specific methods.

These compose cleanly with 5.8 landing first. After 5.8, `emit()` reads from `ControlStack` and calls `hook.generateContractTypes(...)` as before. M6 then refactors the hook calling pattern inside a function whose input interface is already clean.

One area of M6 that 5.8 partially subsumes: M6 task 6.2.1 keeps `validateTypes()` on the hook. After 5.8, `validateTypes()` receives its `operationRegistry` from the `ControlStack` (via the framework-assembled registry), so M6 can take that as a given.

## Consequences

- **One assembly, one place.** The framework assembles `ComponentMetadata` (including operation registry) once. Families don't replicate extraction logic.
- **Family packages are pure configuration.** They provide descriptors, hooks, and family-specific operations — not framework-level orchestration.
- **Layering is clean.** Family packages at layer 9 cannot be imported by family internals. They are entry points, not building blocks.
- **New families are simpler.** A new family defines a descriptor with `ComponentMetadata` (including `operationSignatures`), a hook, and optionally a family instance for database operations. Assembly comes for free.
- **Operations are cross-family.** The framework assembles the operation registry from all descriptors, making it available to any code that needs it (emitter validation, runtime, etc.).
