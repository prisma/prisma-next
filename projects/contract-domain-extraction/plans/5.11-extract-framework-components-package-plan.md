# Extract framework-components into its own package

## Problem

The `@prisma-next/contract` package currently contains three distinct concerns:

1. **Contract data types** — `ContractBase`, `DomainModel`, `ContractIR`, hash types, domain validation. These describe the in-memory contract representation (per ADR 182).
2. **Framework component descriptors** — `ComponentMetadata`, `FamilyDescriptor`, `TargetDescriptor`, `AdapterDescriptor`, `ExtensionDescriptor`, `DriverDescriptor`, pack refs, instances, type renderers, authoring contribution types. These describe the composition system for building a Prisma Next stack.
3. **Assembly functions** — `assembleComponents()`, `extractCodecTypeImports()`, etc. These compose descriptors into assembled state for a specific plane (currently only control-plane/emission).
4. **Emission SPI types** — `TypesImportSpec`, `RenderTypeContext`, `TargetFamilyHook`, `ValidationContext`, `GenerateContractTypesOptions`, `ParameterizedCodecDescriptor`. These describe the emission pipeline interface, not the contract data structure.

Concerns 2–4 do not belong in the contract package. Per ADR 182, the contract package should focus on the in-memory contract representation — the data that authoring surfaces produce and runtime consumers read. Emission is a separate pipeline that consumes the contract; framework components are the composition primitives that describe what each part of the system contributes.

The current placement causes:
- **Conceptual confusion**: importing descriptor types from `@prisma-next/contract/framework-components` implies they're part of the contract, but they're part of the composition system.
- **Incorrect layering**: all `core/shared` packages are treated as architectural peers (`layer: core, plane: shared`), but they have real dependency ordering — `contract` depends on `operations` only because emission types reference `OperationRegistry`.
- **Plane-specific assembly in a shared package**: `assembly.ts` currently extracts control-plane/emission-specific fields (`codecTypeImports`, `parameterizedRenderers`, etc.), but the execution plane needs its own assembly pattern for different facets of the same descriptors.

## Goal

1. Extract framework component descriptors, instances, authoring contribution types, type renderers, emission SPI types, and assembly into a dedicated `@prisma-next/framework-components` package.
2. Introduce a `foundation` layer below `core` in the framework domain, making the contract a true leaf package with zero framework-domain dependencies.

## Architectural layering

### Current state

All `1-core/shared/` packages are `layer: core, plane: shared` — flat peers:

| Package | Directory | Layer | Plane |
|---|---|---|---|
| `@prisma-next/utils` | `1-core/shared/utils` | core | shared |
| `@prisma-next/plan` | `1-core/shared/plan` | core | shared |
| `@prisma-next/contract` | `1-core/shared/contract` | core | shared |
| `@prisma-next/operations` | `1-core/shared/operations` | core | shared |
| `@prisma-next/config` | `1-core/shared/config` | core | shared |
| control-plane | `1-core/migration/control-plane` | core | migration |
| execution-plane | `1-core/runtime/execution-plane` | core | runtime |

Framework layer order: `["core", "authoring", "tooling", "runtime-executor"]`

### Proposed state

Split the leaf packages into a new `foundation` layer. Add `framework-components` at `core`. Move leaf packages to a `0-foundation` directory so the filesystem mirrors the layer hierarchy.

| Package | Directory | Layer | Plane |
|---|---|---|---|
| `@prisma-next/utils` | **`0-foundation`**`/shared/utils` | **foundation** | shared |
| `@prisma-next/plan` | **`0-foundation`**`/shared/plan` | **foundation** | shared |
| `@prisma-next/contract` | **`0-foundation`**`/shared/contract` | **foundation** | shared |
| `@prisma-next/operations` | `1-core/shared/operations` | core | shared |
| `@prisma-next/framework-components` | `1-core/shared/framework-components` **(new)** | core | shared |
| `@prisma-next/config` | `1-core/shared/config` | core | shared |
| control-plane | `1-core/migration/control-plane` | core | migration |
| execution-plane | `1-core/runtime/execution-plane` | core | runtime |

Framework layer order: `["foundation", "core", "authoring", "tooling", "runtime-executor"]`

### Dependency direction (all point downward)

```
foundation (leaf — zero framework-domain deps)
  utils          plan          contract
    ↑              ↑               ↑
core
  operations   framework-components   config   control-plane   execution-plane
```

- `contract` → zero framework-domain deps (drops current dep on `operations`)
- `framework-components` → `contract` (component descriptors reference contract types where needed)
- `operations` → `plan`
- `config` → `contract`, `utils`
- `control-plane` → `contract`, `operations`
- `execution-plane` → `contract`, `control-plane`

### Clean Architecture alignment

The layering follows dependency inversion (DIP):

1. **Contract** (innermost/foundation) — pure domain data: `Contract<S,M>`, `DomainField`, `DomainRelation`, hash types, domain validation. No behavioral dependencies.
2. **Framework Components** (core) — composition primitives: descriptors, instances, pack refs, type renderers, authoring types. Also carries emission SPI interfaces (`TypesImportSpec`, `RenderTypeContext`, `TargetFamilyHook`) as ports — the emission pipeline depends inward on these interfaces, framework-component implementations carry declarative metadata that satisfies them.
3. **Emission Pipeline** (tooling) — consumes both contract data and component descriptors to produce artifacts.

Dependencies point inward. The emission pipeline never leaks into lower layers. `TypesImportSpec` lives in framework-components because it's component-descriptor vocabulary ("what import does this component contribute?"), not contract data. `TargetFamilyHook` lives in framework-components as an SPI port that the outer emission layer calls through — classic DIP.

## What moves

### To `@prisma-next/framework-components` (new package)

**From `@prisma-next/contract` (`framework-components.ts`):**
- `ComponentMetadata`, `ComponentDescriptor`
- `FamilyDescriptor`, `TargetDescriptor`, `AdapterDescriptor`, `DriverDescriptor`, `ExtensionDescriptor`
- `FamilyInstance`, `TargetInstance`, `AdapterInstance`, `DriverInstance`, `ExtensionInstance`
- `TargetBoundComponentDescriptor`
- `PackRefBase`, `FamilyPackRef`, `TargetPackRef`, `AdapterPackRef`, `DriverPackRef`, `ExtensionPackRef`
- Type renderers: `TypeRenderer`, `TypeRendererString`, `TypeRendererRawFunction`, `TypeRendererTemplate`, `TypeRendererFunction`, `NormalizedTypeRenderer`
- `normalizeRenderer()`, `interpolateTypeTemplate()`

**From `@prisma-next/contract` (`framework-authoring.ts`):**
- All authoring contribution types and functions: `AuthoringContributions`, `AuthoringFieldNamespace`, `AuthoringTypeNamespace`, `AuthoringFieldPresetDescriptor`, `AuthoringTypeConstructorDescriptor`, `AuthoringFieldPresetOutput`, `AuthoringStorageTypeTemplate`, `AuthoringColumnDefaultTemplate`, `AuthoringArgRef`, `AuthoringArgumentDescriptor`, `AuthoringTemplateValue`
- `isAuthoringArgRef()`, `isAuthoringFieldPresetDescriptor()`, `isAuthoringTypeConstructorDescriptor()`
- `resolveAuthoringTemplateValue()`, `validateAuthoringHelperArguments()`
- `instantiateAuthoringTypeConstructor()`, `instantiateAuthoringFieldPreset()`

**From `@prisma-next/contract` (`types.ts`) — emission SPI types:**
- `TypesImportSpec`, `RenderTypeContext`
- `TargetFamilyHook`, `ValidationContext`, `GenerateContractTypesOptions`
- `TypeRenderEntry`, `TypeRenderContext`
- `ParameterizedCodecDescriptor`, `TypeRenderer` (the simplified one in `types.ts`)

**From `@prisma-next/contract` (`assembly.ts`) — into plane-specific subpaths:**
- Control-plane assembly (`@prisma-next/framework-components/control`): `AssemblyInput`, `AssembleComponentsInput`, `AssembledComponentState`, `AssembledAuthoringContributions`, `assembleComponents()`, all `extract*()` functions, `assertUniqueCodecOwner()`
- Execution-plane assembly (`@prisma-next/framework-components/runtime`): placeholder for now

**From `@prisma-next/contract` (`framework-components.ts`) — component-vs-contract validation:**
- `ContractComponentRequirementsCheckInput`, `ContractComponentRequirementsCheckResult`, `checkContractComponentRequirements()` — bridges contract data with component IDs ("does this composition satisfy this contract's requirements?"). Every caller already knows about both contracts and framework components; it belongs at the layer that knows both.

### What stays in `@prisma-next/contract`

- Contract IR types (`ir.ts`)
- Domain types: `ContractBase`, `DomainModel`, `DomainField`, `DomainRelation`, hash types (`StorageHashBase`, `ExecutionHashBase`, `ProfileHashBase`)
- Domain validation (`validate-domain.ts`)
- JSON schema (`schemas/`)

After extraction, the contract package drops its dependency on `@prisma-next/operations` (only needed by `ValidationContext`, which moves out) and potentially `@prisma-next/utils` (only needed by `framework-authoring.ts`, which moves out). The contract becomes a true leaf.

## What this enables

- **Plane-specific assembly.** The package exposes `@prisma-next/framework-components/control` for control-plane assembly and `@prisma-next/framework-components/runtime` for execution-plane assembly. Both operate on the same shared descriptors but extract different facets.
- **Clean contract package.** `@prisma-next/contract` becomes purely the in-memory contract representation per ADR 182 — no emission logic, no composition system, no behavioral dependencies.
- **Enforced layering.** The foundation/core split is expressed in `architecture.config.json` and the directory hierarchy, preventing accidental upward dependencies from contract into framework-components.
- **Accurate mental model.** Importing `@prisma-next/framework-components` to describe your stack composition is conceptually distinct from importing `@prisma-next/contract` to work with contract data.

## Steps

### Step 1: Create the foundation directory and move leaf packages

Create `packages/1-framework/0-foundation/shared/` and move:
- `packages/1-framework/1-core/shared/utils` → `packages/1-framework/0-foundation/shared/utils`
- `packages/1-framework/1-core/shared/plan` → `packages/1-framework/0-foundation/shared/plan`
- `packages/1-framework/1-core/shared/contract` → `packages/1-framework/0-foundation/shared/contract`

Update `architecture.config.json`:
- Add glob for `packages/1-framework/0-foundation/shared/**` → `domain: framework, layer: foundation, plane: shared`
- Update existing `packages/1-framework/1-core/shared/**` glob (now covers only `operations`, `config`, `framework-components`)
- Add `"foundation"` to the start of `layerOrder.framework`

### Step 2: Create the framework-components package

Create `packages/1-framework/1-core/shared/framework-components/`:
- `package.json` — `@prisma-next/framework-components`
- `tsconfig.json`, `tsdown.config.ts`
- Dependencies: `@prisma-next/contract` (foundation), `@prisma-next/utils` (foundation, for `ifDefined`)

### Step 3: Move source files

Move from `@prisma-next/contract/src/` to the new package's `src/`:
- `framework-components.ts` → shared descriptor/instance/pack-ref/type-renderer types
- `framework-authoring.ts` → authoring contribution types and helpers
- `assembly.ts` → control-plane assembly (exposed via `/control` subpath)

Move emission SPI types from `@prisma-next/contract/types.ts` to the new package:
- `TypesImportSpec`, `RenderTypeContext`, `TargetFamilyHook`, `ValidationContext`, `GenerateContractTypesOptions`, `TypeRenderEntry`, `TypeRenderContext`, `ParameterizedCodecDescriptor`

Update internal imports in moved files (e.g. `./types` → `@prisma-next/contract/types` for any remaining contract type references like `ContractIR`).

### Step 4: Clean up contract package dependencies

Remove from `@prisma-next/contract`:
- Dependency on `@prisma-next/operations` (no longer needed after `ValidationContext` moves out)
- Dependency on `@prisma-next/utils` if only used by `framework-authoring.ts`

### Step 5: Set up exports

Three subpaths for `@prisma-next/framework-components`:
- `@prisma-next/framework-components` — shared descriptors, instances, pack refs, type renderers, authoring types, emission SPI types
- `@prisma-next/framework-components/control` — control-plane assembly functions and types
- `@prisma-next/framework-components/runtime` — execution-plane assembly (empty placeholder initially)

### Step 6: Update import paths across the codebase

Replace all imports:
- `@prisma-next/contract/framework-components` → `@prisma-next/framework-components`
- `@prisma-next/contract/assembly` → `@prisma-next/framework-components/control`
- Emission SPI type imports from `@prisma-next/contract/types` → `@prisma-next/framework-components`

~70 files import from `@prisma-next/contract/framework-components`. Additional files import emission SPI types from `@prisma-next/contract/types`.

### Step 7: Remove old exports from `@prisma-next/contract`

- Delete moved source files from the contract package
- Remove `./framework-components` and `./assembly` from the contract package's `exports` and `tsdown.config.ts`
- Remove emission SPI types from `types.ts`

### Step 8: Verification

- `pnpm lint:deps` clean (no layering violations — foundation < core enforced)
- `pnpm typecheck` clean
- `pnpm test:packages` pass
- All imports resolve correctly

## Scope boundaries

- **This plan only moves code.** No API changes, no renames, no refactors of assembly logic. The functions and types move verbatim.
- **Execution-plane assembly is a placeholder.** The `/runtime` subpath is created empty; a future task populates it when the runtime needs its own composed state from descriptors.
- **Emission SPI types move verbatim.** `TargetFamilyHook` and related types move to framework-components as-is. Any future refactoring of the emission SPI is out of scope.

## Sequencing

This can be done independently of tasks 5.8 (control flow inversion) and 3.10 (Mongo PSL interpreter). It has no prerequisite beyond the current state of the codebase.

If done before 5.8, the 5.8 plan should reference `@prisma-next/framework-components/control` instead of `@prisma-next/contract/assembly`.

## Risk

Low. This is a pure move with no behavioral changes. The main risks are:
- **Breadth of import path updates** (~70+ files) — mechanical but tedious.
- **Directory moves for foundation packages** — affects `package.json` `repository.directory` fields and any hard-coded paths in tooling configs. Requires updating `architecture.config.json` globs.
