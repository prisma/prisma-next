# Complete `@prisma-next/framework-components` extraction and introduce foundation layer

## Problem

After tasks 3.10, 3.12, and 3.13, the `@prisma-next/framework-components` package exists with component types, assembly, stack creation, execution-plane types, and control-plane types. However, `@prisma-next/contract` still owns **emission SPI types** (`TargetFamilyHook`, `ValidationContext`, `GenerateContractTypesOptions`, `ParameterizedCodecDescriptor`, etc.) that belong with the composition system, not the contract data model. This coupling is the sole reason `@prisma-next/contract` depends on `@prisma-next/operations` — only `ValidationContext` references `OperationRegistry`.

Additionally, the architecture config previously treated all `1-core/shared/` packages as flat peers at the `core` layer, even though `utils` and `contract` are true leaves with no framework-domain dependencies (the `plan` package has since been deleted). Introducing a `foundation` layer below `core` makes this real dependency ordering explicit and enforceable.

## Goal

1. Move emission SPI types from `@prisma-next/contract` to `@prisma-next/framework-components/emission`, breaking the contract → operations coupling and making the contract a true leaf package.
2. Consolidate duplicate type definitions (`TypesImportSpec`, `RenderTypeContext`) that exist in both packages.
3. Introduce a `foundation` layer below `core`; move `utils` and `contract` to `0-foundation/` so the filesystem mirrors the layer hierarchy (the `plan` package has been deleted).

## Prerequisites

**Task 5.A1–A6 (Phase A) must land first.** Phase A modifies `TargetFamilyHook` in `contract/types.ts` (changes `ContractIR` → `Contract` in method signatures) and touches ~163 files across the same consumer domains that 5.11 updates. Running these in parallel would cause merge conflicts in 30+ files where both branches modify import lines.

After Phase A:
- `TargetFamilyHook.validateTypes()` and `.validateStructure()` accept `Contract` (not `ContractIR`)
- `TargetFamilyHook.generateContractTypes()` accepts `Contract` (not `ContractIR`)
- `ContractIR` is removed from the contract package
- `StorageBase` is added to `contract/types.ts`

This plan assumes Phase A is merged. Types described below reflect the post–Phase A state.

## Current state

`@prisma-next/framework-components` has 4 subpath exports:

| Subpath | Contents |
|---|---|
| `./components` | `ComponentMetadata`, descriptors, instances, pack refs, `checkContractComponentRequirements()` |
| `./authoring` | `AuthoringContributions`, field/type presets, template helpers |
| `./control` | `ControlStack`, `createControlStack()`, control descriptors/instances, result types |
| `./execution` | `ExecutionStack`, `createExecutionStack()`, execution descriptors/instances, runtime requirements |

`@prisma-next/contract` still exports these emission SPI types via `./types`:

| Type | Why it's emission SPI, not contract data |
|---|---|
| `TargetFamilyHook` | The main emission SPI interface — describes hooks into the emission pipeline |
| `ValidationContext` | Passed to hook validators; references `OperationRegistry` from `@prisma-next/operations` |
| `GenerateContractTypesOptions` | Options for `.d.ts` generation |
| `TypeRenderContext` | Context for parameterized type rendering |
| `TypeRenderEntry` | Normalized renderer entry for `generateContractTypes` |
| `TypeRenderer` | String \| function type renderer (simplified version in `types.ts`) |
| `ParameterizedCodecDescriptor` | Declarative parameterized codec metadata |
| `TypesImportSpec` | **Duplicated** — identical definition exists in `framework-components/src/types.ts` |
| `RenderTypeContext` | **Duplicated** — identical definition exists in `framework-components/src/type-renderers.ts` |

`@prisma-next/contract` depends on `@prisma-next/operations` solely because `ValidationContext` references `OperationRegistry`. After moving `ValidationContext`, this dependency can be dropped.

`@prisma-next/framework-components` already imports `TargetFamilyHook` from `@prisma-next/contract/types` (in `control-descriptors.ts`). After the move, this becomes an internal import.

## Architectural layering

### Current state

Previously all `1-core/shared/` packages were `layer: core, plane: shared` — flat peers (the `shared/` intermediate has since been removed):

| Package | Directory | Layer | Plane |
|---|---|---|---|
| `@prisma-next/utils` | `0-foundation/utils` | core | shared |
| `@prisma-next/contract` | `0-foundation/contract` | core | shared |
| `@prisma-next/operations` | `1-core/operations` | core | shared |
| `@prisma-next/framework-components` | `1-core/framework-components` | core | shared |
| `@prisma-next/config` | `1-core/config` | core | shared |

Framework layer order: `["core", "authoring", "tooling", "runtime-executor"]`

### Proposed state

| Package | Directory | Layer | Plane |
|---|---|---|---|
| `@prisma-next/utils` | **`0-foundation`**`/utils` | **foundation** | shared |
| `@prisma-next/contract` | **`0-foundation`**`/contract` | **foundation** | shared |
| `@prisma-next/operations` | `1-core/operations` | core | shared |
| `@prisma-next/framework-components` | `1-core/framework-components` | core | shared |
| `@prisma-next/config` | `1-core/config` | core | shared |

Framework layer order: `["foundation", "core", "authoring", "tooling", "runtime"]`

### Dependency direction (all point downward)

```
foundation (leaf — zero framework-domain deps)
  utils          plan          contract
    ↑              ↑               ↑
core
  operations   framework-components   config   control-plane   execution-plane
```

- `contract` → zero framework-domain deps (drops `operations` dependency)
- `framework-components` → `contract`, `utils`, `operations` (via `ValidationContext`)
- `operations` → `plan`
- `config` → `contract`, `utils`

## Scope boundaries

- **This plan only moves code.** No API changes, no renames, no refactors of emission SPI logic.
- **Emission SPI types move verbatim** (post–Phase A versions). Any future narrowing of the `TargetFamilyHook` interface (M6) is out of scope.
- **Shim package removal is deferred.** `@prisma-next/core-execution-plane` (pure re-export shim) and deprecated re-exports in `@prisma-next/core-control-plane` are NOT removed in this task. Their consumer migration (~20 and ~85 files respectively) would inflate the diff and increase conflict risk. A follow-up task handles shim removal.
- **`GenerateContractTypesOptions` moves verbatim** even though M6 may later remove it when narrowing the hook interface. Moving then removing is cleaner than leaving it in the wrong package.

## Sequencing

This task is independent of task 5.8 (control flow inversion) — they can run in any order or in parallel. If 5.8 completes first, some import paths may already reference `@prisma-next/framework-components/control` instead of older paths, reducing the consumer migration scope for workstream A.

M6 (SQL emitter migration to shared generation) should happen AFTER this task. M6 narrows the `TargetFamilyHook` interface — it should modify the types at their canonical location in `@prisma-next/framework-components/emission`, not in `@prisma-next/contract/types`.

## Commit strategy

Workstream A (emission SPI extraction):
1. Add emission types to framework-components + new `./emission` subpath
2. Migrate framework-domain consumers (one commit)
3. Migrate SQL-domain consumers (one commit)
4. Migrate Mongo-domain consumers (one commit)
5. Migrate targets, extensions, examples, tests (one commit)
6. Remove emission SPI types from contract + drop `@prisma-next/operations` dep

Workstream B (foundation layer):
7. Move `utils`, `contract` to `0-foundation/`; update `architecture.config.json`

Each commit should leave the codebase in a passing state (`pnpm typecheck` + `pnpm test:packages`).

## Risk

Low. This is a pure move with no behavioral changes. The main risks are:

- **Breadth of import path updates** — emission SPI types are imported from `@prisma-next/contract/types` by ~100+ files, but most import contract data types (which stay). Only files importing emission SPI symbols need updating; estimated ~30-40 files based on symbol usage.
- **Phase A merge timing** — starting before Phase A lands will cause widespread merge conflicts. Sequence strictly after Phase A.
- **Directory moves for foundation packages** — affects `package.json` `repository.directory` fields and `architecture.config.json` globs. Package names and import paths are unchanged, so no consumer updates needed.
- **M6 follow-up** — `GenerateContractTypesOptions` and `TargetFamilyHook.generateContractTypes()` will be modified/removed by M6. This is expected and not a problem — M6 modifies them at their canonical location.

## What this enables

- **Clean contract package.** `@prisma-next/contract` becomes purely the in-memory contract representation per ADR 182 — no emission logic, no composition system, no dependency on `@prisma-next/operations`.
- **Enforced layering.** The foundation/core split is expressed in `architecture.config.json` and the directory hierarchy, preventing accidental upward dependencies from contract into framework-components or operations.
- **Accurate mental model.** Emission SPI types live alongside the composition system they describe (`@prisma-next/framework-components/emission`), distinct from contract data types (`@prisma-next/contract/types`).
- **M6 readiness.** When M6 narrows the `TargetFamilyHook` interface, it modifies a single canonical location rather than coordinating across packages.
