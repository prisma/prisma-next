# Task 3.9 ‚Äî Extract assembly into framework and build Mongo control plane

## Goal

Extract family-agnostic descriptor assembly functions from the SQL family package into the framework layer, and build a Mongo control plane (family package at layer 9). The Mongo family package is built using the inverted control-flow pattern from the start: the framework/orchestration layer runs assembly and passes the pre-assembled state to the family instance ‚Äî the family provides descriptors and hooks but never calls assembly functions. The SQL family's existing pattern is left in place for now; a follow-up task (5.8) completes the inversion for SQL.

## Context

The descriptor-based assembly machinery in `packages/2-sql/3-tooling/family/src/core/assembly.ts` contains two categories of functions:

**Family-agnostic** ‚Äî operate on `ComponentMetadata` (the framework type every descriptor extends via `DescriptorWithTypes`):
- `extractCodecTypeImports()`
- `extractOperationTypeImports()`
- `extractQueryOperationTypeImports()`
- `extractExtensionIds()`
- `extractParameterizedRenderers()`
- `extractParameterizedTypeImports()`
- `assembleAuthoringContributions()`
- Supporting helpers: `addUniqueId()`, `assertUniqueCodecOwner()`

**SQL-specific** ‚Äî operate on `SqlControlStaticContributions` or SQL-specific descriptor types and must stay in the SQL family:
- `extractCodecControlHooks()` ‚Äî operates on `TargetBoundComponentDescriptor<'sql', string>` (SQL-specific input type)
- `assembleOperationRegistry()` ‚Äî calls `descriptor.operationSignatures()` (SQL-specific method)
- `assembleControlMutationDefaultContributions()` ‚Äî SQL mutation default generators
- `assemblePslInterpretationContributions()` ‚Äî SQL PSL scalar type descriptors

The Mongo demo currently bypasses the control plane entirely ‚Äî it manually specifies `codecTypeImports` and calls `emit()` directly in `examples/mongo-demo/scripts/generate-contract.ts`.

**Design principle (from this task forward):** The framework should drive assembly. Given a list of component descriptors, the framework reads `ComponentMetadata` from each, runs assembly once, and passes the assembled state to whoever needs it (emitter, runtime, family-specific operations). The family package at layer 9 provides the descriptor + hook, not assembly logic. See task 5.8 for the full inversion plan.

## Prerequisite

Alberto's TS contract authoring redesign ([PR #261](https://github.com/prisma/prisma-next/pull/261), branch `feat/contract-ts-revamp`) ‚Äî our base branch. It already:
- Moved `AuthoringContributions`, `AuthoringFieldPresetDescriptor`, `AuthoringTypeConstructorDescriptor`, etc. to `packages/1-framework/0-foundation/contract/src/framework-authoring.ts`
- Added `authoring?: AuthoringContributions` to `ComponentMetadata`
- Added `assembleAuthoringContributions()` to `assembly.ts`

## Steps

### Step 1: Rebase onto latest base branch

Rebase onto `origin/feat/contract-ts-revamp` to pick up the latest changes (ADR renumbering, assembly additions, etc.).

### Step 2: Extract family-agnostic assembly to the framework

**Target location:** `packages/1-framework/0-foundation/contract/src/assembly.ts`, exported via `@prisma-next/contract/assembly`.

This co-locates the assembly functions with the `ComponentMetadata` type they operate on (in `framework-components.ts`).

**What to move:**

1. The `DescriptorWithTypes` interface ‚Äî or retire it and use `ComponentMetadata` (which already has the same `types` shape). The assembly functions should accept `ReadonlyArray<ComponentMetadata>` (or a picked subset with `id`).

2. All family-agnostic functions listed above (extract*, assembleAuthoringContributions, helpers).

3. Define an `AssembledComponentState` interface that captures the assembly output:
   ```ts
   interface AssembledComponentState {
     readonly codecTypeImports: ReadonlyArray<TypesImportSpec>;
     readonly operationTypeImports: ReadonlyArray<TypesImportSpec>;
     readonly queryOperationTypeImports: ReadonlyArray<TypesImportSpec>;
     readonly extensionIds: ReadonlyArray<string>;
     readonly parameterizedRenderers: Map<string, NormalizedTypeRenderer>;
     readonly parameterizedTypeImports: ReadonlyArray<TypesImportSpec>;
     readonly authoringContributions: AssembledAuthoringContributions;
   }
   ```

4. Export a single `assembleComponents(components)` function that runs all extractions and returns `AssembledComponentState`.

**What stays in the SQL package:**
- `SqlControlDescriptorWithContributions` (extends `SqlControlStaticContributions`)
- `assembleOperationRegistry()`, `assembleControlMutationDefaultContributions()`, `assemblePslInterpretationContributions()`
- The SQL-specific parts of `assembly.ts` continue to import the shared functions from the framework

### Step 3: Update the SQL family package to import from framework

Update `packages/2-sql/3-tooling/family/src/core/assembly.ts`:
- Remove the functions that moved to the framework
- Import them from `@prisma-next/contract/assembly`
- `SqlControlDescriptorWithContributions` may need to extend a framework assembly input type
- `createSqlFamilyInstance()` in `control-instance.ts` continues to call the assembly functions as before (just imports changed) ‚Äî the full inversion to framework-driven assembly is deferred to task 5.8

### Step 4: Verify SQL behavior unchanged

- All SQL family tests pass (`packages/2-sql/3-tooling/family/test/`)
- All SQL emitter tests pass
- `pnpm lint:deps` clean
- Integration tests pass

### Step 5: Create the Mongo family package at layer 9

Create `packages/2-mongo-family/9-family/`:
- `package.json` ‚Äî `@prisma-next/family-mongo`
- `tsconfig.json`, `tsdown.config.ts`
- `src/index.ts`

Update `architecture.config.json`:
- Add `{ "glob": "packages/2-mongo-family/9-family/**", "domain": "mongo", "layer": "family", "plane": "migration" }`
- Add `"family"` to the end of `layerOrder.mongo`: `["core", "tooling", "orm", "runtime", "family"]`

### Step 6: Define Mongo descriptor and build control plane

In `packages/2-mongo-family/9-family/src/`:

1. **Mongo target descriptor** ‚Äî a concrete descriptor object (or factory) that carries:
   - `id: 'mongo'`, `familyId: 'mongo'`, `targetId: 'mongo'`
   - `types.codecTypes.import` ‚Äî the `@prisma-next/mongo-core/codec-types` import spec (currently hardcoded in the demo)
   - No operations yet (Mongo operations in SQL terms is separate work)

2. **`createMongoFamilyInstance(assembledState)`** ‚Äî receives pre-assembled `AssembledComponentState` from the orchestration layer, returns a `MongoControlFamilyInstance` implementing `ControlFamilyInstance<'mongo'>`:
   - `emitContract({ contractIR })` ‚Äî calls `emit(ir, emitOptionsFromAssembledState, mongoTargetFamilyHook)`
   - `validateContractIR(contractJson)` ‚Äî calls `validateMongoContract()`
   - `verify()`, `sign()`, `readMarker()`, `introspect()`, `schemaVerify()` ‚Äî stubs that throw "not implemented"

3. **`MongoFamilyDescriptor`** class implementing `ControlFamilyDescriptor<'mongo'>`:
   - `kind: 'family'`, `id: 'mongo'`, `familyId: 'mongo'`, `version: '0.0.1'`
   - `hook: mongoTargetFamilyHook`
   - `create(stack, assembledState)` ‚Äî passes `assembledState` through to `createMongoFamilyInstance(assembledState)`

### Step 7: Wire the Mongo demo through the control plane

Update `examples/mongo-demo/scripts/generate-contract.ts`:
- Import `MongoFamilyDescriptor` and `assembleComponents` from the framework, plus descriptor objects (e.g. `mongoTargetDescriptor`) from `@prisma-next/family-mongo`
- Replace manual `codecTypeImports` and direct `emit()` call with the inverted pattern ‚Äî the script drives assembly and passes the result to the family:
  ```ts
  const assembledState = assembleComponents([mongoTargetDescriptor]);
  const familyDescriptor = new MongoFamilyDescriptor();
  const instance = familyDescriptor.create(stack, assembledState);
  const result = await instance.emitContract({ contractIR: blogIR });
  ```

Update `examples/mongo-demo/package.json`:
- Add dependency on `@prisma-next/family-mongo`

### Step 8: Verification

- All Mongo emitter tests pass
- Mongo demo tests pass (`pnpm test` in `examples/mongo-demo/`)
- SQL family tests pass (imports changed, behavior unchanged)
- `pnpm lint:deps` clean
- `pnpm typecheck` clean
