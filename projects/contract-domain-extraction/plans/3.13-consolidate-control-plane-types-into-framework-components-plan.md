# Task 3.13 — Consolidate control-plane descriptor/instance types into framework-components

## Goal

Move `Control*Instance`, `Control*Descriptor`, result types, and `ControlPlaneStack` from `@prisma-next/core-control-plane` into `@prisma-next/framework-components`. Extend `ControlStack` to carry both the component descriptors and the aggregated metadata (unifying `ControlStack` + `ControlPlaneStack`). Extract migration and schema-view capabilities into dependency-inverted interfaces with type predicates, so `framework-components` has no knowledge of the migration or schema-view domains. Convert `core-control-plane/types.ts` and `config/types.ts` to re-export shims.

## Problem

Three types named "stack" represent overlapping concepts:

- `ControlStack` (framework-components) — aggregated metadata (type imports, renderers, authoring contributions). Discards the descriptors after assembly.
- `ControlPlaneStack` (core-control-plane) — raw descriptor bundle (target, adapter, driver, extensions). No aggregated data.
- `ControlFamilyDescriptor.create()` takes both as separate arguments: `(stack: ControlPlaneStack, controlStack?: ControlStack)`.

This split forces callers to manage two parallel structs and leads to divergent implementations:

- SQL family uses `ControlPlaneStack` (ignores `controlStack`).
- Mongo family requires `controlStack` (ignores `ControlPlaneStack`, casts first arg `as never`).
- CLI call sites only pass `ControlPlaneStack`, never `controlStack`.

By contrast, `ExecutionStack` already carries both descriptors and derived state in a single type. `ControlStack` should follow the same pattern.

Additionally, the `Control*Instance` and `Control*Descriptor` interfaces are defined in `core-control-plane` (migration layer) but are consumed broadly — they belong in the shared framework layer. They are also duplicated verbatim in `@prisma-next/config/types.ts`.

### New dependency

`framework-components` gains a dependency on `@prisma-next/contract` (for `ContractIR`, `ContractMarkerRecord`, `TargetFamilyHook`). This is the reverse direction from the dependency we removed in task 3.12 (`contract` → `framework-components`), so no cycle.

### Duplicate in `config/types.ts`

`@prisma-next/config/types.ts` contains a full copy of all `Control*Instance`, `Control*Descriptor`, `ControlPlaneStack`, `OperationContext`, and result types. This package already depends on `@prisma-next/framework-components`. After step 1, convert `config/types.ts` to re-export from `@prisma-next/framework-components/control`.

## Steps

### Step 1: Move interfaces into `framework-components`

Mechanical movement of types. `framework-components` becomes the canonical source. `core-control-plane/types.ts` and `config/types.ts` become re-export shims.

1. Add `@prisma-next/contract` as a dependency of `@prisma-next/framework-components`.
2. Create new source files in `framework-components/src/`:
   - `control-instances.ts` — `ControlFamilyInstance` (without `toSchemaView`), `ControlTargetInstance`, `ControlAdapterInstance`, `ControlDriverInstance`, `ControlExtensionInstance`.
   - `control-descriptors.ts` — `ControlFamilyDescriptor`, `ControlTargetDescriptor` (without `migrations`), `ControlAdapterDescriptor`, `ControlDriverDescriptor`, `ControlExtensionDescriptor`.
   - `control-result-types.ts` — `OperationContext`, `VerifyDatabaseResult`, `VerifyDatabaseSchemaResult`, `SignDatabaseResult`, `EmitContractResult`, `IntrospectSchemaResult`, `SchemaIssue`, `SchemaVerificationNode`.
3. Update `framework-components/src/exports/control.ts` to re-export all new types.
4. Update `framework-components` build config (`tsdown.config.ts`) if needed — the `./control` entry already exists, so just add the re-exports to the barrel.
5. Convert `core-control-plane/types.ts`:
   - Delete the moved interfaces.
   - Add re-exports from `@prisma-next/framework-components/control` for all moved types.
   - Keep migration types, schema-view types, and capability interfaces (defined in step 1.7).
6. Convert `config/types.ts`:
   - Replace duplicate type definitions with re-exports from `@prisma-next/framework-components/control`.
   - Keep any config-specific types (if any — currently there are none).
7. Define capability interfaces in `core-control-plane`:
   - `MigratableTargetDescriptor<TFamilyId, TTargetId>` — extends `ControlTargetDescriptor` with `readonly migrations: TargetMigrationsCapability<TFamilyId, TTargetId, TFamilyInstance>`.
   - `hasMigrations(target): target is MigratableTargetDescriptor<string, string>` — type predicate.
   - `SchemaViewCapable<TSchemaIR>` — interface with `toSchemaView(schema: TSchemaIR): CoreSchemaView`.
   - `hasSchemaView(instance): instance is ControlFamilyInstance<string> & SchemaViewCapable` — type predicate.
   - Export these from a new or existing `core-control-plane` subpath.
8. Update `core-control-plane` consumers (~60 files import from `@prisma-next/core-control-plane/types`):
   - Files importing only moved types: change import to `@prisma-next/framework-components/control`. OR leave as-is (since `core-control-plane/types.ts` re-exports them). Prefer the direct import for files that don't also import migration types.
   - Files importing `TargetMigrationsCapability` or `MigrationPlanner`/`MigrationRunner`: keep importing from `@prisma-next/core-control-plane/types`.
   - Files checking `target.migrations`: update to use `hasMigrations()` predicate.
   - Files checking `familyInstance.toSchemaView`: update to use `hasSchemaView()` predicate.
9. Build, typecheck, run tests.

**Key constraint:** This step is purely a type-level move. No runtime behavior changes. The re-export shims ensure backward compatibility — no consumer needs to change immediately.

### Step 2: Extend `ControlStack` to carry descriptors

Unify `ControlStack` and `ControlPlaneStack` so callers work with a single struct.

1. Inline `AssembledAuthoringContributions` into `ControlStack` (F05):
   ```typescript
   readonly authoringContributions: {
     readonly field: AuthoringFieldNamespace;
     readonly type: AuthoringTypeNamespace;
   };
   ```
   Delete the `AssembledAuthoringContributions` interface.

2. Add descriptor fields to `ControlStack`, parameterized by family/target:
   ```typescript
   export interface ControlStack<
     TFamilyId extends string = string,
     TTargetId extends string = string,
   > {
     // Component descriptors
     readonly family: ControlFamilyDescriptor<TFamilyId>;
     readonly target: ControlTargetDescriptor<TFamilyId, TTargetId>;
     readonly adapter?: ControlAdapterDescriptor<TFamilyId, TTargetId>;
     readonly driver?: ControlDriverDescriptor<TFamilyId, TTargetId>;
     readonly extensionPacks: readonly ControlExtensionDescriptor<TFamilyId, TTargetId>[];
     // Aggregated metadata
     readonly codecTypeImports: ReadonlyArray<TypesImportSpec>;
     readonly operationTypeImports: ReadonlyArray<TypesImportSpec>;
     readonly queryOperationTypeImports: ReadonlyArray<TypesImportSpec>;
     readonly extensionIds: ReadonlyArray<string>;
     readonly parameterizedRenderers: Map<string, NormalizedTypeRenderer>;
     readonly parameterizedTypeImports: ReadonlyArray<TypesImportSpec>;
     readonly authoringContributions: {
       readonly field: AuthoringFieldNamespace;
       readonly type: AuthoringTypeNamespace;
     };
   }
   ```

3. Update `CreateControlStackInput` to use the specific `Control*Descriptor` types and include `driver?`:
   ```typescript
   export interface CreateControlStackInput<
     TFamilyId extends string = string,
     TTargetId extends string = string,
   > {
     readonly family: ControlFamilyDescriptor<TFamilyId>;
     readonly target: ControlTargetDescriptor<TFamilyId, TTargetId>;
     readonly adapter?: ControlAdapterDescriptor<TFamilyId, TTargetId>;
     readonly driver?: ControlDriverDescriptor<TFamilyId, TTargetId>;
     readonly extensionPacks?: ReadonlyArray<ControlExtensionDescriptor<TFamilyId, TTargetId>>;
   }
   ```
   Now that the `Control*Descriptor` types live in `framework-components`, there's no layering reason to use the lighter `ComponentDescriptor`. The descriptors carry the `types` metadata that `extract*()` functions need, plus identity and factory methods, and pass through directly to the output `ControlStack` without type widening.

4. Update `createControlStack()` to accept the new input shape and carry the descriptors through to the output. The `extract*()` functions (currently accepting `Pick<ComponentMetadata, 'types'>` as a compromise from task 3.12) can be tightened to accept `ReadonlyArray<ControlTargetDescriptor | ControlAdapterDescriptor | ControlExtensionDescriptor>` or kept as-is if the wider signature is still useful for downstream SQL family callers.

5. Update `ControlFamilyDescriptor.create()` signature to accept a single `ControlStack`:
   ```typescript
   create(stack: ControlStack<TFamilyId, string>): TFamilyInstance;
   ```

6. Update family descriptor implementations:
   - `MongoFamilyDescriptor.create(stack)` — reads aggregated metadata from `stack` directly. Remove `as never` cast from the Mongo demo.
   - `SqlFamilyDescriptor.create(stack)` — reads descriptors from `stack.target`, `stack.adapter`, etc.

7. Update CLI call sites:
   - `ControlClient.init()` — call `createControlStack()` instead of `createControlPlaneStack()`. Pass the result to `family.create(stack)`.
   - `contract-emit.ts` — same.
   - `migration-plan.ts` — same.
   - Remove `createControlPlaneStack()` calls.

8. `ControlPlaneStack` becomes either a type alias (`type ControlPlaneStack<F, T> = Pick<ControlStack<F, T>, 'target' | 'adapter' | 'driver' | 'extensionPacks'>`) or is deleted. The re-export shim in `core-control-plane` provides backward compat.

9. Update tests. Update `ControlStack` consumers to use the new generic parameters where needed.

10. Build, typecheck, run tests.

### Step 3: Move `createControlPlaneStack()` factory

1. The factory logic from `core-control-plane/stack.ts` (defaulting `extensionPacks` to `[]`, defaulting `driver` to `undefined`) is already subsumed by `createControlStack()` after step 2.
2. Convert `core-control-plane/stack.ts` to a re-export shim (or delete if no consumers remain after step 2).
3. Build, typecheck, run tests.

## Consumer migration strategy

The ~60 files importing from `@prisma-next/core-control-plane/types` fall into categories:

| Category | Count (approx) | Action |
|---|---|---|
| Import only moved types (Control*Instance, Control*Descriptor, result types) | ~35 | Change to `@prisma-next/framework-components/control` |
| Import moved types + `TargetMigrationsCapability` or migration types | ~15 | Split imports: moved types from `framework-components/control`, migration types from `core-control-plane/types` |
| Import `ControlPlaneStack` only | ~5 | Change to `framework-components/control` (re-exported) |
| Test files with mock types | ~5 | Update to use new imports |

Since `core-control-plane/types.ts` becomes a re-export shim, existing imports continue to work. Consumer migration can happen incrementally.
