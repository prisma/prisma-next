# Task 3.10 — Extract framework components package and unify stack creation

## Goal

Create a `@prisma-next/framework-components` package in the framework core shared layer. Move framework component types, assembly logic, and stack creation there. Merge the current `assembleComponents()` into plane-specific `createControlStack()` / `createExecutionStack()` functions that both bundle descriptors and produce derived state, following the established `create*Stack()` naming convention.

Additionally: export `mongoFamilyDescriptor` as a pre-built value (not a class), and convert `createMongoFamilyInstance()` to use a private class.

## Problem

Task 3.9 added framework component assembly logic (`assembleComponents()`, `AssembledComponentState`) to the `@prisma-next/contract` package. This is wrong for three reasons:

1. **Wrong package.** Framework components (descriptors, metadata, assembly, authoring) are not contract concerns. The contract package should hold contract data types and validation only.
2. **Wrong name.** `assembleComponents()` / `AssembledComponentState` are control-plane-specific (they produce type imports, renderers, authoring contributions for emission) but have generic, unprefixed names. The established convention is `create{Plane}Stack()` — e.g. `createControlPlaneStack()`, `createSqlExecutionStack()`.
3. **Missing execution-plane counterpart.** There is no family-agnostic execution stack creation. The Mongo demo's `db.ts` wires runtime components manually with no family involvement.

## Context

### Established terminology


| Concept                | Control plane                     | Execution plane                     |
| ---------------------- | --------------------------------- | ----------------------------------- |
| Descriptor bundle type | `ControlPlaneStack`               | `ExecutionStack`                    |
| Bundle factory         | `createControlPlaneStack()`       | `createExecutionStack()`            |
| SQL-specific           | —                                 | `createSqlExecutionStack()`         |
| Package                | `@prisma-next/core-control-plane` | `@prisma-next/core-execution-plane` |


Both existing `create*Stack()` functions are typed passthroughs — they bundle descriptors into a struct but do no extraction/assembly. The assembly logic (`assembleComponents()`) is a separate step that should be merged into stack creation.

## Steps

### Step 1: Create the package

Create `packages/1-framework/1-core/framework-components/`:

- `package.json` — `@prisma-next/framework-components`
- `tsconfig.json`, `tsconfig.prod.json`, `tsdown.config.ts`
- Dependencies: `@prisma-next/contract` (for `TypesImportSpec`, `RenderTypeContext` — emission SPI types that haven't moved yet), `@prisma-next/utils`

Update `architecture.config.json`:

- Add package mapping: `{ "glob": "packages/1-framework/1-core/framework-components/**", "domain": "framework", "layer": "core", "plane": "shared" }`

### Step 2: Move framework component types

Move the following source files from `@prisma-next/contract` to `@prisma-next/framework-components`:

- `framework-components.ts` → `src/framework-components.ts`
- `framework-authoring.ts` → `src/framework-authoring.ts`

These files import `TypesImportSpec` and `RenderTypeContext` from the contract package's `types.ts`. These imports stay as-is (they're emission SPI types that move in a later task).

Set up exports:

- Subpath `./components` — exports from `framework-components.ts`
- Subpath `./authoring` — exports from `framework-authoring.ts`

### Step 3: Move and merge assembly + stack creation (control plane)

Move `assembly.ts` from `@prisma-next/contract` to `@prisma-next/framework-components/src/control-stack.ts`.

Move `ControlPlaneStack` type definition and `createControlPlaneStack()` from `@prisma-next/core-control-plane` into the same file.

Merge the two:

- Rename `assembleComponents()` to `createControlStack()`
- The function takes the same input as today (`{ family, target, adapter?, extensionPacks? }`)
- The return type `ControlStack` includes both:
  - The input descriptors (as `ControlPlaneStack` does today)
  - The derived control-plane state (as `AssembledComponentState` does today)
- Rename `AssembledComponentState` → `ControlStack`
- Rename `AssembleComponentsInput` → `CreateControlStackInput`

Set up subpath export: `./control`

The existing `createControlPlaneStack()` in `@prisma-next/core-control-plane/stack` becomes a re-export from the new package (or is deprecated). The `ControlPlaneStack` type in `@prisma-next/core-control-plane/types` stays (it's used by the control-plane descriptor interfaces like `ControlFamilyDescriptor.create(stack)`), but its definition moves.

### Step 4: Move and define execution stack creation

Move `createExecutionStack()`, `instantiateExecutionStack()`, `ExecutionStack`, and `ExecutionStackInstance` from `@prisma-next/core-execution-plane/stack` to `@prisma-next/framework-components/src/execution-stack.ts`.

Define `createExecutionStack()` to follow the same pattern as `createControlStack()`:

- Input: `{ family, target, adapter?, driver?, extensionPacks? }`
- Output: `ExecutionStack` containing the input descriptors plus minimal derived execution-plane state (component IDs for now)

The existing `ExecutionStack` type and `createExecutionStack()` / `instantiateExecutionStack()` in `@prisma-next/core-execution-plane` become re-exports from the new package.

Set up subpath export: `./execution`

### Step 5: Update backward-compat re-exports in contract and plane packages

In `@prisma-next/contract`:

- The `assembly` subpath re-exports from `@prisma-next/framework-components/control`
- The `framework-components` subpath re-exports from `@prisma-next/framework-components/components`

In `@prisma-next/core-control-plane`:

- `stack` subpath re-exports `createControlPlaneStack` (mapped to `createControlStack`) and `ControlPlaneStack` from `@prisma-next/framework-components/control`

In `@prisma-next/core-execution-plane`:

- `stack` subpath re-exports from `@prisma-next/framework-components/execution`

These re-exports are transitional — they prevent a mass rename of every import across the repo. Consumers can migrate incrementally. **Do not update existing consumer imports in this task** (except for the Mongo family and Mongo demo, which should import from the new package directly).

### Step 6: Export `mongoFamilyDescriptor` as a value

In `@prisma-next/family-mongo`:

- Add a const `mongoFamilyDescriptor` singleton in `control-descriptor.ts`
- Export it from the `./control` subpath
- Stop exporting the `MongoFamilyDescriptor` class (the class stays private)

Replace the object literal in `control-instance.ts` with a private class implementing `MongoControlFamilyInstance`.

### Step 9: Update tests

- Move `framework-components.authoring.test.ts` similarly

## Scope boundaries

- **Code move only** — no new business logic except the trivial `createExecutionStack()` that bundles descriptors with component IDs.
- **Backward-compat re-exports** — existing consumers (`@prisma-next/contract/assembly`, `@prisma-next/contract/framework-components`, `@prisma-next/core-control-plane/stack`, `@prisma-next/core-execution-plane/stack`) continue to work via re-exports. Only the Mongo family and Mongo demo switch to direct imports.
- **Emission SPI types stay in contract** — `TypesImportSpec`, `RenderTypeContext`, `TargetFamilyHook`, etc. remain in `@prisma-next/contract/types`. Moving them is deferred (task 5.11).
- **No directory restructuring** — no `0-foundation` layer, no directory renames. The new package goes in `packages/1-framework/1-core/framework-components/`.
- **SQL consumers unchanged** — `createSqlExecutionStack()`, `createExecutionContext()`, and all SQL assembly code stay where they are. They consume the re-exports.
