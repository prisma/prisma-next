# Task 3.11 — Complete control flow inversion for descriptor assembly

## Goal

Complete the inversion of descriptor assembly so that the framework drives assembly, not individual family packages. After this task, descriptor assembly happens once in the framework layer when the user provides their component list, and the assembled state flows to whoever needs it — emitter, runtime, family-specific operations. Family packages at layer 9 provide descriptors and hooks, not assembly orchestration.

This task picks up where 3.9 left off. Task 3.9 extracts the assembly functions to the framework and builds the Mongo family package using the inverted pattern. This task migrates the SQL family to the same pattern and updates the framework orchestration code (CLI, ControlClient) to drive assembly.

## Design

### Current flow (family drives assembly)

```
User config
  → createControlPlaneStack({ target, adapter, driver, extensions })
  → config.family.create(stack)
    └─ SqlFamilyDescriptor.create(stack)
       └─ createSqlFamilyInstance({ target, adapter, extensions })
          → assembleOperationRegistry(descriptors)       ← family calls assembly
          → extractCodecTypeImports(descriptors)          ← family calls assembly
          → extractOperationTypeImports(descriptors)      ← family calls assembly
          → ... all extraction from descriptors
          → returns instance with preassembled state

Later: familyInstance.emitContract({ contractIR })
         → emit(ir, preassembledEmitOptions, sqlTargetFamilyHook)
```

### Target flow (framework drives assembly)

```
User config
  → Framework collects all component descriptors
  → Framework reads ComponentMetadata from each
  → Framework runs assembleComponents() ONCE → AssembledComponentState
  → Framework passes assembled state to:
       emitContract:  emit(ir, assembledState, familyHook)    ← framework calls emit directly
       familyInstance: receives assembled state for family ops
       runtime:       operation registry, codec types, etc.
```

### Key changes

1. **`ControlFamilyDescriptor.create(stack)`** changes signature. Instead of receiving a raw `ControlPlaneStack` and doing its own assembly, the family receives pre-assembled state from the framework. The family may do its own family-specific assembly (SQL: operation registry, mutation defaults) on top.

2. **`ControlClient.emit()`** and CLI `executeContractEmit()` call `emit()` directly using `assembledState` + `family.hook`, rather than delegating to `familyInstance.emitContract()`.

3. **`ControlClient.init()`** runs framework assembly as part of initialization, making `AssembledComponentState` available to all subsequent operations.

4. **`SqlFamilyInstance.emitContract()`** is removed (the framework handles emission). The SQL family instance retains: `validateContractIR()`, `verify()`, `sign()`, `schemaVerify()`, `introspect()`, `readMarker()`, `toSchemaView()`, and family-specific assembly (operation registry, mutation defaults, PSL interpretation).

5. **SQL family package moves to layer 9** (`packages/2-sql/9-family/`). The current `packages/2-sql/3-tooling/family/` package is at the tooling layer, where it can be imported by other SQL tooling. At layer 9 it sits at the top of the import hierarchy — nothing in the SQL domain can import from it.

## Steps

### Step 1: Update the framework orchestration layer

In `ControlClientImpl.init()` (at `packages/1-framework/3-tooling/cli/src/control-api/client.ts`):
- Collect component metadata from `[family, target, adapter, ...extensionPacks]`
- Call `assembleComponents()` from `@prisma-next/contract/assembly`
- Store `AssembledComponentState` on the client instance

### Step 2: Framework-driven emission

Update `ControlClientImpl.emit()`:
- Instead of `this.familyInstance.emitContract({ contractIR: enrichedIR })`, call `emit(enrichedIR, assembledEmitOptions, this.options.family.hook)` directly
- Build `EmitOptions` from `AssembledComponentState`

Update `executeContractEmit()` in `packages/1-framework/3-tooling/cli/src/control-api/operations/contract-emit.ts`:
- Same change: framework assembles and calls `emit()` directly

### Step 3: Update ControlFamilyDescriptor interface

In `packages/1-framework/1-core/migration/control-plane/src/types.ts`:
- `ControlFamilyDescriptor.create()` receives assembled state alongside (or instead of) the raw stack
- Or: the framework creates the family instance by passing assembled state to a factory

### Step 4: Refactor SQL family instance

- Remove `emitContract()` from `SqlControlFamilyInstance`
- `createSqlFamilyInstance()` receives `AssembledComponentState` from the framework, does family-specific assembly (operation registry, mutation defaults) on top
- SQL-specific assembly functions stay in the SQL package but operate on top of framework-assembled state

### Step 5: Move SQL family package to layer 9

- Move `packages/2-sql/3-tooling/family/` → `packages/2-sql/9-family/`
- Update `architecture.config.json`: add family layer to SQL layer order
- Update all import paths
- Verify `pnpm lint:deps` clean

### Step 6: Update Mongo family package

Update `packages/2-mongo-family/9-family/`:
- If task 3.9 built it with a temporary `emitContract()` that calls `emit()` internally, simplify: the framework now calls `emit()` directly
- The Mongo family instance only needs `validateContractIR()` and stubs for database operations

### Step 7: Update runtime assembly (if applicable)

The execution plane (`packages/1-framework/1-core/runtime/execution-plane/`) also assembles component descriptors at runtime. If the same assembly functions are relevant there, wire `assembleComponents()` into the runtime stack creation path.

### Step 8: Verification

- All SQL family, emitter, and integration tests pass
- All Mongo emitter and demo tests pass
- CLI contract emit works end-to-end
- `pnpm lint:deps` clean
- `pnpm typecheck` clean
- Verify that `packages/2-sql/9-family/` and `packages/2-mongo-family/9-family/` are at the top of their respective import hierarchies

## Consequences

- **One assembly, one place.** The framework assembles `ComponentMetadata` once. Families don't replicate extraction logic.
- **Family packages are pure configuration.** They provide descriptors, hooks, and family-specific operations — not framework-level orchestration.
- **Layering is clean.** Family packages at layer 9 cannot be imported by family internals. They are entry points, not building blocks.
- **New families are simpler.** A new family defines a descriptor with `ComponentMetadata`, a hook, and optionally a family instance for database operations. Assembly comes for free.

## Dependencies

- **Task 3.9** must be complete (assembly functions extracted to framework, Mongo family package exists).
- **ADR 179 (Unified contract representation)** context — the unified `Contract<Storage, ModelStorage>` type informs how `validateContractIR` and emission work across families.
