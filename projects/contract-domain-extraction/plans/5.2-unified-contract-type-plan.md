# Task 5.2 — Unified contract representation

## Goal

Replace `ContractIR`, `ContractBase`, and `SqlSemanticContractDefinition` with a single `Contract<TStorage, TModels>` type — model-first, family-parameterized, and structurally identical before and after serialization. See [ADR 182](../../../docs/architecture%20docs/adrs/ADR%20182%20-%20Unified%20contract%20representation.md).

This task defines the new types and proves them with type-level assertions. It does **not** migrate consumers, update the emitter, or change authoring surfaces — those are follow-on tasks (5.3, 5.4, 5.7).

## Scope

### In scope (tasks 5.1, 5.2, 5.6)

1. **Audit** (`ContractIR` vs unified type) — document remaining structural gaps.
2. **Define types** — `Contract<TStorage, TModels>`, `StorageBase`, `ContractModel<ModelStorage>`, rename `Domain*` → `Contract*`, define `SqlStorage`/`SqlModelStorage` family instantiations, update `MongoContract` to adopt unified base.
3. **Framework-level `validateContract`** — two-pass validation with family-provided storage validator.
4. **Type-level proofs** — assertions in `unified-type-design.ts` proving structural compatibility, round-trip properties, and backward compatibility with existing consumers.

## Design decisions

### 1. Emission-time metadata

The unified `Contract` type represents the **in-memory** canonical contract. JSON persistence fields are handled by the serialization boundary:

| Field | Decision |
|-------|----------|
| `schemaVersion` | JSON-only persistence field. Emitter writes it; parser strips it. Not on `Contract`. |
| `sources` | Deprecated. Not on `Contract`. |
| `storageHash` | Moves onto `StorageBase` → accessed as `contract.storage.storageHash`. |
| `executionHash` | Moves onto `ExecutionSection` → accessed as `contract.execution.executionHash`. |
| `profileHash` | Stays top-level on `Contract` until a better home is found. |
| `meta` | Stays on `Contract` (generator metadata, set during emission). |
| `capabilities` | Stays on `Contract` (authoring-time). |
| `extensionPacks` | Stays on `Contract` (authoring-time). |

### 2. Hash generic parameters live on Storage/Execution, not Contract

Today `SqlContract` has five generic parameters (S, M, StorageHash, ExecutionHash, ProfileHash). The unified type has **two**: `TStorage` and `TModels`. Hash literal types are carried by the storage/execution type parameters:

```typescript
// Before: hash generics on the contract
type SqlContract<S, M, TStorageHash, TExecutionHash, TProfileHash> = ...

// After: hash generics on Storage (and Execution)
interface SqlStorage<THash extends string = string> extends StorageBase<THash> { ... }
type SqlContract<S extends SqlStorage, M extends ...> = Contract<S, M>;
```

An emitted `contract.d.ts` carries the hash literal via the storage type parameter:
```typescript
export type Contract = Contract<SqlStorage<'sha256:abc123'>, { ... }>;
```

### 3. `Contract*` prefix replaces `Domain*`

All shared domain types are renamed to use the `Contract` prefix, reflecting that they are the canonical contract representation, not a separate "domain" layer:

| Current | Unified |
|---------|---------|
| `DomainField` | `ContractField` |
| `DomainRelation` | `ContractRelation` |
| `DomainReferenceRelation` | `ContractReferenceRelation` |
| `DomainEmbedRelation` | `ContractEmbedRelation` |
| `DomainRelationOn` | `ContractRelationOn` |
| `DomainDiscriminator` | `ContractDiscriminator` |
| `DomainVariantEntry` | `ContractVariantEntry` |
| `DomainModel` | `ContractModel<ModelStorage>` |

`DomainModel` becomes a type alias for backward compatibility during migration:
```typescript
type DomainModel = ContractModel;  // ContractModel<Record<string, unknown>>
```

### 4. `TModels` vs `ModelStorage`

ADR 182 shows `Contract<Storage, ModelStorage>`. In practice, `TModels` is needed to carry literal model types from the emitted `contract.d.ts` (known keys like `User`, `Post` with exact field/relation/storage shapes).

`Contract` uses `TModels` as the generic param. `ModelStorage` is a concept that lives in family-specific type aliases:

- `SqlContract<S, M>` constrains `M extends Record<string, ContractModel<SqlModelStorage>>`
- `MongoContract<S, M>` constrains `M extends Record<string, ContractModel<MongoModelStorage>>`
- Framework code uses `Contract` (model storage is opaque `Record<string, unknown>`)

### 5. MongoContract adopts emission-time fields

`MongoContract` today lacks `target`, `capabilities`, `extensionPacks`, `execution`, `meta`, `profileHash`. With the unified type, `MongoContract` becomes `Contract<MongoStorage, M>` and gains all these fields. This is a minimal structural change — Mongo already has the right model-first shape.

### 6. Framework-level `validateContract` (task 5.6)

Two-pass validation with family-provided storage validator:

```typescript
type StorageValidator = (contract: Contract) => void;

function validateContract<TContract extends Contract>(
  value: unknown,
  storageValidator: StorageValidator,
): TContract;
```

1. **Parse**: strip `schemaVersion`, `sources`; normalize structure.
2. **Domain validation** (framework): roots, relation targets, variant/base consistency, discriminators, ownership, orphans. Already exists as `validateContractDomain()`.
3. **Storage validation** (family-provided): SQL validates tables/columns/FKs; Mongo validates collections/embedding.

This generalizes the pattern already established by `validateMongoContract()`.

## Type design

### `Contract<TStorage, TModels>`

```typescript
interface Contract<
  TStorage extends StorageBase = StorageBase,
  TModels extends Record<string, ContractModel> = Record<string, ContractModel>,
> {
  readonly target: string;
  readonly targetFamily: string;
  readonly roots: Record<string, string>;
  readonly models: TModels;
  readonly storage: TStorage;
  readonly capabilities: Record<string, Record<string, boolean>>;
  readonly extensionPacks: Record<string, unknown>;
  readonly execution?: ExecutionSection;
  readonly profileHash?: ProfileHashBase<string>;
  readonly meta: Record<string, unknown>;
}
```

### `StorageBase<THash>`

```typescript
interface StorageBase<THash extends string = string> {
  readonly storageHash: StorageHashBase<THash>;
}
```

### `ContractModel<ModelStorage>`

```typescript
interface ContractModel<ModelStorage = Record<string, unknown>> {
  readonly fields: Record<string, ContractField>;
  readonly relations: Record<string, ContractRelation>;
  readonly storage: ModelStorage;
  readonly discriminator?: ContractDiscriminator;
  readonly variants?: Record<string, ContractVariantEntry>;
  readonly base?: string;
  readonly owner?: string;
}
```

### `ExecutionSection<THash>`

```typescript
type ExecutionSection<THash extends string = string> = {
  readonly executionHash: ExecutionHashBase<THash>;
  readonly mutations: {
    readonly defaults: ReadonlyArray<ExecutionMutationDefault>;
  };
};
```

### SQL family

```typescript
interface SqlStorage<THash extends string = string> extends StorageBase<THash> {
  readonly tables: Record<string, StorageTable>;
  readonly types?: Record<string, StorageTypeInstance>;
}

interface SqlModelStorage {
  readonly table: string;
  readonly fields: Record<string, { readonly column: string }>;
}

type SqlContract<
  S extends SqlStorage = SqlStorage,
  M extends Record<string, ContractModel<SqlModelStorage>>
    = Record<string, ContractModel<SqlModelStorage>>,
> = Contract<S, M>;
```

### Mongo family

```typescript
interface MongoStorage<THash extends string = string> extends StorageBase<THash> {
  readonly collections: Record<string, MongoStorageCollection>;
}

type MongoContract<
  S extends MongoStorage = MongoStorage,
  M extends Record<string, ContractModel<MongoModelStorage>>
    = Record<string, ContractModel<MongoModelStorage>>,
> = Contract<S, M>;
```

## Structural audit: `ContractIR` vs `Contract`

| `ContractIR` field | `Contract` equivalent | Gap |
|--------------------|----------------------|-----|
| `schemaVersion` | Not on `Contract` (JSON-only) | Emitter writes; parser strips |
| `target` | `target` | Same |
| `targetFamily` | `targetFamily` | Same |
| `roots?` | `roots` (required) | IR has optional; Contract requires |
| `models` | `models` | IR is `Record<string, unknown>`; Contract is `Record<string, ContractModel<M>>` |
| `relations?` | `model.relations` (per-model) | IR has optional top-level; Contract has per-model (already migrated in M1–M4) |
| `storage` | `storage` | IR is `Record<string, unknown>`; Contract is `StorageBase & family-specific` |
| `execution?` | `execution?` | Same, but `executionHash` moves onto this section |
| `extensionPacks` | `extensionPacks` | Same |
| `capabilities` | `capabilities` | Same |
| `meta` | `meta` | Same |
| `sources` | Not on `Contract` (deprecated) | Dropped |

**Key structural gaps:**
1. IR `storage` is untyped (`Record<string, unknown>`); `Contract.storage` extends `StorageBase`.
2. IR `roots` is optional; `Contract.roots` is required.
3. IR has no hashes (computed during emission); `Contract.storage.storageHash` carries the hash.
4. IR `models` is untyped; `Contract.models` is `Record<string, ContractModel<M>>`.
5. IR has top-level `relations`; `Contract` has per-model relations only (already addressed in M1–M4).

## What this replaces

| Current | Unified |
|---------|---------|
| `ContractBase<Hash, Hash, Hash>` (3 hash params) | `Contract<TStorage, TModels>` (2 params) |
| `SqlContract<S, M, Hash, Hash, Hash>` (5 params) | `SqlContract<S, M>` = `Contract<S, M>` (2 params) |
| `MongoContract` (no ContractBase, missing fields) | `MongoContract<S, M>` = `Contract<S, M>` (unified base) |
| `DomainModel` | `ContractModel<ModelStorage>` |
| `DomainField` / `DomainRelation` / etc. | `ContractField` / `ContractRelation` / etc. |
| `contract.storageHash` | `contract.storage.storageHash` |
| `contract.executionHash` | `contract.execution.executionHash` |
| `schemaVersion` on in-memory type | Stripped during parsing (JSON-only) |
| `sources` on in-memory type | Dropped entirely (deprecated) |

## Properties to prove (type-level assertions)

1. `SqlContract extends Contract` — SQL instantiation satisfies the unified base.
2. `MongoContract extends Contract` — Mongo instantiation satisfies the unified base.
3. `ContractModel<SqlModelStorage> extends ContractModel` — SQL model satisfies base model.
4. `ContractModel<MongoModelStorage> extends ContractModel` — Mongo model satisfies base model.
5. `SqlStorage extends StorageBase` — SQL storage satisfies the storage base.
6. `MongoStorage extends StorageBase` — Mongo storage satisfies the storage base.
7. Framework code can read domain fields from `Contract` (opaque storage).
8. SQL code can read typed storage from `SqlContract` (`model.storage.table`, `model.storage.fields`).
9. Emitted literal types are preserved through `TModels` (known keys, literal codecIds/columns).
10. Hash literal types are carried by `TStorage` (no hash generics on `Contract`).
