## Summary

❌ Fail. The example's schema, adapter, error postures, and cascade behaviour are solid — but the README's "re-running is a no-op" promise is broken: step 2 (`migration plan`) rewrites the committed better-auth space mirrors because they are stale relative to the pack's current contract (F-1, ⚠️). Separately, the example's `db.ts` template crashes the whole Node process when an idle pool connection is terminated (F-3, ⚠️) — observed live when the dev database reaped an idle connection between the sign-up curl and the authenticated request. Five further findings are docs-fidelity and environment-noise items (F-2, F-4–F-7). Every negative control fired correctly; no original-bug regressions exist (greenfield project).

## Findings

### F-1 — ⚠️ High — README step 2 (`migration plan`) dirties the tree: committed space mirrors are stale (missing `scalarList` capability)

**Scenario:** 1 — Stand up the schema with the README's three-step flow
**Step:** 3 (`pnpm exec prisma-next migration plan --name init`), detected at step 6's `git status`
**Oracle:** README: "The committed artifacts are the outputs of these steps — re-running them is a no-op"
**Observed:**

```
$ pnpm exec prisma-next migration plan --name init
│  ✔ No changes detected
│    from: sha256:e1815c286cb6cc4ef6134a27bfc599bf4e8f6159c2abb74289c592f88dd5bb87
│    to:   sha256:e1815c286cb6cc4ef6134a27bfc599bf4e8f6159c2abb74289c592f88dd5bb87
$ git status --short -- .
 M migrations/better-auth/contract.json
 M migrations/better-auth/refs/head.json
```

Structural diff of `migrations/better-auth/contract.json` (committed vs rewritten): exactly one delta — `.capabilities.sql.scalarList: ADDED in working`. The pack's shipped contract (`packages/3-extensions/better-auth/src/contract/contract.json` and `dist/contract-BQ5LtCVN.mjs`) **has** `"scalarList": true`; the example's committed mirror **and the pack's own baseline `end-contract.json`** (`packages/3-extensions/better-auth/migrations/20260713T0717_create_auth_tables/end-contract.json`) do not. The space contract was evidently re-emitted after the baseline migration/mirrors were generated, picking up the new default `scalarList` capability, and the downstream artifacts were never refreshed. `storageHash` is unchanged (capabilities aren't hashed), so hash-based integrity checks and CI's offline `checkIntegrity` pass while the bytes drift.

**Expected (per script):** steps 1 and 6 print identical empty `git status` output.
**Reproduction:**

- `git rev-parse HEAD` → `7c57da046cb41e832c117356c0b2177deee56173`
- `git status` at failure → ` M migrations/better-auth/contract.json` / ` M migrations/better-auth/refs/head.json` (from `examples/better-auth/`)
- Bisection: `git restore` both files → `migration plan` alone re-dirties both; `db init` and `db update` alone dirty neither.
- Exact command: `pnpm exec prisma-next migration plan --name init` (cwd `examples/better-auth`)
  **Notes:** ADR 212 documents that migrate invocations overwrite the pinned mirrors and "the user reviews the resulting diff in their PR" — so the *mechanism* is by design; the *finding* is that at head the diff must be empty and is not, breaking the example README's explicit no-op claim for any consumer who runs the documented flow. Also misleading: the CLI prints "✔ No changes detected" while modifying two files. Fix scope: regenerate/commit the example's `migrations/better-auth/{contract.json, refs/head.json}` (and audit the pack's baseline `end-contract.json` + supabase example for the same staleness) so `migration plan` at head is byte-stable.

### F-3 — ⚠️ High — example server process dies on terminated idle connection: `db.ts` template installs no pool error handler

**Scenario:** 2 — Run the server, sign up, make the authenticated request
**Step:** 3 (authenticated `/api/me` request; server had been idle ~1 minute after sign-up)
**Oracle:** README "Run it" — server stays up across the documented sign-up → authenticated-request sequence.
**Observed:** `curl http://localhost:3000/api/me …` → exit code 7 (connection refused). Server log (verbatim, trimmed):

```
listening on http://localhost:3000
  POST /api/auth/sign-up/email  — create an account
  GET  /api/me                  — session + profile (authenticated)
node:events:487
      throw er; // Unhandled 'error' event
      ^

Error: Connection terminated unexpectedly
    at Connection.<anonymous> (…/node_modules/.pnpm/pg@8.21.0/node_modules/pg/lib/client.js:193:73)
    …
Emitted 'error' event on BoundPool instance at:
    at Client.idleListener (…/pg-pool/index.js:62:10)
    …
Node.js v24.16.0
```

followed by a ~200-line object dump. Process dead; port 3000 released.

**Expected (per script):** failure-mode category "Server fails to start, crashes on first request, or logs an unexplained warning a user would have to triage".
**Reproduction:**

- Trigger: db1 was created with `@prisma/dev` defaults (`databaseIdleTimeoutMillis: 1000` — the same helper the repo's own tests use); the dev server reaped the pool's idle client; `pg.Pool` emitted `'error'` with no listener attached in `src/prisma/db.ts` → unhandled `'error'` event → process crash.
- Re-ran scenario 2 against db2 (idle timeout 1h): full journey passes (sign-up 200 + cookie; `/api/me` 200 `{"session":{…},"profile":null}`).
- Exact command: `pnpm exec tsx src/main.ts` with `DATABASE_URL=postgres://postgres:postgres@127.0.0.1:39233/template1?sslmode=disable`
  **Notes:** a user's vanilla local Postgres won't reap idle connections at 1s, but pgbouncer/serverless Postgres/network blips terminate idle connections routinely, and `pg`'s own docs require a pool `'error'` listener for exactly this reason. `examples/better-auth/src/prisma/db.ts` is the consumer template the package README points at — copying it ships a process-crash footgun. One-line fix (`pool.on('error', …)`) or a README note. The ~200-line unstructured crash dump is itself the "crash with no usable diagnostic" shape, though the first line does name the cause.

## Exploratory notes

- **Completed the profile story:** created `profile-ada` via `appDb.db.orm.public.Profile.create(…)` (tsx one-liner using the example's own `createAppDb`), and `/api/me` then returned the full documented shape — session + profile + nested user. The README's headline claim is true once app code has done its part.
- **Cascades from the consumer's seat:** deleting ada via `authDb.orm.public.User.where({email}).delete()` cascaded the profile (`null` on re-read) *and* the session — the still-held browser cookie instantly yields 401 `{"error":"not authenticated"}`. Excellent, coherent behaviour.
- **`db init` re-run on an initialized database:** idempotent — "Applied 0 operation(s) across 2 space(s), database signed", exit 0.
- **`prisma-next migration status`** (the command `db init`/`db update` output recommends): works, reports the better-auth space's baseline `20260713T0717_create_auth_tables` as `applied`, summary "Up to date".
- **Malformed JSON sign-up** → 400 `{"message":"Invalid JSON in request body","code":"BAD_REQUEST"}`; **unknown route** → 404 JSON. No 5xx anywhere all run.
- **CLI output modes:** the CLI prints the pretty tree on a TTY and JSON when piped — nice for scripting; worth knowing that captured logs look different from what the user sees.
- **No `examples/README.md` index exists** (all 14 examples, not just this one) — discovery of the new example relies on the package README's link. Not a finding against this PR.
- **Un-explored ideas for a future round:** sign-out endpoint; concurrent duplicate sign-ups racing the `user.email` unique; two servers sharing one database; `db verify` surface; `pnpm emit` alias vs README spelling.
