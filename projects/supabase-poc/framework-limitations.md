# Framework Limitations — Supabase PoC

Framework gaps surfaced while building the Supabase example app on Prisma Next. Each entry is a signal that the framework *could* do more — the PoC is designed to expose these, not paper over them. The point of this PoC is precisely to capture this list, so be generous: if something felt awkward, it goes here.

**Format mirrors [`projects/mongo-example-apps/framework-limitations.md`](../mongo-example-apps/framework-limitations.md).**

Status values: `Open` (live in this PoC), `Triaged` (decided what to do with it), `Resolved` (fixed during a follow-up project), `Won't fix` (decided not to address).

---

## RLS / per-request session context

Limitations related to running queries with a per-request RLS / role / GUC context. These are the central problem the PoC explores.

> **Headline result (M2 / phase-2 / T2.4):** the userspace `createSupabaseRuntime` factory ([`examples/supabase-todos/src/server/supabase-runtime.ts`](../../examples/supabase-todos/src/server/supabase-runtime.ts)) ships **with zero `packages/` edits** (`git diff origin/main -- packages/` is empty). PN's existing public surface — `SqlDriver` / `SqlConnection` SPI, `createRuntime` / `createSqlExecutionStack` / `createExecutionContext` from `@prisma-next/sql-runtime`, and the codec / lowering machinery the runtime drives internally — is sufficient to assemble a Supabase-style per-request RLS-scoped runtime in userspace. The frictions filed below are real but **none of them blocked the M2 deliverable**; they are upstreamable improvements to make this pattern less surprising for the next builder.

| ID | Issue | Impact | Workaround in app | Status |
|---|---|---|---|---|
| FL-14 | `RuntimeCore.verifyPlanIfNeeded` ([`packages/1-framework/4-runtime/runtime-executor/src/runtime-core.ts:172`](../../packages/1-framework/4-runtime/runtime-executor/src/runtime-core.ts)) calls `driver.query(readMarkerSql, [])` directly — a back-channel that bypasses `acquireConnection()` and any per-request transactional envelope. The marker is stored in the `prisma_contract` schema, which is **not** readable to downgraded roles (`authenticated` / `anon`). A naive RLS-scoped driver that routes `query()` through the same `BEGIN; SET LOCAL ROLE …` envelope as `execute()` gets `ERROR: permission denied for schema prisma_contract` on the very first user plan. There is no `verify.mode: 'none'` or "trust the wrapped runtime's verification" knob; the three modes (`startup` / `onFirstUse` / `always`) all eventually run the marker query. | Every userspace driver that downgrades the connection role (Supabase RLS, but also any `SET ROLE` pattern, e.g. PostgREST-style multi-tenant tenancy) has to reason about the marker-query back-channel separately from the user-facing `execute()` path. The split is invisible in the public docs — a builder discovers it the first time they try to issue a query and sees an unexpected `permission denied`. | The PoC's [`ScopedDriver`](../../examples/supabase-todos/src/server/supabase-runtime.ts) bifurcates the lifecycles: `execute()` runs the per-plan `BEGIN; SET LOCAL …; <plan>; COMMIT` envelope; `query()` borrows a fresh `PoolClient` and runs the SQL **without** `SET LOCAL ROLE`, executing as the pool's underlying superuser. The split is invisible to user code (the public `Runtime` surface only exposes `execute()` / `connection()`), but it is a workaround — `query()` is not "use this for admin-y back-channel SQL," that's just what we make it mean. **Upstream fix sketches** (any one would resolve this): (a) add `verify.mode: 'none'` so the wrapping factory can opt out of marker verification when the inner admin runtime has already verified; (b) route the marker query through `acquireConnection() + connection.query()` so a driver can scope it via the same envelope as user plans; (c) thread a `markerQueryHook` through `createRuntime` options so the driver can inject `SET LOCAL ROLE postgres` (or equivalent escalation) for marker queries while keeping user plans scoped. Cross-references Sketch 1 (scoped-session SPI) — promoting "runtime scoped to a GUC profile + role" to a first-class concept would make this back-channel question disappear (the SPI would own the marker-vs-user distinction). | Open — surfaced in T2.2. Workaround is small but the design split is non-obvious; document loudly until the SPI lands. |
| FL-17 | `RuntimeCore` ([`packages/1-framework/4-runtime/runtime-executor/src/runtime-core.ts:144`](../../packages/1-framework/4-runtime/runtime-executor/src/runtime-core.ts)) holds the marker-verification state (`#verified` flag) **per runtime instance**. Every `factory.authenticate(...)` call in [`createSupabaseRuntime`](../../examples/supabase-todos/src/server/supabase-runtime.ts) constructs a new `SqlRuntimeImpl` (because the per-session `ScopedDriver` is bound to the per-session `jwtClaims` / `role` config), so the first plan execution per session triggers a fresh marker query (the FL-14 back-channel path). There is no shared "this contract is already verified for this process" cache; the verify state cannot be lifted out of the runtime instance and into the `ExecutionContext` (which *is* shared across sessions). | At burst-traffic scale (N concurrent request-scoped sessions on a connection pool) the marker query fires N times in addition to the N user plans — doubling the back-channel pool pressure relative to "verified once at process start." The behavior is correct (every marker query returns the same cached row); the cost is invisible without instrumentation. The PoC's tests don't exercise this — `factory.test.ts:201` runs 50 parallel sessions but the test asserts identity isolation, not back-channel volume. | The PoC's [`createSupabaseRuntime`](../../examples/supabase-todos/src/server/supabase-runtime.ts) sets `verify: { mode: 'onFirstUse', requireMarker: true }` so each session pays at most one marker query (not one per plan), but cannot share the verified state across sessions — the back-channel `query()` runs N times for N sessions regardless. **Upstream fix sketches:** (a) lift the `#verified` flag from `RuntimeCoreImpl` into the `ExecutionContext` (which is reused across sessions in this PoC) — fast verifications stay fast, repeat verifications become a no-op flag check; (b) expose a `verify.mode: 'shared'` that consults a process-level cache keyed by `(contract.storage.storageHash, target)`; (c) add `verify.mode: 'none'` (the FL-14 sketch a) and let the wrapping admin runtime own verification once at process start, then have per-request runtimes opt out. (a) is the most architecturally honest; (c) is the smallest change; (a) and (c) compose. Cross-references FL-14 (same machinery, different facet — FL-14 is *role* visibility, FL-17 is *frequency*) and Sketch 1 (a first-class scoped-session SPI would naturally inherit verified state from the parent runtime). | Open — surfaced in T2.2; flagged in code-review as "implementer absorbed silently." Cosmetic at low concurrency; load-bearing under burst traffic. |
| FL-16 | `@prisma-next/driver-postgres/runtime` exports the descriptor (`postgresDriverDescriptor`) and the binding helpers (`createBoundDriverFromBinding`, `PostgresBinding`, `PostgresDriverCreateOptions`), but **not** the underlying `PostgresQueryable` / `PostgresConnectionImpl` / `PostgresPoolDriverImpl` / `PostgresDirectDriverImpl` classes ([`packages/3-targets/7-drivers/postgres/src/postgres-driver.ts`](../../packages/3-targets/7-drivers/postgres/src/postgres-driver.ts)). The cursor-based streaming path (`pg-cursor` integration), the buffered-execute fallback, the `normalizePgError` mapping, and the `acquireClient`/`releaseClient` template-method abstraction are all private. A userspace `SqlDriver` author that wants to layer per-request RLS on top of the standard pg-pool semantics either re-implements those pieces (copy-paste) or restricts itself to a strict subset (no cursor, no normalised errors). The spec ([`spec.md` § Migration operation factories for RLS](spec.md) and [plan.md § 2.2 → "If composition turns out to be impossible without copy-paste, that's an `FL-NN`"](plan.md)) called this out as a possible FL ahead of time; this entry confirms it. | The PoC's [`ScopedDriver`](../../examples/supabase-todos/src/server/supabase-runtime.ts) re-implements only the buffered execute path (single `client.query(sql, params)` then yield rows) — sufficient for the small-fixture tests but giving up the streaming semantics and error normalisation that `PostgresQueryable` provides. A userspace driver that wraps `@prisma-next/driver-postgres` on a high-throughput path would either accept the loss or copy-paste the cursor + normalize-error code. | Buffered-only execute, no error normalisation, no `pg-cursor` import. The `try/finally` in the per-plan async generator handles the rollback-on-throw contract that R-FX-7 asserts (the buffered path is sufficient for "iterator gets `.return()`-ed mid-iteration → rollback + evict"). **Upstream fix sketch:** export `PostgresQueryable` (or a public composition helper that accepts `acquireClient` / `releaseClient` callbacks and returns a fully-equipped `SqlConnection` / `SqlDriver` shape) from `@prisma-next/driver-postgres/runtime`. Cross-cuts with Sketch 1 — a first-class scoped-session SPI on the driver would mean the framework owns the BEGIN/SET LOCAL/COMMIT envelope and userspace just provides the GUC profile, sidestepping the composition question entirely. | Open — surfaced in T2.2. Pre-flagged in spec/plan; severity is "would block production-grade rollout" (cursor streaming and error normalisation are not optional at scale), but the PoC's tests prove the buffered subset is sufficient to validate the pattern. |

## Realtime / change streams

Limitations related to streaming database changes to clients. Out of PN's scope today; populated as we discover what would have been nicer to have natively.

| ID | Issue | Impact | Workaround in app | Status |
|---|---|---|---|---|
| <a id="fl-20"></a>FL-20 | Logical-replication publication membership lives **outside** PN's contract IR / migration ops surface. Supabase Realtime broadcasts `postgres_changes` events only for tables that are members of the `supabase_realtime` publication; the table membership is set per database via `ALTER PUBLICATION supabase_realtime ADD TABLE public.<table>`. The contract IR (`SqlContract<SqlStorage>`) describes app schema (tables, columns, indexes, FKs, RLS-via-FL-04) but has no notion of "this table emits change events." The framework's migration op surface (`@prisma-next/target-postgres/migration`) likewise has no `addToPublication` / `createPublication` factory; the only escape hatch for publication-shaped DDL is `rawSql`. Surfaced in T4.11 (Vite SPA realtime): the SPA's `supabase.channel('todos:user:<uid>').on('postgres_changes', { table: 'todos', filter: 'user_id=eq.<uid>' }).subscribe()` returns silently — no error from the channel, no rows from the subscription — when `public.todos` is not in the publication. The cause is invisible without grepping `pg_publication_tables`. | Two-fold: (a) **Discoverability** — a contributor who clones the repo, runs `supabase start && pnpm migrate:up`, opens the SPA, and notices realtime "doesn't work" has no path through the contract / plan / migration files to the answer; the missing publication membership is invisible from the PN-shaped surfaces. (b) **Bootstrap convention** — every PoC / production deployment that wants Realtime has to set publication membership *somewhere*, and that "somewhere" is currently outside the migration system. There is no idiomatic spot to put it: not the contract (no IR for it), not the migration (no op factory), not `supabase/config.toml` (no `[realtime.publications]` section as of CLI v1.x). | The seed script ([`examples/supabase-todos/scripts/seed.ts`](../../examples/supabase-todos/scripts/seed.ts) — `ensureRealtimePublication()`) issues an idempotent `ALTER PUBLICATION supabase_realtime ADD TABLE public.todos` after the user / row fixtures land. Idempotency is via a `pg_publication_tables` membership check before the ALTER. The seed is the bootstrap step that already owns the direct DB URL and runs once after `supabase start && pnpm migrate:up`, so it is the closest thing the example has to a "demo wiring outside the contract" hook — but it's a workaround: a fresh database that skips `pnpm seed` (e.g. a CI integration test that loads its own fixtures) loses Realtime. **Upstream fix sketches:** (a) extend `@prisma-next/target-postgres/migration` with `addToPublication(name, schema, table)` / `createPublication(name, options)` op factories so publication membership lives in the migration like any other DDL — composes cleanly with FL-04 (closed `objectType` union; would want a `'publication'` member). (b) Promote a `realtime` capability bit on the contract IR (Supabase-target-shaped) so the contract emits a `recommendedPublications` hint that an extension pack can lower into ops at apply-time. (a) is the smaller change and the one this PoC's friction directly motivates; (b) is the architecturally cleanest answer if PN ever grows a target-aware publication concept. Cross-references FL-04 (op-class union) and FL-08 (private migration op-authoring helpers — same root cause: user-authored migration ops are second-class). | Open — surfaced in phase-4c (T4.11, Vite SPA realtime). Workaround works for the demo; would be a cliff for any non-PoC deployment that doesn't run the seed. |

## Driver / connection lifecycle

Limitations encountered while wrapping `@prisma-next/driver-postgres` to apply RLS context.

> The frictions encountered while building [`createSupabaseRuntime`](../../examples/supabase-todos/src/server/supabase-runtime.ts) ended up clustering in the **RLS / per-request session context** section above (FL-14 / FL-16 / FL-17). They are recorded there rather than duplicated here because the root cause is the runtime composition story, not the connection-acquisition mechanics. **M3 (`'connection'` mode, stretch) was descoped at the orchestrator's call after phase-2 closed cleanly** — the connection-mode design context lives at [FL-18](#fl-18) below, and the upstream design surface that would replace it lives in [Sketch 1 — Scoped-session SPI](#sketch-1--scoped-session-spi).

| ID | Issue | Impact | Workaround in app | Status |
|---|---|---|---|---|
| <a id="fl-21"></a>FL-21 | `createSupabaseRuntime`'s `'transaction'` scope mode rules out user-initiated nested transactions outright. `session.beginTransaction()` throws synchronously with code `runtime/unsupported-scoped-tx` ([`examples/supabase-todos/src/server/supabase-runtime.ts`](../../examples/supabase-todos/src/server/supabase-runtime.ts) — `unsupportedScopedTxError`). The reason is concrete: every plan already runs inside its own `BEGIN; SET LOCAL request.jwt.claims …; SET LOCAL ROLE …; <plan>; COMMIT` envelope, so a user-initiated transaction would have to compose with that envelope via Postgres `SAVEPOINT`s — and the framework's `Runtime.beginTransaction()` SPI doesn't expose savepoint semantics. Spec'd as R-FX-8 (out of scope for the PoC); not just a current-impl shortfall, but a structural consequence of "transaction-per-plan" being the only scope mode (FL-18) plus the SqlRuntime SPI assuming a single transaction layer. **Surfaced during 5.1 completeness sweep** — should have been filed during phase-2; the [`supabase-runtime.ts`](../../examples/supabase-todos/src/server/supabase-runtime.ts) docblock already had an `(FL-NN)` placeholder pointing here, but the matching entry was never added. Continuous-capture discipline slipped: R-FX-8's spec wording ("out of scope for the PoC") was misread as "not a framework gap" rather than "deferred-and-still-a-gap." | A user wiring a multi-step write that wants atomicity beyond a single plan (e.g. "insert a todo and a denormalised audit row") cannot use `session.beginTransaction()` to wrap them — they have to either (a) drop into raw SQL inside a single plan, (b) issue both writes as separate plans and accept that they are independently committed, or (c) layer their own `SAVEPOINT` machinery on top of an admin runtime (which bypasses RLS, defeating the point). The throw is a hard wall, not a graceful fallback. | The PoC's handlers all happen to be single-plan operations (todos GET / POST / PATCH / DELETE; public messages GET), so the limitation never bites in practice. The factory throws `unsupportedScopedTxError` synchronously from `session.beginTransaction()` so a future handler that reaches for it fails loudly rather than producing an undefined-behaviour result. **Upstream fix sketch:** [Sketch 1 — Scoped-session SPI](#sketch-1--scoped-session-spi) is the architecturally honest answer — a first-class scoped-session SPI can layer user-initiated nested transactions via `SAVEPOINT` (transaction-scope mode) or via the borrowed connection's native transaction stack (connection-scope mode) without the userspace factory having to choose. Cross-references [FL-18](#fl-18) (descope of connection-scope mode — same Sketch-1 promotion path) and FL-14 above (the marker-query back-channel that complicates any "scope this whole runtime" answer). | Open — surfaced during phase-5.1 completeness sweep; should have been filed in phase-2. Cosmetic for the PoC (no handler reaches for `beginTransaction`); load-bearing for any production handler that wants atomic multi-plan writes inside an RLS-scoped session. |
| <a id="fl-19"></a>FL-19 | The phase-4 anon-session path on public routes ([`examples/supabase-todos/src/server/middleware/scoped-runtime.ts`](../../examples/supabase-todos/src/server/middleware/scoped-runtime.ts) — `factory.authenticate({ jwtClaims: {}, role: 'anon' })`) goes through the same `'transaction'`-scope envelope every other request gets: `BEGIN; SELECT set_config('request.jwt.claims', '{}', true); SET LOCAL ROLE "anon"; <plan>; COMMIT`. That's a 4-round-trip setup/teardown cost on **every** public request — including hot endpoints like `GET /api/public/messages` where the underlying SELECT is a sub-millisecond table read. The cost is uniform (not specific to anon) but anon is where it shows up in practice, because public endpoints tend to be the high-traffic ones (a public board page is hit on every visit; the authenticated todos page is hit per signed-in user). | **Performance friction, not a correctness or security gap.** At local-Postgres latencies the envelope cost is sub-millisecond and invisible; against networked Postgres (Supabase Cloud, RDS, etc.) every public request pays an extra ~4× round-trip latency beyond the actual query. For a SELECT that takes 1 ms over a 5 ms-RTT network, the overhead dwarfs the work — and on a hot public endpoint that overhead is what the user-facing latency is dominated by. Cross-references FL-18: connection-scope mode would amortise the `SET ROLE` / `set_config` call once per long-lived connection rather than per request, eliminating most of this cost. | The factory shape is intentionally uniform — handlers call `c.var.db.execute(plan)` regardless of route auth shape, so RLS is the single point of correctness and there is no admin-vs-scoped fork inside handler code (which would itself be a security smell). For the PoC and any Supabase deployment whose public endpoints are low-traffic, accept the overhead. For high-traffic public endpoints, the practical workaround is a layer **above** PN: a small in-process or HTTP cache (Redis, edge cache, etc.) on the `GET /api/public/messages` response keyed by a sensible TTL. The cache would absorb most of the load and the cost is paid only on cache misses. **Upstream fix sketch:** the connection-scope mode design captured at [FL-18](#fl-18) is the right architectural answer — once a process can hold an anon-scoped connection across requests, the envelope cost amortises away. The promotion path is [Sketch 1 — Scoped-session SPI](#sketch-1--scoped-session-spi): a first-class `SqlSessionScope` SPI lets the driver pick connection-vs-transaction scope based on its capability key, and the public-route case naturally falls into the connection-scope bucket (anon claims and role are static; binding once is the obvious shape). Severity is **equivalent to FL-18** — same architectural gap, different facet (FL-18 is the descope; FL-19 is the per-request cost the descope leaves on the table). | Open — surfaced in phase-4a system-design-review § 4 (anon-session lifecycle) and rolled into phase-4b's first commit. Cross-references [FL-18](#fl-18) and [Sketch 1 — Scoped-session SPI](#sketch-1--scoped-session-spi). |
| <a id="fl-18"></a>FL-18 | [`createSupabaseRuntime`](../../examples/supabase-todos/src/server/supabase-runtime.ts) only implements `scopeMode: 'transaction'` (the per-plan `BEGIN; SET LOCAL request.jwt.claims …; SET LOCAL ROLE …; <plan>; COMMIT` envelope; satisfies R-FX-1 / R-FX-2). Connection-scope mode (R-FX-3 — direct/session-mode connections that hold a `PoolClient` per session and run `SET request.jwt.claims` / `SET ROLE` *once* at session start, not `SET LOCAL`) is not implemented. The two modes have meaningfully different lifecycle shapes, so connection mode is a *sibling* implementation rather than a parameter flip on the existing one. The factory accepts `scopeMode: 'transaction'` as the only legal value; passing anything else throws synchronously ([`supabase-runtime.ts`](../../examples/supabase-todos/src/server/supabase-runtime.ts) `createSupabaseRuntime` body). The plan's tasks 3.1–3.3 sketched the design but were descoped after phase-2 closed cleanly — the spec ([`spec.md` § Acceptance Criteria](spec.md) and [`plan.md` § Milestone 3](plan.md)) explicitly accepted this descope path. | **No functional gap** for the PoC: every Postgres URL accepts `SET LOCAL`, so transaction mode works against the local Supavisor pooler (Supabase's default deployment shape) *and* against direct URLs. The gap is a **performance** gap for production deployments that prefer direct/session connections — every plan pays 4 round-trips of setup/teardown overhead (`BEGIN`, `set_config`, `SET LOCAL ROLE`, `COMMIT`) where connection mode would amortise to one-time setup per session. At local-Postgres latencies the overhead is sub-millisecond and invisible; against Supabase Cloud or any networked deployment with a few ms of round-trip latency, the multiplicative cost over a request handler that issues several plans becomes measurable. | Use the existing transaction-mode factory against a pooled URL (Supavisor / pgbouncer transaction mode) — this is the Supabase-default deployment shape and incurs no additional cost beyond what production already pays. For direct-connection deployments that want to claim back the per-plan envelope cost, a userspace fork would implement the sibling design: factory takes an explicit `scopeMode: 'transaction' \| 'connection'` (no auto-detect — the plan's task 3.3 considered detecting from connection-string patterns like `pgbouncer=true` / port `6543`, and the [Open items default](plan.md#open-items) was "drop; require explicit choice," which we apply here); connection mode borrows one `PoolClient` per `authenticate()`, runs `SET request.jwt.claims = $1` and `SET ROLE <role>` once at session start, holds the client until `session.end()` releases it, calls `destroy(err)` on mid-session errors, and lets `session.beginTransaction()` work normally (delegating to the borrowed client's transaction lifecycle, unlike transaction mode's R-FX-8 throw). The `SqlDriver` SPI surface is the same; the difference is purely in the connection-acquisition layer. **Upstream fix** is the right shape rather than a userspace fork: see [Sketch 1 — Scoped-session SPI](#sketch-1--scoped-session-spi), which would promote "runtime scoped to a GUC profile + role" to a first-class concept on `SqlRuntime` / `SqlDriver` and let the driver pick connection-vs-transaction scope based on its capability key, sidestepping the userspace per-mode question entirely. | Open — descoped at orchestrator's call after phase-2 closed cleanly. Severity is below FL-02 (cross-schema FK is a hard production blocker), roughly on par with FL-09 (DDL builder gap) — a real architectural improvement worth lifting upstream, not a papercut. |

## Migration authoring

Limitations encountered while authoring the schema and RLS policies via PN's migration system. This is the second part of PN the PoC exercises (alongside the runtime).

| ID | Issue | Impact | Workaround in app | Status |
|---|---|---|---|---|
| FL-04 | `OperationClass` (the `target.details.objectType` union) is closed: `'dependency' \| 'type' \| 'table' \| 'column' \| 'primaryKey' \| 'unique' \| 'index' \| 'foreignKey'`. There is no `'policy'` value. | Pack-level operation factories that produce kinds outside this set must either fall back to `'dependency'` (losing semantic precision in planner output / target details) or type-cast (a smell). | RLS factories (`enableRowLevelSecurity`, `createRlsPolicy`, `dropRlsPolicy`) set `target.details.objectType = 'dependency'`. | Open — surfaced in T1.5; covered by Sketch 3. |
| FL-05 | `prisma-next migration plan` scaffolds `migration.ts` with `#!/usr/bin/env -S node` plus mode `0755` (executable bit set). The file is TypeScript, and Node's ESM loader doesn't compile `.ts` directly — invoking the scaffolded file as `./migrations/<ts>_<slug>/migration.ts` fails with `ERR_MODULE_NOT_FOUND`. Source: [`packages/1-framework/3-tooling/migration/src/runtime-detection.ts:16`](../../packages/1-framework/3-tooling/migration/src/runtime-detection.ts) emits the shebang unconditionally for `.ts` outputs. Developer-experience papercut, not a correctness issue (canonical invocations don't use the shebang) — the scaffold misleads new contributors into trusting the executable bit. | Surfaced in 1.6 when verifying the re-attest step. The migration's docblock, [`SKILL.md` § 10 step 3](skills/writing-rls-policies-with-pn/SKILL.md#10-where-to-write-it-pn-specific) and [`SKILL.md` § 9](skills/writing-rls-policies-with-pn/SKILL.md#9-common-anti-patterns) all now spell out "use `tsx`, not `node` directly". | Always invoke via `pnpm exec tsx <path>` or `pnpm --filter <example> migrate:up`. The shebang and exec-bit on the scaffolded file are ignored by both invocations, so the workaround needs no source edit. **Upstream fix sketch:** for `.ts` scaffold outputs, either (a) emit `#!/usr/bin/env -S tsx` so `./migration.ts` works directly when `tsx` is on PATH, or (b) drop the shebang+exec-bit entirely (the canonical invocation is `tsx` / `migration apply` and neither needs them). Option (b) is simpler and matches the project convention that migrations are run through the CLI / package script, not invoked directly. | Open — surfaced in 1.6 (CLI-first refactor). Cosmetic; safe to defer. |
| FL-06 | `@prisma-next/target-postgres/migration` exports an op-factory called `rawSql` ([`packages/3-targets/3-targets/postgres/src/exports/migration.ts`](../../packages/3-targets/3-targets/postgres/src/exports/migration.ts)). The runtime SQL lane in `@prisma-next/sql-builder/runtime` (and the planner's "raw SQL" surface) also use the name `rawSql` for an entirely different concept — a builder that produces a *runtime* `Plan` from a SQL string. The two share a bare name across two domains (migration-time DDL escape hatch vs. runtime query builder). | Naming collision is invisible until a file imports from both, at which point `import { rawSql } from '@prisma-next/target-postgres/migration'` and `import { rawSql } from '@prisma-next/sql-builder/runtime'` shadow each other. In a single file (e.g. an integration test that issues a runtime query against a table created via a `rawSql` migration op) the user has to alias one of them, and the resulting code is hard to grep. Even when only one is imported, the name `rawSql` *reads* like a query builder, not an operation factory — see [`SKILL.md` § 10](skills/writing-rls-policies-with-pn/SKILL.md#10-where-to-write-it-pn-specific) where the in-example RLS factories explicitly choose to *not* re-use the name. | Aliasing on import (`import { rawSql as rawSqlOp } from '@prisma-next/target-postgres/migration'`) when both are needed in one file. The PoC's RLS factories ([`examples/supabase-todos/migrations/utils/rls-ops.ts`](../../examples/supabase-todos/migrations/utils/rls-ops.ts)) wrap `rawSql` and re-export under names that read as operations (`enableRowLevelSecurity`, `createRlsPolicy`, `alterRlsPolicy`, `dropRlsPolicy`), avoiding the bare name in user code. **Upstream fix sketch:** rename the migration-side factory to `op()` (the call site reads as `op('CREATE POLICY …')`, which matches its semantic — produces an `Op`) or `rawSqlOp()`. Keep the runtime `rawSql` name as-is (it has the longer-established conceptual claim on the term). | Open — surfaced in round-3 review of `rls-ops.ts`. Cosmetic; coordinate with any future rename of the migration op-factory exports. |
| FL-07 | `PostgresPlanTargetDetails` (the `target.details` shape that every migration op carries; defined in [`packages/3-targets/3-targets/postgres/src/exports/planner-target-details.ts`](../../packages/3-targets/3-targets/postgres/src/exports/planner-target-details.ts)) is opaque to user-authored op factories. Its tri-shape `{ schema, objectType, name, table? }` overloads `name` (sometimes the table name; sometimes the policy/index/constraint name with `table` set as the parent) and the `objectType` discriminant has no `'policy'` value (FL-04), so policy ops fall back to `'dependency'` — which itself has no documented meaning at the call site. # Design sketches — proposed upstream work

Three half-page sketches of changes that *could* be upstreamed if the team decides the patterns established by this PoC are worth promoting from "userspace recipe" to "first-class framework concept." These are documentation of "what we'd want next," not commitments. They are intentionally short; if any prove popular, they become their own project with a proper spec.

## Sketch 1 — Scoped-session SPI

> **Status:** _to be written during 5.2_  
> **Concrete example unblocked by this sketch:** [FL-18](#fl-18) (connection-scope mode of `createSupabaseRuntime`, descoped from M3) — a first-class scoped-session SPI lets the driver pick transaction-vs-connection scope based on its capability key, removing the per-mode userspace fork the PoC would otherwise need.

**Problem.** Today, "run the next N plans against this Postgres connection with these GUCs / this `ROLE`" is not a first-class concept. Apps have to wrap the driver to express it (see this PoC's `createSupabaseRuntime`). Supabase users hit this; multi-tenant apps using session GUCs hit this; anyone who needs `SET LOCAL search_path` per request hits this.

**Sketch.** _(to be written)_

  - Option A: a `runtime.withSession({ guc: Record<string, string>, role?: string }): SqlRuntime` method on `SqlRuntime` that returns a runtime bound to a GUC profile. Implementation chooses connection-scope or transaction-scope based on driver capability.
  - Option B: a dedicated `SqlSessionScope` SPI on the driver. The runtime gains a `scope(opts): SessionScope` method that returns an object owning a connection (or transaction) for its lifetime; plans executed through it inherit the GUCs.

**Trade-offs.** _(to be written)_

  - Connection-scope vs transaction-scope is a property of the deployment (pooler mode), not the application. The SPI must let drivers pick.
  - Interaction with explicit `beginTransaction()` needs to be defined: nested? error?
  - Capability key like `sql.sessionScope.connection` / `sql.sessionScope.transaction` so middleware/lints can branch.

## Sketch 2 — Subscription lane

> **Status:** _to be written during 5.3_

**Problem.** Prisma Next has a great story for streaming the rows of a query. It has no story for streaming changes to a table. Supabase users get this from a separate service; plain Postgres users get this from `LISTEN/NOTIFY`. There's no contract-aware primitive in PN that says "tell me when a row I would have read changed."

**Sketch.** _(to be written)_

  - A new lane `db.subscribe(tables.todos).where(...).select({...}) → AsyncIterable<ChangeEvent<Row>>`.
  - Lowers via adapter capability:
    - Plain Postgres: `LISTEN <channel>` + an outgoing-trigger / replication-slot reader.
    - Supabase: open a Realtime websocket via an adapter-level helper.
  - `ChangeEvent` is `{ kind: 'insert' | 'update' | 'delete', row: Row, oldRow?: Row, source: { commitTs: string, lsn?: string } }`.

**Trade-offs.** _(to be written)_

  - Backpressure / replay semantics need to be defined.
  - RLS enforcement on subscriptions is a separate consideration (Supabase Realtime does this server-side; plain LISTEN/NOTIFY can leak rows the subscriber couldn't otherwise read).
  - Capability gates: `sql.subscribe.listen`, `sql.subscribe.replication`, `supabase.realtime`.

## Sketch 3 — RLS-aware contract metadata + lints + native policy authoring

> **Status:** _to be written during 5.4_

**Problem.** Today PN has no way to know that a table is RLS-protected, so it cannot:

1. warn an author who runs a query against `todos` from an unauthenticated runtime (the query just returns zero rows silently — a class of bug Supabase users will hit constantly), or
2. emit RLS-related operations (`ENABLE ROW LEVEL SECURITY`, `CREATE POLICY`) from the migration planner. This PoC's example has to hand-edit planner-emitted migrations to insert calls to in-example factories.

A related friction is that the postgres target's `OperationClass` union is closed and has no `'policy'` value, so any pack-level factory producing policy ops must fall back to `'dependency'`.

**Sketch.** _(to be written)_

  - Optional contract annotation `@@rls(roles: ["anon", "authenticated"])` (PSL) / `tables.todos.rls(...)` (TS DSL) capturing roles and (optionally) policy expressions.
  - The migration planner reads these annotations and emits `enableRowLevelSecurity` / `createRlsPolicy` operations into the produced migration file (no hand-editing required for the common case).
  - The two factories themselves get promoted from in-example helpers to public exports of `@prisma-next/target-postgres/migration`, and `OperationClass` gains a `'policy'` value (or becomes pack-extensible).
  - A `SqlMiddleware` lint `rls/missing-session` that consults `plan.meta.refs.tables` against the contract; when a referenced table is RLS-annotated and the runtime has no `session` context attached (see Sketch 1), it warns or errors per policy.
  - Optional second lint `rls/expected-role` that validates the active role is in the table's allowlist.

**Trade-offs.** _(to be written)_

  - Depends on Sketch 1 — without a session concept, the lint has nothing to check against.
  - Annotations don't *enforce* anything at query time; Postgres still owns enforcement. They exist for authoring-time feedback and to drive migration-planner output.
  - Embedding the policy *expressions themselves* in the contract is the open design question. Options: (a) only roles in the contract; expressions in a sibling file (looser coupling, less drift signal); (b) full policies in the contract (tightly coupled, planner is fully declarative, but the contract starts encoding operational concerns).
  - Making `OperationClass` extensible is itself a small cross-cutting change to `@prisma-next/target-postgres` planner internals; size with care.
