# System design review — commit `68ebbeb25` (M1-cleanup F4)

## What the commit does

It hoists the contract-space identity / authoring types out of the SQL family (`@prisma-next/family-sql/control`) and into the framework control plane (`@prisma-next/framework-components/control/control-spaces.ts`), renaming them in the process: `ExtensionContractRef` → `ContractSpaceHeadRef`, `ExtensionMigrationPackage` → `AuthoredMigrationPackage`, `ExtensionContractSpace` → `AuthoredContractSpace<TContract extends Contract = Contract>`. It also lifts `MigrationHints` / `MigrationMetadata` from `migration-tools/src/metadata.ts` up into `framework-components/control/control-migration-types.ts` so the new framework-level types can reference them without an upward import, and renames the producer helper `writeExtensionMigrationPackage` → `writeAuthoredMigrationPackage`. There are no backward-compat re-exports for the three space types in the SQL family; the `MigrationMetadata` move retains a re-export shim in `migration-tools/src/metadata.ts`.

## What this does to the system shape

The change is, at root, a bounded-context fix. The project spec is explicit (FRs 3-6, § Approach) that contract spaces are framework-level — "the framework operates per space" and "a Mongo descriptor will eventually consume the same types specialised to its own contract." Pre-commit, the canonical declarations of the space envelope lived in the SQL family. That was a category error: a family-agnostic concept being owned by one family forces every other family that wants to participate either to upward-import from a sibling family (a layering violation) or to redeclare the same shape (a synonym-for-the-same-thing). Putting the types in `framework-components/control` aligns ownership with the concept and lets the SQL family's `SqlControlExtensionDescriptor.contractSpace?:` *specialise* the framework type rather than *define* it.

The new generic parameter `<TContract extends Contract = Contract>` is the seam that makes this possible without giving up typing at the consumption site: the SQL family pins `AuthoredContractSpace<Contract<SqlStorage>>` while the framework type stays family-neutral. The default value `= Contract` keeps framework-level usage ergonomic. This generic earns its keep against the *current* structure (one family in tree, with a clearly typed contract value); it isn't speculative.

The renames are, to my eye, the most consequential part of the commit, more so than the file move. The old names partitioned the typology along the *who authored the value* axis — `Extension*`, as if extension-authored migrations were a different *kind* of thing from app-authored migrations. They aren't. The app-space planner produces in-memory packages with the same shape as an extension descriptor's, and the framework's emitter materialises both into identical on-disk layouts. The real partition is *lifecycle*: in-memory authoring form (no `dirPath`) vs. on-disk emitted form (has `dirPath`, lives at `migrations/<space-id>/<dirName>/`). `Authored*` encodes that partition. `ContractSpaceHeadRef` correctly does *not* carry the prefix because the head ref is identical in shape on both sides of emission. The vocabulary now reads true to a fresh contributor: it tells them the right structural distinction exists, and the right one only.

A subsidiary win: `AuthoredMigrationPackage` collapses two prior synonyms — `ExtensionMigrationPackage` in `family-sql/control` and the lampshaded `MigrationPackageContents` duplicate in `migration-tools/src/io.ts`. Conceptual integrity recovered.

## Concerns

1. **The `migration-tools/metadata` re-export is conceptual debt the commit takes on.** The commit drops the three `Extension*` re-exports from `2-sql/9-family/src/exports/control.ts` "per repo convention" (no backward-compat shims). The same rule did not get applied to `migration-tools/src/metadata.ts`, which is now a one-line re-export of `MigrationHints` / `MigrationMetadata` "so existing `@prisma-next/migration-tools/metadata` consumers (12 files in CLI + tests) continue to work unchanged." That leaves the codebase with two import paths for the same type — exactly the synonym-for-the-same-thing the rest of the commit is fixing. If the convention is no shims, the 12 call sites should be updated and the re-export removed. Surface as F4-followup; tractable, not a blocker.

2. **Spec drift acknowledged in the commit message.** The commit notes that § 3 / § 6 / § 7 of `framework-mechanism.spec.md` still mention the old names in narrative prose. The spec is a surface a fresh contributor reads; three sections of it now contradict the updated § 1. This should land before the cleanup is considered settled — the orchestrator pass mentioned in the message.

3. **`AuthoredMigrationPackage.ops` source type changed silently.** The old `ExtensionMigrationPackage.ops` was typed as `MigrationOps` from `@prisma-next/migration-tools/package` (the on-disk op type). The new `AuthoredMigrationPackage.ops` is `readonly MigrationPlanOperation[]` from `framework-components/control/control-migration-types`. The type-d test asserts `AuthoredMigrationPackage['ops']` equals `MigrationOps`, which suggests the two are aliases today, but conceptually they shouldn't be — `MigrationPlanOperation` is the plan-time op type and `MigrationOps` is the post-emission on-disk type, mirroring the very same in-memory-vs-on-disk distinction that the rest of the rename is built on. Either fold them (and document the alias), or keep them distinct and pick the *plan-time* type for the authoring form deliberately. Right now the choice reads as accidental.

4. **`Authored*` could be doc-clarified at point of declaration.** The new prefix is defensible — "authored" stands in for "the author's in-memory view, pre-emission" — but a fresh contributor coming from "what was wrong with `Extension*`?" needs to read three doc comments to figure out that the new partition is *lifecycle*, not *authorship*. A one-line lead in `control-spaces.ts` ("the `Authored*` prefix denotes in-memory pre-emission form, distinct from the post-emission `MigrationPackage` shape in `migration-tools/package`") would cement the partition the prefix encodes. Cheap; high readability return.

5. **Migration-tools' role recharacterisation deserves a note.** With `MigrationMetadata` now owned in `framework-components/control`, `migration-tools` shifts from "owner of the migration-package vocabulary" to "I/O implementer for vocabulary owned upstream." That is the right positioning, but the package's own README/exports still read as if it owns the types. A short follow-up to migration-tools docs is warranted.

## Verdict

**CONCERNS.** The structural direction is correct and the renames are a meaningful improvement to the system's typology — the `Extension*` prefix was lying about a partition that the types do not in fact make, and `Authored*` cuts along the real axis (in-memory vs. on-disk). The commit also fixes a layering / bounded-context defect: a family-agnostic concept now lives in the family-agnostic layer. The residual issues — the `migration-tools/metadata` re-export contradicting the no-shim convention, the spec § 3/§ 6/§ 7 lag, the silent change in `ops` source type, and a small doc-clarity gap — are tractable follow-ups, not structural blockers. Land the cleanup follow-up before the next round.
