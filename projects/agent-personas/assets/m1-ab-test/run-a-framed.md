# System Design Review — `68ebbeb25` (M1-cleanup F4)

## Summary

The commit hoists three contract-space identity types — `ExtensionContractRef`, `ExtensionMigrationPackage`, `ExtensionContractSpace` — out of the SQL family (`packages/2-sql/9-family/src/core/migrations/types.ts`) and up into the framework's control plane (`packages/1-framework/1-core/framework-components/src/control/control-spaces.ts`), simultaneously renaming them to `ContractSpaceHeadRef`, `AuthoredMigrationPackage`, and `AuthoredContractSpace<TContract extends Contract = Contract>`. To make the move possible without an upward import, it also moves `MigrationHints` and `MigrationMetadata` from `@prisma-next/migration-tools/metadata` into `framework-components/control/control-migration-types.ts` and turns the old `metadata.ts` into a re-export. The producer-side helper `writeExtensionMigrationPackage` is renamed to `writeAuthoredMigrationPackage` in step.

## Walkthrough at the structural level

**Where the types now live.** Three small interfaces and a generic specialisation cross a layer boundary. Before this commit, "what a contract space is" was defined in the SQL family — a structural choice that read cold as "contract spaces are a SQL feature." After this commit, the same concept is defined in `framework-components/control`, which is the framework's family-agnostic control-plane vocabulary. The SQL family's `SqlControlExtensionDescriptor.contractSpace?:` field now consumes `AuthoredContractSpace<Contract<SqlStorage>>` — a specialisation of the framework type rather than a SQL-owned definition. This direction matches the project spec, whose FRs 3–6 are framed in family-neutral terms ("the planner runs per space …", not "the SQL planner runs per space"), and prepares the seam for a future Mongo descriptor to consume the same shape against `Contract<MongoStorage>`. The dependency direction is correct: the more concrete (SQL family, in `2-sql/`) now depends on the more abstract (framework control plane, in `1-framework/1-core/`).

**The new generic parameter.** `AuthoredContractSpace<TContract extends Contract = Contract>` introduces a single type parameter so each family pins its own contract shape. The default of `Contract` keeps the unparameterised reference valid in framework-layer code paths that don't care about the SQL/Mongo distinction. This is the minimum amount of generic surface needed to support both "a planner that operates uniformly across spaces regardless of family" and "a descriptor author who sees a typed contract value." Fine.

**Boundary effect on `framework-components`.** The hoist is not just three identity types — it pulls `MigrationHints` and `MigrationMetadata` (the migration-package on-disk envelope) into `framework-components/control` as well, because `AuthoredMigrationPackage` references `MigrationMetadata`. That is a non-trivial expansion of the core's responsibility: framework-components/control now owns the *shape* of a migration metadata envelope (hash, from/to, fromContract/toContract, hints, providedInvariants, authorship, signature, createdAt). The *write/read code* and the arktype runtime schema that validates `migration.json` correctly stayed in `migration-tools`. The bounded context "what's in a migration directory" is now split across two packages: type definitions in `1-framework/1-core/framework-components`, parsing/writing/canonicalisation in `1-framework/3-tooling/migration`. The split is structurally legitimate (the dependency direction is downward, framework-components carries no migration-tools imports), but it does enlarge framework-components/control's contract.

**How `migration-tools` is rewired.** `migration-tools/src/metadata.ts` collapses to a single line: `export type { MigrationHints, MigrationMetadata } from '@prisma-next/framework-components/control'`. The lampshaded `MigrationPackageContents` interface that previously duplicated the in-memory shape inside `migration-tools/src/io.ts` is removed; `writeAuthoredMigrationPackage` now imports `AuthoredMigrationPackage` directly from framework-components. Net structural effect: one canonical in-memory shape, one canonical metadata shape, both owned by the framework layer; migration-tools is reduced to the I/O concern and the arktype schema.

**Consumer-side updates.** The SQL family's `2-sql/9-family/src/core/migrations/types.ts` drops the three local interface declarations and imports `AuthoredContractSpace` from framework-components; `2-sql/9-family/src/exports/control.ts` drops the three `Extension*` re-exports outright (no shim, per repo convention); the integration-test fixture and the type test update their imports / type names. The fact that no SQL-side functional code other than the descriptor field had to change is encouraging — these were genuinely identity types, not bearers of SQL-specific behaviour, and the move is compatible with the SQL family carrying on as before.

## Concerns

**`Authored*` is an implied taxonomy that does not partition.** The new prefix `Authored*` (`AuthoredContractSpace`, `AuthoredMigrationPackage`, plus the renamed `writeAuthoredMigrationPackage`) reads cold as "as opposed to … some other kind of `ContractSpace` / `MigrationPackage`." There is no other kind. Per the doc-comment and per the spec § 1 resolution, the value is just an in-memory contract-space view; the only structural sibling is `MigrationPackage` in `migration-tools/src/package.ts`, which differs by carrying a `dirPath` (i.e. it is the *on-disk* form). The honest typology is `MigrationPackage` (in-memory canonical) and `OnDiskMigrationPackage extends MigrationPackage` adding `dirPath` — exactly the shape the spec § 1 explicitly endorses ("`MigrationPackage` is the canonical structural shape — `{ dirName, metadata, ops }`, in-memory by default … `@prisma-next/migration-tools/package` exposes the augmented `OnDiskMigrationPackage extends MigrationPackage`"). The commit ships an intermediate state that does not yet match its own spec: framework-components/control defines `AuthoredMigrationPackage` while migration-tools/package.ts continues to export `MigrationPackage` *with* `dirPath`. So we now have two structurally similar types, in two packages, with names that imply a partition along the wrong axis (lifecycle phase / authoring relationship) instead of the real one (presence of `dirPath`). The same critique applies to `AuthoredContractSpace`: app-space and extension-space share the *exact* shape, as the spec § 1 itself notes; the `Authored*` qualifier is not picking out a structural distinction the system makes.

The cost compounds: `writeExtensionMigrationPackage` was renamed once (this commit) and the spec already records that a *third* rename to `materialiseMigrationPackage` is queued under M1-cleanup F6. That is two renames touching every consumer when the second one carries no architectural payoff — the first rename should have moved straight to the final shape (`MigrationPackage` / `OnDiskMigrationPackage`, `materialiseMigrationPackage`).

**Backward-compat re-export shim violates a stated repo rule.** `migration-tools/src/metadata.ts` is preserved as a thin re-export so that the 12 CLI + test consumers of `@prisma-next/migration-tools/metadata` keep working. The repo's golden rules (`AGENTS.md` / `CLAUDE.md`) state explicitly: *"Don't add exports for backwards compatibility unless requested to do so"*, and *"Do not reexport things from one file in another, except in the `exports/` folders"*. The shim is both a backward-compat export and a re-export from a non-`exports/` source file. The right move is to update the 12 call sites to import from `@prisma-next/framework-components/control` and delete `metadata.ts` (or fold it into `exports/`). The commit message acknowledges the shim deliberately; the architect's read is that the shim is debt the team is taking on, not a structural decision earning its keep.

**Spec is now internally inconsistent.** The commit updates `framework-mechanism.spec.md` § 1 to use the new vocabulary (`ContractSpaceHeadRef`, `AuthoredMigrationPackage`, `AuthoredContractSpace`) but leaves §§ 3, 6, 7 referencing the old names (`ExtensionContractRef` and family). The commit message flags this as out of scope for this round. As a structural artefact the spec now teaches a fresh reader two vocabularies side by side. Conceptual integrity requires this to be cleaned up before any further round modifies the spec — not deferred indefinitely. The fact that it is queued surfaces it as known debt; the cost is small *now* and grows the longer the inconsistency persists.

**Migration metadata's home.** Pulling `MigrationMetadata` into `framework-components/control` resolves the upward-import problem cleanly and is justified given current dependency direction. It does, however, mean the framework's core control-plane vocabulary now includes hash semantics, planner-version tracking, content-addressing assertions, and authorship/signature shape — concerns that read more naturally as "migration system internals." The alternative (parameterising `AuthoredMigrationPackage<TMetadata>` over the metadata type, or weakening the framework reference to `unknown`) would have kept framework-components leaner. Either choice is defensible; the chosen one should be considered a bet that "migration metadata envelope" really is a framework-level concept and not a tooling-level one.

## Verdict

**CONCERNS.** The directional move — contract-space identity types belong in the framework layer, not in the SQL family — is correct and sets up the multi-family endgame the spec promises. The dependency direction is right and the SQL family's surface is appropriately reduced. But the chosen names introduce an implied partition (`Authored*`) that the underlying types do not actually make, and the commit knowingly ships an intermediate state with two more renames already queued (`MigrationPackage` / `OnDiskMigrationPackage` per the spec; `materialiseMigrationPackage` per F6). The vocabulary should land at the spec's own end-state in one move rather than across three. Additionally, the `migration-tools/src/metadata.ts` re-export shim is explicit backward-compat that the repo rules forbid, and the spec is left half-converted between the old and new vocabularies. None of these are blockers for the directional intent — they are debt the team is taking on, and the architect's lens reports it as such so the next round can collapse it.
