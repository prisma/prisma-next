# Agent Personas Plan

## Summary

Build a shared agent-persona library — the `drive-agent-personas` skill at `.agents/skills/drive-agent-personas/` — and rewire the existing skill set to declare which persona executes each one. The work plugs the F0–F6-class vocabulary-and-typology leak that surfaced in M1-cleanup of `extension-contract-spaces`. v1 ships seven personas and refactors `/drive-pr-local-review` into a composite + atomic single-persona sub-skills.

**Spec:** `projects/agent-personas/spec.md`

## Shipping Strategy

Each milestone is independently safe to land. The implicit gate that keeps existing skills working through the entire rollout is **the developer-as-default rule**: a skill without an explicit "Adopt the `<id>` persona" instruction executes as `developer`, which is the agent's current default behaviour. Nothing in v1 forces existing skills to opt in; retrofits in M3 are additive (replace inline stance with a persona reference; preserve observable behaviour).

- **M1** lands the architect persona + minimal `drive-agent-personas` scaffold (just enough to resolve one persona ID). No existing skill is touched. If the M1 A/B test fails, the project halts here with no externalised behaviour to roll back.
- **M2** populates the library and completes the convention doc. Still nothing wired in — the library exists but is opt-in.
- **M3** retrofits three existing skills and decomposes `/drive-pr-local-review`. Each retrofit preserves observable behaviour (the persona doc *is* the inline stance, factored out and pointed at by ID). The decomposition replaces a multi-output skill with a composite that produces the same three artefacts.

There are no feature flags. The implicit gate is the load instruction in the skill body — present means "use this persona," absent means "use developer default" (which is what every skill does today).

### Milestone 1: Validate the assumption (kill-the-project gate)

This milestone exists to fail fast. If TC-1 fails, the entire project halts here — no second persona is authored, no skills are retrofitted, no `/drive-pr-local-review` is decomposed. The library shape and the convention are deferred until the construction is proven to work on the case it exists to solve.

### Milestone 2: Populate the library + document the convention

Author the remaining six persona docs and complete the `drive-agent-personas` SKILL.md so the convention is fully documented and the library is ready for skill authors to adopt. Library is opt-in — no existing skill is wired in until M3.

**Authoring constraint (carried from m1 R1 learnings).** Each persona's stance must describe the *lens bias* the persona produces, not the conclusions it guarantees. The m1 A/B test demonstrated that a persona shifts an executor's priorities and the bar at which a class of concern is dismissed; it does not turn the executor into the user during interactive review. Persona docs must avoid promising outcomes ("this persona will catch X") and instead describe what they elevate, what they treat as load-bearing, and what they out-of-scope. See `learnings.md § Personas raise the floor; they do not eliminate the post-implementation interactive-review pass`.

### Milestone 3: Retrofit existing skills + decompose `/drive-pr-local-review`

Wire existing skills to the library and prove multi-persona composition by decomposing the multi-output `/drive-pr-local-review` into atomic single-persona sub-skills behind a tech-lead-orchestrated composite.

**Tasks:**

- [ ] **Task 3.0 (added m3 R1 — user-driven scope extension to T3.1).** Extend `.agents/skills/drive-agent-personas/SKILL.md § Composite skills` to admit *two* composite shapes (the current convention only documents one):
  - **Shape A — Decomposition into atomic sub-skills.** The composite delegates to atomic single-persona sub-skills, in declared order, on declared artefacts. Used when the skill produces multiple distinct artefacts (each by a different lens), the sub-workflows are independent enough to be separable, and cross-pollination between lenses is not load-bearing. *Example post-M3*: `/drive-pr-local-review` produces three artefacts (system-design-review.md / code-review.md / walkthrough.md), each by its own atomic sub-skill.
  - **Shape B — In-skill lens transitions within a continuous workflow.** The composite stays as a single skill that adopts an orchestrator persona (typically `tech-lead`) and *transitions* between persona-loads at declared workflow boundaries within the skill body. Used when the workflow is interactive/conversational, cross-pollination between lenses is load-bearing (one lens's surfaced concern feeds into the next lens's evaluation), and splitting into sub-skills would lose the conversational continuity. *Example post-M3*: `drive-design-discussion` adopts `tech-lead` as orchestrator and sequences `architect` → `principal-engineer` (and is future-extensible to security / QA personas as those are admitted).

  Both shapes preserve the convention's load-bearing principles: every skill (composite or atomic) names its own orchestrator/lead persona explicitly; persona-loads are visible in the skill body at every transition; no silent persona inheritance. The shape choice is the skill author's call based on workflow shape — artefact-producing → A; conversation-continuous → B.

  Mechanic for Shape B: the composite skill's body contains explicit persona-load instructions at workflow boundaries (e.g. *"Begin by adopting the `architect` persona ... When architect-class concerns are settled, adopt the `principal-engineer` persona ..."*). The orchestrator persona (tech-lead) is the meta-frame holding the workflow together; it appears explicitly at synthesis points (start, transitions, end). Each transition is its own load instruction; the agent re-loads at each boundary. Document an explicit "When to choose A vs B" subsection so future skill authors can pick correctly. (No new TC; this convention extension is verified by the post-T3.1 cold-read of the amended SKILL.md and by T3.1 / T3.4 each being instances of the documented shape.)

At synthesis (end of discussion), reload `tech-lead` explicitly to summarise the multi-lens decisions made and note any unresolved tensions for the user to choose. Replace the inline "Core directive" stance paragraph with the explicit persona-load sequence (architect first, PE second, tech-lead at boundaries). The skill is **future-extensible**: as security / QA personas are admitted in v2+, additional persona-load steps slot into the workflow at appropriate points; the user's stated intent is *"I could imagine this growing later to include the security persona, testing/qa persona, etc."* Confirm both architect and PE persona docs carry the relevant stance content; if not, reconcile in the persona docs (any architect-class concerns currently inlined in `drive-design-discussion` that aren't already in `personas/architect.md` get added there; any PE-class concerns that aren't in `personas/principal-engineer.md` get added there). (Satisfies TC-7, partial TC-10.)

## Plan amendments after m1 R1

Three observations surfaced during the m1 R1 reviewer pass; recorded here for the historical record. None re-opens m1 (which is SATISFIED) — they inform downstream milestones and any future re-trial.

7. **TC-1 expected-outcome wording was a-priori.** Plan TC-1 row reads *"the unframed run does not"* surface the typology concern. Actual m1 result: the unframed run engages with naming/typology at lower priority and dismisses the *load-bearing* concern; the persona effect is verdict-shifting and dismissal-resistance, not a clean catches-it/misses-it switch. AC-5 is satisfied under its *"or equivalent"* clause, but if anyone re-runs TC-1 in future they should expect a more nuanced shift. Bias the re-evaluation toward *what the framed run elevates and the unframed run dismisses on identical evidence*, not *what the unframed run misses entirely*.

8. **TC-1 source-pinning convention.** Both initial sub-subagent runs read the spec at HEAD via the workspace path and got the post-F6 retraction. The clean-baseline corrective is to read every contextual document via `git show <sha>:<path>` so the inputs reflect the world *at the commit*, not at HEAD. Pin this for any future commit-based A/B test in this repo. See `learnings.md § Pin source-material provenance to the commit, not to HEAD`.

9. **Personas raise the floor, not the ceiling.** The m1 A/B test demonstrated a verdict shift but did not retrace the F6 inference in a single pass. Implication carried into M2 (personas describe lens bias, not guaranteed conclusions — see the Authoring constraint above) and M3 (composite review skills should keep the human in the loop, not flatten the post-implementation interactive review pass into "the architect persona reviewed it, you're done"). See `learnings.md § Personas raise the floor; they do not eliminate the post-implementation interactive-review pass`.
