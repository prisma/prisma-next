# Slice: substrate (S1.A)

_Parent project: [`projects/contract-ir-planes/`](../../). This slice delivers the foundation every other contract-ir-planes slice consumes ŌĆö see plan ┬¦ S1.A._

## At a glance

Add the framework-level type primitives the rest of the project consumes ŌĆö a `domain` plane on the framework `Contract` type, a narrowed `Namespace` interface, family-shaped namespace types carrying the family-specific slots, and a polymorphic free-function `elementCoordinates(storage)` walk that iterates each namespace's own-enumerable entity-bearing properties structurally (no family-name lookup table) and yields `(plane, namespaceId, entityKind, entityName)` tuples ŌĆö and extend `AuthoringContributions.entityTypes` so target packs can contribute hydration and validator-schema fragments for a new entity kind through a single descriptor, keyed by the kind's existing `discriminator`. Postgres pack contributes `hydrate` + `validatorSchema` for its `'postgres-enum'` kind as proof the wiring works end-to-end. No on-disk contract changes; enum entries continue to flow through the existing framework-shared `storage.<ns>.types` slot (S1.B migrates them to `storage.<ns>.enum` ŌĆö slot name per the *essence + singular* convention recorded in ADR Decision 5).

> **Vocabulary note.** Earlier drafts used the term *"slot key"* to name the property on a namespace object where a kind's entries live (e.g. `tables`, `types`, `collections`). The codebase already names this concept ŌĆö it is the **entity kind**, the same string used as the kind's `discriminator`. The two-name confusion produced a redundant descriptor field (`storageSlotKey?`) and a framework-level family-aware lookup table (`SLOT_KEYS_BY_NAMESPACE_KIND`) that both shipped in D1 + D2 and are removed by D3. The end-state spec below is the cleaned-up shape; the dispatch history is preserved in the slice plan for audit.

## Scope

### In scope

**Framework type primitives** (`packages/1-framework/`):

- `Contract` type ([`packages/1-framework/0-foundation/contract/src/contract-types.ts`](../../../../packages/1-framework/0-foundation/contract/src/contract-types.ts)) gains an optional `domain` field at the root, typed as `Record<NamespaceId, Record<EntityKind, Record<EntityName, unknown>>>` (precise typing layered in by family packs). Existing flat `models` / `valueObjects` siblings stay ŌĆö populating `domain` is S1.C's job; this slice only adds the type.
- `Namespace` interface ([`packages/1-framework/1-core/framework-components/src/ir/namespace.ts`](../../../../packages/1-framework/1-core/framework-components/src/ir/namespace.ts)) narrows to required `{ id: string; kind: string }`. `kind` is promoted from optional on `IRNode` to required on `Namespace`. `NamespaceBase` requires concrete classes to declare `kind` (today they set it via non-enumerable `Object.defineProperty`; that pattern is preserved ŌĆö only the interface shape changes).
- `Storage` interface ([`packages/1-framework/1-core/framework-components/src/ir/storage.ts`](../../../../packages/1-framework/1-core/framework-components/src/ir/storage.ts)) stays unchanged. A new free function `elementCoordinates(storage): Generator<EntityCoordinate>` is exported from the same module. The walk iterates each namespace's **own-enumerable properties** structurally ŌĆö for every property whose value is a non-null entity-bearing map, the walk yields one `(namespaceId, entityKind, entityName)` tuple per entry, where `entityKind` is the property name. `id` is skipped as a known scalar; `kind` is non-enumerable per the IR-class pattern and auto-skips. No framework-level knowledge of family slot vocabularies; the walk is family-agnostic and open to extension ŌĆö a future namespace concretion that declares a new entity-bearing property gets walked automatically. `EntityCoordinate` is a new exported type `{ namespaceId: string; entityKind: string; entityName: string }`.
- New `EntityCoordinate` type co-located with `Storage` (settled per spec OQ1 working position).

> **Why a free function, not a method on `Storage`.** Adding the walk as a required member of the `Storage` interface (the original R1 design) broke structural assignability of emitted `contract.d.ts` storage literals against `Contract<SqlStorage>` consumers: the printed literal carries `storageHash`/`namespaces`/`types?` but no method members, so promoting the interface broke every fixture's structural match. The free function consumes any `Storage`-shaped value and leaves the interface ŌĆö and every emitted artefact's structural conformance ŌĆö untouched. See the slice plan's R2 redirect note and the D1 dispatch brief for the full rationale.

> **Why structural property iteration, not a family-keyed lookup.** The walk's dispatch needs no up-front knowledge of which entity kinds each namespace kind holds. Contract hydration (family-specific) has already enforced the structural shape by the time the walk runs; the walk just enumerates entity-bearing properties. The family-keyed lookup table that shipped in D1 + D2 (`SLOT_KEYS_BY_NAMESPACE_KIND`) hardcoded `'sql-namespace'` and `'mongo-namespace'` into the framework layer ŌĆö a layering violation ŌĆö and silently failed for Postgres contracts whose namespace `kind` is `'schema'` (the namespace-promotion path in `PostgresContractSerializer`), not `'sql-namespace'`. Structural iteration removes the layering violation, removes the lookup-table maintenance burden, fixes the Postgres breakage, and makes future namespace concretions walk for free. The invariant the structural walk relies on: namespace concretions carry `id` + (non-enumerable) `kind` + entity-kind slot maps, and nothing else ŌĆö declared in the `Namespace` interface JSDoc.

**Family-shaped namespace types** (move family-specific slots off the framework interface):

- `SqlNamespace` type alias ([`packages/2-sql/1-core/contract/src/ir/sql-storage.ts`](../../../../packages/2-sql/1-core/contract/src/ir/sql-storage.ts)) was already `Namespace & { tables, types? }`; nothing structural changes here, but the typing now reflects the narrowed framework `Namespace`. `MongoNamespace` ([`packages/2-mongo-family/1-foundation/mongo-contract/src/ir/mongo-storage.ts`](../../../../packages/2-mongo-family/1-foundation/mongo-contract/src/ir/mongo-storage.ts)) similarly unchanged structurally ŌĆö the framework narrowing makes the family extension explicit.
- The framework canonicalizer's `TOP_LEVEL_ORDER` ([`packages/1-framework/0-foundation/contract/src/canonicalization.ts`](../../../../packages/1-framework/0-foundation/contract/src/canonicalization.ts) L17ŌĆōL31) gains `domain` as a top-level key. The SQL-specific `storage.namespaces.*` path checks (L70ŌĆōL167, L231ŌĆōL273) stay unchanged in this slice ŌĆö they're S1.B's problem (when the slot shape moves) and S1.D's problem (when the family-contribution hook lands).

**Descriptor mechanism** (`AuthoringContributions.entityTypes` extension):

- [`packages/1-framework/1-core/framework-components/src/shared/framework-authoring.ts`](../../../../packages/1-framework/1-core/framework-components/src/shared/framework-authoring.ts) ŌĆö extend `AuthoringEntityTypeDescriptor<Input, Output>` to carry **two** new optional fields alongside the existing `kind`, `discriminator`, `args`, `output`:
  - `hydrate?: (raw: unknown) => Output` ŌĆö serializer hydration factory the framework calls generically for any entry whose envelope `kind` matches the descriptor's `discriminator`. Omitted means the kind is authoring-only.
  - `validatorSchema?: ArkSchema` ŌĆö schema fragment the family validator composes into its namespace-entry schema. Mongo validator's `MongoNamespaceEnvelopeSchema` and SQL validator's `NamespaceEntrySchema` learn to compose contributed fragments at startup, keyed by `discriminator`.

  The descriptor's existing `discriminator` field (the kind's `kind:` literal in the IR envelope, e.g. `'postgres-enum'`) is the single coordinate the hydration registry and the validator-composition surface key on. **No separate `storageSlotKey` field** ŌĆö the property name on the namespace envelope where entries land is purely a hydration/persistence concern; the family hydration knows its built-in slots (`tables` for SQL, `collections` for Mongo) and treats every other entity-bearing property structurally, dispatching each entry through the kind-keyed registry.
- Postgres pack ([`packages/3-targets/3-targets/postgres/src/core/authoring.ts`](../../../../packages/3-targets/3-targets/postgres/src/core/authoring.ts)) updates its existing `enum` entry-type descriptor to populate the two new fields: `hydrate: (raw) => new PostgresEnumType(raw)`, `validatorSchema: PostgresEnumTypeSchema` (re-exported from the SQL validator's existing hardcoded schema for now ŌĆö full extraction is S1.B's job). The `discriminator: 'postgres-enum'` (already present) is what the family base serializer and validator both look up the contributions by.

**Generic dispatch through the descriptor registry**:

- SQL family base serializer ([`packages/2-sql/9-family/src/core/ir/sql-contract-serializer-base.ts`](../../../../packages/2-sql/9-family/src/core/ir/sql-contract-serializer-base.ts)) already accepts an `entityTypeRegistry: Map<kind, factory>` for entry hydration keyed by the entry's `kind` discriminator. The descriptor's `hydrate?` callback registers into that same map; **no parallel slot-keyed registry**. The family base's namespace hydration walks the namespace envelope's properties: the family-built-in slot (`tables`) hydrates through the family's hardcoded `StorageTable` construction; every other entity-bearing property iterates its entries and looks each entry up through `entityTypeRegistry.get(entry.kind)`. The Postgres pack's existing ad-hoc `hydrateSqlNamespaceEntry` enum branch ([`packages/3-targets/3-targets/postgres/src/core/postgres-contract-serializer.ts`](../../../../packages/3-targets/3-targets/postgres/src/core/postgres-contract-serializer.ts)) is reachable through this single registry by the end of the slice ŌĆö but is NOT yet the only path; the existing enum-in-`types`-slot path continues to work because no on-disk contracts move.
- Family validator schema composition: SQL and Mongo validators learn to fold contributed `validatorSchema` fragments into their per-namespace entry schemas at framework boot, keyed by `discriminator`. For SQL, today's hardcoded `'types?': type({ '[string]': PostgresEnumTypeSchema })` ([`packages/2-sql/1-core/contract/src/validators.ts`](../../../../packages/2-sql/1-core/contract/src/validators.ts) L168) is preserved as-is in this slice ŌĆö the contributed-fragment composition exists alongside it as a no-op for SQL (the contributed Postgres-enum fragment validates the same shape; the redundancy is the F1 cure). S1.B is where SQL drops the hardcoded `types?` enum slot.
- **No collision validator** ŌĆö the family hardcodes its built-in slot names (`tables` / `collections`) in its hydration path; pack-contributed entries land in other property slots whose names are family-pack-owned at the namespace concretion level (not framework-attributed). Pack-vs-pack slot collisions, if they later become possible, are caught at namespace-concretion construction time, not at descriptor registration.

## Approach

Four dispatches, sequential within this slice. **D1 + D2 shipped under an earlier framing** that introduced an `storageSlotKey` field on the descriptor, a parallel namespace-slot hydration registry on the family serializer base, a `FamilyDescriptor.reservedStorageSlotKeys` field, a slot-key collision validator, and a family-name-keyed lookup table inside `elementCoordinates`. The user surfaced two compounding issues during slice-PR open: (1) the "slot key" vocabulary is redundant with the existing `entityKind` / `discriminator` concept; (2) the walk doesn't need any up-front knowledge of slot names because contract hydration has already enforced the structural shape ŌĆö the walk can iterate the namespace's own-enumerable entity-bearing properties directly. **D3 is the vocabulary + structural-walk cleanup dispatch** that landed those corrections. **D4 is the artefact-review cleanup dispatch** that lands the substrate-level outputs from the architect + principal-engineer artefact reviews ŌĆö the `EntityCoordinate` gains a `plane: 'domain' | 'storage'` axis (A03; settled now while no consumers exist), and the planning-artefact text edits propagate (A05 / A07 / A08 / F09). By the end of the slice the substrate is in place in its cleaned-up shape, the coordinate is honest about what plane it addresses, and the planning artefacts at every altitude tell a single story. No on-disk contract has shifted; the next slice (S1.B) is what consumes the new mechanism for real.

> **Deferred to standalone tickets (`relatedTo: ['TML-2584']`).** [TML-2634](https://linear.app/prisma-company/issue/TML-2634) (A01) ŌĆö rename existing slot keys (`tables` ŌåÆ `table`, `types` ŌåÆ `type`, `collections` ŌåÆ `collection`) into the same essence + singular convention pack contributions follow from S1.B onward. [TML-2636](https://linear.app/prisma-company/issue/TML-2636) (A02(b)) ŌĆö namespace concretions lift entity maps under `.entries` so the structural-walk invariant becomes invariant-by-construction (current invariant is JSDoc + test). Both touch ~150 consumer call sites + ~50ŌĆō80 fixture regens; the cost only pays out when there's downstream leverage. The cleanup convention pack contributions follow is locked in D4 so future pack contributions don't add to the eventual migration size.

### Dispatch 1 ŌĆö Framework primitives _(shipped; one residual change for D3)_

Type-level additions on the framework `Contract`, `Namespace`, and `Storage` interfaces. The interface-narrowing of `Namespace` (`kind` becomes required) is the only behavioural surface this dispatch touched ŌĆö concrete classes already declare `kind`, so the change was mechanical at the type level but exposed any consumer treating `kind` as optional. As shipped:

1. Added `EntityCoordinate` type co-located with `Storage` in `framework-components/ir/storage.ts`. Exported a free `elementCoordinates(storage): Generator<EntityCoordinate>` function from the same module. Initially dispatched on `Namespace.kind` via an inline `Map<namespaceKind, ReadonlyArray<{ slotKey, entityKind }>>` lookup table. **D3 replaces this with the structural property walk** (see ┬¦ Dispatch 3 below); the table goes away entirely.
2. Promoted `kind` from optional on `IRNode` to required on `Namespace`. Updated `NamespaceBase` to require concrete declaration. The existing non-enumerable `Object.defineProperty(this, 'kind', { value: 'ŌĆ”', enumerable: false })` pattern stays in concrete classes. **No D3 change here.**
3. Added `domain` to the framework `Contract` type as an optional field with the shape `Record<string, Record<string, Record<string, unknown>>>`. The framework canonicalizer's `TOP_LEVEL_ORDER` learned the new key. No content is populated. **No D3 change here.**

### Dispatch 2 ŌĆö Descriptor mechanism + Postgres registration _(shipped; residual changes for D3)_

Extended the descriptor; wired dispatch generically; registered Postgres enum through it. As shipped:

1. Extended `AuthoringEntityTypeDescriptor` with `storageSlotKey?`, `hydrate?`, `validatorSchema?`. **D3 drops `storageSlotKey?`** (redundant with `discriminator`); `hydrate?` and `validatorSchema?` stay.
2. Added a parallel `namespaceSlotHydrationRegistry: Map<storageSlotKey, hydrate>` on `SqlContractSerializerBase` and its Mongo sibling, alongside the existing `entityTypeRegistry: Map<kind, factory>`. Added a `FamilyDescriptor.reservedStorageSlotKeys` field and a collision validator. **D3 collapses the parallel registry back into `entityTypeRegistry` keyed by `discriminator`**; drops `reservedStorageSlotKeys`; drops the collision validator.
3. SQL and Mongo family validators learned to compose `validatorSchema` fragments into their per-namespace entry schemas at boot. **D3 re-keys this composition surface on `discriminator`** (was keyed on `storageSlotKey`); the SQL validator's hardcoded `'types?': type({ '[string]': PostgresEnumTypeSchema })` block stays for now ŌĆö the composition surface exists alongside it as a no-op until S1.B.
4. Postgres pack updated its `entityTypes.enum` descriptor with `storageSlotKey: 'types'` + `hydrate` + `validatorSchema`. **D3 drops `storageSlotKey: 'types'`**; the descriptor's `discriminator: 'postgres-enum'` is what the family base looks up the hydrator under. The `hydrate` callback's body is unchanged; the call-site in `postgres-contract-serializer.ts` delegates by looking up `entityTypeRegistry.get(entry.kind)`.

### Dispatch 3 ŌĆö Vocabulary + structural-walk cleanup

Land the corrections from the user's slice-PR-open pushback. Implementation order:

1. **Framework walk.** Replace `SLOT_KEYS_BY_NAMESPACE_KIND` + `kind`-dispatch in `elementCoordinates(storage)` with a structural property walk over each namespace's own-enumerable properties. Skip `id` (known scalar); `kind` is non-enumerable per the IR-class pattern and auto-skips. For every other own-enumerable property whose value is a non-null object, iterate `Object.keys(slot)` and yield `{ namespaceId, entityKind: propertyName, entityName: key }`. Remove the lookup table and the unrecognised-kind throw. Update the `Namespace` interface JSDoc to declare the invariant the walk relies on.
2. **Descriptor surface.** Drop `storageSlotKey?` from `AuthoringEntityTypeDescriptor`. Update the JSDoc on `hydrate?` and `validatorSchema?` to spell out that the descriptor's `discriminator` is the registry key the family base looks up under.
3. **Family base serializers.** Remove the parallel `namespaceSlotHydrationRegistry` constructor parameter, field, and `hydrateNamespaceSlot` helper from both `SqlContractSerializerBase` and `MongoContractSerializerBase`. The existing `entityTypeRegistry` keyed by `kind` is the single registry. Update `hydrateSqlNamespaceEntry` to iterate the raw namespace envelope's properties: `tables` is hydrated by the family hardcode (unchanged); for every other entity-bearing property, iterate its entries and dispatch each through `entityTypeRegistry.get(entry.kind)`. (Mongo symmetric for `collections`.)
4. **Family descriptor + control stack.** Drop `reservedStorageSlotKeys?: readonly string[]` from `FamilyDescriptor` (and from `ControlStack` thread-through). Drop the `reservedStorageSlotKeys: ['tables']` / `['collections']` declarations on the SQL and Mongo family-descriptor sites.
5. **Collision validator.** Delete the `storageSlotKey`-collision validator added in D2 step 8 and its unit test.
6. **Postgres registration.** Drop `storageSlotKey: 'types'` from `postgresAuthoringEntityTypes.enum`. Update the Postgres contract-serializer's enum-branch delegation: where it previously looked up `namespaceSlotHydrationRegistry.get('types')`, it now lets the family base's per-property iteration dispatch through `entityTypeRegistry.get('postgres-enum')`.
7. **Validator-composition keying.** Update `createSqlContractSchema` (and the Mongo equivalent) to key contributed fragments by `discriminator` instead of `storageSlotKey`. The hardcoded `'types?'` SQL slot stays untouched (additive coexistence; F1 cure).
8. **Tests.** Adjust the registry-collision and pack-contribution unit tests (added in D2) to exercise the discriminator-keyed surface. Drop the `'tables'`/`'collections'` rejection assertions; replace with one assertion that pack-contributed descriptors register through the `entityTypeRegistry` keyed by `discriminator`.

After D3 the substrate is in its cleaned-up end-state: structural walk, single kind-keyed registry, descriptor surface that's purely `discriminator` + `hydrate?` + `validatorSchema?` for the pack-contribution path. Fixtures byte-stable throughout.

## Open Questions

Settled in-flight; recorded here for audit.

1. **Where does `EntityCoordinate` live?** Same file as `Storage` interface, or sibling `entity-coordinate.ts`. **Settled D1**: same file as `Storage` ŌĆö they're conceptually coupled; the file is small enough.
2. **Descriptor-registry shape for pack-contributed slots.** ~~Working position: parallel Map keyed by `storageSlotKey`.~~ **Re-settled D3**: single registry keyed by `discriminator`; the parallel slot-keyed map and the `storageSlotKey` descriptor field are both retired. The realization: hydration only needs to know how to construct an entry from its raw envelope, and the entry's `kind` discriminator is sufficient to look up the right constructor. The slot key on the namespace envelope is a per-property iteration concern that the family base handles structurally.
3. **`validatorSchema` composition order.** When multiple packs contribute to the same family's namespace schema, what's the merge order? **Settled D3**: registration order, keyed by `discriminator`. Pack-vs-pack collisions on `discriminator` are caught by the existing `entityTypeRegistry` collision handling (the registry rejects duplicate discriminators). No separate slot-name collision validator needed.
4. **`elementCoordinates(storage)` over the SQL `types?` slot.** Should the walk yield `(ns, 'types', name)` entries for the framework-shared `types` slot? **Settled D3** (re-settled from D1): yes, by structural consequence ŌĆö the walk iterates every entity-bearing own-enumerable property; `types` qualifies if present. S1.B will remove the property when enum entries migrate; the walk naturally stops emitting `types` entries because the property no longer exists.
