# Slice `e2e-demonstration` — Dispatch plan

**Slice spec:** [`spec.md`](spec.md) · **Linear:** TML-2796 (closes TML-2792 alongside)

Four sequential dispatches. Dispatch 1 wires the planner-warning channel through framework + SQL family + CLI so a control-policy suppression becomes user-visible — the substrate that makes Dispatch 2's Postgres fifth-scenario assertion possible. Dispatches 2 and 3 author the per-family CLI integration tests that demonstrate every applicable policy through the public CLI for Postgres and Mongo respectively. Dispatch 4 corrects the architectural shape Dispatch 1 chose (post-filtering the planner's output) to input-side filtering — required so `db update` succeeds when an `external` subject is in a state the SQL diff engine cannot model — and rolls in the PR #711 review hygiene items.

### Dispatch 1: warnings channel — `MigrationPlannerSuccessResult.warnings` → SQL planner population → `db update` CLI surfacing

- **Outcome:** A successful migration plan that suppressed calls under control-policy can carry those suppressions out as warnings, end-to-end through the public CLI: `MigrationPlannerSuccessResult` carries optional `warnings: readonly MigrationPlannerConflict[]`; the SQL family planner populates it with `SqlPlannerConflictKind: 'controlPolicySuppressedCall'` entries (one per call dropped by `filterCallsByControlPolicy`, locating the offending `{ namespace, table, controlPolicy }`); the `db update` CLI command (both apply and `--dry-run`) prints a `Warnings:` block under its existing summary when the plan carries any. The change is additive — `warnings` omitted when empty, and existing tests stay green without reflecting the new field.
- **Builds on:** The spec's chosen design (additive optional field; one new conflict kind; CLI summary extension).
- **Hands to:** A user-visible diagnostic surface for control-policy suppressions through `db update` — the channel Dispatch 2 asserts against.
- **Focus:** In — the framework planner-result type extension, SQL family planner population, `controlPolicySuppressedCall` kind + location shape, CLI summary print, unit-level tests for each layer (framework planner-result shape; SQL family populates warnings on suppressed calls; CLI prints warnings block when present). Out — the integration tests (Dispatches 2–3), Mongo (no Mongo planner-side control gating exists to surface), and any builder-DSL changes (handed off to Dispatches 2/3 if forced by the fixture).

### Dispatch 2: Postgres CLI integration test — `cli.control-policy.postgres.e2e.test.ts` exercising all five Postgres scenarios

- **Outcome:** A new PGlite-backed CLI integration test under `test/integration/test/cli.control-policy.postgres.e2e.test.ts` (fixture under `test/integration/test/fixtures/control-policy/postgres/`) is green and asserts the spec's five Postgres scenarios end-to-end: `managed` table created/altered then a drift causes verifier fail; `tolerated` table created-if-missing with extra columns preserved across reruns and verifier passing on extras / failing on declared-column drops; `external` table receives zero DDL with verifier passing on shape match and failing on declared drift; `observed` table receives zero DDL with verifier emitting warning-only on mismatch; `defaultControlPolicy: 'external'` namespace with a per-object `managed` mis-declaration produces zero DDL into the namespace AND surfaces the suppression warning in `db update --dry-run` and `db update` apply CLI output. The test is pinned by the existing `Integration Tests` CI gate.
- **Builds on:** Dispatch 1's user-visible `db update` warnings surface (the Postgres fifth scenario asserts against it).
- **Hands to:** Postgres-side closure of the project-DoD conditions *"all four policies behave end-to-end (verify + plan) for the SQL family"* and *"the conflict diagnostic surfaces"*.
- **Focus:** In — the Postgres fixture (one TS file authoring all five scenarios through `typescriptContract`), the integration test driving `runDbInit` / `runDbUpdate` / `db verify` and asserting CLI output + DB state via `withClient`, and any narrowly-scoped SQL builder-DSL touch-ups the fixture forces. Out — Mongo coverage (Dispatch 3), an example-app touchpoint, the PSL surface (slice 6), per-column overrides (project non-goal). Halt and surface if the dispatch grows new verifier or planner *behaviour* — those belong in slices 2/3 and were already merged.

### Dispatch 4: architectural correction — input-side control-policy filtering + PR #711 review hygiene

- **Outcome:** The SQL family planner pipeline partitions `external` and `observed` subjects out of the planner's _input_ before any DDL is generated, rather than filtering them from its _output_; warnings for those subjects are constructed directly from the suppressed partition. `tolerated` keeps a planner entry but follows a create-if-absent short-circuit that does not require diffing the existing object's full state. The Postgres e2e test gains a sixth scenario in which an `external` table is pre-seeded into a state the SQL diff engine cannot model (e.g. an extension-provided column type or an exotic constraint) — `db update` succeeds, the planner does not observe the subject at all, and the suppression warning is emitted. The five PR #711 review hygiene items land in the same dispatch (they touch the same files): drop `isPlannerWarningList` for a justified `blindCast` at the read site; replace the bare `as unknown as MongoContractResult<Definition>` in the Mongo `defineContract` return with `blindCast<MongoContractResult<Definition>, "...">`; rename `formatTargetRef`/`defaultTargetRef` and `formatPostgresControlPolicyTargetRef` to use the `subject`/`Label` vocabulary; replace the conditional-spread idiom in `buildSuppressionWarning`'s location/meta construction with the repo's `ifDefined` helper.
- **Builds on:** Dispatch 1's framework planner-result type + warning kind (unchanged); Dispatch 2's Postgres test scaffolding (extended with one fixture and one scenario); the operator review on PR #711 that surfaced the un-plannable-external failure mode and the five hygiene items.
- **Hands to:** A correct end-to-end demonstration: the project DoD's "all four policies behave end-to-end" condition can no longer be invalidated by an un-plannable external schema, and the slice's user-visible surface is free of the cast / naming / predicate smells the operator flagged.
- **Focus:** In — restructuring the planner pipeline in `packages/2-sql/9-family/src/core/migrations/` and `packages/3-targets/3-targets/postgres/src/core/migrations/` so external/observed subjects never enter planning; building warnings from the suppressed partition rather than from dropped planner calls; the `tolerated` create-if-absent short-circuit; the un-plannable-external Postgres fixture + scenario; the five hygiene-item edits. Out — Mongo coverage of un-plannable state (Mongo has no planner pipeline to fix and its verifier already tolerates extras by design); any new verifier or planner behaviours; the PSL surface (slice 6); per-column overrides. Halt and surface if the dispatch finds that the framework's planner-result shape needs to change (the `warnings` field is already the right shape — only its population path moves).
