# Slice `psl-authoring` — Dispatch plan

**Slice spec:** [`spec.md`](spec.md) · **Linear:** TML-2779

Two sequential dispatches. Dispatch 1 closes the PSL per-object surface gap (`@@control(<policy>)`) — the headline of the slice and the surface a PSL-only author needs first. Dispatch 2 closes the contract-default gap by adding `defaultControlPolicy` to the contract specifiers in `prisma-next.config.ts`, achieving full PSL/TS authoring parity for both the per-object override and the contract default. The dispatches are independent at the code level (PSL surface and specifier surface touch disjoint files), but a sequential order is natural because the spec's headline is PSL parity and the specifier work is the clean-up that closes the contract-default gap.

### Dispatch 1: PSL `@@control(<policy>)` model-level attribute

- **Outcome:** A PSL document carrying `@@control(<policy>)` on a `model` block lowers to `storage.namespaces[<ns>].tables[<tableName>].controlPolicy = '<policy>'` in the SQL family's PSL interpreter, with the five validation diagnostics from the slice spec firing on malformed authoring (no argument, multiple arguments, unknown policy, duplicate `@@control`, named arg). A new round-trip parity test asserts that PSL → IR → JSON → IR equality holds for `tolerated`, `external`, and `observed` policies (the `managed` case is the default-behaviour baseline already covered by the substrate slice's coverage). No parser changes are required: `@@control(external)` reuses the existing model-attribute grammar that already accepts bare-identifier args (`@relation(onDelete: Cascade)` is the precedent).
- **Builds on:** The spec's chosen design (single parameterised attribute; lowercase argument; lowering to the existing `controlPolicy` IR slot from slice 1).
- **Hands to:** A PSL-only author who can set per-object control policy on any model. The contract-level default is still `managed` for PSL contracts that don't author a TS-side override; that gap is the dispatch-2 entry point.
- **Focus:** In — the new `@@control` branch in `packages/2-sql/2-authoring/contract-psl/src/interpreter.ts` (placed alongside the existing `@@unique` / `@@index` walk at `interpreter.ts:858+`); the five validation diagnostics; the round-trip parity test alongside the existing PSL interpreter tests; a brief authoring-surface README entry. Out — the contract-default specifier work (Dispatch 2), Mongo PSL (no Mongo PSL exists — Mongo authoring stays TS-only per the spec), per-namespace inheritance (project non-goal), target-only-kind PSL spellings (no surface to extend yet).

### Dispatch 2: `defaultControlPolicy` on contract specifiers — PSL/TS parity at the `prisma-next.config.ts` boundary

- **Outcome:** The four contract specifiers — `prismaContract` (SQL PSL, in `packages/2-sql/2-authoring/contract-psl/src/provider.ts`), `typescriptContract` and `typescriptContractFromPath` (SQL TS, in `packages/2-sql/2-authoring/contract-ts/src/config-types.ts`), `emptyContract` (SQL, same file), and the parallel Mongo specifiers in `packages/2-mongo-family/2-authoring/contract-ts/src/config-types.ts` — accept an optional `defaultControlPolicy: ControlPolicy`. The PSL specifier and `emptyContract` extend their existing options bag with the new field; the two TS specifier functions grow a third positional argument shaped as an options bag (`typescriptContract(contract, output?, options?: { defaultControlPolicy? })`) so existing two-argument call sites under `packages/3-extensions/*/prisma-next.config.ts` and `test/integration/test/fixtures/cli/**` keep working unchanged. Each specifier's `load(...)` callback applies the source-wins precedence rule: when the loaded contract's `defaultControlPolicy` is `undefined`, the specifier writes its value in; when the contract already carries a `defaultControlPolicy`, the specifier value is silently overridden. Unit tests for each specifier pin the precedence rule; one fixture asserts that a PSL-authored contract loaded through `prismaContract(_, { defaultControlPolicy: 'external' })` flows the policy through to the verifier and planner using the existing slice 5 e2e harness shape (no new e2e test — the unit-level fixture and serialiser round-trip are enough; slice 5 already exercises the policy semantics end-to-end).
- **Builds on:** Dispatch 1's PSL surface for per-object policy is unrelated and not a dependency; this dispatch builds only on the spec's chosen specifier-arg shape and TML-2800's source-level field on the IR.
- **Hands to:** PSL/TS authoring parity for the contract default. The project-DoD condition *"the PSL authoring surface ships, **or** is explicitly deferred"* closes; the project's last open authoring-surface gap is closed.
- **Focus:** In — adding `defaultControlPolicy?: ControlPolicy` to `PrismaContractOptions`; adding the third positional options bag to the two TS specifier functions; analogous addition to `emptyContract`'s options bag; analogous addition to the Mongo specifiers; the source-wins precedence rule applied inside each specifier's `load(...)` callback; unit tests pinning the precedence rule per specifier; one fixture demonstrating PSL → loaded contract carries the specifier-set default; brief config-reference README entries for the new option. Out — anything in Dispatch 1's scope; refactoring the TS specifier signatures from positional `output` to options-bag-as-second-arg (deferred — the third-positional shape preserves backwards compatibility); validating the policy literal at config-load time (the TS type enforces it; per the spec's open question 2); a new e2e integration test (slice 5's existing tests cover the policy semantics end-to-end through public CLI; this dispatch only proves the new authoring path delivers the same loaded contract).
