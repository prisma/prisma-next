## The bar you enforce
A faithful port reproduces the SAME upstream test: **the same database schema**, logically the same query, and **the SAME assertions — every one, runtime AND type-level.**

**The DB schema MUST match the original test (first-class, non-negotiable).** Before looking at any query, diff the port's fixture `prisma/functional/<suite>/_fixture/contract.prisma` against the upstream `prisma/_schema.ts` (postgres branch — evaluate the templated string for the postgres provider) **model-by-model, field-by-field**, and confirm EVERY one is preserved: every model/table (`@@map`); every field, its scalar type AND native type (`@db.*` — e.g. `@db.Decimal(p,s)`, `@db.VarChar(n)`, `@db.ObjectId`), nullability (`?`), list-ness (`[]`); `@id`/compound `@@id`; `@unique`/`@@unique` (including multi-column and named); `@default(...)` (incl. `now()`, `autoincrement()`, `dbgenerated`); `@map`; every relation `@relation(fields/references/onDelete/onUpdate)` and its cardinality; `@@index`; enums and their members. **Any model/field/type/constraint/relation that is dropped, weakened, or altered to dodge a prisma-next gap is a SCHEMA-SIMPLIFICATION violation → reject** (the port then belongs in `non-ported`/`it.fails`, not a green built on a simplified schema). The ONLY permitted deltas are prisma-next authoring equivalences that preserve the same storage shape (e.g. `String @id @default(cuid())` → `String @id` when ids are seeded explicitly and the column stays `text`; a `types { X = T @db.Foo(...) }` named type carrying a native attribute) — and each such delta must be justified in your report, not assumed. If you cannot obtain the exact postgres-branch schema, say so; do not pass the fixture on faith.

For EACH ported test, list the upstream assertions (quote them with `tests.ts:LINE`) and map each to the port's assertion (`<file>.test.ts:LINE`). This mapping is how you prove nothing was dropped or weakened. Flag any of these as a VIOLATION with its category:
- **DROPPED-TYPE-ASSERTION** — upstream has `expectTypeOf`/`@ts-expect-error` and the port omits it or replaces it with a runtime `typeof`/`instanceof` check. (Type assertions belong INLINE in the same `it()`, not a separate `.test-d.ts`, not dropped.)
- **DROPPED/WEAKENED-RUNTIME-ASSERTION** — port asserts less than upstream (only "does not throw"; drops an ordering/count/exact-value; asserts a subset).
- **FEATURE-SUBSTITUTION** — the mechanism/input under test was swapped for a different supported one (raw SQL → ORM; Decimal.js input → string; atomic increment → read-modify-write; nested mutation → manual join rows; inclusive cursor → exclusive-with-changed-expectations; **`select:{_count:true}` all-relations shorthand → `include(rel => rel.count())`** — the count-EVERY-relation shorthand has no prisma-next surface (a *specific* relation count IS correctly ported via include-count, so only the `_count:true` shorthand is the violation); **`startsWith`/`endsWith`/`contains`/`mode:'insensitive'` → `.like()`/`.ilike()`** — different operators; **`select:undefined`/`include:undefined`/`Prisma.skip` → omitting the argument** — omission tests a different thing).
- **IN-MEMORY-POSTPROCESSING** — the port reproduces a query-level `orderBy`/`where`/aggregate/groupBy with JS (`result.sort()`, `.filter()`, a hand-rolled sum/count) instead of making the DB do it. (Mirroring an in-JS step upstream itself performs is fine; substituting a DB op is a violation → `test.fails`/`non-ported`.)
- **WRONG-SHAPE-TRANSLATION** — an allowed API-shape mapping done wrong. Note in particular: `findUnique` must map to `.first(pk)` / `.where(pk).first()` (NOT `.all()` + index), and `findUniqueOrThrow`/`findFirstOrThrow` to `.firstOrThrow()`.
- **SCHEMA-SIMPLIFICATION** — fixture drops/changes fields/types/relations/constraints (e.g. `onDelete: Cascade`) vs upstream to dodge something.
- **INPUT-SUBSTITUTION** — seeded/queried values differ in a way that changes the subject.
- **UNDER-PORTED-MATRIX** — upstream parametrizes N cases; port covers fewer with no per-case accounting. Report the upstream count vs the ported count.
- **WRONG-DISPOSITION** — a `non-ported`/`it.fails` that should be a real port, or a passing test that should be `it.fails`/non-ported.

**The litmus (reject the port if any is "yes"):** Did the port change the mechanism, the input, or the asserted result relative to upstream in order to pass? Would this test still pass if prisma-next's missing feature were suddenly added and behaved like Prisma's? (A faithful `it.fails` would flip to green; a workaround would not.)

Be a hard skeptic. Independently confirm every `non-ported` reason against the public API (`packages/3-extensions/sql-orm-client/src/collection.ts`, the ORM types, `filters.ts`, model-accessor, and the package `exports/`). Common wrong non-ports to challenge — these are SUPPORTED, so a non-port citing them is WRONG-DISPOSITION: `.ilike()` (case-insensitive LIKE); callback `.gt()/.lt()` on Numeric (the `order` trait); prisma-next cursor (exclusive → `it.fails`, not non-ported); nested M:N `create`/`connect` through an explicit junction (`test/sql-orm-client/mn-nested-write.test.ts`); **interactive transactions** (high-level facade `postgres(...).transaction(async tx => { … tx.orm … })`; `withTransaction` is the low-level primitive — ports should use the facade; a callback `$transaction(async tx => …)` ports to it; only the array/batch `$transaction([...])` form is genuinely absent); **inline type assertions** (a type-only upstream test ports to inline `@ts-expect-error`/`expectTypeOf` — "type-only"/"type-check-only" is NEVER by itself a non-port reason; only non-portable when it names a *Prisma-generated* type with no re-expressible constraint). Genuinely inexpressible (confirm): raw `$queryRaw`/`$executeRaw` (no raw-SQL-string executor), array/batch `$transaction([...])`, atomic `{increment}`, `_count: true` all-relations shorthand (no all-relations-count surface — but a *specific* relation count IS portable via `include(rel => rel.count())`, so only reject the count-every-relation shorthand), `startsWith`/`endsWith`/`contains` (only `like`/`ilike` exist — and they are NOT the same operator), cross-column/`ScalarFieldRef`, `$on('query')` inspection, JSON-path filters, Decimal.js input interop, `Prisma.skip`/`Prisma.JsonNull` sentinels. **Also reject IN-MEMORY-POSTPROCESSING** (a JS `.sort()`/`.filter()`/aggregate standing in for a query op) even when the test is green.
