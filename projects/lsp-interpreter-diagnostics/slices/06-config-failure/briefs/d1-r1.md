## Task

In `packages/1-framework/3-tooling/language-server/src/server.ts`, change the
project-load lifecycle per the slice spec § Design (read it first —
`projects/lsp-interpreter-diagnostics/slices/06-config-failure/spec.md`):

1. **Failure diagnostic**: catch the `loadProject` rejection at the load-lifecycle
   level (`startProjectLoad`'s chain, guarded by `isCurrentLoad` so superseded loads
   stay silent); publish exactly one diagnostic on `pathToFileURL(configPath)` at
   (0,0)–(0,1) — message = the thrown error's message (String(error) fallback), a
   stable code (e.g. `PRISMA_NEXT_CONFIG_LOAD_FAILED`), severity error — via
   `sendDiagnostics` unconditionally (do NOT gate on `clientCapabilities.pullDiagnostics`;
   the config file belongs to the TS language service — spec's tsserver rationale).
   Track published-config-diagnostic state per configPath; clear (publish `[]`) on
   the next successful load of that config and in `stopManagingProject`.
2. **Retention**: the `loading` entry keeps the previous `ProjectState` (extend the
   `ManagedProject` union — e.g. `lastGood?: ProjectState` replacing/augmenting the
   `hadLoadedProject` boolean, which is derivable from it); on reload failure with a
   last-good, restore the entry to `{status: 'loaded', project: lastGood}` so
   awaiters and subsequent reads keep serving it — do NOT funnel that case into
   `stopManagingProject`. First-load failure (no last-good): today's behavior
   exactly (entry removed, awaiters see the failure as now, documents unmanaged).
3. Success path: no new work beyond clearing a previously-published config
   diagnostic.
