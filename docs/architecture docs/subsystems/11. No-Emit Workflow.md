# No-Emit Workflow

**Purpose:** let developers author the data contract in TypeScript and get full types immediately, with no explicit generate/emit step in day-to-day development, while preserving all guarantees of the contract-first model, including deterministic hashing (coreHash), runtime verification, and compatibility with the migration system.

## Goals

- Let developers author the data contract in TypeScript and get full types immediately, with no explicit generate/emit step in day-to-day development
- Enable environments where executing a separate emit CLI command is not possible (e.g. AI builders)
- Preserve all guarantees of the contract-first model, including deterministic hashing (coreHash), runtime verification, and compatibility with the migration system
- Keep tools and CI artifact-driven by using the canonical JSON form of the contract when needed
- Remain adapter- and lane-agnostic, and keep the door open for PSL-first teams

## Non-goals

- Replace PSL as an authoring mode
- Execute arbitrary code during tooling steps
- Introduce non-determinism into hashing or planning

## Overview

- **TS-first authoring**: a typed builder constructs a pure data contract object
- **Immediate types**: the query DSL reads the contract object directly, so makeT(contract) provides types with no emit
- **Deterministic canonicalization**: canonicalize(contract) produces canonical JSON and a coreHash
- **Runtime accepts either the contract object or the JSON artifact and always computes or verifies the same coreHash**
- **Tools and CI operate on the canonical JSON artifact to avoid evaluating application code**
- **Planner requires canonical JSON for A→B diffs; in TS-first repos the build pipeline can emit on demand in CI**

## Authoring in TS

```typescript
import { defineContract, pg } from "@prisma/contract-core"

export const contract = defineContract({
  target: "postgres",
  tables: {
    user: pg.table({
      columns: {
        id: pg.int4().primaryKey(),
        email: pg.text().unique(),
        active: pg.bool().default(true),
        createdAt: pg.timestamptz().defaultNow()
      },
      indexes: [],
      foreignKeys: []
    })
  }
} as const)
```

### Contract object requirements

- **Pure data only**: plain objects and literals, no functions or getters
- **Serializable**: JSON-serializable shapes, stable key ordering handled by canonicalizer
- **Typed builders**: scalar helpers return branded POJOs that encode type metadata without runtime behavior
- **Immutable after construction**: mutation is discouraged and optionally linted

### Type inference

- `makeT(contract)` reads the TS object generically and yields a typed t and DSL surface
- Result types are inferred from the contract and explicit projections, with dialect-specific nullability rules applied in the SQL adapter

### Canonicalization and coreHash

- `canonicalize(contract)` → `{ json: string, coreHash: string, canonicalVersion: number }`
- Canonicalization rules are stable and versioned per ADR-010
- TS-first and PSL-first yield identical canonical JSON and the same coreHash for the same schema

## Runtime integration

```typescript
import { createRuntime } from "@prisma/runtime"
import { contract } from "./contract"
import { pgAdapter } from "@prisma/adapter-postgres"

const rt = createRuntime({
  contract,            // TS object, no emit needed
  adapter: pgAdapter(),
  verify: "onFirstUse" // or "startup" | "always"
})
```

- Runtime canonicalizes the TS object at startup to compute the coreHash
- Contract marker verification uses the same coreHash as JSON-first mode
- Codecs and adapter capabilities behave identically in both modes

## Planner and migrations

- Planner consumes canonical JSON for both A and B contracts to produce edges
- In TS-first repos, CI runs a fast emit-if-missing step:
  - Import the TS contract in a sandboxed build environment
  - Call canonicalize(contract) to write contracts/<coreHash>.json
  - Validate hash(json) === coreHash
- Migration edges store toContractRef and planner hints, not TS code
- Runner and preflight consume JSON artifacts only

## Dev experience

### No explicit generate

- Author queries and contracts in TS
- Types flow immediately from the contract object
- The runtime verifies contract alignment at first use

### Optional auto-emit for tools

- Vite/Next/esbuild plugin can auto-emit canonical JSON when the contract module is imported or on file change
- This powers Studio, CLI diffs, and agents without requiring the app to run

```typescript
// vite.config.ts
import { prismaNextPlugin } from "@prisma/dev-plugin"

export default {
  plugins: [
    prismaNextPlugin({
      contractPath: "src/contract.ts",
      outFile: "contracts/current.contract.json",
      mode: "watch" // emit on import and on change with debounce
    })
  ]
}
```

### CI pipeline

- Build step compiles TS normally
- A contract step runs canonicalize(contract) or loads the already auto-emitted JSON and validates coreHash
- Planner and preflight run against the canonical JSON and ledger only
- Exit codes and diagnostics follow preflight ADRs

```bash
pnpm build
pnpm prisma-next contract emit --contract src/contract.ts --out contracts
pnpm prisma-next plan --from <hashA> --to <hashB>
pnpm prisma-next preflight --mode shadow
```

## PPg integration

- PPg preflight-as-a-service accepts the canonical JSON blob or computes it by importing the TS contract in a trusted job
- For auditability, PPg retains the canonical JSON used for a preflight job
- Optional policy: PPg runs JSON-only, disallowing TS evaluation in hosted environments

## Safety and determinism

- Lint rules disallow dynamic table/column creation at runtime and contract mutation after construction
- Canonicalization is the source of truth for hashing; its version is pinned and upgraded deliberately
- The runtime fails fast if the TS object includes non-serializable values
- The same guardrails, lints, budgets, and telemetry apply regardless of authoring mode

## Fallback emit on demand

- `prisma-next contract emit` can be run ad hoc to materialize the canonical JSON for review or for downstream tools
- A precommit or prepush hook can ensure the JSON exists and matches the TS contract

## Configuration knobs

```typescript
createRuntime({

  marker: { storeContract: "off" } // "off" | "compressed" | "full"
```


## Edge cases

- **Non-deterministic construction**: forbid Date.now, random IDs, environment-dependent values in the contract
- **Adapter extensions**: TS contract may reference target-specific types by identifier, but the canonical JSON remains code-free
- **Large contracts**: auto-emit can be gated by file size or explicit command to keep hot reload fast
- **Dual authoring**: if both PSL and TS exist, ADR-035 decides canonical source and conflict strategy

## TS-Authored Contract Serialization

TS-authored contracts must serialize extension blocks identically to PSL-first mode:

### Extension Block Serialization
- **Deterministic Ordering**: Extension blocks must be serialized in deterministic order
- **Canonical Format**: Extension data must follow canonical JSON format per ADR 106
- **Version Pinning**: Extension versions must be pinned and immutable per contract
- **Reference Stability**: Extension references must use structured addressing

### Extension Validation
- **Schema Compliance**: Extension data must validate against pack-provided schemas
- **Capability Gating**: Extension usage must be checked against adapter capabilities
- **Namespace Validation**: Extension namespaces must be valid and non-colliding
- **Reference Resolution**: Extension decorations must reference existing core nodes

## Lints Prohibiting Dynamic Extension Registration

Runtime lints prevent dynamic extension registration for security and determinism:

### Prohibited Patterns
- **Dynamic Pack Loading**: Runtime loading of extension packs is prohibited
- **Dynamic Capability Registration**: Runtime registration of capabilities is prohibited
- **Dynamic Codec Registration**: Runtime registration of codecs is prohibited
- **Dynamic Function Registration**: Runtime registration of extension functions is prohibited

### Lint Rules
- **Static Extension Declaration**: Extensions must be declared at build time
- **Immutable Capability Registry**: Capability registry must be immutable at runtime
- **Pre-compiled Extension Code**: Extension code must be pre-compiled and bundled
- **Deterministic Extension Resolution**: Extension resolution must be deterministic

### Security Benefits
- **Sandbox Compliance**: Prevents runtime code injection through extensions
- **Deterministic Behavior**: Ensures consistent behavior across environments
- **Audit Trail**: Enables complete audit trail of extension usage
- **Compliance**: Meets security and compliance requirements for extension execution

## CI Trust Model Unchanged

The CI trust model remains unchanged: tooling runs on canonical JSON, not TS:

### Contract Verification
- **JSON-First Validation**: All validation runs on canonical `contract.json`
- **Hash Verification**: Contract hashes are computed from canonical JSON
- **Extension Validation**: Extension validation runs on serialized extension data
- **Capability Verification**: Capability verification runs on canonical capability declarations

### Build Process
- **TS Compilation**: TS contracts are compiled to canonical JSON
- **Extension Serialization**: Extensions are serialized to canonical format
- **Validation Pipeline**: Validation pipeline operates on canonical artifacts
- **CI Integration**: CI processes canonical artifacts, not TS source

## Performance

- Canonicalization cost is proportional to contract size and happens at startup or compile time
- The runtime caches the canonical string and coreHash, and memoizes validation
- Type checking cost is the same as any TS program using generics and as const

## Security and privacy

- TS evaluation happens in the developer's environment or CI, never in PPg unless explicitly enabled
- Canonical JSON contains schema metadata only; redaction policies apply to comments or sensitive annotations if present
- PPg stores contract blobs with integrity checks and retention limits

## Acceptance criteria

- Queries typecheck and run with no explicit generate in dev when authored in TS
- The same coreHash is produced from TS and from PSL for identical schemas
- Planner and preflight operate on JSON only, with no TS evaluation
- Auto-emit plugin reliably produces canonical JSON on edit with clear error surfacing
- Runtime verification and guardrails behave identically in TS-first and PSL-first workflows

## Open questions

- Do we ship a minimal TS-to-PSL pretty-printer for teams that want to keep PSL files in repo for review
- Do we support hybrid contracts where parts are generated by agent workflows and merged deterministically
- Should the plugin optionally emit diff bundles to speed up Studio and agent UX for large contracts

## ADRs

- **ADR 096** — TS-authored contract parity & purity rules
  - `defineContract()` must produce pure, JSON-serializable data; no functions/getters/dynamic keys; immutability after construction; deterministic ordering. Canonicalization of TS and PSL must yield identical coreHash under ADR-010 rules.
  - **Why**: Guarantees TS-first is as safe/deterministic as PSL-first

- **ADR 097** — Tooling runs on canonical JSON only
  - Planner, preflight, ledger, PPg services must consume canonical JSON; they do not import or evaluate application TS. CI may run a minimal "emit-if-missing" step to materialize JSON from TS.
  - **Why**: Keeps tools deterministic, secure, and language-agnostic

- **ADR 098** — Runtime accepts contract object or JSON
  - Runtime API supports `{ contract }` (TS object) or `{ contractJson }`. On TS input, runtime canonicalizes at startup, caches coreHash, and treats both inputs identically for verification and guardrails.
  - **Why**: Formalizes dual inputs so apps can be no-emit while tools stay artifact-driven

- **ADR 099** — Contract authoring lint rules
  - Ship an ESLint plugin: forbid dynamic property names in tables, forbid post-construction mutation, forbid non-serializable values, flag environment-dependent values. Optional precommit check to assert canonicalize(contract) is stable.
  - **Why**: Prevents non-determinism creeping into TS-authored contracts

- **ADR 100** — CI contract emission trust model
  - Define the sandbox for emitting canonical JSON from TS in CI: no network, restricted import graph, reproducible Node version, pinned canonicalVersion. Fail CI if hash(json) !== coreHash or if emission would change an existing blob.
  - **Why**: Makes TS evaluation in CI predictable and auditable

- **ADR 046–050 family** — No-emit workflow extensions
  - Extension block serialization, dynamic registration lints, and CI trust model for extensions

- **ADR 106** — Canonicalization for extensions
  - Deterministic normalization rules for extension data and reference stability
