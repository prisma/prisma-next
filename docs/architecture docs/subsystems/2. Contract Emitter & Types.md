# Contract Emitter & Types

**Audience:** engineering, PM, DX

**Purpose:** define how we turn PSL or a TS builder into a deterministic `contract.json` plus TypeScript types, how hashing works, and how tools consume these artifacts

## Goals

- Produce a canonical, machine-readable data contract from PSL or a TS builder
- Generate minimal `.d.ts` types that make the DSL ergonomic and fast in editors
- Keep emission deterministic and cross-platform stable for hashing and CI
- Eliminate explicit "generate" in day-to-day dev via watch plugins
- Provide clear errors and actionable diagnostics

## Inputs and Outputs

### Inputs

- `schema.prisma` (PSL authoring mode)
- `contract.ts` TS builder entry point exporting `defineContract(...)` (TS-first mode)

### Outputs

- **contract.json** canonical JSON data contract
  - includes `schemaVersion`, `targetFamily`, `target`, `coreHash`, optional `profileHash`, `models`, `storage`, `capabilities`, `codecs`, `meta`
- **contract.d.ts** TypeScript declaration file
  - exposes table and model types, column accessors, and helper types used by the DSL

### Invariants

- PSL and TS builder produce the same `contract.json` for the same intent
- Emission is deterministic and produces identical hashes on all supported platforms

## Determinism and Hashing

- **Canonicalize JSON**
  - stable key ordering, normalized scalars, normalized defaults, stable array ordering where ordering is semantic
- **Compute two hashes**
  - `coreHash` for logical meaning (models, fields, relations, storage shape)
  - `profileHash` for physical profile and capability toggles that do not change meaning
- **Persist hashes**
  - embedded in `contract.json`, available to the runtime, and written to the DB marker by the migration runner

## Emission Pipeline

1. **Parse**
   - PSL → AST via the PSL parser with extension attribute support
   - TS builder → in-memory AST via `defineContract(...)` with pack registration
2. **Normalize**
   - resolve model↔storage mappings, expand defaults, infer deterministic names where omitted (e.g., PK/UK/FK names under the chosen naming scheme)
   - process extension decorations and constructs from installed packs
3. **Validate**
   - structural validation with zod schemas
   - semantic checks like FK targets exist, uniqueness sets are sound, codec availability
   - extension validation against pack-provided schemas and capability gating
4. **Canonicalize and hash**
   - produce canonical JSON and compute `coreHash` and `profileHash`
   - apply extension canonicalization rules per ADR 106
5. **Write artifacts**
   - `contract.json` and `contract.d.ts` to the configured artifact directory
   - generate branded types for extension values

## Types That Are Generated

The declarations are pure types to avoid runtime coupling and to keep bundles lean:

- **Contract.Tables**
  - map of table names to `{ columnName: Type }` suitable for inferring result shapes
- **Contract.Models**
  - map of model names to model field types, aligned with models → storage mapping
- **Contract.Relations**
  - relation handles `{ to, cardinality, on }` used by higher layers
- **Tables helper and T type**
  - structural types that the DSL uses to type `t.user.id` style access
- **Narrow utility types** for projections, joins, and nullability propagation

**No runtime t object is emitted** — the runtime `t` is constructed by `makeT(contractJson)` in the DSL package, and the generated `.d.ts` supplies types so that `t` is fully typed without shipping code

## Extension Support

### PSL Parser Extension Syntax

The PSL parser supports namespaced extension attributes:

```prisma
extensions {
  pgvector = "1.2.0"
}

model Document {
  embedding Bytes @pgvector.column(dim: 1536, distance: cosine)
}
```

Extension attributes are validated against pack-provided JSON schemas and emit deterministic decorations under `contract.extensions.<namespace>`.

### TS Builder Pack Registration

The TS builder supports pack registration during construction:

```typescript
import { defineContract } from '@prisma/contract-core'
import { pgvector } from '@prisma/pack-pgvector'

const contract = defineContract({
  target: 'postgres',
  tables: {
    document: pgvector.table({
      columns: {
        embedding: pgvector.column({ dim: 1536, distance: 'cosine' })
      }
    })
  }
})
```

Packs provide typed helpers that ensure canonical emission identical to PSL-first mode.

### Branded Types Generation

Extension values receive branded TypeScript types for type safety:

```typescript
// Generated contract.d.ts
export type Vector<N extends number> = Brand<readonly number[], `vector:${N}`>
export type Geography = Brand<GeoJSON.Geometry, 'postgis:geography'>

export interface Tables {
  document: {
    id: number
    embedding: Vector<1536>  // Branded type from pgvector
  }
  place: {
    id: number
    geom: Geography  // Branded type from postgis
  }
}
```

Branded types prevent mixing incompatible extension values and enable compile-time verification.

### Codec Integration

Extension codecs are registered at runtime and provide:

- **Encode**: JavaScript values → database wire format
- **Decode**: Database wire format → branded JavaScript types
- **Validate**: Runtime validation of extension values
- **Losslessness**: Policy control for conversion fidelity

## Error Taxonomy Extensions

New error categories for extension validation:

- **`EMIT_EXT_UNKNOWN_NAMESPACE`** — Extension namespace not pinned in extensions block
- **`EMIT_EXT_UNKNOWN_ATTRIBUTE`** — Extension attribute not recognized by pack
- **`EMIT_EXT_SCHEMA_VIOLATION`** — Extension payload fails pack schema validation
- **`EMIT_EXT_CAPABILITY_UNSUPPORTED`** — Extension requires unavailable adapter capability
- **`EMIT_EXT_CANONICALIZATION_FAILURE`** — Extension data cannot be canonicalized

## Developer Workflow

- **Dev watch**
  - Vite/Next/esbuild plugin auto-emits on import and on file change
  - errors surface inline in the terminal and IDE
- **CI**
  - explicit `prisma-next emit` step verifies determinism and hashes
  - produces artifacts for downstream steps like migration planning and preflight
- **Editor responsiveness**
  - `.d.ts` is small and incremental, keeping TS language server snappy

## Configuration

`prisma-next.config.ts` at the repo root:

```typescript
import { defineConfig } from '@prisma/config'

export default defineConfig({
  authoring: 'psl',               // 'psl' | 'ts'
  schema: './prisma/schema.prisma',
  builder: './contract/contract.ts',
  outDir: './.prisma-next',
  naming: {
    pk: '{table}_pkey',
    uk: '{table}_{cols}_key',
    fk: '{table}_{cols}_fkey',
    idx: '{table}_{cols}_idx',
  },
  target: { family: 'sql', dialect: 'postgres' },
  emit: { dts: true, json: true },
})
```

- If `authoring: 'psl'`, `schema` is required and `builder` is ignored
- If `authoring: 'ts'`, `builder` is required and `schema` is optional for back-generation
- `outDir` is the canonical location for `contract.json` and `contract.d.ts`

## CLI

- `prisma-next emit`
  - parse → validate → canonicalize → hash → write artifacts
- `prisma-next verify`
  - validate an existing `contract.json` and recompute hashes
- `prisma-next diff <from.json> <to.json>`
  - print a human and JSON diff suitable for planner input

## Error Handling and Diagnostics

- Clear messages with source spans for PSL and callsite hints for TS builder
- **Categorized errors**
  - structural validation, semantic validation, capability gaps, naming conflicts
- **Suggested fixes** where possible
  - e.g., add equality index for a foreign key, add default for non-null column

## Performance Targets

- **Small schema**
  - emit < 100 ms on a modern laptop
- **Medium schema (~100 tables)**
  - emit < 700 ms
- **Large schema**
  - emit < 2 s with caching of parser and builder modules

## Consumption by Other Subsystems

- **Runtime**
  - reads `contract.json`, verifies `plan.coreHash` on execute, exposes capability flags to plugins
- **Query DSL**
  - uses `contract.d.ts` for types and `makeT(contractJson)` for runtime column objects
- **TypedSQL**
  - emits plans that carry `contractHash`, validated against `contract.json`
- **Migrations**
  - planner consumes two contracts to compute an edge, runner writes the DB marker to `toHash`
- **Preflight and PPg**
  - read capabilities and storage to run EXPLAIN checks and advisors

## Testing

- **Golden tests** for PSL → `contract.json` and TS builder → `contract.json`
  - include hash stability
- **Round-trip tests** where applicable
  - PSL → JSON → PSL (rendered) sanity checks for information preservation
- **Negative tests** for validation and capability errors
- **Performance regression tests**

## Open Questions

- Boundary between core and profile for collations and encodings
- Minimal surface for model-level computed fields in `.d.ts` without implying runtime codegen
- Optional back-generation of PSL from the TS builder for teams migrating away from PSL authoring

## Minimal Example

**schema.prisma:**

```prisma
model User {
  id        Int      @id @default(autoincrement())
  email     String   @unique
  active    Boolean  @default(true)
  createdAt DateTime @default(now())
}

model Post {
  id        Int      @id @default(autoincrement())
  title     String
  user      User     @relation(fields: [userId], references: [id])
  userId    Int
  @@index([userId])
}
```

**Emitted contract.json (excerpt):**

```json
{
  "schemaVersion": "1",
  "targetFamily": "sql",
  "target": "postgres",
  "coreHash": "sha256:…",
  "profileHash": "sha256:…",
  "models": {
    "User": { "storage": { "table": "user" }, "fields": { "id": { "column": "id" }, "email": { "column": "email" }, "active": { "column": "active" }, "createdAt": { "column": "createdAt" } } },
    "Post": { "storage": { "table": "post" }, "fields": { "id": { "column": "id" }, "title": { "column": "title" }, "userId": { "column": "user_id" } }, "relations": { "user": { "to": "User", "cardinality": "N:1", "on": { "parentCols": ["id"], "childCols": ["user_id"] } } } }
  },
  "storage": {
    "tables": {
      "user": {
        "columns": {
          "id": { "type": "int4", "nullable": false },
          "email": { "type": "text", "nullable": false },
          "active": { "type": "bool", "nullable": false },
          "createdAt": { "type": "timestamp", "nullable": false }
        },
        "primaryKey": { "columns": ["id"], "name": "user_pkey" },
        "uniques": [ { "columns": ["email"], "name": "user_email_key" } ],
        "indexes": []
      }
    }
  },
  "capabilities": { "postgres": { "lateral": true, "jsonAgg": true } },
  "codecs": { "int4": { "ts": "number" }, "text": { "ts": "string" }, "bool": { "ts": "boolean" }, "timestamp": { "ts": "Date" } }
}
```

**Generated contract.d.ts (excerpt):**

```typescript
export declare namespace Contract {
  export interface Tables {
    user: { id: number; email: string; active: boolean; createdAt: Date }
    post: { id: number; title: string; user_id: number }
  }
  export interface Models {
    User: { id: number; email: string; active: boolean; createdAt: Date }
    Post: { id: number; title: string; userId: number }
  }
  export interface Relations {
    post: { user: { to: 'User'; cardinality: 'N:1'; on: { parentCols: ['id']; childCols: ['user_id'] } } }
    user: { posts: { to: 'Post'; cardinality: '1:N'; on: { parentCols: ['id']; childCols: ['user_id'] } } }
  }
}

export type Tables = Contract.Tables
export type Models = Contract.Models
```

This keeps the runtime code minimal, the types precise, and the artifacts stable for CI, agents, and PPg integrations

## ADRs

- **ADR 006** — Dual authoring modes (PSL-first and TS-first) with a single canonical artifact
  - One source of truth per repo, PSL or TS builder, both must emit byte-identical `contract.json`
- **ADR 007** — Types-only emission and no runtime client codegen
  - Emit `.d.ts` only, construct `t` at runtime via `makeT(contractJson)`, never ship generated JS
- **ADR 008** — Dev auto-emit, CI explicit emit
  - Remove the explicit "generate" step in day-to-day via Vite/Next/esbuild plugins, but require an explicit emit in CI
- **ADR 009** — Deterministic naming scheme for constraints
  - Lock in `{table}_{cols}_{suffix}` for PK/UK/FK/IDX so emission remains stable across platforms
- **ADR 010** — Canonicalization rules for contract.json
  - Exact key ordering, scalar normalization, array ordering semantics to guarantee cross-platform identical hashes
- **ADR 104** — PSL extension namespacing & syntax
  - Namespaced attributes (`@pgvector.column()`) and deterministic emission to `contract.extensions`
- **ADR 105** — Contract extension encoding
  - Structured encoding of decorations and constructs under `extensions.<namespace>`
- **ADR 106** — Canonicalization for extensions
  - Deterministic normalization rules for extension data and reference stability
- **ADR 114** — Extension codecs & branded types
  - Branded TypeScript types and codec integration for extension values
