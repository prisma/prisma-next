# Runtime & Plugin Framework

**Purpose:** describe the executable core of Prisma Next: how Plans move from authoring lanes to the database, how plugins observe and enforce guardrails, and how we keep safety guarantees without compromising performance.

## Goals

- Execute a Plan with strong guarantees and low overhead
- Provide built-in contract verification as a non-negotiable safety gate
- Make safety and observability composable via a small, stable Hook API
- Keep Plans immutable end to end so hashing, caching, and audit stay trustworthy
- Keep the core thin and lane-agnostic, with dialect specifics handled by adapters

## Non-goals

- Multi-statement orchestration or unit-of-work semantics
- Dialect logic in the runtime core
- A universal SQL parser or result mapper beyond configured codecs
- Making contract verification optional (it's a safety gate, not a feature)

## Key Concepts

- **Plan**
  - lane-agnostic execution unit with sql, params, and meta that includes target, coreHash, optional refs and projection, and annotations for policy
- **Adapter profile**
  - deterministic lowerer for a target dialect that takes a relational AST and emits sql and params, declaring capabilities like lateral, jsonAgg
- **Runtime**
  - the minimal executor that accepts a Plan, runs the hook pipeline, calls the driver, and returns results
- **Contract Verification**
  - built-in safety gate that compares database contract hash with application contract hash before execution
- **Plugin**
  - a module that implements one or more hooks to extend verification behavior, lint, budget, or record telemetry

## Connection Lifecycle

The runtime never owns pooling logic or sockets directly. It delegates to a driver for the target:

1. **Create**
   - `createRuntime({ contract, adapter, driver, verify, plugins, mode })`
   - Validates `contract.json` and caches `coreHash`
   - Loads adapter profile and driver
   - Performs capability discovery and negotiation, yielding `profileHash`
   - Registers plugins in order
2. **Warmup**
   - optional `driver.warmup()` and built-in contract verification if configured
   - Loads and composes codecs from adapter and contract extension codecs
3. **Acquire**
   - on first execute, driver acquires a pooled connection or session
4. **Execute**
   - runtime executes a single Plan through the pipeline and the driver's `query(sql, params)`
5. **Release**
   - driver returns the connection to the pool
6. **Shutdown**
   - `runtime.end()` flushes telemetry, closes driver, and disposes plugin resources

## Built-in Contract Verification

Contract verification is a **non-negotiable safety gate** built into the runtime core. It ensures the database schema matches the application's data contract before any query execution.

### Configuration

```typescript
const runtime = createRuntime({
  ir: contract,
  driver,
  verify: {
    mode: 'startup' | 'onFirstUse' | 'always' | 'off', // off only with allowUnsafe: true
    onMismatch: 'throw' | 'warn',                      // core behavior
    requireMarker: true                                 // fail fast if marker table missing
  },
  plugins: [
    // Optional: extend verification behavior
    driftReporter({ sink: 'otlp' }),
    preflightOnMismatch({ mode: 'explain' })
  ]
})
```

### Verification Modes

- **`startup`**: Verify once at runtime creation (recommended for production)
- **`onFirstUse`**: Verify per table when first referenced (development default)
- **`always`**: Verify every Plan execution (debugging)
- **`off`**: Skip verification (only with `allowUnsafe: true`)

### Built-in Behavior

1. **Hash Comparison**: Compares `plan.meta.coreHash` with database marker
2. **Caching**: Per-pool caching; invalidates on reconnect
3. **Multi-tenant**: Runs when tenant context is bound (per ADR 033)
4. **Error Handling**: Throws `RuntimeError(CONTRACT_MISMATCH)` on mismatch

### Pluggable Extensions

While the core verification is built-in, plugins can extend behavior:

- **Drift Reporting**: Log mismatches, send metrics, Slack notifications
- **Auto-fix Advisors**: Suggest migration paths or open PRs
- **Preflight Runners**: Run shadow/explain on mismatch
- **Policy Variations**: Different behavior per environment
- **Audit Events**: Attach evidence to CI/PPg

## Execution Pipeline

```
Plan (immutable)
  └─▶ beforeCompile       // optional, AST-present lanes only
        └─ adapter.lower  // if AST is present and not yet lowered
             └─ ▶ lowered SQL + params
  └─▶ contract verification // built-in: compare coreHash with database marker
        └─▶ beforeExecute  // lints, budgets, annotations checks (plugins)
             └─▶ driver.query  // single statement, blocking
  └─▶ afterExecute        // telemetry, result post-processing, sampling
     ↘ onError            // invoked on any failure path with context
```

### Immutability of Plan

- Plans are immutable inputs
- Plugins may derive a new Plan by returning `{ plan: next }` but must never mutate in place
- The runtime assigns a `planId` and `sqlFingerprint` before `beforeExecute` and uses them for caching and telemetry

## Hook API v1

Hooks are async and run in registration order. Any hook may throw or escalate a violation to block execution

```typescript
type Plan = {
  sql: string
  params: unknown[]
  meta: {
    target: string
    coreHash: string
    lane?: string
    refs?: { tables: string[]; columns: Array<{ table: string; column: string }> }
    projection?: Record<string, string>
    annotations?: Record<string, unknown>
  }
}

type CompileContext = {
  contract: ContractJson
  adapter: AdapterProfile
}

type ExecuteContext = {
  contract: ContractJson
  adapter: AdapterProfile
  driver: Driver
  mode: 'strict' | 'permissive'
  now: () => number
}

interface Plugin {
  name: string

  beforeCompile?(
    plan: Plan,
    ctx: CompileContext
  ): Promise<{ plan?: Plan } | void>

  beforeExecute?(
    plan: Plan,
    ctx: ExecuteContext
  ): Promise<{ plan?: Plan } | void>

  afterExecute?(
    plan: Plan,
    result: { rows: unknown[]; rowCount?: number; latencyMs: number },
    ctx: ExecuteContext
  ): Promise<void>

  onError?(
    plan: Plan,
    err: RuntimeError,
    ctx: ExecuteContext
  ): Promise<void>
}
```

**Notes:**

- `beforeCompile` is called only when the lane supplies an AST and defers lowering to the adapter
- `beforeExecute` always runs and is where verification and guardrails live
- Hooks must be idempotent and deterministic with respect to inputs

## Extension Support

### Capability Discovery and Negotiation

At `connect()` time, the runtime performs capability discovery and negotiation:

1. **Adapter Advertisement**: Adapter declares supported capabilities per ADR 065
2. **Contract Requirements**: Runtime checks contract capabilities against adapter capabilities
3. **Extension Pack Loading**: Loads installed packs and validates capability requirements
4. **ProfileHash Computation**: Computes `profileHash` from negotiated capabilities and adapter configuration
5. **Early Failure**: Missing required capabilities cause connection failure with stable error codes

```typescript
const runtime = createRuntime({
  contract,
  adapter: postgresAdapter,
  driver,
  // Capability negotiation happens automatically
  // Missing capabilities cause E_CAPABILITY_MISSING
})
```

### Codec Composition

The runtime loads and composes codecs from multiple sources with strict precedence:

1. **App-provided codecs** (highest priority)
2. **Pack codecs** from installed extension packs
3. **Adapter built-ins** for core SQL types
4. **Driver decoding hints** (fallback)

```typescript
// Extension codecs are automatically registered
const results = await runtime.execute(query)
// results.embedding is typed as Vector<1536> (branded type)
```

### Extension Guardrails and EXPLAIN Policies

Extension-specific guardrails enforce policies for extension usage:

- **Capability Gating**: Reject queries using unavailable extension functions/operators
- **Budget Enforcement**: Apply extension-specific cost models and row limits
- **EXPLAIN Policies**: Extension operators may have different EXPLAIN behavior
- **Security Sandboxing**: Extension code runs under strict constraints

```typescript
const runtime = createRuntime({
  contract,
  adapter,
  plugins: [
    extensionGuardrails({
      rules: {
        'pgvector/require-ivfflat-index': 'warn',
        'postgis/no-volatile-in-predicate': 'error'
      }
    })
  ]
})
```

## Built-in Plugins

### Lints

**Default rules:**

- `no-select-star`
- `mutation-requires-where`
- `no-missing-limit` for read Plans without pagination intent
- `no-unindexed-predicate` using `meta.refs` and contract indexes when available

Each rule has a level `off | warn | error` and an error code for on-error handling

### Budgets

- **Row budget**
  - uses EXPLAIN if fast and available, else applies heuristics
- **Latency budget**
  - measured wall clock with `now()`
- **SQL length budget**
  - protects accidental query explosions

Budgets produce structured violations with the observed value and configured threshold

### Telemetry

- Emits `planId`, `sqlFingerprint`, `latencyMs`, `rowCount`, `errorCode` to a pluggable sink
- Redacts params unless configured otherwise
- Supports sampling and per-rule counters for safety dashboards

## Error Taxonomy and Configurability

Errors are structured and machine readable

```
category/code
  contract/hash-mismatch
  contract/target-mismatch
  contract/marker-missing
  policy/annotations-missing
  lint/no-select-star
  lint/mutation-missing-where
  budget/rows-exceeded
  budget/latency-exceeded
  adapter/capability-missing
  compile/lowering-failed
  driver/query-failed
  runtime/unexpected
```

**Behavior:**

- Each violation maps to a level — `error` blocks, `warn` logs and continues, `off` ignores
- Global mode tunes defaults — `strict` upgrades selected warnings to errors, `permissive` downgrades selected errors to warnings
- `onError` receives the normalized error object and context to publish or transform messages

### Error Envelope

```typescript
type RuntimeError = {
  code: string                // category/code
  message: string
  details?: unknown
  phase: 'beforeCompile' | 'beforeExecute' | 'execute' | 'afterExecute'
  cause?: unknown
}
```

## Overhead Budget and Benchmarks

### Targets

- < 5% overhead on p95 latency for CRUD-class Plans with lints and budgets enabled
- < 1 ms median overhead for the hook pipeline when all built-ins are on and no EXPLAIN is issued
- Zero-plugin runtime adds ~0.2–0.4 ms median on a local driver baseline

### Strategies

- Keep Plan verification O(1) — use a single `coreHash` comparison and short-circuit when matching
- Cache adapter lowering per `(sqlFingerprint, adapter.version)` when lanes supply AST repeatedly
- Make EXPLAIN checks opt-in and cache their cardinality hints by `sqlFingerprint`
- Precompute refs during emission where possible to avoid SQL parsing
- Short-circuit hook chain on first blocking violation

### Benchmark Suite

- **Microbench**
  - no-plugin baseline vs full stack on simple SELECT and UPDATE
  - report median, p95, p99 on local Postgres
- **Macrobench**
  - end-to-end example app with typical read + write mix
  - report overhead percentage and violation catch rate under synthetic regressions

## Example Usage

```typescript
import { createRuntime } from '@prisma/runtime'
import { postgresAdapter } from '@prisma/adapter-postgres'
import { driver } from './pg-driver'
import contract from './contract.json'
import { lints, budgets, telemetry, driftReporter } from '@prisma/runtime/plugins'

const rt = createRuntime({
  ir: contract,
  adapter: postgresAdapter,
  driver,
  verify: {
    mode: process.env.NODE_ENV === 'production' ? 'startup' : 'onFirstUse',
    onMismatch: 'throw',
    requireMarker: true
  },
  mode: 'strict',
  plugins: [
    lints({
      rules: {
        'no-select-star': 'error',
        'mutation-requires-where': 'error',
        'no-missing-limit': 'warn'
      }
    }),
    budgets({ maxRows: 10_000, maxLatencyMs: 150 }),
    telemetry({ sink: console }),
    driftReporter({ sink: 'otlp' })
  ]
})

// plan produced by any lane
const plan = sql().from('user').where(t.user.active.eq(true)).select({ id: t.user.id }).build()

const rows = await rt.execute(plan)
```

## Extensibility Guide

- New plugins implement any subset of hooks and declare a name
- Keep plugins pure and deterministic with respect to inputs
- Prefer reading annotations and refs to inspecting SQL text
- For dialect or lowering concerns, contribute via an adapter profile, not a runtime plugin

## Testing Strategy

- Hook API contract tests with fake plugins and error injection
- Golden SQL and hash stability tests to ensure Plan immutability and identity do not drift
- Violation matrix ensuring each built-in rule behaves consistently under strict and permissive
- Benchmarks run in CI on tagged commits with budget alerts

## Open Questions

- Adaptive budgets that learn typical latency and row counts per `planId`
- Pluggable EXPLAIN interpreters per adapter to standardize cost hints
- Optional server-side guardrails in PPg that mirror on-prem defaults without configuration drift

## ADRs

- **ADR 021** — Contract marker storage & verification modes
  - Where and how we store the `coreHash` in the DB, table name and schema, startup/onFirstUse/always verification behavior, and cross-env drift handling

- **ADR 022** — Lint rule taxonomy & configuration model
  - Canonical rule IDs, levels (off|warn|error), defaults by mode (strict|permissive), and how lanes attach rule context to violations

- **ADR 023** — Budget evaluation & EXPLAIN policy
  - When to run EXPLAIN, caching by `sqlFingerprint`, safe fallbacks, and time/row/size budget precedence

- **ADR 024** — Telemetry schema & privacy
  - Event envelope (`planId`, `sqlFingerprint`, latency, rows), param redaction, sampling, sinks, and PII controls

- **ADR 025** — Plan caching & memoization in runtime
  - What can be cached, keys, invalidation on adapter/contract change, and memory limits

- **ADR 026** — Conformance kit & certification levels
  - L0/L1/L2 definitions for producers and runtimes, fixture formats, and publishing results

- **ADR 027** — Error envelope & stable codes
  - Final shape of `RuntimeError`, code namespaces, mapping rule levels to outcomes, and upgrade policy

- **ADR 028** — Migration structure & operations
  - Migration file structure, on-disk formats, schemas, and operations for working with migration DAGs

- **ADR 029** — Shadow DB preflight semantics
  - Creation, seeding, isolation guarantees, teardown, and limits for CI and PPg preflight-as-a-service

- **ADR 030** — Result decoding & codecs registry
  - How drivers, contract codecs, and lane hints compose, precedence rules, and extension points

- **ADR 031** — Adapter capability discovery & negotiation
  - Static vs runtime capability surfacing, feature flags, and how lanes react to missing capabilities

- **ADR 032** — Dev auto-emit integration (Vite/Next/esbuild)
  - Invalidation triggers, debounce/backoff, error surfacing in IDE/console, and "no explicit generate" contract

- **ADR 033** — Multi-tenant contract enforcement
  - Per-tenant markers, contract pinning in connection/session, and guardrails for cross-tenant drift

- **ADR 034** — Raw Plan factory manifest
  - Optional sidecar manifest for factories (origin SQL path, target profile, annotations), useful for CI and agents

- **ADR 035** — Dual authoring conflict resolution
  - When PSL and TS builder both exist, which is canonical, diff strategy, and errors vs auto-reconciliation

- **ADR 036** — TypedSQL header annotation spec
  - Formal grammar for `-- @param`, `-- @intent`, sensitivity, and optional refs/projection hints

- **ADR 065** — Adapter capability schema & negotiation v1
  - Canonical capability keys, optional features, negotiation flow at connect time, and how lanes/plugins branch on capabilities rather than target strings

- **ADR 114** — Extension codecs & branded types
  - Branded TypeScript types and codec integration for extension values

- **ADR 115** — Extension guardrails & EXPLAIN policies
  - Extension-specific policy enforcement, capability gating, and EXPLAIN behavior

- **ADR 122** — Database Initialization & Adoption
  - Initialization operations, adoption strategies (greenfield, brownfield-conservative, brownfield-incremental), multi-service namespacing, and incremental contract expansion

- **ADR 123** — Drift Detection, Recovery & Reconciliation
  - Comprehensive drift taxonomy (marker, schema, DAG, capability, transactional, cache, canonicalization), detection mechanisms by environment, and recovery operations with idempotency-based strategies
