# Runtime & Plugin Framework

This document describes the executable core of Prisma Next: how Plans move from authoring lanes to the database, how plugins observe and enforce guardrails, and how we keep safety guarantees without compromising performance

## Goals

- Execute a Plan with strong guarantees and low overhead
- Make safety and observability composable via a small, stable Hook API
- Keep Plans immutable end to end so hashing, caching, and audit stay trustworthy
- Keep the core thin and lane-agnostic, with dialect specifics handled by adapters

## Non-goals

- Multi-statement orchestration or unit-of-work semantics
- Dialect logic in the runtime core
- A universal SQL parser or result mapper beyond configured codecs

## Key Concepts

- **Plan**
  - lane-agnostic execution unit with sql, params, and meta that includes target, coreHash, optional refs and projection, and annotations for policy
- **Adapter profile**
  - deterministic lowerer for a target dialect that takes a relational AST and emits sql and params, declaring capabilities like lateral, jsonAgg
- **Runtime**
  - the minimal executor that accepts a Plan, runs the hook pipeline, calls the driver, and returns results
- **Plugin**
  - a module that implements one or more hooks to verify, lint, budget, or record telemetry

## Connection Lifecycle

The runtime never owns pooling logic or sockets directly. It delegates to a driver for the target:

1. **Create**
   - `createRuntime({ contract, adapter, driver, plugins, mode })`
   - Validates `contract.json` and caches `coreHash`
   - Loads adapter profile and driver
   - Performs capability discovery and negotiation, yielding `profileHash`
   - Registers plugins in order
2. **Warmup**
   - optional `driver.warmup()` and `contractVerifier` startup check if configured
   - Loads and composes codecs from adapter and contract extension codecs
3. **Acquire**
   - on first execute, driver acquires a pooled connection or session
4. **Execute**
   - runtime executes a single Plan through the pipeline and the driver's `query(sql, params)`
5. **Release**
   - driver returns the connection to the pool
6. **Shutdown**
   - `runtime.end()` flushes telemetry, closes driver, and disposes plugin resources

## Execution Pipeline

```
Plan (immutable)
  ‚îî‚îÄ‚ñ∂ beforeCompile       // optional, AST-present lanes only
        ‚îî‚îÄ adapter.lower  // if AST is present and not yet lowered
             ‚îî‚îÄ ‚ñ∂ lowered SQL + params

  ‚îî‚îÄ‚ñ∂ afterExecute        // telemetry, result post-processing, sampling

```

### Immutability of Plan

- Plans are immutable inputs
- Plugins may derive a new Plan by returning `{ plan: next }` but must never mutate in place
- The runtime assigns a `planId` and `sqlFingerprint` before `beforeExecute` and uses them for caching and telemetry

## Hook API v1

Hooks are async and run in registration order. Any hook may throw or escalate a violation to block execution

```typescript
type Plan = {
  sql: string
  params: unknown[]
  meta: {
    target: string
    coreHash: string
    lane?: string
    refs?: { tables: string[]; columns: Array<{ table: string; column: string }> }
    projection?: Record<string, string>
    annotations?: Record<string, unknown>
  }
}

type CompileContext = {
  contract: ContractJson
  adapter: AdapterProfile
}

type ExecuteContext = {
  contract: ContractJson
  adapter: AdapterProfile
  driver: Driver
  mode: 'strict' | 'permissive'
  now: () => number
}

interface Plugin {
  name: string

  beforeCompile?(
    plan: Plan,
    ctx: CompileContext
  ): Promise<{ plan?: Plan } | void>

  beforeExecute?(
    plan: Plan,
    ctx: ExecuteContext
  ): Promise<{ plan?: Plan } | void>

  afterExecute?(
    plan: Plan,
    result: { rows: unknown[]; rowCount?: number; latencyMs: number },
    ctx: ExecuteContext
  ): Promise<void>

  onError?(
    plan: Plan,
    err: RuntimeError,
    ctx: ExecuteContext
  ): Promise<void>
}
```

**Notes:**

- `beforeCompile` is called only when the lane supplies an AST and defers lowering to the adapter
- `beforeExecute` always runs and is where verification and guardrails live
- Hooks must be idempotent and deterministic with respect to inputs

## Extension Support

### Capability Discovery and Negotiation

At `connect()` time, the runtime performs capability discovery and negotiation:

1. **Adapter Advertisement**: Adapter declares supported capabilities per ADR 065
2. **Contract Requirements**: Runtime checks contract capabilities against adapter capabilities
3. **Extension Pack Loading**: Loads installed packs and validates capability requirements
4. **ProfileHash Computation**: Computes `profileHash` from negotiated capabilities and adapter configuration
5. **Early Failure**: Missing required capabilities cause connection failure with stable error codes

```typescript
const runtime = createRuntime({
  contract,
  adapter: postgresAdapter,
  driver,
  // Capability negotiation happens automatically
  // Missing capabilities cause E_CAPABILITY_MISSING
})
```

### Codec Composition

The runtime loads and composes codecs from multiple sources with strict precedence:

1. **App-provided codecs** (highest priority)
2. **Pack codecs** from installed extension packs
3. **Adapter built-ins** for core SQL types
4. **Driver decoding hints** (fallback)

```typescript
// Extension codecs are automatically registered
const results = await runtime.execute(query)
// results.embedding is typed as Vector<1536> (branded type)
```

### Extension Guardrails and EXPLAIN Policies

Extension-specific guardrails enforce policies for extension usage:

- **Capability Gating**: Reject queries using unavailable extension functions/operators
- **Budget Enforcement**: Apply extension-specific cost models and row limits
- **EXPLAIN Policies**: Extension operators may have different EXPLAIN behavior
- **Security Sandboxing**: Extension code runs under strict constraints

```typescript
const runtime = createRuntime({
  contract,
  adapter,
  plugins: [
    extensionGuardrails({
      rules: {
        'pgvector/require-ivfflat-index': 'warn',
        'postgis/no-volatile-in-predicate': 'error'
      }
    })
  ]
})
```

## Built-in Plugins

### Contract Verifier

- Compares `plan.meta.coreHash` to the database marker and `contract.coreHash`
- Ensures `plan.meta.target` matches the configured adapter
- **Modes:**
  - `startup`: verify once at runtime creation
  - `onFirstUse`: verify per table when first referenced
  - `always`: verify every Plan

### Lints

**Default rules:**

- `no-select-star`
- `mutation-requires-where`
- `no-missing-limit` for read Plans without pagination intent
- `no-unindexed-predicate` using `meta.refs` and contract indexes when available

Each rule has a level `off | warn | error` and an error code for on-error handling

### Budgets

- **Row budget**
  - uses EXPLAIN if fast and available, else applies heuristics
- **Latency budget**
  - measured wall clock with `now()`
- **SQL length budget**
  - protects accidental query explosions

Budgets produce structured violations with the observed value and configured threshold

### Telemetry

- Emits `planId`, `sqlFingerprint`, `latencyMs`, `rowCount`, `errorCode` to a pluggable sink
- Redacts params unless configured otherwise
- Supports sampling and per-rule counters for safety dashboards

## Error Taxonomy and Configurability

Errors are structured and machine readable

```
category/code
  contract/hash-mismatch
  contract/target-mismatch
  policy/annotations-missing
  lint/no-select-star
  lint/mutation-missing-where
  budget/rows-exceeded
  budget/latency-exceeded
  adapter/capability-missing
  compile/lowering-failed
  driver/query-failed
  runtime/unexpected
```

**Behavior:**

- Each violation maps to a level ‚Äî `error` blocks, `warn` logs and continues, `off` ignores
- Global mode tunes defaults ‚Äî `strict` upgrades selected warnings to errors, `permissive` downgrades selected errors to warnings
- `onError` receives the normalized error object and context to publish or transform messages

### Error Envelope

```typescript
type RuntimeError = {
  code: string                // category/code
  message: string
  details?: unknown
  phase: 'beforeCompile' | 'beforeExecute' | 'execute' | 'afterExecute'
  cause?: unknown
}
```

## Overhead Budget and Benchmarks

### Targets

- < 5% overhead on p95 latency for CRUD-class Plans with lints and budgets enabled
- < 1 ms median overhead for the hook pipeline when all built-ins are on and no EXPLAIN is issued
- Zero-plugin runtime adds ~0.2‚Äì0.4 ms median on a local driver baseline

### Strategies

- Keep Plan verification O(1) ‚Äî use a single `coreHash` comparison and short-circuit when matching
- Cache adapter lowering per `(sqlFingerprint, adapter.version)` when lanes supply AST repeatedly
- Make EXPLAIN checks opt-in and cache their cardinality hints by `sqlFingerprint`
- Precompute refs during emission where possible to avoid SQL parsing
- Short-circuit hook chain on first blocking violation

### Benchmark Suite

- **Microbench**
  - no-plugin baseline vs full stack on simple SELECT and UPDATE
  - report median, p95, p99 on local Postgres
- **Macrobench**
  - end-to-end example app with typical read + write mix
  - report overhead percentage and violation catch rate under synthetic regressions

## Example Usage

```typescript
import { createRuntime } from '@prisma/runtime'
import { postgresAdapter } from '@prisma/adapter-postgres'
import { driver } from './pg-driver'
import contract from './contract.json'
import { contractVerifier, lints, budgets, telemetry } from '@prisma/runtime/plugins'

const rt = createRuntime({
  contract,
  adapter: postgresAdapter,
  driver,
  mode: 'strict',
  plugins: [
    contractVerifier({ verify: 'onFirstUse' }),
    lints({
      rules: {
        'no-select-star': 'error',
        'mutation-requires-where': 'error',
        'no-missing-limit': 'warn'
      }
    }),
    budgets({ maxRows: 10_000, maxLatencyMs: 150 }),
    telemetry({ sink: console })
  ]
})

// plan produced by any lane
const plan = sql().from('user').where(t.user.active.eq(true)).select({ id: t.user.id }).build()

const rows = await rt.execute(plan)
```

## Extensibility Guide

- New plugins implement any subset of hooks and declare a name
- Keep plugins pure and deterministic with respect to inputs
- Prefer reading annotations and refs to inspecting SQL text
- For dialect or lowering concerns, contribute via an adapter profile, not a runtime plugin

## Testing Strategy

- Hook API contract tests with fake plugins and error injection
- Golden SQL and hash stability tests to ensure Plan immutability and identity do not drift
- Violation matrix ensuring each built-in rule behaves consistently under strict and permissive
- Benchmarks run in CI on tagged commits with budget alerts

## Open Questions

- Adaptive budgets that learn typical latency and row counts per `planId`
- Pluggable EXPLAIN interpreters per adapter to standardize cost hints
- Optional server-side guardrails in PPg that mirror on-prem defaults without configuration drift

  - Final shape of `RuntimeError`, code namespaces, mapping rule levels to outcomes, and upgrade policy
