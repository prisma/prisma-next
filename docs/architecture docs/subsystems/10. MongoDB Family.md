# MongoDB Family

## Overview

MongoDB is a database family in Prisma Next. The contract, ORM, execution pipeline, and plugin framework all generalize across fundamentally different data models â€” the same `ContractBase` domain structure, the same `Collection` chaining API, and the same plugin lifecycle work for both SQL and MongoDB. Family-specific concerns (query plan compilation, driver dispatch, storage mapping) are cleanly bounded behind shared interfaces.

**Responsibilities:** MongoDB-specific contract types (`MongoContract`), execution pipeline (`MongoQueryPlan`, `MongoDriver`, `MongoRuntimeCore`), codec registry (BSON codecs), ORM client, contract validation (`validateMongoContract()`).

**Non-goals:** Aggregation pipeline DSL (the ORM covers CRUD; pipelines are a lower-level escape hatch â€” see [Aggregation pipelines](#aggregation-pipelines)), PSL authoring for document schemas, migration runner (MongoDB collections are schemaless in the DDL sense â€” schema evolution uses data invariants, see [ADR 176](../adrs/ADR%20176%20-%20Data%20migrations%20as%20invariant-guarded%20transitions.md)), streaming subscriptions via change streams.

## How PN accommodates MongoDB

MongoDB's data model differs from SQL in ways that stress every layer of the stack. Each difference has a corresponding architectural response.

### Embedded documents

The idiomatic MongoDB pattern stores related data inside the parent document â€” a User contains an Address, a Post contains its Comments. There's no SQL equivalent. This changes how the ORM loads and mutates related data.

In the contract, embedding is a property of the *relation*, not the model. A relation with `"strategy": "embed"` means the related model lives nested inside the parent's document. A relation with `"strategy": "reference"` means the related model lives in its own collection, resolved at query time via `$lookup` or application-level stitching. The same model can be embedded in one parent and referenced from another. See [ADR 174](../adrs/ADR%20174%20-%20Aggregate%20roots%20and%20relation%20strategies.md).

An explicit `roots` section in the contract declares which models are aggregate roots â€” ORM entry points with their own collections. Models not in `roots` are accessed through relations (embedded) or through a base model (polymorphic variants). This distinction, which is invisible in SQL where every model has its own table, is essential for MongoDB.

In the ORM, embedded relations auto-project into the parent row without a separate query. Referenced relations use `$lookup` pipeline stages or multi-query stitching.

### No schema enforcement

MongoDB doesn't enforce types at the storage level. A field declared as `number` in the contract might contain a string in the database. The contract's guarantees describe intended structure, not database-enforced constraints.

`validateMongoContract()` performs three-layer validation at load time: structural (Arktype schema), domain (family-agnostic â€” validates roots, models, fields, discriminator/variants consistency, relation bidirectionality), and storage (Mongo-specific â€” STI enforcement for polymorphic variants). The domain validation layer is reusable across families.

### Mutation operators

SQL mutations assign values to columns (`SET col = value`). MongoDB mutations use operators (`$set`, `$inc`, `$push`, `$pull`, `$addToSet`) that express fine-grained, atomic patches. The ORM compiles plain data updates to `$set` and exposes Mongo-native atomic operations as family-specific extensions on the mutation surface.

### No referential integrity

MongoDB has no foreign keys, no cascading deletes, no database-level guarantees about cross-collection references. Referential integrity â€” cascading deletes, restrict constraints â€” must be enforced in application code to provide a symmetric experience across families.

### Polymorphism

Storing different document shapes in the same collection (single-collection inheritance) is a standard MongoDB pattern. The contract represents this with `discriminator` + `variants` on the base model and `base` on each variant. Persistence strategy (single-table vs. multi-table inheritance) is emergent from storage mappings, not declared explicitly. Discriminator narrowing produces polymorphic return types â€” querying a base model returns a union of variant row types. See [ADR 173](../adrs/ADR%20173%20-%20Polymorphism%20via%20discriminator%20and%20variants.md).

### Aggregation pipelines

Beyond basic CRUD, MongoDB uses aggregation pipelines for joins (`$lookup`), computed fields (`$project`), grouping (`$group`), and reshaping. These fill the same architectural role as the SQL query builder â€” a lower-level escape hatch for everything the ORM can't express. A type-safe pipeline builder is a future goal; raw pipeline execution is available now.

### Change streams

MongoDB change streams (built on the oplog) let applications subscribe to real-time data changes â€” inserts, updates, deletes. In SQL, real-time is typically an add-on. The runtime's lifecycle needs to accommodate unbounded streaming subscriptions alongside request-response queries.

## Family and target separation

MongoDB is its own family, not a target under a generic "document" family. The SQL family abstraction earns its keep because SQL databases genuinely share a common interface: the SQL query language, the relational model, and query semantics. There is no equivalent shared interface for "document databases" â€” MongoDB and Firestore have different query languages, different data organization models, and different capabilities.

The contract hierarchy follows the domain/storage separation from [ADR 172](../adrs/ADR%20172%20-%20Contract%20domain-storage%20separation.md):

```
ContractBase (shared domain: roots, models with fields/discriminator/variants, relations)
â”śâ”€â”€ SqlContract (model.storage: field â†’ column; top-level storage: tables, columns, indexes)
â”‚   â””â”€â”€ Targets: Postgres, MySQL, SQLite...
â””â”€â”€ MongoContract (model.storage: collection name; top-level storage: collections)
    â””â”€â”€ Target: MongoDB
```

Splitting family from target (even with a single Mongo target) keeps extension points and seams explicit â€” the interface/concretion boundary stays clean rather than collapsing into a monolith.

## Contract

The contract's domain/storage separation enables a shared `ContractBase` across families. A Mongo contract looks like this:

```json
{
  "roots": {
    "users": "User",
    "posts": "Post"
  },
  "models": {
    "User": {
      "fields": {
        "id": { "nullable": false, "codecId": "mongo/objectId@1" },
        "email": { "nullable": false, "codecId": "mongo/string@1" }
      },
      "relations": {
        "posts": {
          "to": "Post", "cardinality": "1:N", "strategy": "reference",
          "on": { "localFields": ["id"], "targetFields": ["authorId"] }
        },
        "addresses": { "to": "Address", "cardinality": "1:N", "strategy": "embed" }
      },
      "storage": { "collection": "users" }
    },
    "Address": {
      "fields": {
        "street": { "nullable": false, "codecId": "mongo/string@1" },
        "city": { "nullable": false, "codecId": "mongo/string@1" }
      },
      "relations": {},
      "storage": {}
    }
  }
}
```

**Domain level** (`roots`, `model.fields`, `model.relations`, `model.discriminator`/`variants`) is family-agnostic in structure. `model.fields` is a record of `{ nullable: boolean, codecId: string }`. The domain section is self-describing: reading it gives a complete picture of the application's data model without consulting storage. "Family-agnostic" describes the *structure*, not the *values* â€” a Mongo field says `"mongo/string@1"` and a SQL field says `"pg/text@1"`, but the shape is identical. See [ADR 172](../adrs/ADR%20172%20-%20Contract%20domain-storage%20separation.md).

**Storage bridge** (`model.storage`) is family-specific. SQL maps fields to columns because SQL has genuine field-to-column indirection. Mongo needs only the collection name â€” document fields match domain fields directly. An embedded model like `Address` above has empty storage (`{}`) because it doesn't own a collection; its data lives inside the parent's document. See [contract-symmetry.md](../../planning/mongo-target/1-design-docs/contract-symmetry.md) for the full convergence/divergence analysis.

## ORM

The ORM uses the same `Collection` chaining API across families. Query state accumulates through immutable method chaining; terminal methods compile to family-specific query plans.

```typescript
const db = mongoOrm({ contract, runtime });

class UserCollection extends Collection<Contract, 'User'> {
  admins() { return this.where({ kind: 'admin' }); }
  byEmail(email: string) { return this.where({ email }); }
}

const tasks = await db.tasks
  .where((task) => task.assigneeId.eq('u1'))
  .include('assignee')
  .orderBy([(task) => task.createdAt.desc()])
  .take(10)
  .all();

const user = await db.users.byEmail('alice@example.com').first();
```

The consumer-facing API is identical to the SQL ORM. What differs is the internal plumbing:

| Concern | SQL | Mongo |
|---|---|---|
| **Terminal compilation** | `CollectionState` â†’ `SqlQueryPlan` (SQL AST) | `CollectionState` â†’ `MongoQueryPlan` (`FindCommand` / `AggregateCommand`) |
| **Include resolution** | Lateral joins, correlated subqueries, multi-query stitching | `$lookup` pipeline stages; embedded relations auto-projected |
| **Where expressions** | SQL AST nodes (`AnyWhereExpr`) | Mongo filter documents (`MongoExpr`) |
| **Field mapping** | Column remapping via `model.storage.fields` | Identity (domain fields = document fields) |
| **Mutations** | `INSERT...RETURNING`, `ON CONFLICT`, FK cascades | `insertOne`, `updateOne` with operators (`$set`, `$inc`, `$push`) |

Row type inference, custom collection subclasses, and the `include` interface are shared patterns across families. See [ADR 175](../adrs/ADR%20175%20-%20Shared%20ORM%20Collection%20interface.md).

## Execution pipeline

Each family gets its own plan type, driver, and runtime. The plugin lifecycle (`beforeExecute â†’ onRow â†’ afterExecute`) and metadata (`PlanMeta`) are shared across families; the plan content and driver dispatch are family-specific. The framework layer requires no family-specific changes.

- **`MongoQueryPlan`** â€” pairs a `MongoCommand` (discriminated union: `FindCommand`, `InsertOneCommand`, `UpdateOneCommand`, `DeleteOneCommand`, `AggregateCommand`) with `PlanMeta`.
- **`MongoDriver`** â€” wraps the `mongodb` Node.js driver, dispatches commands to the correct driver method, returns `AsyncIterable<Document>`.
- **`MongoRuntimeCore`** â€” validates the plan, calls the driver, wraps results in `AsyncIterableResult<Row>`.
- **`MongoCodecRegistry`** â€” base codecs (`objectId`, `string`, `int32`, `boolean`, `date`) following the same registry shape as SQL.

### Package layout

Mongo packages split across two domains â€” family (abstractions) and target (concretions):

```
packages/2-mongo-family/
  1-core/            -- MongoContract types, MongoQueryPlan, MongoCodec interfaces
  4-orm/             -- Mongo ORM client (findMany, include, polymorphic queries)
  5-runtime/         -- MongoRuntime, MongoPlugin interface

packages/3-mongo-target/
  1-mongo-target/    -- Target pack definition
  2-mongo-adapter/   -- Concrete codecs (objectId, string, int32, boolean, date)
  3-mongo-driver/    -- MongoDriver (wraps the mongodb Node.js driver)
```

Package boundaries are enforced by `pnpm lint:deps` â€” `2-mongo-family` and `3-mongo-target` must not import `2-sql` or `3-extensions` packages.

## Open questions

These are design refinements, not architectural risks. The full analysis is in [design-questions.md](../../planning/mongo-target/1-design-docs/design-questions.md).

- **Update operators**: How the ORM mutation surface accommodates `$inc`, `$push`, `$pull` alongside plain data updates. No SQL equivalent.
- **Referential integrity**: Application-level enforcement of cascading deletes, restrict constraints when the database provides no foreign key guarantees.
- **Read-time validation**: What happens when a document doesn't match the contract â€” reject, coerce, or warn?
- **Aggregation pipeline DSL**: The escape-hatch query surface for Mongo, symmetric to the SQL query builder. Type-safe pipeline builder vs. raw pipeline passthrough.
- **Change streams**: How the runtime's lifecycle accommodates unbounded streaming subscriptions alongside request-response queries.
- **Value types / composites**: Embedded value objects (e.g., Address) currently sit in `models` with `storage: {}`. A separate `types`/`composites` section would distinguish entities (have identity) from value types (interchangeable by field values).
- **Polymorphic associations**: Polymorphism on relations (a Comment belonging to Post or Video). No contract representation yet.
- **Discriminator values are untyped strings**: The contract stores discriminator values as strings, but something in the system must convert them to native DB values (e.g., SQL column values).

## Testing

- **Provisioning**: [mongodb-memory-server](https://github.com/typegoose/mongodb-memory-server) â€” runs a real `mongod` binary in-process, no container orchestration. Configured as a replica set.
- **Type-level tests**: Contract and ORM API verified with TypeScript files that must typecheck against hand-crafted contract types.
- **Integration tests**: Query execution against a running MongoDB instance via the `mongodb` Node.js driver.
