# MongoDB Family

## Overview

Prisma Next supports multiple database families. SQL was the first; MongoDB is the second. Supporting MongoDB matters because its data model is fundamentally different from SQL — documents instead of rows, embedded subdocuments instead of joins, no schema enforcement, no foreign keys. If the architecture can accommodate MongoDB without breaking its core abstractions, it generalizes to other non-SQL databases too.

This document explains what makes MongoDB different from SQL, how each difference is handled in the contract, ORM, and execution pipeline, and where the boundaries of the current design are.

**Responsibilities:** MongoDB-specific contract types (`MongoContract`), execution pipeline (`MongoQueryPlan`, `MongoDriver`, `MongoRuntimeCore`), codec registry (BSON codecs), ORM client, and contract validation (`validateMongoContract()`).

**Migrations are supported.** MongoDB does have server-side schema state — indexes, JSON Schema validators, collection options (capped, time-series, collation) — and Mongo participates in the migration system on the same footing as SQL targets. `prisma-next migration new --target mongo` and `prisma-next migration plan --target mongo` scaffold a runnable class-flow `migration.ts` (extending `Migration` from `@prisma-next/family-mongo/migration`). Three paths produce identical `ops.json` artifacts — the authoritative migration contract (see [ADR 192](../adrs/ADR%20192%20-%20ops.json%20is%20the%20migration%20contract.md)): inline emit at the end of `migration plan`, self-emit via `./migration.ts`, and the transitional `migration emit` CLI command (see [ADR 193](../adrs/ADR%20193%20-%20Class-flow%20as%20the%20canonical%20migration%20authoring%20strategy.md) for why CLI emit is transitional). Data migrations are authored as `dataTransform({ check, run })` operations whose pipelines invariant-guard the transition (see [ADR 176](../adrs/ADR%20176%20-%20Data%20migrations%20as%20invariant-guarded%20transitions.md)). See the [Migration System](7.%20Migration%20System.md) subsystem doc for the full canonical pipeline.

## At a glance

A Mongo contract describes the same things as a SQL contract — models, fields, relations — but with MongoDB-specific storage details. Here's an example to ground the concepts that follow:

```json
{
  "roots": {
    "users": "User",
    "posts": "Post"
  },
  "models": {
    "User": {
      "fields": {
        "_id": { "nullable": false, "codecId": "mongo/objectId@1" },
        "email": { "nullable": false, "codecId": "mongo/string@1" }
      },
      "relations": {
        "posts": {
          "to": "Post", "cardinality": "1:N",
          "on": { "localFields": ["_id"], "targetFields": ["authorId"] }
        }
      },
      "storage": { "collection": "users" }
    },
    "Post": {
      "fields": {
        "_id": { "nullable": false, "codecId": "mongo/objectId@1" },
        "title": { "nullable": false, "codecId": "mongo/string@1" },
        "authorId": { "nullable": false, "codecId": "mongo/objectId@1" }
      },
      "relations": {
        "author": {
          "to": "User", "cardinality": "N:1",
          "on": { "localFields": ["authorId"], "targetFields": ["_id"] }
        },
        "comments": { "to": "Comment", "cardinality": "1:N" }
      },
      "storage": {
        "collection": "posts",
        "relations": {
          "comments": { "field": "comments" }
        }
      }
    },
    "Comment": {
      "owner": "Post",
      "fields": {
        "_id": { "nullable": false, "codecId": "mongo/objectId@1" },
        "text": { "nullable": false, "codecId": "mongo/string@1" },
        "createdAt": { "nullable": false, "codecId": "mongo/date@1" }
      },
      "relations": {},
      "storage": {}
    }
  }
}
```

A few things to notice before reading further:

- **`roots`** declares which models are ORM entry points — `db.users` and `db.posts`. Comment is not in `roots` because it's only reachable through Post.
- **`owner: "Post"`** on the Comment model means Comment belongs to Post's aggregate. Its data lives inside Post documents. The relation from Post to Comment (`{ "to": "Comment", "cardinality": "1:N" }`) is a plain graph edge — no storage annotation. The physical location is in Post's `storage.relations`: `"comments": { "field": "comments" }`.
- **Comment has `storage: {}`** — empty, because it doesn't own a collection. Its data lives within Post's storage. Note that Comment is still an entity with its own `_id` — it has identity even when embedded. Value objects (data without identity, like an address or a money amount) are a separate concept with their own contract section — see [ADR 178](../adrs/ADR%20178%20-%20Value%20objects%20in%20the%20contract.md).
- **Field structure is identical to SQL.** Each field is `{ nullable, codecId }`. A Mongo field says `"mongo/string@1"` where SQL would say `"pg/text@1"`, but the shape is the same.
- **Storage is much thinner than SQL.** SQL needs field-to-column mappings; Mongo just needs the collection name, because document fields match domain fields directly.

The rest of this document explains each of these concepts in detail.

## Design principles

These principles shape all contract and ORM design decisions — not just for MongoDB, but across families. MongoDB forced each of them to be explicit because its data model doesn't let any of them stay implicit the way SQL does.

**1. Domain and persistence are separated.** The contract has two layers. The domain layer (`roots`, `models` with `fields`/`relations`/`discriminator`/`variants`) describes what the application models and how they relate. The storage layer (`model.storage`, top-level `storage`) describes how things persist. In the example above, `User.fields` says "a User has an id and an email" (domain); `User.storage` says "Users live in the `users` collection" (persistence). See [ADR 172](../adrs/ADR%20172%20-%20Contract%20domain-storage%20separation.md).

**2. The domain model is self-describing.** Reading `roots`, `models`, and `relations` gives a complete picture of the application domain without consulting the storage block. A reader can understand what the application models, how they relate, and which are entry points — all before knowing anything about tables, columns, or collections.

**3. Family-specific details are scoped.** The domain structure is family-agnostic — it has the same shape for SQL and MongoDB. Family-specific persistence details are confined to `model.storage` and the top-level `storage` section. "Family-agnostic" describes the *structure*, not the *values*: codec IDs like `"mongo/string@1"` are family-specific values in a family-agnostic structure.

**4. The contract describes facts, not instructions.** The contract states what exists — models, fields, discriminators, storage units. The ORM decides how to represent these at runtime. Persistence strategies like single-table inheritance vs. multi-table inheritance are emergent from the combination of domain declarations and storage mappings, not declared explicitly.

**5. Each model has one canonical storage location.** A model (entity) belongs to exactly one aggregate. It either owns its own collection/table (aggregate root) or it lives inside another model's storage (embedded entity). This is a consistency boundary: if a model could live in two places, two consumers could independently mutate it, breaking invariant guarantees. When you need data from one aggregate root available inside another for read performance, that's denormalization — the embedded copy is a snapshot (a value object), not the model itself. The canonical model lives in one place.

**6. Ownership is a domain fact stated on the model.** An owned model declares `"owner": "ParentModel"` — a domain fact about aggregate membership. This mirrors how `base` declares polymorphic specialization. Relations to owned models are plain graph edges (no storage annotations); the parent's `storage.relations` maps them to physical locations. See [ADR 177](../adrs/ADR%20177%20-%20Ownership%20replaces%20relation%20strategy.md).

## How PN accommodates MongoDB

MongoDB's data model stresses every layer of the PN stack in ways that SQL doesn't. This section walks through each major difference and explains how the architecture handles it.

### Embedded documents

In SQL, related data lives in separate tables and is joined at query time. In MongoDB, the idiomatic pattern is to store related data *inside* the parent document — a Post document contains an array of Comment subdocuments. This means the ORM needs to know the difference between "this relation points to another collection" and "this relation is stored inside the parent document."

The contract captures this with the `owner` property on the owned model:

- An owned model declares `"owner": "ParentModel"` — a domain-level fact stating that the model belongs to the parent's aggregate. Its `storage` block is empty (`{}`), meaning "I don't own a collection; my data lives within my owner's storage."
- The parent's `storage.relations` maps the relation to its physical storage location: `"comments": { "field": "comments" }` for Mongo, `"comments": { "column": "comments_data" }` for SQL JSONB.
- Relations to owned models are plain graph edges: `{ "to": "Comment", "cardinality": "1:N" }` — no storage annotation on the relation itself.

A referenced relation (to a model with its own collection) carries `on` join details: `{ "to": "Post", "cardinality": "1:N", "on": { "localFields": ["_id"], "targetFields": ["authorId"] } }`. The absence of `owner` on the target model — plus the presence of `on` — distinguishes references from owned relations. See [ADR 177](../adrs/ADR%20177%20-%20Ownership%20replaces%20relation%20strategy.md).

This also introduces the concept of **aggregate roots**: models that own their own collection and can be queried directly. In SQL, every model has its own table, so this distinction is invisible. In MongoDB, some models are owned by a parent and can only be reached through it. The contract makes this explicit with a top-level `roots` section that maps ORM accessor names (like `db.users`) to model names (like `User`). If a model isn't in `roots`, it's not an independent entry point. See [ADR 174](../adrs/ADR%20174%20-%20Aggregate%20roots%20and%20relation%20strategies.md).

Each model has exactly one canonical storage location — either its own collection (aggregate root) or inside a parent's document (owned). A model can't be both. When developers need data from one aggregate root available inside another for read performance (the MongoDB "extended reference" pattern), the embedded copy is a denormalized snapshot — a value object — not the model itself. The canonical Post lives in the `posts` collection; a `{ title, slug }` summary embedded in a User document is a different thing, even though it shares some fields. Value objects are defined in the contract's `valueObjects` section — see [ADR 178](../adrs/ADR%20178%20-%20Value%20objects%20in%20the%20contract.md). See also [design principle #5](#design-principles).

In the ORM, owned relations auto-project into the parent row — no separate query is issued. Referenced relations use `$lookup` pipeline stages or multi-query stitching, similar to how SQL handles joins.

### No schema enforcement

SQL databases enforce types at the storage level: if a column is declared as `integer`, you can't store a string in it. MongoDB has no such enforcement by default. A field the contract says is a number might contain a string in the actual database. The contract describes intended structure, not database-enforced constraints. However, the migration planner can generate `$jsonSchema` validators for collections, providing opt-in type enforcement at the database level — see [Schema validation](#schema-validation-jsonschema).

`validateMongoContract()` validates the contract itself (not the data in the database) at load time. It performs three layers of validation:

1. **Structural** — does the contract JSON match the expected shape? (Arktype schema validation)
2. **Domain** — are the roots, models, fields, relations, and discriminator/variants internally consistent? Are relations bidirectional? This layer is family-agnostic and reusable for SQL.
3. **Storage** — are Mongo-specific rules respected? For example, polymorphic variants using single-table inheritance must share their base model's collection.

### Mutation operators

SQL mutations assign values to columns: `SET name = 'Alice'`. MongoDB mutations use *operators* that express fine-grained, atomic patches: `$set` (assign a value), `$inc` (increment a number), `$push` (append to an array), `$pull` (remove from an array), `$addToSet` (append only if not already present).

The ORM supports three mutation forms, each with clear semantics:

```typescript
// Plain object update — partial: only specified fields change
db.posts.where({ id }).update({ title: "New Title" })
// Compiles to: $set: { title: "New Title" }

// Field accessor — per-field Mongo-native operations
db.posts.where({ id }).update(u => [
  u("stats.views").inc(1),
  u("tags").push("featured"),
  u("metadata.lastEdited").set(new Date()),
])
// Compiles to: { $inc: { "stats.views": 1 }, $push: { "tags": "featured" }, $set: { "metadata.lastEdited": ... } }
```

The dot-path field accessor ([ADR 180](../adrs/ADR%20180%20-%20Dot-path%20field%20accessor.md)) maps directly to MongoDB's native update operators — `u("field").inc(1)` becomes `$inc: { "field": 1 }`. Mutation operators are capability-gated: Mongo gets the full suite (`inc`, `push`, `pull`, `addToSet`, `pop`, etc.); SQL is limited to `set` and `unset` for JSONB paths.

### No referential integrity

MongoDB has no foreign keys, no cascading deletes, no database-level guarantees that a referenced document actually exists. If a Post references an Author by `authorId`, nothing stops the Author from being deleted while the Post still points to it.

To provide a symmetric experience across families, referential integrity — cascading deletes, restrict constraints — is enforced in application code rather than relying on database features.

### Polymorphism

Storing different document shapes in the same collection is a standard MongoDB pattern. A `notifications` collection might contain email notifications, SMS notifications, and push notifications — each with different fields, but all sharing an `_id` and a `type` discriminator field.

The contract represents this with three concepts:

- A **discriminator** on the base model: the field name and its possible values (`type: "email" | "sms" | "push"`)
- **Variants**: the specialized models with their type-specific fields
- A **base** reference on each variant, pointing back to the base model

Persistence strategy (whether variants share a collection or each have their own) is emergent from the storage mappings, not declared explicitly. When variants share a base collection (single-collection polymorphism via `@@base`), the interpreter suppresses separate collection entries for variant models and merges any variant-specific indexes into the base collection. Discriminator narrowing produces polymorphic return types — querying a base model returns a union of variant row types, and filtering by discriminator value narrows the type. See [ADR 173](../adrs/ADR%20173%20-%20Polymorphism%20via%20discriminator%20and%20variants.md).

### Schema validation ($jsonSchema)

MongoDB collections can enforce document structure using `$jsonSchema` validators. The migration planner generates these validators from the contract so that documents inserted or updated in a collection must conform to the expected field types.

BSON types in validators are derived from codec metadata, not hardcoded maps. When the interpreter generates a validator for a collection, it resolves each field's codec ID through the `CodecLookup` and reads the codec's `targetTypes[0]` to get the BSON type name (e.g., `mongo/double@1` → `double`, `mongo/string@1` → `string`). This means any codec registered by an adapter or extension automatically gets correct validator support without modifying the interpreter.

For polymorphic collections (models with `@@discriminator` and `@@base`), the validator uses a `oneOf` pattern: base fields appear in the top-level `properties` and `required`, while variant-specific fields are grouped into `oneOf` branches, each constrained by the discriminator value. This ensures that a document in the collection must match exactly one variant shape.

### Aggregation pipelines

MongoDB uses aggregation pipelines — ordered sequences of transformation stages — for joins (`$lookup`), computed fields (`$project`), grouping (`$group`), and reshaping. These are MongoDB's equivalent of complex SQL queries.

**All MongoDB read queries compile to aggregation pipelines exclusively.** The older `find()` API is not used. This is a binding design constraint — see [ADR 183](../adrs/ADR%20183%20-%20Aggregation%20pipeline%20only,%20never%20find%20API.md). The rationale: `find()` is a strict subset of `aggregate()`, maintaining two code paths doubles compilation and lowering logic for no architectural benefit, and the typed pipeline stage representation is the natural foundation for both the ORM and the future pipeline query builder.

A typed query builder that produces the same pipeline stages ships as `@prisma-next/mongo-query-builder`. Its surface is wider than "pipeline builder" suggests — it covers writes, find-and-modify, and `$merge` / `$out` in addition to reads — and its chain shape is documented below in [Query builder](#query-builder).

### Change streams

MongoDB change streams (built on the oplog) let applications subscribe to real-time data changes — inserts, updates, deletes — on a collection, database, or entire deployment. In SQL, real-time notifications are typically an add-on (e.g., Postgres `LISTEN`/`NOTIFY`). In MongoDB, they're a first-class primitive.

The runtime's lifecycle needs to accommodate these unbounded streaming subscriptions alongside normal request-response queries.

## Family and target separation

MongoDB is its own *family*, not a target under a generic "document" family. The SQL family abstraction earns its keep because SQL databases genuinely share a common interface: the SQL query language, the relational model, and standard query semantics. There is no equivalent shared interface among "document databases" — MongoDB and Firestore have different query languages, different data organization models, and different capabilities. If Firestore support were added, it would be its own family too.

The contract hierarchy reflects the domain/storage separation described in [ADR 172](../adrs/ADR%20172%20-%20Contract%20domain-storage%20separation.md):

```text
ContractBase (shared domain: roots, models with fields/discriminator/variants, relations)
├── SqlContract (model.storage: field → column; top-level storage: tables, columns, indexes)
│   └── Targets: Postgres, MySQL, SQLite...
└── MongoContract (model.storage: collection name; top-level storage: collections)
    └── Target: MongoDB
```

Even though there's currently only one Mongo target (MongoDB itself), splitting family from target keeps extension points and seams explicit. If a MongoDB-compatible database with different driver requirements appeared, it would slot in as a second target under the Mongo family without changing the family-level abstractions.

## Contract

Refer to the [contract example above](#at-a-glance) for a concrete illustration. The contract's domain/storage separation (described in [ADR 172](../adrs/ADR%20172%20-%20Contract%20domain-storage%20separation.md)) enables a shared `ContractBase` across families. The key design points:

**The domain level is family-agnostic in structure.** `roots`, `model.fields`, `model.relations`, `model.discriminator`/`variants` — these all have the same shape regardless of whether the contract is for SQL or MongoDB. "Family-agnostic" describes the *structure* of the domain section, not the *values* — codec IDs like `"mongo/string@1"` vs `"pg/text@1"` are family-specific, but the `{ nullable, codecId }` shape is identical. This means consumer code that reads the domain level (ORM clients, validation, tooling) can work against `ContractBase` without knowing which family produced the contract.

**The storage section is much thinner than SQL.** In SQL, `model.storage` maps each field to a column name, because SQL has genuine field-to-column indirection (a field called `email` might map to a column called `user_email`). In MongoDB, document fields match domain fields directly, so `model.storage` only needs the collection name. An embedded model like `Comment` has empty storage (`{}`) because it doesn't own a collection — its data lives inside the parent's document.

For a detailed analysis of where Mongo and SQL contracts converge and diverge, see the contract symmetry analysis linked in [References](#references).

## ORM

The ORM presents the same chaining API to consumers regardless of database family. A developer querying MongoDB uses the same `where()`, `select()`, `include()`, `orderBy()`, `take()`, and `all()` methods as one querying Postgres:

```typescript
const db = mongoOrm({ contract, runtime });

class UserCollection extends Collection<Contract, 'User'> {
  admins() { return this.where({ kind: 'admin' }); }
  byEmail(email: string) { return this.where({ email }); }
}

const tasks = await db.tasks
  .where((task) => task.assigneeId.eq('u1'))
  .include('assignee')
  .orderBy([(task) => task.createdAt.desc()])
  .take(10)
  .all();

const user = await db.users.byEmail('alice@example.com').first();
```

Each method call returns a new `Collection` with accumulated state (filters, includes, ordering, limits). Nothing hits the database until a terminal method like `.all()`, `.first()`, or a mutation terminal is called. At that point, the accumulated `CollectionState` is compiled into a family-specific query plan — `SqlQueryPlan` for SQL, `MongoQueryPlan` for MongoDB (reads and writes both use the same plan type and runtime `execute(plan)` path).

The consumer-facing API is identical. What differs is the internal plumbing:


| Concern                  | SQL                                                         | Mongo                                                                     |
| ------------------------ | ----------------------------------------------------------- | ------------------------------------------------------------------------- |
| **Terminal compilation** | `CollectionState` → `SqlQueryPlan` (SQL AST)                | `CollectionState` → `MongoQueryPlan` with an `AnyMongoCommand` (`AggregateCommand` for reads — [ADR 183](../adrs/ADR%20183%20-%20Aggregation%20pipeline%20only,%20never%20find%20API.md); write commands for mutations) |
| **Include resolution**   | Lateral joins, correlated subqueries, multi-query stitching | `$lookup` pipeline stages; embedded relations auto-projected              |
| **Where expressions**    | SQL AST nodes (`AnyWhereExpr`)                              | Mongo filter AST (`MongoFilterExpr` in `mongo-query-ast`)                 |
| **Field mapping**        | Column remapping via `model.storage.fields`                 | Identity (domain fields = document fields)                                |
| **Mutations**            | `INSERT...RETURNING`, `ON CONFLICT`, FK cascades            | `insertOne`, `updateOne` with operators (`$set`, `$inc`, `$push`)         |


Row type inference, custom collection subclasses, and the `include` interface all work the same way across families. For the full design rationale, see [ADR 175](../adrs/ADR%20175%20-%20Shared%20ORM%20Collection%20interface.md).

## Query builder

Below the ORM and above the raw command AST sits `@prisma-next/mongo-query-builder`: a contract-bound, typed CRUD surface that produces the same `MongoQueryPlan` values the ORM emits. The ORM is model-centric and multi-collection (roots, includes, repositories); the query builder is collection-centric and single-command (reads, writes, find-and-modify, `$merge` / `$out`). Migrations, raw tooling, and other non-ORM callers author plans through the query builder rather than hand-building AST nodes.

```typescript
import { acc, mongoQuery } from '@prisma-next/mongo-query-builder';

const q = mongoQuery<Contract>({ contractJson });

// Read — aggregation pipeline
const analytics = q
  .from('orders')
  .match((f) => f.status.eq('completed'))
  .group((f) => ({ _id: f.department, total: acc.sum(f.amount) }))
  .build();

// Write — filtered update
const updated = q
  .from('orders')
  .match((f) => f.total.gt(100))
  .updateMany((f) => [f.status.set('shipped')]);
```

### Three states

The chain is a state machine over three concrete classes. Each class exposes a distinct vocabulary; transitions are encoded in return types.

- **`CollectionHandle<TContract, ModelName>`** — entered via `q.from(name)`. Inserts, unqualified writes (`updateAll` / `deleteAll`), unqualified `upsertOne(filter, updater)`. Transitions out via `.match(...)` into `FilteredCollection`, or into `PipelineChain` via any pipeline-stage call.
- **`FilteredCollection<TContract, ModelName>`** — one or more `.match(...)` calls on a handle (or chained match). Filtered writes (`updateMany` / `updateOne` / `deleteMany` / `deleteOne`), `upsertOne`, `findOneAndUpdate` / `findOneAndDelete`, further `.match(...)` (AND-folded), or a transition into `PipelineChain`.
- **`PipelineChain<TContract, Shape, U, F>`** — any pipeline-stage call on the previous two states. Pipeline stage methods, `$merge` / `$out` write terminals, and the read terminals `.build()` / `.aggregate()`. Pipeline-style writes and find-and-modify are *conditional* — available only when the phantom markers `U` and `F` are still in their "enabled" state (see the table below).

### Marker pattern

Within `PipelineChain`, two phantom type parameters gate the conditional terminals:

- **`UpdateEnabled`** (`'update-ok'` | `'update-cleared'`) — gates `.updateMany()` / `.updateOne()` no-arg form (the chain consumed as an update-with-pipeline spec).
- **`FindAndModifyEnabled`** (`'fam-ok'` | `'fam-cleared'`) — gates `.findOneAndUpdate(...)` / `.findOneAndDelete(...)` (the chain deconstructed into the wire command's `{ filter, sort }` slots).

Every pipeline-stage method declares what it does to the markers. The full table:

| Stage method                | `UpdateEnabled` | `FindAndModifyEnabled` |
| --------------------------- | --------------- | ---------------------- |
| `.match(...)`               | preserve        | preserve               |
| `.sort(...)`                | clear           | preserve               |
| `.skip(...)`                | clear           | clear                  |
| `.limit(...)`               | clear           | clear                  |
| `.sample(...)`              | clear           | clear                  |
| `.addFields(...)`           | preserve        | clear                  |
| `.project(...)`             | preserve        | clear                  |
| `.replaceRoot(...)`         | preserve        | clear                  |
| `.redact(...)`              | preserve        | clear                  |
| `.group(...)`               | clear           | clear                  |
| `.lookup(...)`              | clear           | clear                  |
| `.unwind(...)`              | clear           | clear                  |
| `.facet(...)`               | clear           | clear                  |
| other shape-changing stages | clear           | clear                  |

The rationale for each entry is the same: a marker is cleared iff the stage produces content the downstream wire command cannot lower. For example, `.sort(...)` preserves `FindAndModifyEnabled` because `findAndModify` has a `sort` slot; `.limit(...)` clears it because `findAndModify` already implies single-document semantics and `.limit(10).findOneAndUpdate(...)` is ambiguous rather than useful.

See [ADR 201](../adrs/ADR%20201%20-%20State-machine%20pattern%20for%20typed%20DSL%20builders.md) for the decision record, including a worked example and the preconditions for reusing this pattern in other typed DSL builders (e.g. a future typed SQL query builder).

### Terminals

Each terminal returns a `MongoQueryPlan<Result>`, where `Result` is the resolved row shape for reads, the affected-count shape for writes, and the returned-document shape for find-and-modify.

- **Insert terminals** — `insertOne`, `insertMany` on `CollectionHandle`. Typecheck the input against the contract's input types.
- **Unqualified writes** — `updateAll`, `deleteAll` on `CollectionHandle`. Take no filter argument; the method name is the filter's declaration that scope is "every document". Grep-visible on purpose.
- **Filtered writes** — `updateMany`, `updateOne`, `deleteMany`, `deleteOne` on `FilteredCollection`. Consume the AND-folded accumulated filter.
- **Upserts** — `upsertOne` on `CollectionHandle` (takes filter and updater) and `FilteredCollection` (takes updater). `upsertMany` is explicitly excluded — Mongo's `updateMany` with `upsert: true` inserts at most one document, so the name would mislead.
- **Find-and-modify** — `findOneAndUpdate`, `findOneAndDelete` on `FilteredCollection` (always) and on `PipelineChain` (gated by `FindAndModifyEnabled`).
- **Pipeline-style updates** — no-arg `updateMany()` / `updateOne()` on `PipelineChain`, gated by `UpdateEnabled`. The chain *is* the update pipeline.
- **`$merge` / `$out`** — `merge(opts)` / `out(collection)` on `PipelineChain`, always available. Pipeline-terminal writes.
- **Read terminals** — `.build()` / `.aggregate()` on `PipelineChain`. Every collection handle and filtered collection lazily materialises into a pipeline if `.aggregate()` is called directly.
- **`rawCommand` escape hatch** — `q.rawCommand(cmd: AnyMongoCommand)` packages a hand-constructed command into a plan. For non-CRUD wire commands (`collStats`, `dbStats`) and for commands the typed API doesn't yet cover.

### Field accessor

Stage callbacks (`match`, `addFields`, `project`, `group`, `update*`, `upsertOne`, `findOneAndUpdate`) receive a single `FieldAccessor<Shape>` — conventionally named `f` — that exposes both filter operators (`eq`, `ne`, `gt`, `exists`, `regex`, …) and Mongo update operators (`set`, `unset`, `inc`, `mul`, `push`, `addToSet`, …). The accessor supports both property access (`f.status`) for flat contract fields and a callable form (`f("stats.views")`) for value-object dot-paths. Per [ADR 180](../adrs/ADR%20180%20-%20Dot-path%20field%20accessor.md), this is one accessor across filters and updates, which is what let the Mongo builder collapse the earlier `FieldProxy` + `FilterProxy` split.

For the user-facing surface — every terminal, every stage, update-operator catalogue, examples — see the [package README](../../../packages/2-mongo-family/5-query-builders/query-builder/README.md).

## Execution pipeline

Each family provides its own plan type, driver, and runtime core. The framework layer — the plugin lifecycle (`beforeExecute → onRow → afterExecute`) and plan metadata (`PlanMeta`) — is shared across families and requires no family-specific changes.

The Mongo-specific components are:

- **`MongoQueryPlan`** — defined in `@prisma-next/mongo-query-ast` as `{ collection, command: AnyMongoCommand, meta: PlanMeta }`. `AnyMongoCommand` covers reads and writes: `AggregateCommand`, `InsertOneCommand`, `UpdateOneCommand`, `DeleteOneCommand`, `InsertManyCommand`, `UpdateManyCommand`, `DeleteManyCommand`, `FindOneAndUpdateCommand`, and `FindOneAndDeleteCommand`. Reads still use `AggregateCommand` with typed pipeline stages ([ADR 183](../adrs/ADR%20183%20-%20Aggregation%20pipeline%20only,%20never%20find%20API.md)); `AggregateCommand.pipeline` accepts `ReadonlyArray<MongoReadStage | Record<string, unknown>>` so stages can be fully typed or raw BSON documents. Filter fields on commands accept `MongoFilterExpr` (typed AST); the adapter lowers them to wire-level filter documents.
- **`MongoAdapter.lower(plan)`** — single entry point `lower(plan: MongoQueryPlan): AnyMongoWireCommand` that lowers any plan to a wire command for the driver.
- **`MongoRuntime.execute(plan)`** — single entry point `execute<Row>(plan: MongoQueryPlan<Row>): AsyncIterableResult<Row>`. Both read and write operations flow through this path; the ORM’s `compileMongoQuery` returns a `MongoQueryPlan` (for reads, wrapping an `AggregateCommand`).
- **`MongoDriver`** — wraps the `mongodb` Node.js driver, dispatches each wire command type to the correct driver method, and returns results as `AsyncIterable<Document>`.
- **`MongoRuntime`** — validates the plan, calls the driver, and wraps results in `AsyncIterableResult<Row>` (the same streaming result type used by SQL).
- **`MongoCodecRegistry`** — provides base codecs for BSON types (`objectId`, `string`, `int32`, `boolean`, `date`, `double`), following the same registry shape as SQL codecs. Each codec declares its `targetTypes` (e.g., `['double']` for the double codec), which are used to derive BSON type names for `$jsonSchema` validators via `codecLookup.get(codecId).targetTypes[0]`.

### Package layout

Mongo packages are split across two domains, mirroring the family/target separation:

```text
packages/2-mongo-family/
  1-foundation/       -- MongoContract types, MongoCodec interfaces, MongoValue primitives
  2-authoring/
    contract-psl/     -- PSL interpretation into Mongo contract input
    contract-ts/      -- TypeScript `defineContract(...)` authoring surface
  3-tooling/emitter   -- Contract emitter
  4-query/query-ast/  -- MongoQueryPlan, filter expressions, command AST nodes
  5-query-builders/
    query-builder/    -- Typed mongoQuery surface: reads, writes, find-and-modify, $merge/$out; produces MongoQueryPlan
    orm/              -- Mongo ORM client: root-driven collections, includes, repositories; also produces MongoQueryPlan
  6-transport/        -- Wire commands, MongoAdapter/MongoDriver interfaces
  7-runtime/          -- MongoRuntime
  9-family/           -- Mongo family descriptor and family pack

packages/3-mongo-target/
  1-mongo-target/     -- Target pack definition and codec type surface
  2-mongo-adapter/    -- Concrete codecs, MongoAdapter implementation
  3-mongo-driver/     -- MongoDriver (wraps the mongodb Node.js driver)
```

Package boundaries are enforced by `pnpm lint:deps` — `2-mongo-family` and `3-mongo-target` must not import `2-sql` or `3-extensions` packages.

## Testing

- **Provisioning**: [mongodb-memory-server](https://github.com/typegoose/mongodb-memory-server) runs a real `mongod` binary in-process with no container orchestration needed. Configured as a replica set to support transactions and change streams.
- **Type-level tests**: Contract types and ORM API are verified with TypeScript files that must typecheck against hand-crafted contract types. If the types are wrong, the test file fails to compile.
- **Integration tests**: Query execution against a running MongoDB instance via the `mongodb` Node.js driver.
