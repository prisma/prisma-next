# MongoDB Family

## Overview

MongoDB is Prisma Next's second database family тАФ the architectural proof that the system generalizes beyond SQL. Every interface a consumer touches (contracts, query plans, plugins, the ORM client) was originally built for relational databases. The MongoDB workstream validated that these interfaces can accommodate a fundamentally different data model without breaking core abstractions.

MongoDB is also a strategic target. Prisma ORM's current Mongo support has significant gaps: no embedded document support, polymorphic fields fall back to untyped `Json`, no change streams, no aggregation pipeline access. PN has an opportunity to offer a genuinely Mongo-native experience.

**Responsibilities:** MongoDB-specific contract types (`MongoContract`), execution pipeline (`MongoQueryPlan`, `MongoDriver`, `MongoRuntimeCore`), codec registry (BSON codecs), ORM client, contract validation (`validateMongoContract()`).

**Non-goals (current):** Production-quality driver with connection pooling, aggregation pipeline DSL, PSL authoring for document schemas, migration runner (Mongo is schemaless in the DDL sense), streaming subscriptions (validated in the SQL runtime workstream via Supabase Realtime).

## What makes MongoDB different

MongoDB's data model stresses every layer of the PN stack in ways that SQL doesn't:

**Embedded documents replace joins.** The idiomatic MongoDB pattern stores related data inside the parent document тАФ a User contains an Address, a Post contains its Comments. This blurs the line between "relation" and "field" in the contract model, and changes how the ORM loads and mutates related data.

**No schema enforcement by default.** MongoDB doesn't enforce types at the storage level. A field declared as `number` in the contract might contain a string in the database. The contract's guarantees are aspirational, not structurally enforced тАФ the runtime must decide what to do when reality doesn't match.

**Mutations use operators, not assignment.** SQL mutations assign values to columns. MongoDB mutations use operators (`$set`, `$inc`, `$push`, `$pull`, `$addToSet`) that express fine-grained, atomic patches. The ORM mutation surface needs to accommodate both plain data updates (compiled to `$set`) and these Mongo-native atomic operations.

**No referential integrity.** MongoDB has no foreign keys, no cascading deletes, no database-level guarantees about cross-collection references. An ORM offering a symmetric experience across families must provide configurable referential integrity in application code.

**Polymorphism is idiomatic.** Storing different document shapes in the same collection is a standard MongoDB pattern. In SQL, single-table inheritance exists but is less common. In MongoDB it's everywhere. The contract and ORM need discriminated unions as a first-class concept.

**Aggregation pipelines are the native query language.** Beyond basic CRUD, MongoDB uses aggregation pipelines for joins (`$lookup`), computed fields (`$project`), grouping (`$group`), and reshaping. These fill the same architectural role as the SQL query builder тАФ a lower-level escape hatch for everything the ORM can't express.

**Real-time streaming is built in.** MongoDB change streams (built on the oplog) let applications subscribe to data changes. In SQL, real-time is an add-on. The runtime must accommodate streaming subscriptions alongside request-response queries.

## Architecture

### Mongo is its own family, not a target under "document"

The SQL family abstraction earns its keep because SQL databases genuinely share a common interface: the SQL query language, the relational model, and query semantics. There is no equivalent shared interface for "document databases" тАФ MongoDB and Firestore have different query languages, different data organization models, and different capabilities. If Firestore came later, it would be its own family.

The contract hierarchy follows the domain/storage separation from [ADR 172](../adrs/ADR%20172%20-%20Contract%20domain-storage%20separation.md):

```
ContractBase (shared domain: roots, models with fields/discriminator/variants, relations)
тФЬтФАтФА SqlContract (model.storage: field тЖТ column; top-level storage: tables, columns, indexes)
тФВ   тФФтФАтФА Targets: Postgres, MySQL, SQLite...
тФФтФАтФА MongoContract (model.storage: collection name; top-level storage: collections)
    тФФтФАтФА Target: MongoDB
```

Splitting family from target (even with a single Mongo target) keeps extension points and seams explicit тАФ the interface/concretion boundary stays clean rather than collapsing into a monolith.

### Execution pipeline

Each family gets its own plan type, driver, and runtime. The plugin lifecycle (`beforeExecute тЖТ onRow тЖТ afterExecute`) and metadata (`PlanMeta`) are shared; the plan content and driver dispatch are family-specific. No changes to the framework layer were needed to accommodate Mongo.

- **`MongoQueryPlan`** тАФ pairs a `MongoCommand` (discriminated union: `FindCommand`, `InsertOneCommand`, `UpdateOneCommand`, `DeleteOneCommand`, `AggregateCommand`) with `PlanMeta`.
- **`MongoDriver`** тАФ wraps the `mongodb` Node.js driver, dispatches commands to the correct driver method, returns `AsyncIterable<Document>`.
- **`MongoRuntimeCore`** тАФ validates the plan, calls the driver, wraps results in `AsyncIterableResult<Row>`.
- **`MongoCodecRegistry`** тАФ base codecs (`objectId`, `string`, `int32`, `boolean`, `date`) following the SQL registry shape.

### Package layout

Mongo packages are split across two domains тАФ family (abstractions) and target (concretions):

```
packages/2-mongo-family/
  1-core/            -- MongoContract types, MongoQueryPlan, MongoCodec interfaces
  4-orm/             -- Mongo ORM client (findMany, include, polymorphic queries)
  5-runtime/         -- MongoRuntime, MongoPlugin interface

packages/3-mongo-target/
  1-mongo-target/    -- Target pack definition
  2-mongo-adapter/   -- Concrete codecs (objectId, string, int32, boolean, date)
  3-mongo-driver/    -- MongoDriver (wraps the mongodb Node.js driver)
```

Package boundaries are enforced by `pnpm lint:deps` тАФ `2-mongo-family` and `3-mongo-target` must not import `2-sql` or `3-extensions` packages.

## Contract

The contract redesign separated domain-level structure from persistence, enabling a shared `ContractBase` across families.

**Domain level** (`roots`, `model.fields`, `model.relations`, `model.discriminator`/`variants`) тАФ family-agnostic structure. `model.fields` is a record of `{ nullable: boolean, codecId: string }`. The domain section is self-describing: reading it gives a complete picture of the application's data model without consulting storage. "Family-agnostic" describes the *structure*, not the *values* тАФ a Mongo field says `"mongo/string@1"` and a SQL field says `"pg/text@1"`, but the structure is identical. See [ADR 172](../adrs/ADR%20172%20-%20Contract%20domain-storage%20separation.md).

**Storage bridge** (`model.storage`) тАФ family-specific. SQL maps fields to columns because SQL has genuine field-to-column indirection. Mongo needs only the collection name тАФ document fields match domain fields directly. See [contract-symmetry.md](../../planning/mongo-target/1-design-docs/contract-symmetry.md) for the detailed convergence/divergence analysis.

**Polymorphism** тАФ `discriminator` + `variants` on the base model, `base` on each variant. Persistence strategy (STI vs MTI) is emergent from storage mappings. Uses specialization/generalization terminology. See [ADR 173](../adrs/ADR%20173%20-%20Polymorphism%20via%20discriminator%20and%20variants.md).

**Aggregate roots** тАФ explicit `roots` section mapping ORM accessor names to models. Embedding is a relation property (`"strategy": "embed"`), not a model property. See [ADR 174](../adrs/ADR%20174%20-%20Aggregate%20roots%20and%20relation%20strategies.md).

**Contract validation** тАФ `validateMongoContract()` performs three-layer validation: structural (Arktype schema), domain (family-agnostic тАФ validates roots, models, fields, discriminator/variants consistency, relation bidirectionality), and storage (Mongo-specific тАФ STI enforcement for polymorphic variants). The domain validation layer is reusable for SQL.

## ORM

The `Collection` class with fluent chaining (`.where().select().include().take().all()`) is the shared ORM interface across families. `CollectionState` accumulates query state through immutable method chaining; terminal methods compile to family-specific query plans. Row type inference, custom collection subclasses, and the `include` interface are shared patterns. Family-specific concerns are cleanly bounded to terminal compilation, include resolution strategy, where expression output, field mapping, and mutation compilation.

See [ADR 175](../adrs/ADR%20175%20-%20Shared%20ORM%20Collection%20interface.md).

## PoC results

The PoC validated that the Prisma Next architecture generalizes to a non-SQL family:

- **Execution pipeline generalizes.** Each family gets its own plan type, driver, and runtime. No framework-layer changes needed.
- **Contract structure generalizes.** Domain/storage separation works: the domain level is structurally identical across families. Only `model.storage` differs.
- **ORM consumer surface is a shared pattern.** Fluent chaining, `CollectionState`, row type inference, custom collections, and the `include` interface are all shared.
- **Polymorphism works end-to-end.** Discriminator narrowing, polymorphic return types, and STI enforcement validated with type-level and integration tests.
- **Embedded documents work without loading.** Relations with `"strategy": "embed"` auto-project into the parent row.
- **Contract validation works with redundancy.** Three-layer validation handles the intentional redundancy in the contract.

## Open questions

These are design refinement, not existential risks. The full analysis is in [design-questions.md](../../planning/mongo-target/1-design-docs/design-questions.md).

- **Update operators**: How the ORM mutation surface accommodates `$inc`, `$push`, `$pull`. No SQL equivalent.
- **Referential integrity**: Application-level enforcement of cascading deletes, restrict constraints.
- **Read-time validation**: What happens when a document doesn't match the contract.
- **Aggregation pipeline DSL**: The escape-hatch query surface for Mongo, symmetric to the SQL query builder.
- **Change streams**: Streaming lifecycle. Being validated in the SQL runtime workstream.
- **Value types / composites**: Embedded value objects (e.g., Address) currently sit in `models` with `storage: {}`. Needs a separate `types`/`composites` section.
- **Polymorphic associations**: Polymorphism on relations (a Comment belonging to Post or Video). No contract representation yet.
- **Discriminator values are untyped strings**: The contract stores discriminator values as strings, but something must convert them to native DB values.

## Testing

- **Provisioning**: [mongodb-memory-server](https://github.com/typegoose/mongodb-memory-server) тАФ runs a real `mongod` binary in-process, no container orchestration. Configured as a replica set.
- **Type-level tests**: Contract and ORM API validated with TypeScript files that must typecheck against hand-crafted contract types.
- **Integration tests**: Query execution against a running MongoDB instance via the `mongodb` Node.js driver.
