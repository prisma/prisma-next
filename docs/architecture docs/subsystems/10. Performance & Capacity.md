# 10. Performance & Capacity

This document sets explicit performance budgets, load expectations, caching strategies, timeout policies, and backpressure controls for Prisma Next runtimes and PPg preflight services.

## Scope

- In-app runtime for executing Plans against a target adapter
- CI/PPg preflight for EXPLAIN and shadow runs
- Adapter obligations that affect latency, throughput, and determinism

## Target Budgets

### Build/Compile Budgets

- Plan construction from DSL AST p95 ≤ 1 ms for simple single-table SELECT, ≤ 3 ms for moderate SELECT with 1 join, ≤ 6 ms for heavy SELECT with 3 joins
- Lowering to wire payload p95 ≤ 0.7 ms simple, ≤ 2 ms moderate, ≤ 4 ms heavy
- Hook overhead (beforeCompile) p95 ≤ 0.3 ms when only lints are enabled
- Cold type import and contract load amortized at process start; hot path must be allocation-light

### Execute Path Budgets

- Runtime overhead relative to the raw driver p95 ≤ 5% for CRUD on warmed connections
- Param encoding and result decoding p95 ≤ 1.5 ms for result sets ≤ 100 rows, scales sub-linearly with rows via streaming decoders
- Hook overhead (beforeExecute + afterExecute) p95 ≤ 1 ms with lints and budgets enabled, ≤ 0.3 ms when only telemetry is enabled
- Connection acquisition p95 ≤ 2 ms on a warmed pool under target concurrency

### Preflight Budgets

- EXPLAIN-only preflight for ≤ 200 Plans completes p95 ≤ 90 s end-to-end
- Per-Plan EXPLAIN p95 ≤ 50 ms on cached schema, ≤ 120 ms on cold
- Shadow DB apply for ≤ 5 additive edges p95 ≤ 8 min including provision, migrate, lint, teardown
- Artifact write (NDJSON + summary) p95 ≤ 300 ms for ≤ 10K events

## Extension-Related Overhead Budgets

Extensions introduce additional overhead that must be accounted for in performance budgets:

### Capability Negotiation Overhead
- **Connect Time**: Capability discovery and negotiation adds <50ms to connection establishment
- **Memory Overhead**: Capability registry and pack metadata consume <1MB per connection
- **CPU Overhead**: Capability checks add <1ms per query execution

### Codec Decoding Overhead
- **Extension Codecs**: Extension-specific codecs add <5ms per result set
- **Branded Type Creation**: Branded type instantiation adds <0.1ms per value
- **Validation Overhead**: Extension value validation adds <1ms per operation

### Extension EXPLAIN Passes
- **Extension Operators**: Extension operators may require additional EXPLAIN passes
- **Capability-Specific Analysis**: Capability-specific query analysis adds <10ms per query
- **Policy Enforcement**: Extension guardrails add <2ms per query execution

### Bundle Processing Overhead
- **Bundle Validation**: Bundle signature verification adds <100ms to preflight startup
- **Pack Loading**: Pack code loading and validation adds <50ms per pack
- **Sandbox Initialization**: Sandbox environment setup adds <20ms per execution

## EXPLAIN Normalization Across Extension Operators

Extension operators require consistent EXPLAIN normalization:

### Extension-Specific Fields
- **Cost Estimates**: Extension operators must provide normalized cost estimates
- **Row Estimates**: Extension operators must provide normalized row estimates
- **Execution Time**: Extension operators must provide normalized execution time estimates
- **Resource Usage**: Extension operators must report normalized resource usage

### Adapter Obligations
- **EXPLAIN Integration**: Adapters must integrate extension operators into EXPLAIN output
- **Normalization Rules**: Adapters must apply consistent normalization rules to extension operators
- **Budget Integration**: Adapters must integrate extension operator costs into budget calculations
- **Performance Monitoring**: Adapters must monitor extension operator performance

Load characteristics
	•	Online requests are latency-sensitive and bursty, dominated by small SELECTs and occasional mutations
	•	CI preflight is batchy, CPU-heavy, with parallel EXPLAIN calls and occasional shadow DBs
	•	Typical concurrency targets: 4–16 per service instance for runtime, 8–32 workers for preflight jobs
	•	Mixed workloads require resource isolation between Plan compilation and driver I/O via micro-queues and semaphores

Caching

Plan cache
	•	Keyed by planHash with value { lowered, annotations, profileHash }
	•	Size target 5k–20k entries per process, LRU with TTL 10–30 min
	•	Invalidation on contract or profile hash change

EXPLAIN cache
	•	Keyed by sqlFingerprint + contractHash + profileHash + adapterVersion
	•	TTL 1–24 h configurable, salt rotation supported to flush stale estimates
	•	Stores normalized cost/row estimates and plan shape digest only, no raw SQL

Connection and prepared statements
	•	Pool size defaults 2× CPU cores, capped by DB limits
	•	Prepared statement cache per connection keyed by fingerprint, size 500–2k entries
	•	Evict on server errors that indicate invalidation

Codec and projection cache
	•	Column codec resolution keyed by {table, column, type}
	•	Projection type inference cached by {selectKeys, joinNullability}

Statement timeouts

Runtime
	•	Default statement timeout 30 s, per-Plan override via annotations or policy
	•	Mutations inherit stricter defaults in production, e.g. 10–15 s
	•	Long-running analytics pushed to background lanes or PPg via advisory policies

Preflight
	•	EXPLAIN timeout default 50 ms with sampling, up to 200 ms for escalated checks
	•	Shadow migrations apply per op timeouts and global wall-clock budget
	•	Hard cap per preflight job to avoid CI overruns

Backpressure and shedding

Runtime backpressure
	•	Global execution semaphore limits concurrent driver calls
	•	Per-connection queue length caps; enqueue timeout produces RUNTIME_BACKPRESSURE error with retry hint
	•	Shedding tiers
	•	Drop noncritical hooks first (e.g., telemetry sampling)
	•	Downgrade budgets to cached EXPLAIN only
	•	Refuse new Plans once queue thresholds are exceeded

Preflight backpressure
	•	Job queue concurrency caps and per-repo rate limits
	•	Cancel superseded jobs on new pushes to the same PR
	•	Degrade to EXPLAIN-only when shadow provisioning is starved, with a clear “needs shadow” flag

Adapter performance obligations
	•	Deterministic lowering with O(n) complexity in AST size
	•	Parameter binding avoids per-row string concatenation
	•	EXPLAIN integration returns normalized shape quickly or times out predictably
	•	DDL executors publish txSemantics and avoid full-table locks when profile supports safer alternatives

Observability

Track and alert on
	•	Build and lowering latency percentiles
	•	Driver execute latency, rows, and error codes
	•	Pool utilization, queue lengths, enqueue timeouts
	•	Plan cache hit rate, EXPLAIN cache hit rate
	•	Timeouts and budget violations by rule
	•	Shadow provisioning time, migration op timings, and failure classifications

Dashboards
	•	Runtime latency by sqlFingerprint and lane
	•	Preflight throughput and pass/fail by repo
	•	Heatmap of slowest projections and most frequent lints

Capacity planning

Rules of thumb
	•	Runtime QPS ≈ concurrency / p95(latency) with 30–50% headroom
	•	Pool size ≈ min(DB max_connections * 0.2, 2 × cores) per service instance
	•	Plan cache memory ≈ entries × 1–2 KB
	•	Preflight workers sized by plansPerJob × explainTime / wallBudget with I/O headroom for shadow DBs

Scaling levers
	•	Increase worker concurrency for EXPLAIN and cap shadow runs
	•	Split preflight into shards by repo or directory
	•	For runtime, scale horizontally before raising pool sizes

Degradation modes
	•	Switch lints to warn and budgets to cached estimates during incidents
	•	Reduce telemetry sampling, drop high-cardinality tags
	•	For CI, skip noncritical Plans by path filters and raise shadow thresholds temporarily

Benchmarks and acceptance gates

Profiles
	•	Tiny CRUD: single table, point filters, 10–50 rows
	•	Moderate joins: 2–3 joins, selective predicates, 50–200 rows
	•	Heavy read: window or aggregate, 1–3k rows

Release gates
	•	Meet compile and execute p95 budgets on all profiles
	•	Runtime overhead ≤ 5% vs raw driver on Tiny and Moderate profiles
	•	EXPLAIN cache hit rate ≥ 80% on warm CI reruns for unchanged Plans
	•	Shadow apply within 8 min for the MVP edge set

## ADRs

	•	ADR 087 — Execution concurrency & pooling model
Define the runtime’s concurrency primitives (global exec semaphore, per-connection queues), default pool sizing, queue limits, enqueue timeouts, and error semantics for backpressure
Complements ADR 082 (shedding) but focuses on pool math and limits
	•	ADR 088 — EXPLAIN cache key & TTL policy
Canonical fingerprint for EXPLAIN caching (sqlFingerprint + contractHash + profileHash + adapterVersion), TTL defaults, salt rotation, and invalidation triggers
Pairs with ADR 023 (budgets) and makes the cache spec explicit
	•	ADR 089 — Performance budgets & release gates
Hard p95 targets for compile/lower/execute/preflight and the CI gates that enforce them with golden perf profiles (Tiny, Moderate, Heavy)
	•	ADR 090 — Degradation & feature throttling order
Formalize the “graceful degradation” sequence under load: drop high-cardinality telemetry → fall back to cached EXPLAIN → downgrade lint levels → finally refuse Plans
	•	ADR 091 — Statement timeout negotiation
Client- and server-side timeout defaults by lane and mutation class, per-Plan overrides via annotations, and cancellation propagation guarantees
Complements ADR 083 but captures the server-negotiation and escalation rules
	•	ADR 092 — SQL fingerprinting algorithm v1
Normalization rules (whitespace, aliases, literals to binds, identifier quoting) used for caching, telemetry, and change detection without colliding with ADR 013’s Plan hashing
	•	ADR 093 — Capacity planning & autoscaling signals
Standard metrics to drive scaling (queue depth, pool utilization, p95 latency), recommended headroom, and alert thresholds for runtime and preflight workers
	•	ADR 094 — Hook overhead contract
No-alloc guidance and p95 overhead ceilings for beforeCompile, beforeExecute, afterExecute; measurement method and failure actions
	•	ADR 095 — Prepared statement cache sizing & eviction
Per-connection PS cache key, size defaults, eviction policy, and invalidation on DDL or specific server errors
Complements ADR 084 but focuses on sizing and eviction strategy
	•	ADR 076 — EXPLAIN normalization across targets
Minimal normalized explain schema for budgets, mapping from target-specific explain outputs, and adapter obligations for units and fields
Complements ADR 023
