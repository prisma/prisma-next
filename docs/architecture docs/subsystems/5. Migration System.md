# Migration System

**Purpose:** specify Prisma Next's migration model and the first production slice. The system is contract-centric: every migration is an edge from one data contract to another, and all execution safety flows from preconditions, postconditions, and a verifiable contract marker stored in the database.

## Goals

- Make migrations deterministic and verifiable from the data contract alone
- Remove reliance on filesystem order while keeping migrations human and agent friendly
- Provide strong guardrails by default and clear extension points for custom operations
- Keep a straightforward rollback story that favors roll-forward and verification
- Operate with low operational risk in CI and production

## Model

### Edges and Nodes

- **Edge**
  - A state transition from `fromCoreHash` → `toCoreHash`
  - Optional `fromProfileHash` and `toProfileHash` if lowering depends on adapter profile capabilities
- **Node tasks**
  - Optional tasks associated with the edge with explicit pre and post checks
  - Used for data backfills, one-off transforms, and validations that are not pure DDL
- **Contract marker**
  - A singleton table row that stores the current `coreHash` and `profileHash`
  - Always exactly one row with `id=1`, updated via UPSERT on each migration
  - Runner compares this marker with the edge's `from*` before apply
- **DAG**
  - The set of edges on disk forms a directed acyclic graph from the empty contract to current contract
  - The runner finds a path from the DB's marker to the desired contract and applies edges in order

```
[ ∅ ] --e1--> [ H1 ] --e2--> [ H2 ] --e3--> [ H3 ]
                 \                         ^
                  \--e4--> [ H2a ] --e5---/
```

## Planner MVP

### Scope for the First Slice

**Supported diffs (additive only):**

- create table
- add nullable column
- add index or unique
- add foreign key
- set column default
- comment/description changes treated as metadata only

**Unsupported initially:**

- renames and drops
- type changes requiring data migration
- composite changes that reorder constraints

**Output:**

- A single migration folder containing `migration.json` with a list of JSON operations and optional node tasks
- Preconditions and postconditions are generated mechanically from the diff

**Hints later:**

- PSL hints such as `@hint(was: ...)` enable stable rename/drop planning without heuristics

## Runner

### Responsibilities

- Compute the path in the migration DAG from DB marker to target contract
- Acquire an advisory lock to prevent concurrent applies
- Validate each edge's applicability via preconditions and the contract marker
- Execute operations idempotently with postcondition checks
- Update the migration ledger and contract marker on success

### Advisory Locks

- Abstracted via the admin driver
- On Postgres, use `pg_advisory_lock` with a deterministic key derived from `coreHash` namespace and database OID
- Locks are reentrant per session and released on commit or connection close

### Idempotency

- Every operation must be safe to re-run
  - if postcondition already holds, treat as no-op
  - if precondition fails and postcondition holds, treat as already applied
  - if both fail, halt with a structured error

## Database Schema

The migration system uses two distinct tables with different purposes:

### Contract Marker (Singleton)
- **prisma_contract.version** - Current database state
  - `id integer primary key` (always 1)
  - `hash text not null` (current contract hash)
  - `updated_at timestamptz not null`
- **Purpose**: Track current database contract state
- **Behavior**: Always exactly one row with `id=1`, updated via UPSERT
- **Query**: `SELECT hash FROM prisma_contract.version WHERE id = 1`

### Migration Ledger (Historical Record - Tooling Only)
- **prisma_contract.ledger** - Complete migration history
  - `edge_id text primary key`
  - `from_core_hash text not null`
  - `to_core_hash text not null`
  - `from_contract jsonb`
  - `to_contract jsonb`
  - `hints jsonb`
  - `applied_at timestamptz not null`
  - `applied_by text`
  - `duration_ms integer`
  - `result jsonb`
- **Purpose**: Audit trail for tooling (PPg Console, Studio) and historical visualization - NOT required for migration operations
- **Behavior**: One row per applied migration edge, append-only
- **Query**: `SELECT * FROM prisma_contract.ledger ORDER BY applied_at`

### Key Distinction

- **Contract marker**: "What is the current state?" (singleton, always updated) - REQUIRED for pathfinding
- **Migration ledger**: "What migrations have been applied?" (historical, append-only) - For UI and audit only

The contract marker alone is sufficient for all migration pathfinding and application. The ledger is useful for presenting migration history in UIs but is not consulted by the runner or planner.

### Contract Marker Update Mechanism

The contract marker uses UPSERT to maintain singleton behavior:

```sql
-- First migration (empty database)
INSERT INTO prisma_contract.version (id, hash) VALUES (1, 'sha256:abc123');

-- Subsequent migrations (update existing row)
INSERT INTO prisma_contract.version (id, hash) VALUES (1, 'sha256:def456')
ON CONFLICT (id) DO UPDATE SET
  hash = EXCLUDED.hash,
  updated_at = now();
```

This ensures:
- **Always one row**: Exactly one record with `id=1`
- **Always current**: The hash reflects the latest applied migration
- **Query consistency**: `SELECT hash FROM prisma_contract.version WHERE id = 1` always returns current state

## Rollback Story

- Prefer roll-forward via a new edge from current marker to the desired contract
- For catastrophic incidents, an edge may include explicit reversible operations, but reversal is not guaranteed for destructive steps
- Shadow verification and backups are the primary safeguard before apply

## Squash and Baselines

The migration system supports two types of squashing to keep DAGs manageable:

### Range Squash (Intermediate Compression)
- **What**: Collapse a contiguous path of migration files into a single migration
- **Example**: `[A] --e1--> [B] --e2--> [C]` becomes `[A] --squash--> [C]`
- **Purpose**: Reduce migration chain length while preserving audit trail
- **Safety**: Original migrations marked as `archived: true` but preserved for history
- **Usage**: `migrate squash --range 2025-10-01..2025-10-13`

### Baseline Squash (Empty → Latest)
- **What**: Create a single migration from empty database to latest contract
- **Example**: `[∅] --baseline--> [Hlatest]` (replaces entire migration history)
- **Purpose**: Bootstrap fresh environments quickly
- **Safety**: Only applies to empty databases (production has contract marker)
- **Usage**: `migrate squash --baseline` (creates `from: { kind: "empty" }`)

### Squash-First Policy (ADR 102)
Teams are encouraged to regularly squash via advisory system:
- Suggest when >20 migrations or >14 days since last baseline
- Preflight validation ensures squashed migration produces same result as original chain
- Original migrations archived but preserved for audit
- Database ledger tracks what was applied; migration files define what's available

See ADR 101 (Advisors framework) and ADR 102 (Squash-first policy) for the advisory system that guides teams toward healthy DAG hygiene.

## Drift Detection and Preflight

- **Drift**
  - Detected when the DB contract marker differs from the planner's expected `from*`
  - Runner refuses to apply and reports the marker so the planner can re-diff
- **Preflight**
  - Apply candidate edges in a shadow database using the same runner and advisory lock
  - Run postconditions and lints
  - Optionally run EXPLAIN on critical queries to surface index issues
  - Produce a machine-readable report consumable by CI and PPg

## File Layout

```
migrations/
  2025-10-18T1205_add-users-table/
    migration.json            # Contains from/to hashes, edge metadata
    tasks.pre.ts              # optional
    tasks.post.ts             # optional
  2025-10-19T0830_add-email-index/
    migration.json
  graph.index.json            # optional performance cache
```

- **Migration folder name**
  - Timestamp and slug for human readability
  - The migration's identity is inside `migration.json` and not derived from the folder name
- **graph.index.json** (optional performance cache)
  - Optional lockfile-style cache for path finding performance
  - Can be regenerated deterministically from migration files
  - Default mode reconstructs graph on demand from migration manifests
  - Most teams won't need this when following squash-first policy (ADR 102)
  - See ADR 039 for when to enable the committed index

### Contract Storage Benefits

Storing the complete destination contract alongside each migration enables:

- **Complete State Reconstruction**: Any migration point can be reconstructed from the stored contract, enabling migration splitting and intermediate state inference
- **Enhanced Tooling**: Dev CLI and PPg can visualize contract evolution over time, showing exactly what changed between any two states
- **Migration Splitting**: The planner can infer intermediate contract states and generate new edges between any two historical states
- **Agent-Friendly Analysis**: Agents get complete context for migration analysis with full contract state at each step
- **Audit and Debugging**: Complete contract context makes debugging migration failures and auditing changes much easier

### Planner Hints Storage

The `hints` section captures:
- **used**: PSL hints that were applied during planning (e.g., `@hint(was: old_table_name)`)
- **applied**: Planner strategies and heuristics used (e.g., `stable_rename_detection`, `additive_only`)
- **plannerVersion**: Version of the planner that generated this edge
- **planningStrategy**: High-level strategy used (e.g., `additive`, `hint_driven`, `destructive`)

This enables:
- Reproducible planning decisions
- Debugging planner behavior
- Agent understanding of planning context
- Migration splitting with consistent planning strategies

### Core Operations v1

- **createTable**
  - table, columns, primary key, uniques, indexes, foreign keys
- **addColumn**
  - table, column, default, nullability
- **addIndex**
  - table, columns, method, unique
- **addForeignKey**
  - table, columns, references, onDelete, onUpdate
- **setDefault**
  - table, column, default

Every operation supports pre and post with a small vocabulary of checks

### Preconditions and Postconditions

- Executed inside the same transaction when the adapter supports transactional DDL
- If transactional DDL is not available, runner orders ops to minimize partial application and rechecks after each op

## Extension Mechanism

- **Custom operations**
  - Name-spaced ops `vendor/opName` loaded from a registry at runtime
  - Must implement plan-time schema validation and apply with pre/post checks
  - Extension-aware operations are gated by capability keys per ADR 116
- **Custom tasks**
  - Node tasks can be `.ts` modules exporting named functions
  - Runner executes them with an admin connection and a context providing contract and adapter
- **Safety bar**
  - Custom ops and tasks must declare `idempotent: true|false` and `requiresTx: true|false`
  - Non-idempotent or non-transactional tasks are permitted only in permissive mode
- **Capability Gating**
  - Operations requiring unavailable capabilities fail with `E_CAPABILITY_MISSING`
  - Planner guardrails prevent operations requiring unavailable packs
  - Runner behavior when capabilities are missing or downgraded

## Runner Phases

1. **Resolve**
   - Read DB contract marker
   - Compute a path from marker to target `coreHash` by reconstructing graph from migration.json headers (or using optional graph.index.json)
2. **Preflight**
   - Optional shadow apply driven by CI or PPg
   - Produce a report with pass/fail per pre/post check and op timing
3. **Apply**
   - Acquire advisory lock
   - For each edge in path:
     - verify `from*` against marker
     - execute `tasks.pre`
     - apply ops with pre/post checks and idempotency rules
     - execute `tasks.post`
     - update marker and ledger
   - Release lock
4. **Report**
   - Emit telemetry for PPg and CI including total duration, per-op timings, and any warnings

## Initialization & Adoption

Prisma Next supports three adoption paths for bringing databases under contract management:

**Greenfield:** Start with empty database, define contract, create baseline, deploy.

**Brownfield-Conservative:** Introspect existing database, generate initial contract, create baseline, adopt all tables at once.

**Brownfield-Incremental:** Introspect existing tables, generate initial contract, then gradually expand contract with new tables over time. See below for incremental expansion details.

For detailed adoption strategies, initialization operations, and cross-environment deployment patterns, see **ADR 122 — Database Initialization & Adoption**.

## Incremental Contract Expansion

The most common brownfield scenario: gradually adding new tables and features to an existing contract.

**Workflow:**
1. Update contract (PSL or TypeScript) to add new table definition
2. Emit new contract → H1 (includes existing + new tables)
3. Planner diffs H0 → H1, identifies new table
4. Generator creates migration: "add table"
5. Runner applies: marker updated to H1

**Version Skew Safety:** Applications deployed for H1 can safely run against H0 database temporarily:
- Queries against existing tables (in both H0 and H1) work normally
- Queries against new tables fail with clear error
- Runtime verification catches mismatch before execution
- No hidden schema changes that would be dangerous

**Edge Case 1: Adopting Manually-Created Tables**

Use idempotent migrations (per ADR 038):
- Migration includes: CREATE TABLE IF NOT EXISTS
- Works whether table exists or not
- Trade-off: Less strict verification, simpler workflow
- Caveat: Requires manual schema validation after

**Edge Case 2: Multi-Stage Adoption**

Adding tables incrementally (week 1, week 2, week 3):

Option A (Recommended): Regular migrations
- H0 → H1 (add posts), H1 → H2 (add comments), H2 → H3 (add likes)
- Clear history, each stage validated independently
- Matches existing mental model

Option B (Optional): Intermediate baselines
- Create baselines for H1, H2, H3 in addition to regular migrations
- New environments can bootstrap to any intermediate state
- Worth it only if stages represent stable milestones

For detailed edge case guidance, see **ADR 122 — Database Initialization & Adoption**.

## Multi-Service Namespacing

Multiple services in the same database can maintain independent contracts using schema-level namespacing (Postgres):

```typescript
const db_a = connect({
  markerLocator: { schema: 'service_a' },
  connectionString: process.env.DATABASE_URL
});

const db_b = connect({
  markerLocator: { schema: 'service_b' },
  connectionString: process.env.DATABASE_URL
});
```

- Each service: independent contract, migrations, DAG
- Marker location: service_a.prisma_contract.marker
- Permissions: Service A cannot see Service B schema
- Benefits: Schema isolation, simple model, leverages native features

For MySQL/SQLite without schema support, use separate databases.

Why not multi-contract in single schema?
- Multiple overlapping contracts too complex
- Version conflicts (which contract wins?)
- Query verification ambiguity
- Marker hash conflicts

Schema-level isolation is simpler and cleaner.

See **ADR 122 — Database Initialization & Adoption** for detailed configuration.

## Drift Detection and Preflight

Database can diverge from expected contract state in many ways:
- Manual DDL changes not via migrations
- Partial migration failures
- Replica lag, caching issues
- Marker table corruption or loss
- Concurrent deployments

**Detection mechanisms:**
- At startup: read and validate marker
- Before query execution: verify query against current contract
- Before migration apply: validate marker matches edge from hash
- After apply: verify postconditions and marker update
- On error: detect schema-related errors and optionally re-verify

**Recovery strategies depend on drift type:**

Marker-level drifts (missing, corrupt, stale):
- Initialization or manual reconciliation
- See ADR 123 for detailed recovery

Schema-level drifts (manual DDL, partial-apply):
- If idempotent migration: safe re-run
- If not idempotent: manual remediation or reset
- Introspection can detect and plan corrections

DAG-level drifts (orphaned database, no path):
- Create adoption baseline or new migration edge
- See ADR 123 for detailed taxonomy

**Idempotency Bridge:**
Idempotent operations (ADR 038) enable safe drift recovery:
- Partial-apply can be recovered by re-running
- CREATE TABLE IF NOT EXISTS: no-op if exists
- Operations already completed: no-op
- Failed operations now succeed

See **ADR 123 — Drift Detection, Recovery & Reconciliation** for comprehensive drift taxonomy and recovery procedures by environment (dev, staging, production).

## Error Taxonomy

- **contract/hash-mismatch**
  - DB marker does not match edge `from*`
- **precondition/failed**
  - A pre check failed for an operation
- **postcondition/failed**
  - A post check failed, indicating a partial apply or unexpected state
- **lock/acquire-failed**
  - Advisory lock could not be obtained
- **adapter/transaction-unsupported**
  - Operation requires transactional DDL but adapter cannot provide it
- **op/unsupported**
  - Edge references an op unknown to the registry
- **task/error**
  - A node task threw or returned a failed status

Each error carries structured details and a remediation hint for agents

## Preflight in CI and PPg

- Shadow database lifecycle is managed by the platform or a local helper — create, seed optional fixtures, apply path, run checks, tear down
- Reports include path found, edges applied, timing, violations, index advisor hints, and an overall status
- PPg can surface these reports inline in PRs and gate promotion on success
