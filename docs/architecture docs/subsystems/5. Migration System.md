# Migration System

**Purpose:** specify Prisma Next's migration model and the first production slice. The system is contract-centric: every migration is an edge from one data contract to another, and all execution safety flows from preconditions, postconditions, and a verifiable contract marker stored in the database.

## Goals

- Make migrations deterministic and verifiable from the data contract alone
- Remove reliance on filesystem order while keeping migrations human and agent friendly
- Provide strong guardrails by default and clear extension points for custom operations
- Keep a straightforward rollback story that favors roll-forward and verification
- Operate with low operational risk in CI and production

## Model

### Edges and Nodes

- **Edge**
  - A state transition from `fromCoreHash` → `toCoreHash`
  - Optional `fromProfileHash` and `toProfileHash` if lowering depends on adapter profile capabilities
- **Node tasks**
  - Optional tasks associated with the edge with explicit pre and post checks
  - Used for data backfills, one-off transforms, and validations that are not pure DDL
- **Contract marker**
  - A small table in the target DB that stores the current `coreHash` and `profileHash`
  - Runner compares this marker with the edge's `from*` before apply
- **DAG**
  - The set of edges on disk forms a directed acyclic graph from the empty contract to current contract
  - The runner finds a path from the DB's marker to the desired contract and applies edges in order

```
[ ∅ ] --e1--> [ H1 ] --e2--> [ H2 ] --e3--> [ H3 ]
                 \                         ^
                  \--e4--> [ H2a ] --e5---/
```

## Planner MVP

### Scope for the First Slice

**Supported diffs (additive only):**

- create table
- add nullable column
- add index or unique
- add foreign key
- set column default
- comment/description changes treated as metadata only

**Unsupported initially:**

- renames and drops
- type changes requiring data migration
- composite changes that reorder constraints

**Output:**

- A single edge folder containing `edge.json` with a list of JSON operations and optional node tasks
- Preconditions and postconditions are generated mechanically from the diff

**Hints later:**

- PSL hints such as `@hint(was: ...)` enable stable rename/drop planning without heuristics

## Runner

### Responsibilities

- Compute the path in the migration DAG from DB marker to target contract
- Acquire an advisory lock to prevent concurrent applies
- Validate each edge's applicability via preconditions and the contract marker
- Execute operations idempotently with postcondition checks
- Update the migration ledger and contract marker on success

### Advisory Locks

- Abstracted via the admin driver
- On Postgres, use `pg_advisory_lock` with a deterministic key derived from `coreHash` namespace and database OID
- Locks are reentrant per session and released on commit or connection close

### Idempotency

- Every operation must be safe to re-run
  - if postcondition already holds, treat as no-op
  - if precondition fails and postcondition holds, treat as already applied
  - if both fail, halt with a structured error

## Ledger Schema

Minimal tables managed by the runner:

- **prisma_contract.marker**
  - `core_hash text not null`
  - `profile_hash text not null`
  - `updated_at timestamptz not null`
- **prisma_contract.ledger**
  - `edge_id text primary key`
  - `from_core_hash text not null`
  - `to_core_hash text not null`
  - `from_contract jsonb`
  - `to_contract jsonb`
  - `hints jsonb`
  - `applied_at timestamptz not null`
  - `applied_by text`
  - `duration_ms integer`
  - `result jsonb`

Optional rows even if the contract marker alone would suffice — useful for audit, PPg UI, and restart behavior. The stored contracts enable complete historical reconstruction and enhanced tooling capabilities.

## Rollback Story

- Prefer roll-forward via a new edge from current marker to the desired contract
- For catastrophic incidents, an edge may include explicit reversible operations, but reversal is not guaranteed for destructive steps
- Shadow verification and backups are the primary safeguard before apply

## Squash and Baselines

Under the squash-first policy (ADR 102), teams regularly squash their migration history to keep the DAG small and manageable:

- **Squash (primary hygiene mechanism)**
  - Collapse a path of edges into a single baseline edge from ∅ → `Hlatest`
  - Mark original edges as archived: true (retained for audit, ignored for pathfinding)
  - Use squashed edge to bootstrap ephemeral or developer databases quickly
  - Default policy suggests squashing when >20 edges or >14 days since last baseline
- **Baselines**
  - A baseline is an edge with fromCoreHash = ∅ embedding the full destination contract
  - Baselines are only applied to fresh databases (runner treats as no-op if marker exists)
  - Runner verifies from* strictly to ensure safety

See ADR 101 (Advisors framework) and ADR 102 (Squash-first policy) for the advisory system that guides teams toward healthy DAG hygiene.

## Drift Detection and Preflight

- **Drift**
  - Detected when the DB contract marker differs from the planner's expected `from*`
  - Runner refuses to apply and reports the marker so the planner can re-diff
- **Preflight**
  - Apply candidate edges in a shadow database using the same runner and advisory lock
  - Run postconditions and lints
  - Optionally run EXPLAIN on critical queries to surface index issues
  - Produce a machine-readable report consumable by CI and PPg

## File Layout

```
migrations/
  edges/
    2025-10-18T1205_add-users-table/
      edge.json
      tasks.pre.ts            # optional
      tasks.post.ts           # optional
    2025-10-19T0830_add-email-index/
      edge.json
  dag.json                    # optional index cache, derived from edges
```

- **Edge folder name**
  - Timestamp and slug for human readability
  - The edge's identity is inside `edge.json` and not derived from the folder name
- **dag.json** (graph index, optional)
  - Optional lockfile-style cache for path finding performance
  - Can be regenerated deterministically from edge files
  - Default mode reconstructs graph on demand from edge manifests
  - Most teams won't need this when following squash-first policy (ADR 102)
  - See ADR 039 for when to enable the committed index

### Contract Storage Benefits

Storing the complete destination contract alongside each migration enables:

- **Complete State Reconstruction**: Any migration point can be reconstructed from the stored contract, enabling migration splitting and intermediate state inference
- **Enhanced Tooling**: Dev CLI and PPg can visualize contract evolution over time, showing exactly what changed between any two states
- **Migration Splitting**: The planner can infer intermediate contract states and generate new edges between any two historical states
- **Agent-Friendly Analysis**: Agents get complete context for migration analysis with full contract state at each step
- **Audit and Debugging**: Complete contract context makes debugging migration failures and auditing changes much easier

### Planner Hints Storage

The `hints` section captures:
- **used**: PSL hints that were applied during planning (e.g., `@hint(was: old_table_name)`)
- **applied**: Planner strategies and heuristics used (e.g., `stable_rename_detection`, `additive_only`)
- **plannerVersion**: Version of the planner that generated this edge
- **planningStrategy**: High-level strategy used (e.g., `additive`, `hint_driven`, `destructive`)

This enables:
- Reproducible planning decisions
- Debugging planner behavior
- Agent understanding of planning context
- Migration splitting with consistent planning strategies

### Core Operations v1

- **createTable**
  - table, columns, primary key, uniques, indexes, foreign keys
- **addColumn**
  - table, column, default, nullability
- **addIndex**
  - table, columns, method, unique
- **addForeignKey**
  - table, columns, references, onDelete, onUpdate
- **setDefault**
  - table, column, default

Every operation supports pre and post with a small vocabulary of checks

### Preconditions and Postconditions

- Executed inside the same transaction when the adapter supports transactional DDL
- If transactional DDL is not available, runner orders ops to minimize partial application and rechecks after each op

## Extension Mechanism

- **Custom operations**
  - Name-spaced ops `vendor/opName` loaded from a registry at runtime
  - Must implement plan-time schema validation and apply with pre/post checks
  - Extension-aware operations are gated by capability keys per ADR 116
- **Custom tasks**
  - Node tasks can be `.ts` modules exporting named functions
  - Runner executes them with an admin connection and a context providing contract and adapter
- **Safety bar**
  - Custom ops and tasks must declare `idempotent: true|false` and `requiresTx: true|false`
  - Non-idempotent or non-transactional tasks are permitted only in permissive mode
- **Capability Gating**
  - Operations requiring unavailable capabilities fail with `E_CAPABILITY_MISSING`
  - Planner guardrails prevent operations requiring unavailable packs
  - Runner behavior when capabilities are missing or downgraded

## Runner Phases

1. **Resolve**
   - Read DB contract marker
   - Compute a path from marker to target `coreHash` by reconstructing graph from edge headers (or using optional dag.json index)
2. **Preflight**
   - Optional shadow apply driven by CI or PPg
   - Produce a report with pass/fail per pre/post check and op timing
3. **Apply**
   - Acquire advisory lock
   - For each edge in path:
     - verify `from*` against marker
     - execute `tasks.pre`
     - apply ops with pre/post checks and idempotency rules
     - execute `tasks.post`
     - update marker and ledger
   - Release lock
4. **Report**
   - Emit telemetry for PPg and CI including total duration, per-op timings, and any warnings

## Error Taxonomy

- **contract/hash-mismatch**
  - DB marker does not match edge `from*`
- **precondition/failed**
  - A pre check failed for an operation
- **postcondition/failed**
  - A post check failed, indicating a partial apply or unexpected state
- **lock/acquire-failed**
  - Advisory lock could not be obtained
- **adapter/transaction-unsupported**
  - Operation requires transactional DDL but adapter cannot provide it
- **op/unsupported**
  - Edge references an op unknown to the registry
- **task/error**
  - A node task threw or returned a failed status

Each error carries structured details and a remediation hint for agents

## Preflight in CI and PPg

- Shadow database lifecycle is managed by the platform or a local helper — create, seed optional fixtures, apply path, run checks, tear down
- Reports include path found, edges applied, timing, violations, index advisor hints, and an overall status
- PPg can surface these reports inline in PRs and gate promotion on success

### Migration Bundle for Hosted Preflight

For hosted preflight (PPg), users must submit a **migration bundle** artifact that contains:

- `bundle.json` index with coreHash, profile, and edge list
- `contract.json` canonical data contract
- `migrations/*.json` edges with pinned op refs and arguments
- `ops/*.js` self-contained ESM files for each op used by edges
- `ops/*.manifest.json` corresponding manifests
- `fixtures/*` optional seed or fixtures for shadow DB

**Local vs Hosted Apply Paths:**
- **Local apply**: Runner loads ops directly from filesystem or node_modules
- **Hosted apply**: PPg consumes pre-built bundles, never resolves user repos or package managers
- Bundles ensure deterministic execution with all dependencies inlined and no network access

## Performance and Reliability

- The runner issues one BEGIN per edge when transactional DDL is supported
- Advisory locks are held at edge granularity to minimize contention
- Pre/post checks are compiled to fast metadata queries using adapter APIs rather than generic SQL where possible
- Idempotency allows safe retries after driver failures

## UX Summary

- Edit `schema.psl` or the TS builder, then emit `contract.json` and types
- `pnpm migrate plan` generates an edge from the last known contract to the new contract
- `pnpm migrate preflight` applies in a shadow database and reports diagnostics
- `pnpm migrate apply` applies to the target database with locks and ledger updates
- `pnpm migrate squash` collapses a path to a single edge for fresh environments

## Open Questions

- Policy for destructive changes once rename/drop land — require explicit flags or two-phase edges
- Partial adapter support for transactional DDL — best-effort fallback ordering and compensation strategies
- Standard library of data migration tasks — backfills, column copy with cast, JSON denormalization

## Success Criteria

- Deterministic planning for additive diffs with zero manual edits
- Idempotent, verifiable applies with contract and ledger updates
- Squash produces a single edge that can bootstrap a fresh DB from empty
- Preflight catches missing indexes and unsafe changes before apply
- Extension points allow teams to add custom ops and tasks without forking core

## ADRs

- **ADR 037** — Transactional DDL fallback & compensation
  - When adapters lack full transactional DDL: op ordering, commit boundaries, compensation steps, and how we surface partial-apply risk

- **ADR 038** — Operation idempotency classification & enforcement
  - Per-op pre/post invariants, what "already applied" means, and how the runner treats replays vs true conflicts

- **ADR 039** — DAG path resolution & integrity
  - How we compute a path (A* vs BFS), cycle detection, orphan edge detection, and deterministic tie-breaking

- **ADR 040** — Node task execution environment & sandboxing
  - Privilege model, timeouts, memory limits, logging, and how tasks declare `requiresTx`/`idempotent` safely

- **ADR 041** — Custom operation registry & versioning
  - Namespacing (`vendor/op@v`), schema validation, compatibility gates, and policy for loading third-party ops

- **ADR 042** — Contract marker evolution
  - Marker table schema, forward-compatible fields (e.g., `profileHash`), and migration of the marker itself

- **ADR 043** — Advisory lock domain & key strategy
  - How we derive lock keys (per-DB vs per-tenant), cross-runner behavior, and failure handling/backoff

- **ADR 101** — Advisors framework
  - Uniform API for computing and surfacing hygiene advisories across CLI, CI, and PPg

- **ADR 102** — Squash-first policy & squash advisor
  - Default policy for keeping migration DAGs small through regular baselines, thresholds, and actionable suggestions
- **ADR 116** — Extension-aware migration ops
  - Extension operations gated by capability keys, planner guardrails, and runner behavior
- **ADR 117** — Extension capability keys
  - Canonical capability keys and reserved namespaces for extension gating
