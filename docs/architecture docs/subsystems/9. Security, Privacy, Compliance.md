# Security, Privacy, Compliance

**Purpose:** define baseline controls for Prisma Next and PPg as a contract-aware data platform. It covers secrets handling, least-privilege access, what gets written to artifacts, redaction rules, multi-tenant isolation, and auditability.

## Security Model

- **Contract-first safety:** Every runtime and PPg workflow validates Plans and migrations against a signed data contract, preventing class-of-errors where the app and DB disagree
- **One call → one statement:** Plans execute a single statement, simplifying authorization, auditing, and blast-radius analysis
- **Deny by default for sensitive paths:** Raw SQL capture, parameter logging, and schema introspection are off by default and require explicit opt-in

## Secrets Handling

- **Sources:** Prefer ambient credentials and managed identities over long-lived secrets when adapters support it. Else fetch short-lived credentials from a secret manager at start and on rotation
- **Storage:** No secrets in repo or artifacts. In memory only, zeroed on close, never written to disk or logs
- **Rotation:** Watch rotation hints from the platform and re-establish pools without dropping in-flight queries. Expose a health endpoint that fails hard when credentials are invalid or expired
- **Transport:** Enforce TLS for all DB connections and outbound telemetry. Support mTLS when available on PPg
- **Webhooks:** Use signed webhooks with timestamped HMAC, replay protection, and key rotation

## Bundle Signature Requirements

Preflight bundles require cryptographic signatures for integrity and authenticity:

### Bundle Signing
- **Integrity Verification**: SHA-256 integrity hashes for all bundle contents
- **Authenticity Signatures**: Cryptographic signatures using Ed25519 or RSA-PSS
- **Signature Chain**: Support for signature chains and certificate validation
- **Key Rotation**: Automated key rotation with backward compatibility

### Bundle Validation
- **Signature Verification**: Verify bundle signatures before execution
- **Certificate Validation**: Validate signing certificates against trusted CAs
- **Timestamp Validation**: Check signature timestamps for freshness
- **Revocation Checking**: Check certificate revocation lists (CRLs)

### Production Requirements
- **Mandatory Signing**: All production bundles must be signed
- **Trusted Publishers**: Only bundles from trusted publishers are accepted
- **Audit Trail**: Complete audit trail of bundle signatures and validation
- **Compliance Reporting**: Automated compliance reporting for signed bundles

## Sandbox Constraints for Custom Operations

Extension code in bundles runs under strict sandbox constraints:

### Execution Constraints
- **No Network Access**: Extension code cannot make HTTP requests or connect to external services
- **No WASM Loading**: Extension code cannot load WebAssembly modules
- **No Dynamic Imports**: Extension code cannot use dynamic imports
- **No Shell Execution**: Extension code cannot execute shell commands
- **No File System Access**: Extension code has limited file system access

### Resource Limits
- **CPU Limits**: CPU usage limits for extension operations
- **Memory Limits**: Memory usage limits for extension operations
- **Execution Timeouts**: Maximum execution time for extension operations
- **Concurrent Limits**: Limits on concurrent extension operations

### Security Monitoring
- **Policy Violations**: Real-time monitoring of policy violations
- **Resource Usage**: Monitoring of resource usage against limits
- **Audit Logging**: Comprehensive logging of extension operations
- **Alert Systems**: Automated alerting for security violations

Principle of least privilege
	•	Roles and scopes
Separate app, migration-runner, and observability roles
App role can read and minimal write for its tables, cannot mutate schema
Migration role can perform DDL and controlled DML tasks, cannot read application secrets
Observability role can EXPLAIN and read PG catalogs, no DDL or DML
	•	Contract marker access
ppg_contract.marker is writeable only by orchestrator
App role can read marker or call ppg_contract.assert to verify alignment
	•	Advisory locks
Namespaced lock IDs prevent cross-project collisions and reduce privilege escalation risk
	•	Plugins
Runtime plugins run under the app process but must not gain wider DB privileges than the host runtime

What goes into artifacts and diagnostics
	•	Plans
Store planId, sqlFingerprint, coreHash, profileHash, annotations, and execution metadata
Do not store raw SQL or params by default
	•	Preflight outputs
Diagnostics NDJSON stream and a summary JSON with rule violations, budgets, and EXPLAIN excerpts
No raw rows, no raw params, and EXPLAIN normalized with cost and row estimates only
	•	Migration ledger
Edge id, fromHash, toHash, op list with pre/post outcomes, timestamps, and runner identity
No arbitrary user data beyond operation parameters already present in the migration package
	•	Telemetry
Latency, rows, error codes, sampling key, and fingerprints
No PII values, no schema payloads unless explicitly enabled
	•	Golden snapshots
SQL and pipeline goldens live in test fixtures only, never in runtime artifacts

Redaction rules
	•	Default redaction
Mask all bind parameters in logs and diagnostics
Mask raw SQL text in telemetry and CI artifacts
	•	Contract sensitivity tags
Columns may carry sensitivity tags in the contract (pii.email, pii.name, secret.token, financial.amount)
Diagnostics and PPg UI replace sensitive identifiers with opaque labels while preserving structural context
	•	Structured EXPLAIN
Keep cost, row estimates, and plan shape but strip literals and object names when sensitivity demands it
	•	Optional dev-mode relaxations
Projects may locally enable raw SQL echo and sampled params for debugging, never in CI or production
	•	Escalation controls
Redaction policy is centrally configured, immutable in production by default, and versioned alongside the contract

Multi-tenant considerations
	•	Contract markers
Support per-tenant markers in shared databases when RLS or schema-per-tenant is used
	•	Isolation
Enforce tenant scoping on runtime Plans via annotations checked by guardrails
PPg preview DBs are per PR and isolated per project with TTL and quotas
	•	Policy and budgets
Budgets can scope to tenant, role, or Plan class to prevent noisy neighbors
Preflight and promotion gates evaluate per environment, not globally
	•	Secrets
Tenant-scoped secrets are fetched and cached independently with separate rotation cycles

Audit logs
	•	Event envelope
timestamp, eventType, planId, sqlFingerprint, coreHash, profileHash, tenantId?, principal, role, outcome, latency, rows, errorCode?
	•	Coverage
Log plan execution, budget or lint violations, migration apply and resume, preflight runs, policy enforcement decisions, and marker changes
	•	Integrity
Append-only storage with write-once semantics and clock drift correction
Optional hashing or signing of batches for tamper evidence
	•	Access
Read access gated by project and environment roles
PII is already redacted; additional column-name masking applies when configured
	•	Retention
Defaults 30–90 days for high-volume logs and longer retention for migration ledger and promotion decisions
Export APIs support customer archiving to their storage

Compliance posture
	•	Data minimization
Default configurations avoid collection of personal data and secrets in any artifact
	•	Residency and sovereignty
PPg previews and artifacts are stored in the selected region
Cross-region replication is opt-in and encrypted
	•	Subject access requests
Because params and rows are not logged by default, DSR scope is limited to user-managed data stores
If projects opt into capturing params, they must enable per-field PII tags in the contract for targeted scrubbing
	•	Third-party adapters and plugins
Adapters declare capability and telemetry behavior in a manifest
Untrusted adapters are disabled by default in PPg and can be allowed per project
	•	SOC 2 and GDPR alignment
Controls above map to logical access, logging, change management, and data protection domains
Policy documents reference this spec and associated ADRs for implementation evidence

Threats and mitigations
	•	Drift injection
Mitigated by marker verification and edge pre/post checks before schema changes
	•	Credential leakage
Mitigated by ambient credentials, rotation, no-disk policy, and structured redaction of all logs
	•	Privilege escalation via migrations
Mitigated by separate runner role, advisory locks, and compensation on failure boundaries
	•	Query exfiltration via diagnostics
Mitigated by redaction defaults, sensitivity tags, and no raw SQL in artifacts
	•	Noisy neighbor and resource starvation
Mitigated by per-tenant budgets, backpressure controls, and PPg quotas

Operational checklist
	•	Secrets come from a manager or managed identity, not from files in the repo
	•	Contract marker verification is enabled in all environments
	•	Raw SQL and param capture are off in CI and production
	•	Migration runner uses a dedicated role with advisory locks and ledger journaling
	•	Preflight artifacts are stored with region pinning and redaction enabled
	•	Audit log export is wired to the customer’s sink if required
	•	Incident runbooks define how to raise redaction, disable nonessential plugins, and rotate keys quickly

Open questions
	•	Do we allow per-Plan opt-out from redaction in non-prod for debugging, and how is that enforced safely
	•	Should PPg support customer-managed secret stores directly for preview DBs
	•	How to standardize sensitivity tagging across SQL and Mongo families without leaking target specifics

## ADRs

	•	ADR 077 — Secrets sourcing & rotation interface
Pluggable secret providers (env, cloud KMS, workload identity), hot-reload semantics, zero-disk policy, pool rebind behavior, and health signaling on rotation
	•	ADR 078 — Least-privilege role model
Canonical roles (app, migrator, observability), allowed operations per role, contract-marker read/write gates, and adapter guidance for privilege minimization
Complements ADR 021 & ADR 042
	•	ADR 079 — Sensitivity taxonomy & redaction levels v1
Contract-level sensitivity tags (pii, secret, financial), masking rules across telemetry, diagnostics, and PPg UI, plus dev-mode relaxations and org-wide policy pins
Complements ADR 024 (telemetry) and ADR 055 (CI redaction)
	•	ADR 080 — Tenant budgets & isolation policy
Per-tenant limits for time/rows/size, Plan class caps, noisy-neighbor protections, and how budgets interact with multi-tenant markers
Complements ADR 033 (multi-tenant enforcement)
	•	ADR 081 — Audit log schema & retention
Stable envelope for audit events, tamper-evidence options, retention tiers, export API, and mapping from runtime/PPg actions to audit categories
Complements ADR 027 (error codes) and ADR 024 (telemetry) but is audit-specific
	•	ADR 082 — Backpressure & shedding policy
Runtime and preflight queue limits, enqueue timeouts, shedding tiers (telemetry → budgets → refuse), and user-visible error semantics
Complements performance targets in the Perf & Capacity doc
	•	ADR 083 — Statement timeout & cancellation policy
Defaults by lane and mutation class, per-Plan overrides via annotations, server-side timeout negotiation, and cancellation propagation guarantees
Pairs with ADR 023 (budgets) but focuses on timeout semantics
	•	ADR 084 — Prepared statement management & invalidation
Fingerprint keys, cache sizing, invalidation triggers (DDL, server errors), and adapter responsibilities for re-prepare strategies
	•	ADR 085 — EXPLAIN & diagnostics privacy guardrails
Minimal normalized fields allowed in artifacts, stripping of literals and object names under sensitivity, and escalation knobs for secure orgs
Complements ADR 096 (EXPLAIN cache) and ADR 047 (diagnostics formats)
	•	ADR 086 — PPg API identity & webhook signing
AuthN/AuthZ for PPg preflight/promote APIs, HMAC signing format, replay protection, key rotation, and client verification libraries
Complements ADR 056 (PPg preflight-as-a-service)
	•	ADR 051 — PPg preflight-as-a-service contract
Server-side checks tied to contract hashes, branch DB lifecycle, promotion gates, and parity guarantees with client-side preflight
	•	ADR 041 — Custom operation loading via local packages + preflight bundles
Namespacing (`vendor/op@v`), schema validation, compatibility gates, and policy for loading third-party ops, forbidding network/WASM
	•	ADR 118 — Bundle inclusion policy for packs
Bundle format including pack code and manifests, security constraints, and execution policies
