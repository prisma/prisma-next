# Query Lanes

**Purpose:** define the query surfaces that produce Plans, how they relate to the data contract, and how extensions plug in.

## What's a Lane?

Think of query lanes like highway lanes - different parallel paths that all lead to the same destination (the unified `Plan` model). Each lane offers different ergonomics and use cases, but they all converge on the same execution pipeline with shared verification, guardrails, and policies. You can choose the lane that best fits your needs, or even switch between lanes in the same codebase.

## Goals

- Keep one small, well-typed SQL DSL built from the data contract
- Provide a first-class escape hatch for raw SQL that still emits a Plan with enough annotations for policy checks and auditing
- Layer an optional ORM extension on top of the DSL without coupling the DSL to ORM concerns
- Treat TypedSQL as a separate, optional extension delivered by a CLI that outputs Plan factories
- Ensure everything reduces to the same Plan model that the default runtime can consume
- Let the community compose new lanes or even alternate runtimes so long as they honor the Plan contract

## The Lanes

- **SQL DSL lane**
  - Minimal TypeScript builder constructed from `contract.json`, producing a relational AST → Plan
  - best for day-to-day CRUD and programmatic composition
- **Escape hatch lane (Raw SQL with annotations)**
  - Raw SQL string plus typed params and explicit annotations that enable verification and guardrails → Plan
  - best for advanced SQL or new engine features while retaining safety rails
- **ORM lane (extension over the DSL)**
  - Relations and convenience APIs compiled down to the same relational AST → Plan
  - best for ergonomic traversal where one call maps to one statement
- **TypedSQL lane (separate CLI extension, later)**
  - Author `.sql` files, CLI validates against a live DB and emits Plan factories
  - best when the SQL itself is the most readable artifact

All lanes emit Plans with a shared shape and behavior and are verified and executed by the same runtime pipeline

## Unified Plan Model

```typescript
export interface Plan<Row = unknown> {
  // Optional dialect-agnostic AST when available (DSL, ORM)
  ast?: QueryAST

  // Always present
  sql: string
  params: unknown[]

  meta: {
    target: 'postgres' | 'mysql' | 'sqlite'
    coreHash: string            // from contract.json
    profileHash?: string
    lane: 'dsl' | 'orm' | 'raw-sql' | 'typed-sql'
    createdAt: string
    // References and projection help guardrails, change detection, and DX
    refs?: { tables: string[]; columns: Array<{ table: string; column: string }> }
    projection?: Record<string, string> // alias → table.column
    // Annotations power policies when AST is missing or incomplete
    annotations?: {
      intent?: 'read' | 'write' | 'admin'
      isMutation?: boolean
      requiresWhereForMutation?: boolean
      hasWhere?: boolean
      hasLimit?: boolean
      sensitivity?: 'none' | 'pii' | 'phi' | 'secrets'
      ownerTag?: string
      budget?: { maxRows?: number; maxLatencyMs?: number }
      // freeform for extensions to add structured claims
      ext?: Record<string, unknown>
    }
    // Optional codecs to typecheck params and rows at the boundary
    codecs?: { params?: unknown; row?: unknown }
  }
}
```

Plans are immutable in practice and in tests (per ADR 011). **All lanes produce Plans that execute as `AsyncIterable<Row>` regardless of authoring method** — consumers choose between incremental streaming and collection.

## No Lane Choice — Same Result Type

All query lanes (DSL, ORM, raw SQL, TypedSQL) return async iterables by default:

```typescript
// DSL lane
for await (const user of sql().from(t.user).select({ id: t.user.id }).build()) { }

// ORM lane
for await (const user of orm(contract).from(t.user).select({ id: t.user.id }).build()) { }

// Raw SQL lane
for await (const row of raw({ sql: '...', params: [...], annotations: { ... } })) { }

// All three return AsyncIterable<Row> with the same streaming semantics
```

This eliminates false choices: you don't have to pick between "streaming lane" and "ORM lane" — streaming is the default for all lanes.

## SQL DSL Lane

### Surface

- Constructed from `contract.json` at runtime via `makeT(contract)`
- Column-centric API with fluent expressions

```typescript
import { sql, makeT } from '@prisma/sql'
import contract from './contract.json'

const t = makeT(contract)

const plan = sql()
  .from('user')
  .where(t.user.active.eq(true))
  .select({ id: t.user.id, email: t.user.email })
  .limit(100)
  .build()
// lane = 'dsl', ast present, refs and projection derived automatically
```

### Type Inference

- Result type inferred from select projection
- Nullability propagates across joins
- Predicate types validated from contract codecs

## Escape Hatch Lane

### Intent

- Let developers or agents ship hand-written SQL while still participating in verification, guardrails, budgets, and auditing
- Do not parse SQL in the core path — rely on required annotations to supply minimal structure for policy checks

### Minimal API

```typescript
import { raw } from '@prisma/sql/raw'

const plan = raw({
  sql: `select u.id, u.email from "user" u where u.active = $1 limit 100`,
  params: [true],
  // Required annotations for guardrails when AST is absent
  annotations: {
    intent: 'read',
    isMutation: false,
    hasWhere: true,
    hasLimit: true
  },
  // Strongly-typed projection and refs unlock better linting and DX
  refs: { tables: ['user'], columns: [{ table: 'user', column: 'id' }, { table: 'user', column: 'email' }] },
  projection: { id: 'user.id', email: 'user.email' },
  // Optional codecs for runtime value checks
  codecs: { row: z.object({ id: z.number(), email: z.string() }) }
})
```

### Guardrails with Raw SQL

- The runtime enforces policy based on annotations when no AST is present — `mutation-requires-where`, `limit-required`, `maxRows`, `maxLatencyMs`
- Optional adapters may best-effort parse or run EXPLAIN to enrich refs and projection — not required for MVP

## ORM Lane

### Intent

- Provide ergonomic include and relations on top of the DSL without coupling the DSL to ORM
- Lower to the same relational AST then to SQL → Plan

### Surface Sketch

```typescript
import { orm } from '@prisma/orm'
const r = orm(contract, { t })

const plan = r.from(t.user)
  .include(r.user.posts, posts => posts
    .select({ id: t.post.id, title: t.post.title })
    .limit(10),
    { asArray: true, alias: 'posts' }
  )
  .select({ id: t.user.id, email: t.user.email })
  .build()
// lane = 'orm', ast present, refs/projection derived
```

### Lowering in Postgres MVP

- **1:N nested** via LEFT JOIN LATERAL (...) and json_agg per parent row
- **N:1** via LEFT JOIN with prefixed child projection
- **One call → one statement** remains the rule

ORM ships later than DSL and raw lanes

## TypedSQL Lane (Separate Extension, Later)

- A CLI that reads `.sql` files, validates param and result types against a live DB or the contract, and emits Plan factories
- Generated factories return Plans with `lane = 'typed-sql'`, `ast` omitted, `refs` and `projection` populated either from a light parser or from developer hints
- Fits teams who prefer SQL as source while retaining the same runtime safety

## Selecting from Views

Views and other read-only sources (ADR 127) are exposed through the DSL identically to tables, with full type safety and read-only enforcement.

### View as Source

Pack-owned view blocks project as read-only sources in `contract.sources`:

```typescript
import { sql, makeT } from '@prisma/sql'

const t = makeT(contract)

// Views appear in the same `t` namespace as tables
const activeUsersView = t['public.active_users']
const dailyStatsView = t['analytics.daily_stats']

// Query views like tables, with full type inference
const plan = sql()
  .from(t['public.active_users'])
  .select({
    id: t['public.active_users'].id,
    email: t['public.active_users'].email
  })
  .limit(100)
  .build()

// Result type is inferred from the source projection
for await (const row of db.execute(plan)) {
  console.log(row.id, row.email)  // fully typed
}
```

### Read-Only Enforcement

Mutations on read-only sources are blocked at two levels:

**Build-time (lints):**
```typescript
// This fails at build time with a lint error
const badPlan = sql()
  .from(t['public.active_users'])
  .update({ email: 'new@example.com' })  // ❌ Lint error: mutation-on-read-only-source
  .build()
```

**Runtime (guardrails):**
```typescript
// If somehow a mutation Plan reaches the runtime, it's blocked
const plan = raw({
  sql: `update public.active_users set email = $1`,
  params: ['new@example.com'],
  annotations: { isMutation: true }
})

await db.execute(plan)  // ❌ Runtime error: mutations on read-only source not allowed
```

### Joins with Views

Lanes can JOIN queries with views:

```typescript
const plan = sql()
  .from(t.user)
  .join(
    t['public.active_users'],
    t.user.id.eq(t['public.active_users'].id)
  )
  .select({
    id: t.user.id,
    email: t['public.active_users'].email,
    active: t.user.active
  })
  .build()
```

### Adapter Lowering

Adapters lower queries against views based on capabilities:

- **Native view support** (Postgres, MySQL): Inline view name in FROM clause
- **No native support** (or via CTE): Expand view definition as CTE and reference

Lanes don't need to know which strategy is used; adapters choose transparently.

## Pack Operators

In addition to pack-provided data constructs (views, enums), packs can register domain-specific functions and operators that integrate with the DSL (ADR 113).

### Operator Registry

Packs register functions and operators at initialization:

```typescript
// pgvector pack registers distance and similarity operators
registry.registerOperator({
  ns: 'pgvector',
  symbol: '<->',  // SQL operator
  name: 'distance',
  signature: (left: Vector, right: Vector) => number,
  lowerFn: (ast: BinaryOp, context: LowerContext) => string
})

registry.registerFunction({
  ns: 'pgvector',
  name: 'cosine_similarity',
  signature: (left: Vector, right: Vector) => number,
  lowerFn: (ast: FunctionCall, context: LowerContext) => string
})
```

### DSL Usage

Lanes expose operators and functions through the registry:

```typescript
import { sql, op, fn, makeT, param } from '@prisma/sql'

const t = makeT(contract)
const query = param.vector('q')

const plan = sql()
  .from(t.document)
  .select({
    id: t.document.id,
    // Use pgvector distance operator
    distance: op('pgvector', '<->', t.document.embedding, query)
  })
  .orderBy(op('pgvector', '<->', t.document.embedding, query))
  .limit(10)
  .build()

// TypeScript knows the operator signature:
// distance: number ✓

// Build fails if types don't match
const badPlan = sql()
  .from(t.document)
  .select({
    // TypeError: string is not compatible with Vector
    distance: op('pgvector', '<->', t.document.embedding, 'not-a-vector')
  })
  .build()
```

### Plan Annotations

Plans track extension operators for guardrails and advisors:

```typescript
interface Plan {
  meta: {
    refs?: {
      operators: [
        { ns: 'pgvector', symbol: '<->', columns: ['embedding'] }
      ],
      functions: [
        { ns: 'pgvector', name: 'cosine_similarity', columns: ['embedding'] }
      ]
    }
  }
}
```

Guardrails can enforce policies like:
- "pgvector operators require `pgvector.ivfflat` capability"
- "Distance queries over unindexed embeddings warn with cost estimate"

### Capability Branching

Lanes branch on capabilities to enable/disable features:

```typescript
// In the DSL
if (adapter.supports('pgvector.ivfflat')) {
  // Use IVFFlat index for faster similarity search
  return sql()
    .from(t.document)
    .orderBy(op('pgvector', '<->', t.document.embedding, query))
    .limit(10)
} else {
  // Fall back to full table scan with sort
  return sql()
    .from(t.document)
    .orderBy(t.document.embedding)  // Sorts as bytea, not semantic
    .limit(10)
}
```

See **ADR 113** and **ADR 117** for the operator registry and capability model.

## Runtime Pipeline

All lanes share the same execution model:

1. **compile**
   - For DSL and ORM, lower AST → SQL with adapter and fill refs and projection
   - For raw and TypedSQL, use provided sql, params, and annotations
2. **beforeExecute**
   - Verify `coreHash`, apply lints and budgets, resolve policy config
3. **execute**
   - Driver streams rows back as `AsyncIterable<Row>`, returning immediately
4. **onRow (per-row, optional)**
   - Plugins observing each row as it streams (for real-time telemetry or sampling)
5. **afterExecute**
   - Called when stream completes or consumer stops iterating; aggregated stats and telemetry
6. **onError**
   - Diagnostics and redaction

Result is always `AsyncIterable<Row>` where `Row` is inferred from the Plan's projection. Plugins observe the same Plan irrespective of lane.

## Extension Functions and Operators

Query lanes surface extension functions and operators through a registry system that enables type-safe usage of domain-specific capabilities.

### Function Registry Integration

Extension packs register functions and operators with signatures and rendering hooks:

```typescript
// Extension function usage in DSL
const dist = fn('pgvector.fn.distance')  // typed factory
const sim = op('pgvector.op.<->')        // typed operator

sql().from(t.item)
  .where(dist(t.item.embedding, param.vector('v')).lt(0.8))
  .orderBy(sim(t.item.embedding, param.vector('q')))
```

### Plan Annotations for Extensions

Plans include structured references to extension functions/operators in `meta.refs`:

```typescript
interface Plan {
  meta: {
    refs?: {
      functions?: Array<{ ns: string, symbol: string, signatureId: string }>
      operators?: Array<{ ns: string, symbol: string, signatureId: string }>
    }
  }
}
```

This enables policy enforcement and capability gating without parsing SQL.

### TypedSQL Header Extensions

TypedSQL files support extension intent and parameter annotations:

```sql
-- @param vector: pgvector/vector(dim=1536)
-- @intent read
-- @extension pgvector.fn.distance
SELECT pgvector_distance(embedding, $1) as dist
FROM items
WHERE pgvector_distance(embedding, $1) < 0.8
ORDER BY embedding <-> $1
LIMIT 10;
```

### Capability Branching Policy

Lanes branch on capabilities rather than target strings:

- **Capability checks**: `adapter.supports('pgvector.ivfflat')`
- **Early failure**: Missing capabilities cause build-time or connect-time errors
- **Graceful degradation**: Optional capabilities enable fallback behavior
- **Policy enforcement**: Guardrails can reject certain operators based on capability metadata

## Extensibility Model

- Extensions can create new lanes by compiling to a Plan and tagging `meta.lane` — the only hard requirement is honoring the Plan contract
- Extension functions/operators integrate through the registry system with type safety and capability gating
- Alternate runtimes are allowed — they must verify `coreHash`, respect immutability, and document hook semantics
- Adapters keep SQL lowering out of lane code — dialect specifics live behind the adapter SPI

## Capability-gated query features

Some query features are always available at the AST level but require explicit capability keys for lowering on specific targets. Lanes branch on capabilities rather than target strings.

- Joins
  - Always available in the AST.
  - Lowered features are gated by `join.lateral`, `join.semi`, `join.anti` capability keys.

- Projection features
  - `.distinct()` requires `projection.distinct`.
  - `.distinctOn()` requires `projection.distinctOn` (e.g., Postgres-only).

- Advisors and lints
  - Index coverage checks for equality joins when a corresponding FK exists in the contract.
  - Rules consult `meta.refs` and adapter capabilities.

- Extension operators
  - Packs register operators and functions through the registry (ADR 113).
  - DSL and TypedSQL lanes consume these with type-safe signatures and capability gating.

## Package Map

- **@prisma/relational-ir**
  - data contract types and validation
- **@prisma/sql**
  - DSL, AST, compiler, makeT, Plan helpers
- **@prisma/sql/raw**
  - escape hatch helpers and annotation schema
- **@prisma/runtime**
  - default runtime with plugin hooks and guardrails
- **@prisma/orm**
  - optional ORM extension that compiles to the same AST
- **@prisma/typed-sql-cli (later)**
  - reads SQL files and emits Plan factories that the runtime can execute

This keeps the core small, makes extensions first-class, and ensures every path ends in the same verifiable Plan that our runtime and PPg can reason about

## ADRs

- **ADR 011** — Unified Plan model across lanes
  - One Plan contract for all lanes with optional ast and optional annotations
  - Plan carries sql, params, meta.coreHash, refs, projection, lane
  - Immutability reaffirmed and enforced by tests
  - Why an ADR: cements the integration surface every lane and runtime must honor

- **ADR 012** — Raw SQL escape hatch with required annotations
  - Define the minimal annotations a raw Plan must provide for policy checks (intent, isMutation, hasWhere, hasLimit)
  - Optional structured refs, projection, and codecs for stronger guardrails
  - No core SQL parsing required; adapters may enrich as an optional add-on
  - Why an ADR: sets safety bar and prevents "raw mode" from bypassing verification

- **ADR 013** — Lane-agnostic Plan identity and hashing
  - Plan hashing and change detection ignore lane and focus on (sql, params, normalized meta)
  - What metadata participates and what is excluded for stability
  - Why an ADR: avoids accidental hash churn when authors swap lanes

- **ADR 014** — Runtime hook API v1 (lane-neutral)
  - Hooks: beforeCompile, beforeExecute, afterExecute, onError with Plan in/out guarantees
  - Error semantics, budgets, and lint levels
  - Why an ADR: locks the extension surface for guardrails and telemetry independent of lane

- **ADR 015** — ORM as an optional extension over the DSL
  - One call → one statement rule
  - Lowering strategy responsibility lives in adapter profiles, not in the DSL core
  - Why an ADR: prevents ORM concerns from leaking into the base builder

- **ADR 016** — Adapter SPI for lowering relational AST
  - Capability flags (e.g., lateral, jsonAgg) and deterministic lowering requirements
  - Golden SQL testing obligations and stability guarantees
  - Why an ADR: formalizes where dialect logic lives so lanes stay portable

- **ADR 017** — Extension and alternate runtime compatibility policy
  - What third-party lanes and runtimes must do: honor Plan contract, verify coreHash, respect immutability, document hooks
  - Versioning and compatibility matrix
  - Why an ADR: encourages ecosystem contributions without fragmenting safety

- **ADR 018** — Plan annotations schema and validation
  - Canonical JSON schema for annotations and validation rules at build time and runtime
  - Reserved keys vs annotations.ext for custom claims
  - Why an ADR: keeps raw Plans verifiable and lets policies evolve safely

- **ADR 019** — TypedSQL as a separate CLI that emits Plan factories
  - Out-of-tree tool, not a core lane
  - Validates params/result types, stamps coreHash, produces factories returning Plans
  - Why an ADR: captures the integration contract without re-introducing client codegen

- **ADR 020** — Result typing and projection inference rules
  - How DSL and ORM compute result types from projections and joins
  - Nullability propagation rules for LEFT JOIN and aggregates
  - Why an ADR: stabilizes type inference so agents and users can rely on it
- **ADR 018** — Plan annotations schema and validation
  - Canonical JSON schema for annotations and validation rules at build time and runtime
  - Reserved keys vs annotations.ext for custom claims
  - Why an ADR: keeps raw Plans verifiable and lets policies evolve safely
- **ADR 036** — TypedSQL header annotation spec
  - Formal grammar for `-- @param`, `-- @intent`, sensitivity, and optional refs/projection hints
  - Extension parameter and function annotations
- **ADR 065** — Adapter capability schema & negotiation v1
  - Canonical capability keys, optional features, negotiation flow at connect time
  - How lanes/plugins branch on capabilities rather than target strings
- **ADR 113** — Extension function & operator registry
  - Registry assembly from packs, type-safe function/operator usage, capability gating
  - Deterministic rendering hooks scoped to adapter profiles
