# 6. Preflight & CI Integration

This document specifies how Prisma Next validates changes in CI before they reach production. Preflight consumes Plans and contract edges to detect safety, performance, and correctness issues early, and reports structured diagnostics that humans and agents can act on.

## Goals

- Catch unsafe queries and migrations before merge or deploy
- Provide deterministic, machine-readable diagnostics for agents and CI
- Offer two modes with predictable cost: Shadow DB and EXPLAIN-only
- Keep privacy-first defaults with redaction and minimal surface area
- Be idempotent and reproducible across CI reruns

## Modes

### Shadow DB mode

Spin up a temporary database that mirrors the target engine and version, apply the proposed migration path (edges), seed optional fixtures, and execute Plans with budgets and lints enabled:
- Highest confidence for schema changes and complex queries
- Validates pre/post checks on operations and verifies the marker write sequence
- Executes queries without touching real data
- **Migration Bundle Required**: For hosted preflight, consumes pre-built migration bundles containing compiled ops and manifests

**Trade-offs:**
- Slower and costlier than EXPLAIN-only
- Requires DDL privileges and ephemeral database orchestration

### EXPLAIN-only mode

Do not execute queries or mutate schema. For each Plan, compute sqlFingerprint, run adapter EXPLAIN with timeout budgets, apply static lints and heuristics, and evaluate budgets:
- Fast and cheap for PRs that adjust queries or add minor schema elements
- Safe default for forks and external contributors
- **No Migration Bundle**: Uses static analysis only, no custom ops execution

**Trade-offs:**
- Estimates may differ from observed runtime behavior
- Cannot validate data-dependent tasks or backfills

### Migration Bundle Acceptance Checks

For hosted preflight using migration bundles, the following acceptance checks are enforced:

### Bundle Validation
- **Bundle Integrity**: Verify bundle.json index matches actual contents
- **Contract Alignment**: Ensure contract.json matches expected coreHash and profileHash
- **Op Compilation**: Validate all ops/*.js files are valid ESM JavaScript
- **Manifest Compliance**: Check all ops/*.manifest.json files declare required fields
- **No Network Dependencies**: Verify no dynamic imports or network operations in ops
- **Node Version Compatibility**: Confirm bundle targets supported Node runtime version

### Security Constraints
- **No WASM Engines**: Reject bundles that attempt to load WebAssembly modules
- **No Network Egress**: Enforce sandbox prevents network operations from ops
- **Signed Bundles**: Require cryptographic signatures for production preflight
- **Resource Limits**: Enforce declared timeout, memory, and CPU constraints

### Acceptance Outcomes
- **PASS**: Bundle valid, all checks pass, ready for shadow apply
- **FAIL**: Bundle invalid or security constraints violated, preflight blocked
- **WARN**: Bundle valid but with non-blocking issues (e.g., deprecated Node version)

## Diagnostics Schema

All preflight outcomes are emitted as newline-delimited JSON (NDJSON) and a summary JSON for CI uploads. Events align with telemetry envelopes but include PR-local context.

### Common fields

- `ts`: ISO timestamp
- `env`: ci
- `service`: repo slug or configured app name
- `commit`: git SHA
- `pullRequest`: provider key and PR number when available
- `contractHash`, `profileHash`
- `lane`: dsl|orm|typed-sql|raw
- `planId`, `sqlFingerprint`

### Artifacts

- `preflight.summary.json` overall status and counts
- `preflight.events.ndjson` stream of detailed events
- Optional SARIF for code scanning integrations

### Example summary

{
  "status": "fail",
  "totals": { "plans": 42, "errors": 3, "warnings": 7 },
  "budgets": {
    "rowCount": { "violations": 2 },
    "latency": { "violations": 1 },
    "sqlSize": { "violations": 0 }
  },
  "lints": {
    "no-select-star": { "errors": 1, "warnings": 0 },
    "mutation-requires-where": { "errors": 1, "warnings": 0 },
    "no-missing-limit": { "errors": 0, "warnings": 5 }
  },
  "mode": "explain",
  "shadow": { "used": false }
}

## GitHub/GitLab App Integration

### Capabilities

- Check runs with pass/fail and summary counts
- Inline annotations on changed files for Plans with source maps or lane-provided file ranges
- Comment bot with grouped diagnostics and suggested fixes
- Required checks gating merge based on budgets and lints

### Exit codes

- `0` success, no errors and budgets within thresholds
- `1` failure due to lint at level error or budget violation configured as blocking
- `2` infrastructure error such as adapter startup or shadow DB provisioning failure
- `3` policy violation like missing marker or disallowed irreversible ops

CLI respects `--soft-fail` to return 0 but still post annotations for exploratory branches

### Budgets in CI

- Row-count and SQL-size budgets enforced pre-execute
- Latency budget uses EXPLAIN-derived estimates or shadow observed latencies
- Budget precedence follows ADR 023 and CI config may tighten thresholds

### Example workflow snippet

```yaml
- name: Preflight
  run: |
    npx prisma-next preflight \
      --mode=explain \
      --budgets.maxRows=10000 \
      --budgets.maxLatencyMs=200 \
      --lint.mode=strict \
      --out=.artifacts/preflight
- name: Upload diagnostics
  uses: actions/upload-artifact@v4
  with:
    name: preflight
    path: .artifacts/preflight/*
```

## Idempotency

- The preflight CLI must be rerunnable with identical inputs yielding identical results except for timestamps
- Shadow DB names are derived from contractHash, commit, and PR number to avoid collisions
- sqlFingerprint caching is shared across steps within the job to avoid duplicate EXPLAIN calls
- For migrations, the shadow runner is always reset to a clean baseline before applying edges

## Data Redaction

- No raw SQL or parameter values are stored or emitted by default
- Column names are included only when privacy mode allows and not tagged sensitive in the contract
- Result samples are never emitted in CI
- Shadow DBs use synthetic seed data where needed, not production dumps

## Cost Controls

- Global time budget for preflight step with graceful degradation
- EXPLAIN timeouts per query with sampling in EXPLAIN-only mode
- Shadow DB TTL with automatic teardown on completion or cancellation
- Max Plans checked per run with include/exclude globs for lanes or folders
- Concurrency cap for query checks to avoid blowing CI CPU quotas

### Example configuration

```json
{
  "privacy": "standard",
  "mode": "explain",
  "budgets": { "maxRows": 10000, "maxLatencyMs": 200, "maxSqlBytes": 200000 },
  "lint": { "mode": "strict" },
  "limits": {
    "maxPlans": 500,
    "explainTimeoutMs": 50,
    "concurrency": 8,
    "totalBudgetMs": 120000
  },
  "sampling": {
    "explain": 0.1,
    "elevateOnViolation": true
  }
}
```

## Planner and Migration Preflight

- In Shadow DB mode, apply the migration path from marker hash to desired contractHash and halt on any failing pre/post check
- Emit migration.edge events with risk from ADR 037 and ledger preview
- Post-apply, run query preflight to ensure queries match the new contract

## Red-Green Author Flow
- **Red:** preflight fails with actionable annotations and hints from lints and budgets
- **Green:** after fixes, preflight passes with cached EXPLAIN outcomes and minimal overhead
- **Merge:** PPg can run the same preflight server-side as a promotion gate

## CLI

```bash
prisma-next preflight \
  --mode=shadow|explain \
  --shadow-url=postgres://... \
  --budgets.maxRows=... \
  --lint.mode=strict|permissive \
  --out=.artifacts/preflight \
  --format=json|sarif|both \
  --soft-fail
```

## Agent Ergonomics

- Agents can read `.artifacts/preflight/preflight.events.ndjson` and propose fixes based on rule IDs and hints
- The contract and Plan shapes are stable, enabling automated re-planning or index suggestions
- Structured exit codes enable agent-controlled retries or fallbacks

## Observability

- Optional emission to OTLP or PPg sink for flake investigation
- Counters for cached vs fresh EXPLAIN, violations by rule, and mode usage
- Fingerprint-level trend charts to spot regressions

## Open Questions

- Should we add a result payload size budget in preflight for JSON-heavy apps
- How to share EXPLAIN caches across CI jobs safely without risking leakage across branches
- Minimum fixtures API for Shadow DB to make complex checks useful without full seed scripts

## ADRs

- **ADR 045** — Preflight mode selection policy
  When to use Shadow DB vs EXPLAIN-only, repo/org overrides, PR labels and path-based routing, and default policies per branch type
  Complements: ADR 029 (shadow semantics), ADR 023 (EXPLAIN policy)

- **ADR 046** — EXPLAIN cache key & invalidation
  Fingerprint construction beyond sqlFingerprint (e.g., param-shape buckets), TTLs, adapter/profile/contract participation, and salt rotation hooks
  Complements: ADR 023 (budgets & EXPLAIN), ADR 042 (marker evolution if hash participates)

- **ADR 047** — Diagnostics artifacts & formats
  Canonical NDJSON event stream and summary JSON for preflight, optional SARIF for code scanning, versioning guarantees for agents and CI
  Complements: ADR 024 (telemetry schema), ADR 027 (error envelope)

- **ADR 048** — CI exit codes & policy mapping
  Stable exit codes, how lint levels and budget violations map to pass/fail, --soft-fail semantics, and retry/backoff guidance for CI runners
  Complements: ADR 022 (lint taxonomy), ADR 023 (budgets)

- **ADR 049** — Plan source maps for CI annotations
  Lane responsibilities for attaching file/line ranges to Plans, minimal schema, and how preflight surfaces inline annotations in GitHub/GitLab
  Complements: ADR 014 (hook API), ADR 031 (adapter capabilities, for lane support)
- **ADR 050** — Preflight redaction policy
  Privacy modes specific to CI artifacts (column name masking via contract sensitivity, no raw SQL, no params), provider-safe defaults
  Complements: ADR 024 (privacy), focuses on CI outputs rather than runtime telemetry

- **ADR 051** — PPg preflight-as-a-service contract
  Server-side checks tied to contract hashes, branch DB lifecycle, promotion gates, and parity guarantees with client-side preflight
  Complements: ADR 029 (shadow semantics), positions PPg differentiation

- **ADR 052** — Result payload size budget
  Optional budget for serialized response size, evaluation timing, interaction with latency/row budgets, and adapter support for streaming hints
  Complements: ADR 023 (budgets)

- **ADR 053** — Golden SQL & Plan snapshots policy
  How we record and compare golden outputs for stability, allowed whitespace/alias normalization, and when to intentionally bump goldens
  Complements: ADR 016 (lowering determinism), ADR 020 (typing rules)

- **ADR 054** — Shadow DB fixtures API
  Minimal, deterministic seeding contract for preflight (datasets, generators, privacy constraints), and caching of seeded shadows across jobs
  Complements: ADR 029 (shadow semantics)
