# PPg Integration

**Purpose:** describe how Prisma Postgres (PPg) becomes a contract-aware platform that pairs with Prisma Next to deliver safer changes, faster feedback, and measurable reliability.

## Goals

- Make PPg understand data contracts and Plans so it can gate promotion, orchestrate migrations, and advise on performance
- Provide a zero-friction preview workflow per PR with automatic contract alignment
- Offer managed orchestration for risky DDL and data backfills with strong resumability
- Expose server-side guardrails and observability that complement the in-app runtime

## Architecture at a Glance

- **Control plane:** APIs and workers that parse contracts, resolve migration paths, orchestrate edges, and post diagnostics to VCS
- **Data plane:** Customer Postgres clusters with a small PPg system schema for markers, ledgers, and advisory locks
- **Integration surfaces:** GitHub/GitLab app, CLI, and REST endpoints used by CI and the Prisma Next runtime

## Contract Markers in PPg

PPg stores and verifies contract identity inside the database so app and DB can prove alignment:

- **Marker table** `ppg_contract.marker`
  - `core_hash` core data contract hash
  - `profile_hash` adapter profile hash (dialect/lowering profile)
  - `updated_at` timestamp
  - `optional tenant_id` for multi-tenant apps
  - one row per database or tenant

- **Functions**
  - `ppg_contract.get_marker()` → current { core_hash, profile_hash }
  - `ppg_contract.set_marker(core_hash, profile_hash)` guarded by orchestration role only
  - `ppg_contract.assert(core_hash, profile_hash)` raises error on mismatch

- **Verification modes**
  - PPg can enforce checks on connection via RLS or login triggers for platform-managed services
  - Apps can run startup or on-first-use verification using the same functions

**References:** ADR 021, ADR 042, ADR 033

## Preview Databases per PR

Every pull request can have an isolated preview DB tied to its proposed contract:

- **Lifecycle**
  - Create on PR open or first preflight request
  - Name and tag include repo, prNumber, core_hash, profile_hash
  - TTL with auto-teardown on PR close or inactivity

- **Provisioning**
  - Base from a project template snapshot or empty baseline
  - Apply migration path from baseline hash to PR's core_hash
  - Optional seed packs or deterministic fixtures
  - **Migration Bundle Only**: PPg accepts only pre-built migration bundles, never resolves user repos or package managers

- **Usage**
  - Preflight run in CI hits the PR preview DB for shadow mode
  - App preview deployments can point at the same DB for end-to-end validation

- **Promotion**
  - On merge, PPg computes path from production marker to target core_hash
  - Promotion gate ensures preflight passed and path is contiguous and verified

**References:** ADR 029, ADR 028, ADR 039

## Promotion Tied to Hashes

Deploys and promotions are gated on contract alignment:
	•	Gate checks
	•	Production DB marker equals expected core_hash before migration
	•	Path exists from current marker to target core_hash in the migration DAG
	•	All edges in path verified in preflight for the same profile_hash
	•	Outcomes
•	If checks pass, orchestration runs edges then updates marker via ppg_contract.set_marker
•	If checks fail, PPg posts diagnostics to the PR and blocks promotion

## Pack enablement and service parity

- PPg declares which packs are supported server-side and how they are enabled for preflight bundles.
- Bundles may include pack code and manifests (ADR 118) or reference a hosted catalog entry by namespace and version.
- Constraints apply to bundles including pack code: no network, no dynamic imports, no WASM; integrity and signature required.

## Hosted Pack Catalog

PPg maintains a hosted catalog of extension packs per region with capability negotiation:

### Pack Registry
- **Regional Availability**: Packs available per region with compliance requirements
- **Version Management**: Semantic versioning with deprecation policies
- **Capability Indexing**: Packs indexed by required capabilities for fast discovery
- **Security Scanning**: Automated security scanning of pack code and dependencies

### Capability Negotiation Flow
1. **Client Request**: Client requests packs for specific capabilities
2. **Regional Filtering**: PPg filters packs by region and compliance requirements
3. **Capability Matching**: Matches client requirements against pack capabilities
4. **Version Resolution**: Resolves compatible pack versions
5. **Bundle Generation**: Generates capability-specific bundles for download

### Pack Distribution
- **CDN Distribution**: Pack bundles distributed via CDN for fast download
- **Integrity Verification**: SHA-256 integrity hashes for pack verification
- **Update Notifications**: Webhook notifications for pack updates
- **Rollback Support**: Previous pack versions available for rollback

## Server-Side Guardrails for Extensions

PPg enforces server-side guardrails for extensions:

### Extension Security
- **Sandbox Execution**: Extension code runs in isolated sandbox environments
- **Resource Limits**: CPU, memory, and execution time limits for extension operations
- **Network Isolation**: No network access for extension code
- **File System Restrictions**: Limited file system access for extension operations

### Policy Enforcement
- **Capability Gating**: Server-side enforcement of capability requirements
- **Budget Enforcement**: Server-side enforcement of query budgets and limits
- **Audit Logging**: Comprehensive logging of extension usage and policy violations
- **Compliance Monitoring**: Real-time monitoring of extension compliance

### Extension Monitoring
- **Performance Metrics**: Extension-specific performance monitoring
- **Error Tracking**: Extension error tracking and alerting
- **Usage Analytics**: Extension usage analytics and reporting
- **Security Monitoring**: Security monitoring for extension operations

Managed migrations orchestration

PPg runs migrations with best-practice sequencing, resumability, and safety checks
	•	Runner semantics
	•	Advisory lock per database or tenant
	•	Partition ops into transactional groups and non-transactional boundaries
	•	Journal to a migration ledger for resume and auditing
	•	Enforce pre/post checks per op for idempotency
	•	**Custom Operations Constraints**
	•	Only bundled operations accepted - no dynamic loading from repos or registries
	•	No network egress from custom ops - enforced by sandbox
	•	No WASM engines - JavaScript/TypeScript only
	•	Node version pinned to match PPg runtime environment
	•	Built-in strategies
	•	Concurrent index
	•	Use CREATE INDEX CONCURRENTLY when safe
	•	Backfill in chunks with progress journaling when needed
	•	Phased defaults
	•	Add nullable column → backfill → set default → set NOT NULL
	•	Each phase has compensations and clear checkpoints
	•	Chunked backfills
	•	Range or cursor chunking with adaptive batch sizes
	•	Rate limits and kill switches to protect production SLOs
	•	Irreversible ops
	•	Require explicit allow list and high-risk classification
	•	Preflight in a preview DB is mandatory

Index and plan advisor

PPg analyzes Plans and observed workloads to suggest changes
	•	Inputs
	•	Query Plans and sqlFingerprint from app telemetry
	•	EXPLAIN results from preflight or server-side sampling
	•	Contract refs for table and column usage
	•	Advisories
	•	Missing equality index on frequent predicates or FKs
	•	Composite index order improvements for common filters/sorts
	•	Redundant or unused indexes
	•	Join cardinality hotspots that warrant schema or index changes
	•	Delivery
	•	Inline PR comments with suggested DDL as machine-readable ops
	•	Console surfaced hints in PPg UI with impact estimates and risk notes

Optional server-side guardrails

PPg can enforce guardrails at the platform edge for production apps
	•	Policies
	•	Block SELECT * or require LIMIT caps by role
	•	Cap estimated cost or rows by Plan fingerprint class
	•	Deny mutations without predicates for certain tables or roles
	•	Require contract alignment for connections
	•	Enforcement modes
	•	Observe log and alert only
	•	Warn return structured warnings to the client alongside results
	•	Block reject with stable error codes and remediation hints
	•	Compatibility
	•	Mirrors in-app lint taxonomy and budgets so teams get consistent outcomes across environments

Observability surfaces

PPg provides first-class visibility for contract health, migrations, and performance
	•	Dashboards
	•	Contract alignment across environments and tenants
	•	Migration runs with boundary timelines, risk classification, and outcomes
	•	Query latency distributions by sqlFingerprint with trend lines
	•	Index advisor suggestions and adoption rate
	•	Logs and events
	•	Structured NDJSON streams and OTLP metrics/traces
	•	Redaction by default for SQL text and params, with column sensitivity masking
	•	Correlation via planId, sqlFingerprint, and core_hash
	•	APIs
	•	REST and webhook endpoints for preflight results, advisor output, and promotion status
	•	Artifact downloads for diagnostics and ledgers

SLOs and budgets

Target service levels for PPg contract-aware features
	•	Preflight-as-a-service
	•	P95 end-to-end under 90 s for EXPLAIN-only PRs of ≤200 Plans
	•	P95 under 8 min for preview DB runs with ≤5 additive edges
	•	Orchestration
	•	Resume within 30 s after worker crash or deploy
	•	Zero partial apply without high-risk acknowledgement
	•	Advisory lock acquisition P95 under 500 ms
	•	Availability
	•	99.9% for APIs and UI
	•	99.95% for promotion gates during configured deployment windows
	•	Data safety
	•	Ledger durability with at least once event delivery to storage
	•	Marker updates are atomic with edge completion

⸻

Security and compliance
	•	Principle of least privilege with dedicated orchestration roles
	•	No raw SQL or params persisted in PPg central logs by default
	•	Customer-managed keys and endpoints for telemetry sinks
	•	Signed webhooks with rotation and replay protection

⸻

Integration with Prisma Next
	•	Prisma Next runtime verifies markers and sends Plans with annotations
	•	CI uses client-side preflight, PPg can mirror server-side for parity
	•	TypedSQL and raw lanes emit Plans that PPg can analyze and gate the same way
	•	Migration edges authored in-repo are executed by PPg with ledger and diagnostics

## ADRs

	•	ADR 055 — PPg server-side guardrails policy model
Canonical policy schema, precedence with client-side lints/budgets, enforcement modes (observe|warn|block), and escalation rules
	•	ADR 056 — Preview DB lifecycle, quotas, and TTLs
Naming, isolation guarantees, per-org/project quotas, TTL defaults/overrides, and teardown reliability
	•	ADR 057 — Promotion gate protocol
Required checks to move from preview to prod, hash alignment rules, evidence bundle format from preflight, and failure remediation paths
	•	ADR 058 — Index & plan advisor recommendation format
Machine-readable suggestion schema (confidence, impact, DDL ops), deduping and suppression rules, and how PR comments are generated
	•	ADR 059 — Managed backfill orchestration
Chunking strategies, pacing controls, pause/resume semantics, and SLO-aware throttling with progress journaling
	•	ADR 060 — Orchestration RBAC & least-privilege roles
Role set for marker, ledger, DDL, and read-only operations, rotation policy, and per-tenant scoping
	•	ADR 061 — PPg telemetry parity with client runtime
Required fields, redaction alignment, sampling, and merging client Plan events with server EXPLAIN/preflight events
	•	ADR 062 — Evidence artifacts for compliance
What PPg stores for audits: ledger snapshots, pre/post check results, Plan fingerprints, retention policy, and export API
	•	ADR 063 — Preview seed fixtures contract
Minimal deterministic seeding API, privacy constraints, caching of seeded snapshots, and invalidation triggers
	•	ADR 064 — Cross-region placement & data sovereignty for previews
Region selection policy, replication options, and constraints for jurisdictional compliance
	•	ADR 051 — PPg preflight-as-a-service contract
Server-side checks tied to contract hashes, branch DB lifecycle, promotion gates, and parity guarantees with client-side preflight
	•	ADR 118 — Bundle inclusion policy for packs
Bundle format including pack code and manifests, security constraints, and execution policies
