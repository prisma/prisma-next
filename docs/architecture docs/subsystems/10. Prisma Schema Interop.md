# Prisma Schema Interop (.prisma)

## Overview

Prisma Next supports Prisma schema files (`.prisma`) as a first-class contract authoring source for SQL/Postgres workflows.

The subsystem has three responsibilities:

1. Parse `.prisma` input and convert it into Prisma Next contract IR for `contract emit`.
2. Apply a `.prisma` schema to an empty or existing database with Prisma 7 semantics (`db push` parity path).
3. Introspect a live database and print a formatted `.prisma` schema (`db pull` parity path).

This design intentionally reuses Prisma 7 parser/CLI behavior for syntax and formatting parity while keeping Prisma Next contract artifacts deterministic.

## Architecture

### Core package: `@prisma-next/contract-psl`

Implemented modules:

- `src/schema-normalize.ts`
  - Removes datasource URL keys (`url`, `directUrl`, `shadowDatabaseUrl`) before Prisma 7 parsing/CLI use.
- `src/contract-converter.ts`
  - Loads schema source from path or inline text.
  - Parses and resolves provider via `@prisma/internals` (`getConfig`, `getDMMF`).
  - Converts DMMF into Prisma Next SQL contract IR (`models`, `storage.tables`, relations, defaults, execution generators, enum types).
  - Records unsupported/partially represented features in `contract.meta.prismaPsl.missingFeatures`.
- `src/prisma-cli.ts`
  - `prismaDbPush()` wrapper for `prisma db push`.
  - `prismaDbPull()` wrapper for `prisma db pull --print`.
  - `generatePrismaDiffSql()` helper for schema diff SQL generation.

### CLI integration

`@prisma-next/cli` supports `.prisma` sources in two places:

- `contract emit`
  - Accepts `.prisma` source from:
    - `config.contract.source: './schema.prisma'`
    - `config.contract.source: { kind: 'prisma', schemaPath }`
    - `config.contract.source: { kind: 'prisma', schema }`
    - Inline schema text (string containing PSL blocks)
  - Converts to contract IR before delegating to family emit.
- `db push` and `db pull`
  - New commands under `prisma-next db`.
  - Resolve schema path from `--schema` or `.prisma` contract source in config.
  - Use Prisma CLI for exact push/pull behavior.

## Data Flows

### `contract emit` with `.prisma`

1. Load config.
2. Resolve `.prisma` source.
3. Parse and convert using `convertPrismaSchemaToContract()`.
4. Emit canonical `contract.json` + `contract.d.ts` through family emitter.
5. Write artifacts.

### `db push`

1. Resolve `.prisma` schema path.
2. Resolve database URL (`--db` or `config.db.connection`).
3. Run Prisma CLI `db push` with sanitized schema.
4. Return structured CLI output.

### `db pull`

1. Resolve `.prisma` schema path.
2. Resolve database URL.
3. Run Prisma CLI `db pull --print` with sanitized schema.
4. Print introspected schema text to stdout.

## Parity Strategy

- **PSL syntax support:** delegated to Prisma 7 parser (`@prisma/internals`) and Prisma CLI, so Prisma grammar compatibility tracks Prisma 7.
- **Push/pull behavior:** delegated to Prisma CLI, so database apply and introspection formatting follows Prisma behavior directly.
- **Contract representation:** features not representable in current Prisma Next contract schema are explicitly surfaced in `meta.prismaPsl.missingFeatures` instead of silently dropped.

## Current Limits

Not all Prisma PSL constructs currently have lossless representation in Prisma Next contract IR/migration ops. Examples include:

- Referential action metadata in contract foreign key objects.
- Advanced index options (sort/operator class/algorithm/partial/fulltext semantics).
- Multi-schema placement metadata.
- View blocks and some unsupported/composite constructs.

See `docs/reference/Prisma-Schema-Parity-and-Gaps.md` for the canonical status and follow-ups.
